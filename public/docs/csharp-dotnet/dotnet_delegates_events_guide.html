Delegates and Events — Complete Guide

1. What is a Delegate?

A delegate is a type-safe reference to a method. It allows you to pass methods as parameters, store them in variables, and call them later — just like any other variable.
int x = 5;         // variable holding an integer
string s = "Hi";   // variable holding a string
MyDelegate d = SomeMethod; // variable holding a METHOD ← this is a delegate

Raw Delegate — The Old Way

// Step 1 — declare the delegate type
public delegate void PrintDelegate(string message);

// Step 2 — create a method matching the signature
static void PrintMessage(string message)
{
    Console.WriteLine("Message: " + message);
}

// Step 3 — assign and call
PrintDelegate del = PrintMessage;
del("Hello!"); // calls PrintMessage("Hello!")

Action and Func — The Modern Way

Action and Func are built-in delegate types provided by .NET as shortcuts so you don't need to declare your own delegate type every time.
// Action — delegate that returns void
Action<string> print = message => Console.WriteLine(message);
print("Hello!");

// Func — delegate that returns a value (last type = return type)
Func<int, int, int> add = (a, b) => a + b;
int result = add(3, 5); // returns 8

// Func with no parameters
Func<string> getName = () => "Omar";
Console.WriteLine(getName()); // Omar

Passing Methods as Parameters

static void ProcessData(List<int> numbers, Func<int, bool> filter)
{
    foreach (int n in numbers)
    {
        if (filter(n))
            Console.WriteLine(n);
    }
}

List<int> numbers = new List<int> { 1, 2, 3, 4, 5, 6 };
ProcessData(numbers, n => n > 3);       // prints 4, 5, 6
ProcessData(numbers, n => n % 2 == 0);  // prints 2, 4, 6

Multicast Delegate — One Call, Many Methods

Action<string> del = null;
del += message => Console.WriteLine("Logger 1: " + message);
del += message => Console.WriteLine("Logger 2: " + message);
del += message => Console.WriteLine("Logger 3: " + message);

del("Error occurred!");
// Output:
// Logger 1: Error occurred!
// Logger 2: Error occurred!
// Logger 3: Error occurred!

2. What is an Event?

An event is a wrapper around a delegate that implements the publisher-subscriber pattern. It allows a class to notify other classes when something happens — without those classes knowing about each other directly.

Why Not Just Use a Delegate Directly?

Raw delegates are dangerous — anyone can fire or clear them from outside the class:

class Button
{
    public Action OnClick; // raw delegate — dangerous!
}

button.OnClick = null;        // ❌ wipes ALL subscriptions from outside
button.OnClick("hacked");     // ❌ fires event from outside
button.OnClick += MyHandler;  // ✅ this is fine

With event keyword — these restrictions are enforced:

class Button
{
    public event Action OnClick; // safe — event wrapper
}

button.OnClick = null;       // ❌ COMPILE ERROR — cannot assign from outside
button.OnClick("hacked");    // ❌ COMPILE ERROR — cannot fire from outside
button.OnClick += MyHandler; // ✅ only += and -= allowed from outside

⚠️ Important: The event keyword restricts outside code to only subscribe (+=) or unsubscribe (-=). Only the class that owns the event can fire it.

Full Event Example — Publisher Subscriber Pattern

class OrderService
{
    public event Action<string> OnOrderPlaced; // declare event

    public void PlaceOrder(string orderNumber)
    {
        Console.WriteLine("Order placed: " + orderNumber);
        OnOrderPlaced?.Invoke(orderNumber); // fire event safely
    }
}

class EmailService
{
    public void SendConfirmationEmail(string orderNumber)
        => Console.WriteLine("Email sent for: " + orderNumber);
}

class SmsService
{
    public void SendSms(string orderNumber)
        => Console.WriteLine("SMS sent for: " + orderNumber);
}

// Wire everything together
var orderService = new OrderService();
var emailService = new EmailService();
var smsService   = new SmsService();

orderService.OnOrderPlaced += emailService.SendConfirmationEmail; // subscribe
orderService.OnOrderPlaced += smsService.SendSms;                 // subscribe

orderService.PlaceOrder("ORD-001");
// Output:
// Order placed: ORD-001
// Email sent for: ORD-001
// SMS sent for: ORD-001
OrderService has no idea that EmailService or SmsService exist. It just fires the event. This is loose coupling. ✅

4. The 3 Levels Summary

Raw Delegate → most control, most danger, old style

Action/Func → cleaner, built-in, what you use daily

Event → safest, publisher-subscriber, services and UI

5. Interview Questions

Q: What is a delegate in C#?

🎯 Interview Answer: A delegate is a type-safe reference to a method that allows methods to be passed as parameters, stored in variables, and called later. Action and Func are built-in delegate types provided by .NET as shortcuts. Multicast delegates can hold references to multiple methods and call them all in sequence with a single invocation.

Q: What is the difference between a delegate and an event?

🎯 Interview Answer: An event is a wrapper around a delegate that implements the publisher-subscriber pattern. While a raw delegate allows anyone to fire it or clear all subscriptions from outside the class, the event keyword restricts outside code to only subscribing or unsubscribing. Only the class that owns the event can fire it. This creates loose coupling where publishers and subscribers don't need to know about each other.

Q: What is the publisher-subscriber pattern?

🎯 Interview Answer: The publisher-subscriber pattern allows a class to notify other classes when something happens without knowing anything about them. The publisher defines an event and fires it when something occurs. Subscribers register their methods with the event using +=. When the event fires all registered methods are called automatically. This promotes loose coupling between classes.
