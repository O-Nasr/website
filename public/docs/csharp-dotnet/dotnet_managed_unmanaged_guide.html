1. Managed vs Unmanaged Code

What is Managed Code?

Managed code is code that runs under the control of the CLR (Common Language Runtime). The CLR handles memory allocation, garbage collection, type safety, and security for you automatically.

Examples of managed objects:

  • string, int, bool, double — basic C# types
  • List<T>, Dictionary<K,V> — collection classes
  • Any class you create in C#
  • SqlConnection, FileStream — .NET wrapper classes
string s = "Hello";       // Managed — GC handles this
List<int> nums = new List<int>();  // Managed — GC handles this

What is Unmanaged Code?

Unmanaged code refers to resources that live outside the CLR's control. These are typically OS-level resources or native memory that the .NET runtime doesn't track.

Examples of unmanaged resources:

  • File handles (OS-level)
  • Database connections (network sockets)
  • Native memory via Marshal.AllocHGlobal()
  • COM objects, GDI handles, Win32 API handles

The Key Rule

The GC only frees managed memory. It does NOT free unmanaged resources. This is one of the most important concepts in .NET internals.

🎯 Interview Answer: The Garbage Collector only tracks objects inside the managed heap under CLR control. Unmanaged resources like file handles, sockets, or native memory are allocated by the operating system and are not tracked by the GC. Therefore, we must release them manually.

3. IDisposable and the Dispose Pattern

Why We Need Dispose()

The Garbage Collector is nondeterministic — it runs when it decides, not when you want it to. For scarce resources like database connections and file handles, we cannot afford to wait. We need deterministic cleanup, and that's exactly what Dispose() gives us.

The using Statement — Best Practice

using (SqlConnection con = new SqlConnection(connectionString))
{
    // do your work here
} // Dispose() is automatically called here
The using block guarantees that Dispose() is called even if an exception is thrown inside the block.

Manual Dispose — When using Is Not Possible

SqlConnection con = new SqlConnection(connectionString);
try
{
    // do work
}
finally
{
    con.Dispose(); // Always dispose in finally
}

🎯 Interview Answer: GC manages memory. IDisposable manages resources. These are two completely different things. Never rely on GC to release unmanaged resources — always use Dispose().

4. Finalizer (~ClassName) — The Safety Net

What is a Finalizer?

A finalizer (also called destructor) is a method the GC calls when it collects an object. It acts as a safety net in case the developer forgets to call Dispose().
~MyClass()
{
    // This is called by GC automatically
    // You don't control WHEN this runs
}

Why Finalizers are Expensive

When a class has a finalizer, the GC cannot clean it immediately. Instead:

  • The object is moved to the Finalization Queue
  • It survives one full GC cycle (moves to Gen 1 or Gen 2)
  • The finalizer thread processes it separately
  • Only then the object becomes eligible for collection

This means objects with finalizers live longer in memory, consume more resources, and slow down the application. Add finalizers only when absolutely necessary.

5. The Full Dispose + Finalizer Pattern

Use this pattern ONLY when your class directly owns unmanaged resources (like IntPtr or native memory):
using System;
using System.Runtime.InteropServices;

class MyUnmanagedClass : IDisposable
{
    private IntPtr _nativeMemory;   // unmanaged resource
    private bool _disposed = false; // prevent double-dispose

    public MyUnmanagedClass()
    {
        _nativeMemory = Marshal.AllocHGlobal(100); // allocate unmanaged memory
    }

    // FINALIZER — safety net if developer forgets Dispose()
    ~MyUnmanagedClass()
    {
        Dispose(false);
    }

    // Core cleanup logic
    protected virtual void Dispose(bool disposing)
    {
        if (!_disposed)
        {
            if (disposing)
            {
                // Free managed resources here (e.g. other IDisposable objects)
            }

            // Free unmanaged resources — always runs
            if (_nativeMemory != IntPtr.Zero)
            {
                Marshal.FreeHGlobal(_nativeMemory);
                _nativeMemory = IntPtr.Zero;
            }

            _disposed = true;
        }
    }

    // Public Dispose — called by developer or using statement
    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this); // Tell GC: no need to run finalizer
    }
}

Why GC.SuppressFinalize(this)?

When Dispose() is called properly, all cleanup is already done. Calling GC.SuppressFinalize(this) tells the GC to skip the finalizer for this object, avoiding the expensive finalization queue process.

7. SafeHandle — The Modern Approach

SafeHandle is a special .NET class designed to wrap unmanaged handles safely. It has its own built-in finalizer and is specifically designed to prevent resource leaks.

🎯 Interview Answer: If your class uses SafeHandle (or classes that use it internally like FileStream), you do NOT need to write your own finalizer. SafeHandle already implements a critical finalizer internally. Your class only needs to implement IDisposable and call Dispose() on the SafeHandle.

8. Quick Rules to Remember

  • GC handles managed memory — you handle unmanaged resources
  • Always use using(...) for IDisposable objects
  • Dispose() releases resources. GC.Collect() releases memory. They are different.
  • Add a finalizer ONLY if your class owns raw unmanaged memory (IntPtr)
  • Call GC.SuppressFinalize(this) inside Dispose() to avoid double cleanup
  • If wrapping SafeHandle or FileStream — no finalizer needed
  • Setting a reference to null does NOT immediately close the resource

9. Common Interview Questions

Q: Why does GC not clean unmanaged objects?

🎯 Interview Answer: The GC only tracks objects inside the managed heap. Unmanaged resources like file handles and sockets are allocated by the OS, not the CLR. The GC has no knowledge of them, so we must use IDisposable and Dispose() to release them.

Q: Why implement both Dispose() and Finalizer?

🎯 Interview Answer: Dispose() provides deterministic, immediate cleanup when the developer calls it correctly. The finalizer acts as a safety net for cases where the developer forgets to call Dispose(). Inside Dispose(), we call GC.SuppressFinalize() to skip the finalizer when it's no longer needed.

Q: What happens if I set SqlConnection con = null?

🎯 Interview Answer: Setting the reference to null only makes the object eligible for garbage collection. It does NOT close the database connection. The GC is nondeterministic, so it may run seconds or minutes later. The underlying OS connection is only released when Dispose() is called explicitly.

Q: When should you add a finalizer to a class?

🎯 Interview Answer: Only when the class directly owns unmanaged resources like native memory (IntPtr). If the class uses SafeHandle, FileStream, SqlConnection, or other IDisposable wrappers, no finalizer is needed because those classes already handle their own cleanup.
