Interface vs Abstract Class — Complete Guide

1. Abstract Class — Partial Blueprint

An abstract class provides a partial implementation that related classes can inherit from. It can have both implemented methods (shared code) and abstract methods (forced override).
abstract class Shape
{
    public string Color { get; set; }   // shared property — all shapes have color

    public void DisplayColor()           // implemented — shared by all shapes
    {
        Console.WriteLine("Color: " + Color);
    }

    public abstract double GetArea();    // abstract — no body, child MUST override
}

class Circle : Shape
{
    public double Radius { get; set; }
    public override double GetArea() => Math.PI * Radius * Radius; // MUST implement
}

class Rectangle : Shape
{
    public double Width { get; set; }
    public double Height { get; set; }
    public override double GetArea() => Width * Height; // MUST implement
}

⚠️ Important: Abstract methods have NO body — child class MUST override them. Regular methods in abstract class HAVE a body — child class CAN override but doesn't have to.

2. Interface — Full Blueprint (Contract)

An interface is a pure contract that defines what a class must do without specifying how. Any unrelated class can implement it.
interface INotificationService
{
    void Send(string message);        // contract — no implementation
    bool IsAvailable();               // contract — no implementation
}

class EmailNotification : INotificationService
{
    public void Send(string message)
    {
        Console.WriteLine("Email: " + message);
    }
    public bool IsAvailable() => true;
}

class SmsNotification : INotificationService
{
    public void Send(string message)
    {
        Console.WriteLine("SMS: " + message);
    }
    public bool IsAvailable() => CheckSmsGateway();
}

⚠️ Important: Since C# 8.0, interfaces CAN have default implementations. However in interviews, interfaces are still primarily described as blueprints/contracts. Mention C# 8.0 as a bonus point.

3. The Biggest Difference — Multiple Inheritance

A class can only inherit from ONE abstract class but can implement MANY interfaces:

// ❌ C# does NOT allow multiple class inheritance
class Dog : Animal, Pet  // COMPILE ERROR!
{
}

// ✅ C# DOES allow multiple interface implementation

class Duck : IFlyable, ISwimmable, IRunnable // perfectly fine ✅

{
    public void Fly()  { Console.WriteLine("Duck flying"); }
    public void Swim() { Console.WriteLine("Duck swimming"); }
    public void Run()  { Console.WriteLine("Duck running"); }
}

5. When to Use Which

Use Abstract Class When:

  • Classes share common code or state
  • You want to provide a base implementation
  • Classes are closely related (same family)
  • Example: Shape → Circle, Rectangle, Triangle

Use Interface When:

  • Classes are completely unrelated but share behavior
  • You need multiple inheritance
  • You want to define a contract for any class to follow
  • Example: IDisposable → SqlConnection, FileStream, any class needing cleanup

7. Interview Questions

Q: What is the difference between interface and abstract class?

🎯 Interview Answer: An abstract class provides a partial implementation that related classes inherit from — it can have implemented methods, constructors, fields, and access modifiers. An interface is a pure contract that defines what a class must do without how, and any unrelated class can implement it. The key practical difference is that a class can only inherit from one abstract class but can implement multiple interfaces.

Q: When would you choose interface over abstract class?

🎯 Interview Answer: Use interface when completely unrelated classes need to follow the same contract, or when you need multiple inheritance behavior. Use abstract class when related classes share common code and state. For example IDisposable is an interface because completely unrelated classes like SqlConnection and FileStream need cleanup behavior — they share no common code.

Q: Can an interface have implementation in C#?

🎯 Interview Answer: Since C# 8.0, interfaces can have default implementations. However interfaces are still primarily used as pure contracts. Default interface implementations are mainly used for versioning — adding new methods to an interface without breaking existing implementations.
