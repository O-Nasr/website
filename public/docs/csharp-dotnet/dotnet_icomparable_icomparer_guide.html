IComparable vs IComparer — Complete Guide

1. Quick Summary

IComparable = The object knows how to sort itself. Sorting logic lives INSIDE the class.
IComparer = A separate external class that knows how to sort from outside. Sorting logic lives OUTSIDE the class.

⚠️ Important: IComparable = one fixed default sort. IComparer = multiple flexible sort strategies. Think of IComparable as the object judging itself, and IComparer as hiring an external judge.

2. IComparable — Internal Default Sorting

The class implements sorting logic inside itself using CompareTo(). One default sort criteria.
class Student : IComparable<Student>
{
    public string Name { get; set; }
    public int Age { get; set; }

    // Sorting logic lives INSIDE the class
    public int CompareTo(Student other)
    {
        return this.Age.CompareTo(other.Age); // sort by Age — fixed default
    }
}

List<Student> students = new List<Student>

{
    new Student { Name = "Ali",   Age = 25 },
    new Student { Name = "Omar",  Age = 20 },
    new Student { Name = "Ahmed", Age = 22 },
};

students.Sort(); // uses CompareTo() internally — sorts by Age

// Output:
// Omar  — 20
// Ahmed — 22
// Ali   — 25

The Problem with IComparable

What if sometimes you want to sort by Name and other times by Age? With IComparable you are stuck — the logic is hardcoded inside the class.
students.Sort(); // ALWAYS sorts by Age — no flexibility ❌
// Cannot change sort criteria without modifying the class

3. IComparer — External Flexible Sorting

Create separate classes for each sort criteria. The logic lives outside the Student class. Zero modifications to original class.
// Sort by Name — external class
class SortByName : IComparer<Student>
{
    public int Compare(Student x, Student y)
    {
        return string.Compare(x.Name, y.Name); // alphabetical
    }
}

// Sort by Age — external class
class SortByAge : IComparer<Student>
{
    public int Compare(Student x, Student y)
    {
        if (x.Age > y.Age) return 1;  // x is greater
        if (x.Age < y.Age) return -1; // x is smaller
        return 0;                      // equal
    }
}

// Now sort any way you want — zero changes to Student class
students.Sort(new SortByName()); // alphabetical by name
students.Sort(new SortByAge());  // ascending by age

// Sort by Name output:
// Ahmed — 22
// Ali   — 25
// Omar  — 20

// Sort by Age output:
// Omar  — 20
// Ahmed — 22
// Ali   — 25

4. CompareTo Return Values

Both CompareTo and Compare follow the same return value convention:

Return Value

Meaning

Negative number (e.g. -1)

Current object is LESS THAN the other

Zero (0)

Current object is EQUAL TO the other

Positive number (e.g. 1)

Current object is GREATER THAN the other

⚠️ Important: Open/Closed Principle — IComparer keeps the class closed for modification but open for extension. Mentioning this in interviews instantly shows mid-level thinking.

6. Real World Example — Multiple Sort Strategies

// Employees sorted different ways depending on context
class SortByDepartment : IComparer<Employee>
{
    public int Compare(Employee x, Employee y)
        => string.Compare(x.Department, y.Department);
}

class SortBySalaryDescending : IComparer<Employee>
{
    public int Compare(Employee x, Employee y)
        => y.Salary.CompareTo(x.Salary); // descending — note y vs x
}

class SortByNameThenAge : IComparer<Employee>
{
    public int Compare(Employee x, Employee y)
    {
        int nameComp = string.Compare(x.Name, y.Name);
        if (nameComp != 0) return nameComp;  // sort by name first
        return x.Age.CompareTo(y.Age);       // then by age if names equal
    }
}

// Usage
employees.Sort(new SortByDepartment());
employees.Sort(new SortBySalaryDescending());
employees.Sort(new SortByNameThenAge());

7. Modern Alternative — LINQ OrderBy

In modern C# code, LINQ is often used instead of IComparer for sorting — cleaner and more readable:

// Instead of IComparer classes — use LINQ
var byName = students.OrderBy(s => s.Name).ToList();
var byAge  = students.OrderBy(s => s.Age).ToList();
var byAgeThenName = students
    .OrderBy(s => s.Age)
    .ThenBy(s => s.Name)
    .ToList();

// IComparer is still important for:
// - List.Sort() which sorts in-place (no new list created)
// - Custom complex sorting logic
// - Older codebases

8. Interview Questions

Q: What is the difference between IComparable and IComparer?

🎯 Interview Answer: IComparable defines a default sort implementation inside the class itself using CompareTo(). It is limited to one fixed sorting criteria and the sorting logic is coupled to the class. IComparer separates sorting logic into external classes using Compare(), allowing multiple different sort strategies without modifying the original class. IComparer is more flexible and follows the Open/Closed principle — the class is closed for modification but open for extension.

Q: When would you use IComparer over IComparable?

🎯 Interview Answer: Use IComparer when you need multiple sort strategies for the same class — for example sorting employees by name in one context and by salary in another. Use IComparable when there is one obvious natural default sort order for the class that will never change — for example sorting dates chronologically or numbers ascending. In modern code, LINQ OrderBy is often preferred over both for its readability.

Q: What do the return values of CompareTo mean?

🎯 Interview Answer: CompareTo returns a negative number when the current object is less than the compared object, zero when they are equal, and a positive number when the current object is greater. This convention is shared by both IComparable.CompareTo() and IComparer.Compare(). The Sort method uses these return values to determine the correct ordering.
