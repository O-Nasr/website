Dependency Injection — Complete Guide

1. The Problem — Tight Coupling

When a class creates its own dependencies internally, it becomes tightly coupled. Changing one class forces changes in others, and testing becomes impossible without dealing with all dependencies.
// ❌ Tight coupling — OrderService creates its own dependency
public class OrderService
{
    private EmailService _emailService = new EmailService();

    public void PlaceOrder(string orderNumber)
    {
        _emailService.SendEmail(orderNumber);
    }
}
// Problems:
// 1. Changing EmailService to SmsService requires modifying OrderService
// 2. Cannot mock EmailService for unit testing

2. The Solution — Dependency Injection

// ✅ Loose coupling — dependency injected from outside
public class OrderService
{
    private readonly INotificationService _notificationService;

    public OrderService(INotificationService notificationService)
    {
        _notificationService = notificationService;
    }

    public void PlaceOrder(string orderNumber)
    {
        _notificationService.Send(orderNumber);
    }
}

// Swap implementations — zero changes to OrderService
var orderService = new OrderService(new EmailNotification());
var orderService = new OrderService(new SmsNotification());   // ✅
var orderService = new OrderService(new MockNotification());  // ✅ testing

3. Three Types of DI

1. Constructor Injection — Most Common

public class OrderService
{
    private readonly INotificationService _notificationService;
    private readonly ILogger _logger;

    // All required dependencies declared upfront — clear and honest
    public OrderService(INotificationService notificationService, ILogger logger)
    {
        _notificationService = notificationService;
        _logger = logger;
    }
}

2. Property Injection — Optional Dependencies

public class OrderService
{
    // Required dependency — stays in constructor
    private readonly INotificationService _notificationService;

    // Optional dependency — can be null, class works without it
    public ILogger Logger { get; set; }

    public OrderService(INotificationService notificationService)
    {
        _notificationService = notificationService;
    }

    public void PlaceOrder(string orderNumber)
    {
        Logger?.Log("Placing order: " + orderNumber); // null-safe
        _notificationService.Send(orderNumber);
    }
}

// Without logger — fine
var service = new OrderService(new EmailNotification());

// With logger — just set the property
var service = new OrderService(new EmailNotification());
service.Logger = new ConsoleLogger(); // inject after creation

3. Method Injection — Per-Method Dependency

public class OrderService
{
    // Dependency passed per method call — not stored in class
    public void PlaceOrder(string orderNumber, INotificationService notificationService)
    {
        notificationService.Send(orderNumber);
    }
}

4. Service Lifetimes in ASP.NET Core

// In Program.cs
builder.Services.AddTransient<INotificationService, EmailNotification>(); // new every time
builder.Services.AddScoped<IOrderService, OrderService>();                 // once per request
builder.Services.AddSingleton<ILogger, Logger>();                          // once forever

Transient — New Instance Every Injection

// Request 1:
//   Controller needs INotificationService → new EmailNotification(#1)
//   Service needs INotificationService    → new EmailNotification(#2)
// Request 2:
//   Controller needs INotificationService → new EmailNotification(#3)
//
// Use for: lightweight stateless services — email sender, validators

Scoped — One Instance Per HTTP Request

// Request 1:
//   Controller needs IOrderService → creates OrderService(#1)
//   Repository needs IOrderService → gets SAME OrderService(#1)
//   Request ends → OrderService(#1) disposed
//
// Request 2:
//   Everything in Request 2 gets NEW OrderService(#2)
//
// Use for: DbContext, Unit of Work — all operations in one request share same context

Singleton — One Instance For App Lifetime

// App starts → creates Logger(#1)
// Request 1 → gets Logger(#1)
// Request 2 → gets Logger(#1)
// Request N → gets Logger(#1)
// App stops → Logger(#1) disposed
//
// MUST be thread-safe!
// Use for: Logger, Configuration, Cache, HttpClient factory

5. Captive Dependency Problem

⚠️ Important: A captive dependency happens when a longer-lived service holds a Scoped dependency. This prevents the Scoped service from being properly disposed after each request. ASP.NET Core throws an exception at runtime when this is detected.

// ❌ WRONG — Singleton holding Scoped dependency
public class MySingletonService
{
    private readonly IOrderService _orderService; // Scoped — should die per request!

    public MySingletonService(IOrderService orderService)
    {
        _orderService = orderService; // captured forever in Singleton 😱
    }
}

// ASP.NET Core throws:
// InvalidOperationException: Cannot consume scoped service 'IOrderService'
// from singleton 'MySingletonService'

// Lifetime Rules:
// Singleton  → can use → Singleton only
// Scoped     → can use → Singleton + Scoped
// Transient  → can use → Singleton + Scoped + Transient

6. DI Makes Testing Easy

// Mock for testing
public class MockNotificationService : INotificationService
{
    public List<string> SentMessages = new List<string>();
    public void Send(string message) { SentMessages.Add(message); }
}

// Unit test — no real notifications, no DB
[Test]
public void PlaceOrder_ShouldSendNotification()
{
    var mockService = new MockNotificationService();
    var orderService = new OrderService(mockService); // inject mock

    orderService.PlaceOrder("ORD-001");

    Assert.AreEqual(1, mockService.SentMessages.Count);
    Assert.AreEqual("ORD-001", mockService.SentMessages[0]);
}

7. Interview Questions

Q: What is Dependency Injection and why do we use it?

🎯 Interview Answer: Dependency Injection is a design pattern where a class receives its dependencies from outside rather than creating them itself. This promotes loose coupling — the class depends on abstractions not implementations, making it easy to swap implementations and test in isolation using mocks without touching production code.

Q: What are the three service lifetimes in ASP.NET Core?

🎯 Interview Answer: Transient creates a new instance on every injection — best for lightweight stateless services. Scoped creates one instance per HTTP request shared across that request — best for DbContext and Unit of Work. Singleton creates one instance for the entire application lifetime — best for Logger, Cache, and Configuration. Singletons must be thread-safe since they are shared across all requests.

Q: What is the captive dependency problem?

🎯 Interview Answer: A captive dependency occurs when a Singleton service holds a reference to a Scoped service, preventing the Scoped service from being properly disposed after each request. The Scoped service becomes trapped in the Singleton forever. ASP.NET Core detects this and throws an InvalidOperationException at runtime.
