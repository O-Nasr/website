Expression Trees — Complete Guide

1. The Fundamental Difference

Both Func and Expression look identical as lambdas — but internally they are completely different things.
// Func — compiled code — executable
Func<int, bool> func = x => x > 5;

// Expression — data structure — inspectable and translatable
Expression<Func<int, bool>> expr = x => x > 5;

⚠️ Important: Think of Func like a cooked meal — ready to eat but you cannot see the recipe. Expression is the written recipe — you can read it, modify it, and translate it to any language.

2. What is an Expression Tree?

An Expression Tree is a tree data structure that represents code as data. The compiler does NOT compile the lambda to machine code — instead it builds a tree of nodes.
Expression<Func<int, bool>> expr = x => x > 5;

// The compiler builds this tree internally:
//
//         GreaterThan (>)
//        /              \
//   Parameter(x)     Constant(5)
//

// Every piece of the expression is a node — accessible as an object

Inspecting the Tree

Expression<Func<int, bool>> expr = x => x > 5;

Console.WriteLine(expr.Body);     // (x > 5)
Console.WriteLine(expr.NodeType); // Lambda

// Get the operation node
BinaryExpression body = (BinaryExpression)expr.Body;
Console.WriteLine(body.NodeType); // GreaterThan
Console.WriteLine(body.Left);     // x
Console.WriteLine(body.Right);    // 5

// You can read EVERY part of the expression as data!

3. Why This Matters — Entity Framework and SQL Translation

This is where everything clicks. When you write a LINQ query with Entity Framework:
var employees = dbContext.Employees
    .Where(e => e.Salary > 5000)
    .ToList();

Entity Framework receives the Expression tree and walks it to build SQL:

// EF receives this expression tree:
//
//         GreaterThan (>)
//        /               \
//  Property(Salary)   Constant(5000)
//

// EF walks the tree and translates each node:

//   Property(Salary) → "Salary"
//   GreaterThan      → ">"
//   Constant(5000)   → "5000"
//
// Builds SQL:
//   WHERE Salary > 5000
//
// Sends to database — only matching rows returned ✅

Why IQueryable Takes Expression — Not Func

// IEnumerable.Where — takes Func — executes in C# memory
IEnumerable<T>.Where(Func<T, bool> predicate)
// SQL: SELECT * FROM Employees  ← transfers ALL rows first!

// IQueryable.Where — takes Expression — translates to SQL
IQueryable<T>.Where(Expression<Func<T, bool>> predicate)
// SQL: SELECT * FROM Employees WHERE Salary > 5000  ← efficient!

⚠️ Important: This is the real reason IQueryable is more efficient than IEnumerable with Entity Framework. IQueryable uses Expression trees to translate filters to SQL. IEnumerable uses Func which runs in memory after all data is loaded.

4. Func vs Expression — Side by Side

// Func — filters in C# memory — pulls all data first
Func<Employee, bool> func = e => e.Salary > 5000;
var result = employees.Where(func);
// SQL: SELECT * FROM Employees  ← all 1M rows transferred 😱

// Expression — filters in database — only matching data
Expression<Func<Employee, bool>> expr = e => e.Salary > 5000;
var result = dbContext.Employees.Where(expr);
// SQL: SELECT * FROM Employees WHERE Salary > 5000  ← 10 rows ✅

5. Building Expressions Dynamically

The most powerful feature — building queries at runtime when filter conditions are not known at compile time:

// Build expression dynamically — equivalent to: e => e.Salary > threshold
int threshold = 5000; // could come from user input at runtime

// Step 1 — parameter: e
ParameterExpression param = Expression.Parameter(typeof(Employee), "e");

// Step 2 — property access: e.Salary
MemberExpression property = Expression.Property(param, "Salary");

// Step 3 — constant: 5000
ConstantExpression constant = Expression.Constant(threshold);

// Step 4 — comparison: e.Salary > 5000
BinaryExpression comparison = Expression.GreaterThan(property, constant);

// Step 5 — full lambda: e => e.Salary > 5000

Expression<Func<Employee, bool>> lambda =

    Expression.Lambda<Func<Employee, bool>>(comparison, param);

// Use it — translated to SQL automatically
var result = dbContext.Employees.Where(lambda).ToList();
// SQL: SELECT * FROM Employees WHERE Salary > 5000

Dynamic Filter — Real World Use

// Build filter based on field name provided at runtime
public IQueryable<Employee> Filter(string fieldName, int value)
{
    var param      = Expression.Parameter(typeof(Employee), "e");
    var property   = Expression.Property(param, fieldName); // dynamic!
    var constant   = Expression.Constant(value);
    var comparison = Expression.GreaterThan(property, constant);
    var lambda     = Expression.Lambda<Func<Employee, bool>>(comparison, param);

    return dbContext.Employees.Where(lambda);
}

// Call with different fields at runtime — all translated to SQL
Filter("Salary", 5000); // WHERE Salary > 5000
Filter("Age", 30);      // WHERE Age > 30
// Impossible to do with hardcoded LINQ ✅

6. Generic Repository With Expressions

The most common real-world pattern you will see in backend projects:

public interface IRepository<T> where T : class
{
    // Takes Expression — translates to SQL automatically
    IQueryable<T> Find(Expression<Func<T, bool>> predicate);
}

public class Repository<T> : IRepository<T> where T : class
{
    private readonly DbSet<T> _dbSet;

    public Repository(AppDbContext context)
    {
        _dbSet = context.Set<T>();
    }

    public IQueryable<T> Find(Expression<Func<T, bool>> predicate)
    {
        return _dbSet.Where(predicate); // translated to SQL ✅
    }
}

// Usage — all SQL — nothing loaded into memory until ToList()
var highEarners = repo.Find(e => e.Salary > 5000);
var seniors     = repo.Find(e => e.Age > 40 && e.Department == "IT");
var recentOrders = repo.Find(o => o.CreatedAt > DateTime.Today.AddDays(-7));

7. Compiling an Expression to Execute It

You can also compile an Expression back to a Func and execute it — best of both worlds:

Expression<Func<int, bool>> expr = x => x > 5;

// Inspect as Expression
BinaryExpression body = (BinaryExpression)expr.Body;
Console.WriteLine(body.NodeType); // GreaterThan

// Compile to Func — now executable
Func<int, bool> compiled = expr.Compile();

// Execute it like any Func
Console.WriteLine(compiled(6)); // true
Console.WriteLine(compiled(3)); // false

9. Interview Questions

Q: What is the difference between Func and Expression?

🎯 Interview Answer: Func is compiled to executable machine code immediately — you can call it directly but cannot inspect or translate it. Expression stores the lambda as a data tree that represents the code structure — you can inspect every node, translate it to SQL or other languages, and build it dynamically at runtime. Expression is what allows Entity Framework to translate LINQ queries to SQL instead of executing them in C# memory.

Q: Why does IQueryable use Expression instead of Func?

🎯 Interview Answer: IQueryable needs to translate filter conditions to SQL to execute them in the database. This is only possible with Expression trees because the code is stored as data that can be inspected and translated. If IQueryable used Func like IEnumerable does, the filter would be compiled code that cannot be translated — all data would have to be loaded into memory first and filtered in C#, which is exactly what IEnumerable does.

Q: When would you build an Expression tree manually?

🎯 Interview Answer: You build Expression trees manually when filter conditions are not known at compile time — for example building a search query where the user selects which fields to filter by at runtime. Instead of writing dozens of if/else statements to build different LINQ queries, you build the expression tree dynamically based on the user input and pass it to IQueryable.Where() — it is still translated to SQL automatically.

Q: What is the connection between Expression trees and IQueryable vs IEnumerable?

🎯 Interview Answer: IEnumerable.Where takes Func — the predicate is compiled code that runs in C# memory after all data is loaded from the database. IQueryable.Where takes Expression — the predicate is a data tree that Entity Framework translates to a SQL WHERE clause so filtering happens in the database. This is why IQueryable is dramatically more efficient for large datasets — only matching rows are transferred instead of the entire table.
