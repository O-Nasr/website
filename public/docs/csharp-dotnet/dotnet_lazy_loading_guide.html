Lazy Loading — Complete Guide

1. What is Lazy Loading?

Lazy Loading is a design pattern where you delay the loading of an object until the exact moment you actually need it. Instead of loading everything upfront when the parent object is created, you load child data on demand.
Simple rule: Don't load data until you actually need it.
The opposite of Lazy Loading is Eager Loading — where everything loads immediately when the object is created, whether you need it or not.

Real Life Analogy

Think of a restaurant menu. You don't cook every dish in the kitchen the moment a customer walks in. You wait until they actually order something. That's lazy loading — prepare only what is asked for.

Quick Code Example

Customer o = new Customer();        // Orders NOT loaded yet
Console.WriteLine(o.CustomerName);  // Orders still NOT loaded

foreach (Order o1 in o.Orders)      // Orders load HERE — on demand
{
    Console.WriteLine(o1.OrderNumber);
}

2. Advantages ✅

1. Faster Application Startup

When your app starts, it only loads essential data. Heavy or optional data waits until needed. This means users see the app faster.
Real Example: An e-commerce app loads the customer's basic profile on login. Their 500 order history records only load when they click 'My Orders'.
public Customer()
{
    _CustomerName = "Ali";  // loads immediately — lightweight
    _Orders = new Lazy<List<Order>>(() => LoadOrders()); // NOT fired yet
}

2. Less Memory Usage

Objects that are never accessed are never loaded into memory. Your application uses only what it actually needs at any given moment.
Real Example: A user opens an article, reads it, and closes the app. Without lazy loading, ALL articles and comments would have been loaded. With lazy loading, only the one article was ever touched.

3. Unnecessary Database Queries Are Avoided

Every database call has a cost — network time, query execution, data transfer. Lazy loading means queries only fire when data is truly needed.
// Without Lazy Loading — DB query fires immediately every time
public Customer()
{
    _Orders = LoadOrders(); // always hits DB even if Orders never used
}

// With Lazy Loading — DB query only fires when Orders is accessed

_Orders = new Lazy<List<Order>>(() => LoadOrders());
If you create 100 Customer objects but only access Orders for 10 of them, you saved 90 unnecessary database calls.

4. Better Scalability

Because you use less memory and fewer database queries, your application can handle more concurrent users without performance degrading quickly.

3. Disadvantages ❌

1. Code Becomes More Complex

You now have to check if data is loaded or not before using it. This adds extra logic to every lazy property.
// Without Lazy Loading — simple and clean
public List<Order> Orders { get; set; }

// With Lazy Loading — extra null check needed everywhere
public List<Order> Orders
{
    get
    {
        if (_Orders == null)
        {
            _Orders = LoadOrders();
        }
        return _Orders;
    }
}

2. Slight Performance Overhead on First Access

Every time a lazy property is accessed for the first time, there is an extra check: "Is this loaded yet? No? Then load it now." That check adds a tiny delay on first access. In most cases it is negligible, but in performance-critical systems it matters.

3. The N+1 Problem — Most Dangerous Disadvantage

This is the most dangerous disadvantage, especially with Entity Framework. When you loop through a collection and each item triggers its own lazy load, you end up with hundreds of hidden database queries.
// Looks clean — but very dangerous
var customers = db.Customers.ToList(); // 1 query — loads 100 customers

foreach (var customer in customers)
{
    // Each access to Orders fires a SEPARATE DB query!
    Console.WriteLine(customer.Orders.Count);
}

// Result:
// 1 query to load 100 customers
// + 100 separate queries for each customer's orders
// = 101 total DB queries fired silently 😱

⚠️ Important: The N+1 problem is one of the most common performance killers in .NET applications. Always be aware of it when using lazy loading inside loops.

4. Harder to Debug

Because data loads at unpredictable times, bugs appear in unexpected places. You might see a NullReferenceException or a database connection error deep inside a loop, far from where the object was originally created.

5. How to Implement Lazy Loading Manually

Two simple changes to your class:

  • Remove the object loading from the constructor
  • Add a null check inside the property getter — load only on first access

Before — Eager Loading (No Lazy)

public class Customer
{
    private List<Order> _Orders = null;

    public Customer()
    {
        _CustomerName = "Ali";
        _Orders = LoadOrders(); // ❌ Always loads — even if never needed
    }
}

After — Manual Lazy Loading

public class Customer
{
    private List<Order> _Orders = null;

    public Customer()
    {
        _CustomerName = "Ali";
        // ✅ Nothing loaded here
    }

    public List<Order> Orders
    {
        get
        {
            if (_Orders == null)       // First access?
            {
                _Orders = LoadOrders(); // Load and cache
            }
            return _Orders;            // Return cached result every time after
        }
    }
}
First access triggers the load. Every access after that returns the cached result. No repeated database calls.

6. The Lazy<T> Class — Built-in Solution

Writing the null-check pattern manually every time is repetitive. .NET provides the built-in Lazy<T> class which does everything automatically — and adds thread safety for free.

How to Use It — 3 Steps

Step 1 — Declare using Lazy<T>:

private Lazy<List<Order>> _Orders = null;

Step 2 — Attach to your loading method in the constructor:

_Orders = new Lazy<List<Order>>(() => LoadOrders());
Step 3 — Access data via the .Value property:
public List<Order> Orders
{
    get { return _Orders.Value; } // Lazy<T> handles null check automatically
}

Full Clean Example with Lazy<T>

public class Customer
{
    private Lazy<List<Order>> _Orders = null;
    private string _CustomerName;

    public string CustomerName
    {
        get { return _CustomerName; }
    }

    public List<Order> Orders
    {
        get { return _Orders.Value; } // clean — no null check needed
    }

    public Customer()
    {
        _CustomerName = "Ali";
        _Orders = new Lazy<List<Order>>(() => LoadOrders()); // attached, not fired
    }

    private List<Order> LoadOrders()
    {
        return new List<Order>
        {
            new Order { OrderNumber = "ORD001" },
            new Order { OrderNumber = "ORD002" }
        };
    }
}

8. Common Interview Questions

Q: What is Lazy Loading?

🎯 Interview Answer: Lazy Loading is a pattern where data is not loaded until it is actually needed. Instead of loading everything when an object is created, child data loads on demand when first accessed. This improves startup performance and reduces unnecessary database queries.

Q: What are the advantages of Lazy Loading?

🎯 Interview Answer: Lazy loading improves startup performance by only loading essential data at launch, reduces memory usage by never loading data that isn't accessed, and avoids unnecessary database queries. This leads to better scalability as the application can handle more concurrent users with the same resources.

Q: What are the disadvantages of Lazy Loading?

🎯 Interview Answer: Lazy loading adds code complexity through null checks and loading logic. It can cause the N+1 problem where looping through a lazily loaded collection triggers a separate database query for each item, severely hurting performance. It also makes debugging harder because errors appear far from where the object was created.

Q: What is the difference between Lazy<T> and manual lazy loading?

🎯 Interview Answer: Both achieve the same goal — loading data on demand. The manual pattern uses a null check inside the property getter. Lazy<T> is the built-in .NET solution that eliminates the null check boilerplate and adds thread safety by default, ensuring data is only loaded once even when multiple threads access the property simultaneously.

Q: What is the N+1 problem?

🎯 Interview Answer: The N+1 problem occurs when you load a list of N objects and then loop through them accessing a lazily loaded property on each one, triggering N additional database queries — one per object. This results in N+1 total queries instead of just 1 or 2. The solution is to use eager loading when you know you will need child data for every item in the list.
