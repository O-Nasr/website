Records vs Classes vs Structs — Complete Guide

1. Class — Mutable Reference Type

Classes are reference types stored on the heap. Variables hold a reference (pointer) to the object — not the object itself. Equality compares references by default.
public class PersonClass
{
    public string Name { get; set; }
    public int Age { get; set; }
}

// Reference semantics — both variables point to SAME object
var p1 = new PersonClass { Name = "Omar", Age = 25 };
var p2 = p1; // p2 is NOT a copy — same reference

p2.Name = "Ali"; // modifies the SAME object
Console.WriteLine(p1.Name); // "Ali" — p1 also changed! 😱

// Equality compares REFERENCES — not content
var p3 = new PersonClass { Name = "Omar", Age = 25 };
var p4 = new PersonClass { Name = "Omar", Age = 25 };
Console.WriteLine(p3 == p4); // false — different objects in memory

2. Struct — Value Type

Structs are value types. Assignment creates a complete independent copy. Equality compares values by default.
public struct PersonStruct
{
    public string Name { get; set; }
    public int Age { get; set; }
}

// Value semantics — s2 is a COMPLETE COPY — independent
var s1 = new PersonStruct { Name = "Omar", Age = 25 };
var s2 = s1; // full copy

s2.Name = "Ali"; // modifies s2 ONLY
Console.WriteLine(s1.Name); // "Omar" — s1 unchanged ✅

// Equality compares VALUES — content
var s3 = new PersonStruct { Name = "Omar", Age = 25 };
var s4 = new PersonStruct { Name = "Omar", Age = 25 };
Console.WriteLine(s3 == s4); // true — same content ✅

Where Struct is Actually Stored

// Local variable → stored on STACK ✅
void MyMethod()
{
    PersonStruct p = new PersonStruct(); // stack
}

// Field inside a class → stored on HEAP with the class
public class Order
{
    public PersonStruct Customer; // heap — lives inside Order object
}

// Inside an array → stored on HEAP
PersonStruct[] people = new PersonStruct[10]; // heap

// Rule: struct is a VALUE TYPE — WHERE it lives depends on context

⚠️ Important: Microsoft guideline: Struct should be 16 bytes or less. Larger structs are expensive to copy — use a class instead.

Struct Performance Warning

// ❌ Large struct — every assignment copies ALL fields — expensive
public struct BigData
{
    public double F1, F2, F3, F4;
    public string S1, S2, S3, S4; // 8 fields!
}
BigData a = new BigData();
BigData b = a; // copies all 8 fields 😱

// ✅ Small struct — cheap copy
public struct Point
{
    public int X, Y; // 2 fields — tiny and fast
}

3. Record — Immutable Reference Type with Value Equality (C# 9+)

A record is a reference type like a class BUT with value equality like a struct. Designed for immutable data. The compiler auto-generates equality, ToString, constructor, and with-expression support.

One Line — Compiler Generates Everything

// One line — positional syntax
public record PersonRecord(string Name, int Age);

// Equivalent to writing ALL of this manually:
public class PersonRecord
{
    public string Name { get; init; }  // init — set only at creation
    public int Age { get; init; }

    public PersonRecord(string Name, int Age)
    {
        this.Name = Name;
        this.Age = Age;
    }

    // Value equality — generated automatically
    public override bool Equals(object obj) { ... }
    public override int GetHashCode() { ... }

    // Clean ToString — generated automatically
    public override string ToString()
        => $"PersonRecord {{ Name = {Name}, Age = {Age} }}";
}

Value Equality — The Biggest Difference from Class

var r1 = new PersonRecord("Omar", 25);
var r2 = new PersonRecord("Omar", 25);

Console.WriteLine(r1 == r2);       // ✅ TRUE — value equality
Console.WriteLine(r1.Equals(r2));  // ✅ TRUE

// Compare with class — same content, different result
var c1 = new PersonClass { Name = "Omar", Age = 25 };
var c2 = new PersonClass { Name = "Omar", Age = 25 };
Console.WriteLine(c1 == c2); // ❌ FALSE — reference equality

Immutability — init Keyword

var person = new PersonRecord("Omar", 25);

person.Name = "Ali"; // ❌ COMPILE ERROR — init only!
                     // Properties can only be set during construction

// Perfect for DTOs — data should not change after creation

with Expression — Non-Destructive Mutation

Since records are immutable — create a new one with modified values using with:

var original = new PersonRecord("Omar", 25);

// Create new record — only Age changes — original untouched
var updated = original with { Age = 26 };

Console.WriteLine(original); // PersonRecord { Name = Omar, Age = 25 }
Console.WriteLine(updated);  // PersonRecord { Name = Omar, Age = 26 }
// original is NOT modified ✅

Auto-Generated ToString

var person = new PersonRecord("Omar", 25);
Console.WriteLine(person);
// Output: PersonRecord { Name = Omar, Age = 25 }
// No need to override ToString manually ✅

Record Types — C# 9 and 10

// Record class (default) — reference type with value equality
public record PersonRecord(string Name, int Age);

// Record struct (C# 10) — value type with value equality
public record struct Point(int X, int Y);

// Mutable record — possible but NOT recommended
public record MutablePerson
{
    public string Name { get; set; } // set not init — mutable
    public int Age { get; set; }
}

Record Inheritance

public record Animal(string Name);
public record Dog(string Name, string Breed) : Animal(Name);

var dog = new Dog("Rex", "Husky");
Console.WriteLine(dog); // Dog { Name = Rex, Breed = Husky }

// Equality considers full type — not just content
var a1 = new Animal("Rex");
var d1 = new Dog("Rex", "Husky");
Console.WriteLine(a1 == d1); // false — different types

4. Side by Side Comparison

// Class — mutable reference type
var c1 = new PersonClass { Name = "Omar", Age = 25 };
var c2 = c1;           // same reference — NOT a copy
c2.Name = "Ali";
Console.WriteLine(c1.Name); // "Ali" — both changed 😱

// Struct — mutable value type
var s1 = new PersonStruct { Name = "Omar", Age = 25 };
var s2 = s1;           // COPY — independent
s2.Name = "Ali";
Console.WriteLine(s1.Name); // "Omar" — unchanged ✅

// Record — immutable reference type
var r1 = new PersonRecord("Omar", 25);
var r2 = r1 with { Name = "Ali" }; // new copy — original untouched
Console.WriteLine(r1.Name); // "Omar" — unchanged ✅
Console.WriteLine(r1 == new PersonRecord("Omar", 25)); // true ✅

6. When to Use Which

Use Class When:

  • Object has complex behavior and methods
  • Object is mutable — changes state over time
  • You need inheritance from non-record types
  • Example: OrderService, UserRepository, HttpClient

Use Struct When:

  • Small lightweight data — 2 to 4 fields maximum
  • Short-lived — created and discarded quickly
  • Frequently created in very large numbers
  • 16 bytes or less — Microsoft guideline
  • Example: Point(X,Y), Color(R,G,B), DateTime, Guid

Use Record When:

  • Immutable data that represents a value
  • DTOs for API requests and responses
  • Value objects in Domain Driven Design
  • Equality should be based on content not reference
  • Example: OrderDto, AddressValue, ApiResponse, SearchQuery

7. Real World Examples

// ✅ Class — mutable business logic
public class OrderService
{
    private readonly IRepository<Order> _repo;
    private int _processedCount = 0; // mutable state

    public OrderService(IRepository<Order> repo) => _repo = repo;

    public void Process(Order order)
    {
        _processedCount++; // state changes
        _repo.Save(order);
    }
}

// ✅ Struct — small value used heavily
public struct Money
{
    public decimal Amount { get; }
    public string Currency { get; }
    public Money(decimal amount, string currency)
    {
        Amount = amount;
        Currency = currency;
    }
}

// ✅ Record — immutable DTO
public record CreateOrderRequest(
    string CustomerName,
    string ProductId,
    int Quantity,
    Money Price
);

// API controller — record equality useful for caching/deduplication
[HttpPost]
public IActionResult Create(CreateOrderRequest request)
{
    // request is immutable — safe to pass anywhere
    _orderService.Process(request);
    return Ok();
}

8. Interview Questions

Q: What is the difference between class and struct?

🎯 Interview Answer: Classes are reference types stored on the heap — assignment copies the reference so multiple variables can point to the same object. Structs are value types usually stored on the stack — assignment creates a complete independent copy. Classes use reference equality by default so two objects with the same content are not equal unless they are the same instance. Structs use value equality so two structs with the same content are equal.

Q: What is a record in C#?

🎯 Interview Answer: A record introduced in C# 9 is a reference type like a class but with value equality like a struct. The compiler automatically generates equality methods, GetHashCode, ToString, and with-expression support. Records use init-only properties making them immutable by default. They are ideal for DTOs, API models, and value objects where equality should be based on content not reference identity.

Q: What is the with expression in records?

🎯 Interview Answer: The with expression creates a new record based on an existing one with specific properties changed — the original record is not modified. For example: var updated = original with { Age = 26 } creates a new record identical to original except Age is 26. This is called non-destructive mutation — you get a modified copy while the original stays immutable.

Q: When would you choose record over class?

🎯 Interview Answer: Use record when the type represents immutable data where equality is based on content — like API request and response DTOs, value objects in domain driven design, or configuration objects. Use class when the object has mutable state that changes over time or complex behavior. The key question is: should two objects with the same data be considered equal? If yes — record. If no — class.

Q: Why should structs be kept small?

🎯 Interview Answer: Structs are value types — every assignment, method call, and return creates a complete copy of all fields. A small struct with 2 to 4 fields copies cheaply. A large struct with many fields copies all of them every time it is used, consuming CPU and memory. Microsoft recommends keeping structs at 16 bytes or less. If data is larger or complex, a class is more appropriate because only the reference is copied — not the entire object.
