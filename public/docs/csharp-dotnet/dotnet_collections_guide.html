Collections Deep Dive — Complete Guide

1. List<T> — Ordered Resizable Array

List<T> is backed by an internal array. When it gets full it automatically doubles in capacity — this is called amortized growth.
List<int> numbers = new List<int>();
numbers.Add(1); // internal array: [1, _, _, _]  capacity: 4
numbers.Add(4); // internal array: [1, 2, 3, 4]  capacity: 4
numbers.Add(5); // FULL! → doubles to capacity 8 → copies all elements

Performance Characteristics

List<int> list = new List<int> { 1, 2, 3, 4, 5 };

int item  = list[2];        // O(1) — direct index access — fast ✅
bool ex   = list.Contains(4); // O(n) — scans every element — slow 😐
list.Add(6);                // O(1) amortized — fast ✅
list.Insert(2, 99);         // O(n) — shifts all elements after index — slow 😐
list.Remove(3);             // O(n) — shifts all elements after removed — slow 😐

When to Use List<T>

  • You need an ordered collection
  • You access items by index
  • You need duplicate values
  • You iterate through all items frequently
  • ❌ Avoid for frequent Contains checks — use HashSet instead
  • ❌ Avoid for frequent middle insert/remove — use LinkedList instead

2. Dictionary<TKey, TValue> — Hash Map

Dictionary uses hashing to achieve O(1) key lookups. Understanding why it's fast requires understanding how hashing works internally.

How Hashing Works Internally

// Adding "Ali" → 25
// Step 1: Calculate hash → "Ali".GetHashCode() → 12345678
// Step 2: Map to bucket → 12345678 % bucketCount → bucket 3
// Step 3: Store ("Ali", 25) in bucket 3

// Reading ages["Ali"]
// Step 1: Calculate hash → 12345678
// Step 2: Go directly to bucket 3
// Step 3: Return 25
// No scanning. No looping. Direct jump. That's why O(1). ✅

Performance Characteristics

Dictionary<string, int> dict = new Dictionary<string, int>();

dict["Ali"] = 25;              // O(1) — add
int age = dict["Ali"];         // O(1) — read by key — fast ✅
bool ex = dict.ContainsKey("Ali");   // O(1) — fast ✅
bool ex = dict.ContainsValue(25);    // O(n) — scans all values — slow 😐
dict.Remove("Ali");            // O(1) — remove

Always Use TryGetValue for Safe Reading

// ❌ Dangerous — throws KeyNotFoundException if key missing
int age = dict["NonExistent"];

// ✅ Safe — no exception
if (dict.TryGetValue("Ali", out int age))
{
    Console.WriteLine("Age: " + age);
}
else
{
    Console.WriteLine("Key not found");
}

Iterating a Dictionary

foreach (KeyValuePair<string, int> pair in dict)
    Console.WriteLine($"{pair.Key}: {pair.Value}");

// Modern deconstruction syntax
foreach (var (key, value) in dict)
    Console.WriteLine($"{key}: {value}");

3. HashSet<T> — Unique Values with Fast Lookup

HashSet<T> is essentially a Dictionary<T, nothing>. It uses the same hashing mechanism but only stores keys — no values. Duplicate adds are silently ignored.
HashSet<int> ids = new HashSet<int>();
ids.Add(1);  // ✅ added
ids.Add(2);  // ✅ added
ids.Add(1);  // ❌ duplicate — silently ignored, returns false
ids.Add(3);  // ✅ added
Console.WriteLine(ids.Count); // 3 — not 4

HashSet vs List — Contains Performance

// With 1 million items, checking Contains 1000 times:
List<int> list = new List<int>(/* 1M items */);
bool ex = list.Contains(999999); // O(n) — scans up to 1M elements 😱
// Total: 1,000 × 1,000,000 = 1 billion checks

HashSet<int> set = new HashSet<int>(/* 1M items */);
bool ex = set.Contains(999999);  // O(1) — instant hash lookup ✅
// Total: 1,000 × 1 = 1,000 checks

Set Operations — Unique to HashSet

HashSet<int> setA = new HashSet<int> { 1, 2, 3, 4, 5 };
HashSet<int> setB = new HashSet<int> { 3, 4, 5, 6, 7 };

setA.UnionWith(setB);      // all elements from both → 1,2,3,4,5,6,7
setA.IntersectWith(setB);  // only in BOTH sets    → 3,4,5
setA.ExceptWith(setB);     // in A but NOT in B    → 1,2
bool sub = setA.IsSubsetOf(setB); // is A fully inside B?

// Real world — finding common users between two lists (O(n) not O(n²))
var commonIds = new HashSet<int>(listA);
commonIds.IntersectWith(listB); // fast intersection ✅

4. Queue<T> — First In First Out (FIFO)

Queue works like a real queue at a ticket counter — first person to join is first to be served.
Queue<string> printQueue = new Queue<string>();

// Enqueue — add to the BACK
printQueue.Enqueue("Document1.pdf");
printQueue.Enqueue("Document2.pdf");
printQueue.Enqueue("Document3.pdf");
// State: [Document1 ← front ... Document3 ← back]

// Dequeue — remove from the FRONT
string next = printQueue.Dequeue(); // returns "Document1.pdf"
// State: [Document2 ← front ... Document3 ← back]

// Peek — see front WITHOUT removing
string front = printQueue.Peek(); // returns "Document2.pdf"

Console.WriteLine(printQueue.Count); // 2

Real World Use Cases

  • Print queues — first document sent = first printed
  • Message queues — first message sent = first processed
  • Task scheduling — process tasks in order received
  • BFS (Breadth First Search) — graph traversal algorithm

5. Stack<T> — Last In First Out (LIFO)

Stack works like a stack of plates — last plate placed on top is first to be taken off.
Stack<string> browserHistory = new Stack<string>();

// Push — add to the TOP
browserHistory.Push("google.com");
browserHistory.Push("github.com");
browserHistory.Push("stackoverflow.com");
// State: stackoverflow.com ← TOP
//        github.com
//        google.com        ← BOTTOM

// Pop — remove from the TOP
string lastPage = browserHistory.Pop(); // returns "stackoverflow.com"
// State: github.com ← TOP
//        google.com

// Peek — see top WITHOUT removing
string current = browserHistory.Peek(); // returns "github.com"

Real World Use Cases

  • Browser back button — last page visited = first to go back to
  • Undo/Redo operations — last action = first to undo
  • Call stack — last method called = first to return
  • DFS (Depth First Search) — graph traversal algorithm
  • Expression parsing — compilers use stacks

6. LinkedList<T> — Frequent Middle Insert/Remove

LinkedList stores elements as nodes — each node points to both the next and previous node. No index-based access but extremely fast insert/remove anywhere in the list.
// Structure: null ← [1] ↔ [2] ↔ [3] ↔ [4] → null

LinkedList<int> linked = new LinkedList<int>();
linked.AddFirst(1);  // [1]
linked.AddLast(2);   // [1, 2]
linked.AddLast(3);   // [1, 2, 3]

// Insert after specific node — O(1) — just change pointers
LinkedListNode<int> node = linked.Find(2);
linked.AddAfter(node, 99); // [1, 2, 99, 3]

// Remove — O(1) if you have the node
linked.Remove(node);  // [1, 99, 3]

7. Thread-Safe Collections

⚠️ Important: Standard collections are NOT thread-safe. If multiple threads access them simultaneously data corruption can occur. Use System.Collections.Concurrent for multi-threaded scenarios.

// ❌ Not thread-safe — data corruption in multi-threaded code
Dictionary<string, int> dict = new Dictionary<string, int>();

// ✅ Thread-safe alternatives
ConcurrentDictionary<string, int> dict = new ConcurrentDictionary<string, int>();
dict.TryAdd("key", 1);
dict.TryGetValue("key", out int value);
dict.AddOrUpdate("key", 1, (k, v) => v + 1);

// Other thread-safe collections

ConcurrentQueue<T> // thread-safe Queue — FIFO

ConcurrentStack<T> // thread-safe Stack — LIFO

ConcurrentBag<T> // thread-safe unordered collection

BlockingCollection<T> // producer-consumer pattern

9. Quick Decision Guide

Need key-value pairs with fast lookup?

→ Dictionary<K, V>

Need unique values with fast existence check?

→ HashSet<T>

Need ordered list with index access?

    → List<T>

Need FIFO processing (task queue)?
    → Queue<T>

    → Stack<T>

→ LinkedList<T>

→ SortedDictionary<K, V>

→ ConcurrentDictionary / ConcurrentQueue / ConcurrentStack

10. Common Interview Questions

Q: What is the difference between List and Dictionary?

🎯 Interview Answer: List<T> is an ordered collection backed by an array — it provides O(1) index access but O(n) search by value. Dictionary<K,V> uses hashing to provide O(1) lookup by key — it is best when you need to quickly find items by a specific identifier. Use List when order matters and you iterate frequently, use Dictionary when fast key-based lookup is the priority.

Q: What is the difference between List and HashSet?

🎯 Interview Answer: List<T> allows duplicates and maintains order — Contains is O(n) because it scans every element. HashSet<T> stores only unique values using hashing — Contains is O(1) instant lookup. If you only need to check existence and don't need order or duplicates, HashSet is dramatically more efficient for large datasets.

Q: When would you use Queue vs Stack?

🎯 Interview Answer: Queue is FIFO — first item added is first to be processed. Use it for task queues, message processing, and BFS graph traversal where order of arrival matters. Stack is LIFO — last item added is first to be processed. Use it for undo/redo operations, browser history, call stack simulation, and DFS graph traversal.

Q: Why are standard collections not thread-safe?

🎯 Interview Answer: Standard collections like List and Dictionary are not synchronized — if multiple threads read and write simultaneously, operations like internal array resizing in List or bucket reorganization in Dictionary can leave the collection in a corrupted state. For multi-threaded scenarios, use the ConcurrentDictionary, ConcurrentQueue, and ConcurrentStack classes from System.Collections.Concurrent which are designed for safe concurrent access.

Q: How does Dictionary achieve O(1) lookup?

🎯 Interview Answer: Dictionary uses hashing — when you add a key it calls GetHashCode() on the key, maps the hash to a bucket index, and stores the value there. When you read, it performs the same hash calculation and jumps directly to the correct bucket without scanning. This makes lookups O(1) on average. Hash collisions where two keys map to the same bucket are handled with chaining and are rare in practice.
