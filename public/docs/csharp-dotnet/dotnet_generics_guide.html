Generics — Complete Guide

1. What Are Generics?

Generics allow you to write type-safe reusable code without repeating the same logic for each type. The type parameter T is a placeholder that gets replaced with the actual type at compile time.

The Problem Without Generics — Code Duplication:

// Without generics — must repeat for every type

public class NumberBox
{
    private int _value;
    public void Set(int value) { _value = value; }
    public int Get() { return _value; }
}

public class StringBox
{
    private string _value;
    public void Set(string value) { _value = value; }
    public string Get() { return _value; }
}
// Same logic — duplicated for every type 😱

The Solution — One Generic Class:

// With generics — one class handles ALL types
public class Box<T>
{
    private T _value;
    public void Set(T value) { _value = value; }
    public T Get() { return _value; }
}

// Usage — T replaced at compile time
Box<int>    intBox    = new Box<int>();
Box<string> stringBox = new Box<string>();
Box<Order>  orderBox  = new Box<Order>();

intBox.Set(42);
Console.WriteLine(intBox.Get()); // 42

⚠️ Important: T is just a convention. You can name it anything: TItem, TEntity, TKey, TValue. Descriptive names are preferred for clarity.

2. Generic Methods

You can make individual methods generic without making the whole class generic:

public class Utilities
{
    // Generic method — works with any type
    public T GetFirst<T>(List<T> items)
    {
        if (items == null || items.Count == 0)
            throw new ArgumentException("List is empty");

        return items[0];
    }

    // Generic method with multiple type parameters
    public Dictionary<TKey, TValue> Zip<TKey, TValue>(
        List<TKey> keys,
        List<TValue> values)
    {
        var result = new Dictionary<TKey, TValue>();
        for (int i = 0; i < keys.Count; i++)
            result[keys[i]] = values[i];
        return result;
    }
}

Type inference — compiler determines T from the argument automatically:

var utils = new Utilities();

// Compiler infers T = int
int first = utils.GetFirst(new List<int> { 1, 2, 3 });

// Compiler infers T = string
string name = utils.GetFirst(new List<string> { "Ali", "Omar" });

3. Generic Constraints

Constraints restrict what types T is allowed to be. Without constraints, T is completely unknown and you cannot call any methods on it.

1. where T : class — Reference Types Only

public class Repository<T> where T : class
{
    public void Add(T item)
    {
        if (item == null)                    // ✅ can check null — T is reference type
            throw new ArgumentNullException();
    }
}

var repo = new Repository<Order>();   // ✅ Order is a class
var repo = new Repository<int>();     // ❌ compile error — int is value type

2. where T : struct — Value Types Only

public class ValueWrapper<T> where T : struct
{
    private T _value;
    public ValueWrapper(T value) { _value = value; }
    public T Value => _value;
}

var wrapper = new ValueWrapper<int>(42);     // ✅ int is value type
var wrapper = new ValueWrapper<string>("");  // ❌ string is reference type

3. where T : new() — Must Have Parameterless Constructor

public class Factory<T> where T : new()
{
    public T Create()
    {
        return new T(); // ✅ can create instance — guaranteed constructor exists
    }
}

class Order   { public Order() { } }            // ✅ has parameterless constructor
class Invoice { public Invoice(int id) { } }    // ❌ no parameterless constructor

var factory = new Factory<Order>();    // ✅
var factory = new Factory<Invoice>();  // ❌ compile error

4. where T : BaseClass — Must Inherit From Class

public class AnimalShelter<T> where T : Animal
{
    private List<T> _animals = new List<T>();

    public void Add(T animal)
    {
        animal.Feed();  // ✅ can call Animal methods — T is guaranteed to be Animal
        _animals.Add(animal);
    }
}

var shelter = new AnimalShelter<Dog>(); // ✅ Dog inherits Animal
var shelter = new AnimalShelter<Car>(); // ❌ Car doesn't inherit Animal

5. where T : IInterface — Must Implement Interface

public class Sorter<T> where T : IComparable<T>
{
    public T FindMax(List<T> items)
    {
        T max = items[0];
        foreach (T item in items)
        {
            if (item.CompareTo(max) > 0) // ✅ can call CompareTo — T implements IComparable
                max = item;
        }
        return max;
    }
}

var sorter = new Sorter<int>();
int max = sorter.FindMax(new List<int> { 3, 1, 4, 1, 5, 9 }); // 9

6. Multiple Constraints Together

public class Repository<T>
    where T : class,        // must be reference type
              IEntity,      // must implement IEntity interface
              new()         // must have parameterless constructor
{
    public T CreateAndSave()
    {
        T entity = new T();          // ✅ new() constraint
        entity.Id = Guid.NewGuid();  // ✅ IEntity constraint
        Save(entity);
        return entity;
    }
}

4. Generic Repository Pattern — Real World

The most common real-world use of generics in backend development:

// Generic repository interface
public interface IRepository<T> where T : class
{
    T GetById(int id);
    IEnumerable<T> GetAll();
    void Add(T entity);
    void Update(T entity);
    void Delete(int id);
}

// Generic base implementation — write once, use for all entities
public class Repository<T> : IRepository<T> where T : class
{
    protected readonly AppDbContext _context;
    protected readonly DbSet<T> _dbSet;

    public Repository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }

    public T GetById(int id) => _dbSet.Find(id);
    public IEnumerable<T> GetAll() => _dbSet.ToList();
    public void Add(T entity) { _dbSet.Add(entity); _context.SaveChanges(); }
    public void Update(T entity) { _dbSet.Update(entity); _context.SaveChanges(); }
    public void Delete(int id)
    {
        var entity = GetById(id);
        if (entity != null) { _dbSet.Remove(entity); _context.SaveChanges(); }
    }
}

// Specific repositories — just inherit, zero repeated code
public class OrderRepository : Repository<Order>
{
    public OrderRepository(AppDbContext context) : base(context) { }

    // Add Order-specific methods only
    public IEnumerable<Order> GetPendingOrders()
        => _dbSet.Where(o => o.Status == "Pending").ToList();
}

public class CustomerRepository : Repository<Customer>
{
    public CustomerRepository(AppDbContext context) : base(context) { }
}

5. Covariance and Contravariance

The Problem — Why Generics Are Invariant By Default

Even though Dog inherits from Animal, List<Dog> cannot be assigned to List<Animal>. This is called invariance.
class Animal { }
class Dog : Animal { }

Animal animal = new Dog();            // ✅ normal inheritance — works fine
List<Animal> animals = new List<Dog>(); // ❌ compile error — invariant!

Why? Because this dangerous scenario would be possible:

List<Animal> animals = new List<Dog>(); // imagine this worked
animals.Add(new Cat()); // Cat IS an Animal — compiler allows it
                        // BUT internally it's a List<Dog>
                        // Cat inside a Dog list = type safety broken 😱

Covariance — out Keyword (Producers)

Covariance allows a more derived type to be used where a base type is expected. Safe when T only comes OUT as a return value — never goes IN as a parameter.
// IEnumerable is covariant — T only comes out
public interface IEnumerable<out T>
{
    IEnumerator<T> GetEnumerator(); // T only returned — never a parameter
}

// Because of "out" — this works
IEnumerable<Animal> animals = new List<Dog>(); // ✅ covariance!

// Your own covariant interface
public interface IProducer<out T>
{
    T Produce();   // T only comes OUT ✅
}

class DogProducer : IProducer<Dog>
{
    public Dog Produce() => new Dog();
}

IProducer<Animal> producer = new DogProducer(); // ✅
// DogProducer.Produce() returns Dog
// Dog IS an Animal — perfectly safe

Contravariance — in Keyword (Consumers)

Contravariance allows a less derived type to be used where a more derived type is expected. Safe when T only goes IN as a parameter — never comes OUT as a return value.
// Your own contravariant interface
public interface IProcessor<in T>
{
    void Process(T item); // T only goes IN ✅
}

class AnimalProcessor : IProcessor<Animal>
{
    public void Process(Animal animal)
    {
        Console.WriteLine("Processing: " + animal.GetType().Name);
    }
}

IProcessor<Dog> processor = new AnimalProcessor(); // ✅ contravariance!
// AnimalProcessor.Process() accepts ANY Animal
// Dog IS an Animal — so it can definitely process a Dog ✅
processor.Process(new Dog());

Mental Model Summary

COVARIANCE  (out) → Producer → T comes OUT → use MORE derived type (Dog for Animal)
CONTRAVARIANCE (in) → Consumer → T goes IN  → use LESS derived type (Animal for Dog)
INVARIANCE        → both in/out             → exact type only (List<T>)

6. default(T) Keyword

Returns the default value for any type — very useful in generic code:

public T GetValueOrDefault<T>(T? value) where T : struct
{
    return value ?? default(T);
}

// default values:
// default(int)      = 0
// default(bool)     = false
// default(double)   = 0.0
// default(string)   = null
// default(Order)    = null

8. Common Interview Questions

Q: What is the difference between generics and using object type?

🎯 Interview Answer: Using object requires boxing and unboxing for value types which hurts performance and loses type safety at compile time. Generics are type-safe at compile time — the type is checked when you write the code, not at runtime. They also avoid boxing completely because the actual type is used directly.

Q: What are generic constraints and why do we use them?

🎯 Interview Answer: Generic constraints restrict what types T can be. Without constraints T is completely unknown — you cannot call any methods on it. Constraints let you guarantee what capabilities T has. For example where T : IComparable<T> lets you call CompareTo(), and where T : new() lets you create instances with new T().

Q: What is the difference between covariance and contravariance?

🎯 Interview Answer: Covariance uses the out keyword and allows a more derived type to be used where a base type is expected — safe because T only flows out as a return value. IEnumerable<T> is covariant so List<Dog> can be assigned to IEnumerable<Animal>. Contravariance uses the in keyword and allows a less derived type to be used where a more derived type is expected — safe because T only flows in as a parameter. A processor that handles Animals can safely be used where a Dog processor is expected because Dog is an Animal.

Q: Where do you use generics in real projects?

🎯 Interview Answer: The most common real-world use is the generic repository pattern where a single base repository handles all CRUD operations for any entity type. This eliminates code duplication across repositories — specific repositories just inherit from the generic base and only add entity-specific methods.
