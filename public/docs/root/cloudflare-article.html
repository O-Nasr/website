Here’s a summary of the Cloudflare article “Counting Things: A Lot of Different Things” — which is referenced in System Design Interview Vol. 2 to illustrate scalable rate limiting and distributed counting. [blog.cloudflare.com]

🧠 Core Problem

Cloudflare needed to build a rate limiter that could:

  • Scale to millions of domains
  • Handle billions of requests per day
  • Operate at the edge of their global network
  • Be fast, accurate, and resilient under attack

🔧 Key Challenges

  1. Distributed Traffic: Requests are spread across many servers and data centers (PoPs).
  2. Latency Sensitivity: Centralized counting is too slow due to network delays.
  3. Availability: Central services are harder to keep online during high traffic or attacks.
  4. Efficiency: Counting must be lightweight and fast, even under load.

🛠️ Design Solutions

1. Local Counting in PoPs

  • Each Point of Presence (PoP) maintains its own counters.
  • This reduces latency and improves fault tolerance.

2. Memcached-Based Sharded Counters

  • Twemproxy clusters are used to shard counters across servers.
  • Consistent hashing ensures minimal disruption when resizing clusters.

3. Rate Limiting Algorithms

  • Fixed Window Counter: Simple but allows spikes at window boundaries.
  • Sliding Window Approximation: Uses previous and current counters to smooth traffic and improve accuracy.
  • Leaky Bucket: Accurate but harder to implement with Memcached due to atomicity limitations.

4. Asynchronous Mitigation

  • Counting is done asynchronously to avoid slowing down legitimate requests.
  • Once a client exceeds the threshold, mitigation is triggered and cached locally.
  • Servers don’t re-check the counter for every request — they just enforce the mitigation.

📊 Performance Results

  • Handles billions of requests/day
  • Mitigated attacks with 400,000 requests/sec to a single domain
  • Achieved 0.003% error rate in rate estimation across 400 million requests

🌍 What Is the Edge of the Network?

The edge of the network refers to the geographically distributed servers or nodes that are closest to the end users — often part of a Content Delivery Network (CDN) or edge computing infrastructure.

🧠 Think of it like:

Instead of sending every request to a central data center, you handle it as close to the user as possible — at the “edge.”

📌 Why Is This Important?

In the article, Cloudflare talks about building a rate limiter that runs at the edge. This means:

  • The rate limiter is deployed on edge servers around the world.
  • It can process and enforce limits locally, without needing to contact a central server.
  • This allows Cloudflare to handle millions of domains and billions of requests efficiently.

🔍 Your Questions:

1. Does the edge server only handle rate limiting?

Not just rate limiting — edge servers can do much more.

In Cloudflare’s architecture, edge servers:

  • Inspect and filter traffic (e.g., rate limiting, bot detection, firewall rules)
  • Serve cached content (e.g., images, HTML, JS)
  • Terminate TLS connections
  • Run edge logic (e.g., Cloudflare Workers)

So yes, rate limiting is one task, but edge servers can also serve content directly if it's cached or processed locally.

2. What does “serves the response” mean? Does it redirect to the data center?

No — it does not redirect unless necessary.

“Serves the response” means the edge server itself sends the final response back to the user.

🔧 Two Possibilities:

✅ If the content is cached or handled at the edge:

  • The edge server responds directly.
  • No need to contact the origin server (e.g., in the US).

❌ If the content is not cached or needs origin data:

  • The edge server forwards the request to the origin (e.g., US data center).
  • Gets the response → caches it → sends it to the user.

So, redirecting is not the right term — it’s more like proxying or fetching from the origin if needed.
