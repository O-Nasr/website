Milvus

✅ 1. Vector

A vector is a list/array of numbers (float32 or float16 or binary) that represents meaning from unstructured data (text, images, audio, video).
Milvus documentation explains that vectors/embeddings are the heart of Milvus, converting unstructured data into searchable numeric representations.

Examples:

  • Text: “cat” → [0.11, -0.42, 0.87, ...]
  • Image: an image of a dog → [0.01, 0.19, -0.55, ...]

Vectors allow semantic search — you search for similar meaning, not keywords.

Milvus supports:

  • dense vectors (float)
  • binary vectors
  • sparse vectors

✅ 2. Collection

A collection is like a table in SQL.
It has:

  • Fields (columns)
  • One or more vector fields
  • Optional metadata fields (int, string, JSON, array, etc.)

Official docs: Milvus organizes data into collections to store multi‑modal vectors & metadata in a structured way.

Example schema:

  • id (primary key)
  • embedding (vector[768])
  • text (string metadata)
  • timestamp (int metadata)

✅ 3. Field

A field is a column inside a collection. Types include:

  • FloatVector
  • BinaryVector
  • Int64, Float, VarChar
  • JSON
  • Array

Milvus supports mixed types to allow hybrid search (vectors + filters).

✅ 4. Primary Key (PK)

Every collection has exactly one primary key field.

  • Can be auto‑ID (Milvus generates IDs)
  • Or a custom ID (you provide IDs)

This is similar to SQL table primary keys.

✅ 5. Partition

A partition is a logical grouping of data inside a collection.

Why use partitions?

  • To split data by category (e.g., “news”, “sports”, “finance”)
  • Improve search performance by searching only a subset

Milvus recommends partitions when doing hybrid metadata filtering.

✅ 6. Segment

Segments are physical data storage units in Milvus.

  • When you insert data, Milvus groups it into segments.
  • Segments are persisted into MinIO/S3.
  • Segments get indexed (HNSW, IVF, DiskANN) later.

Segments are part of Milvus’s internal architecture and are important for performance.

✅ 7. Index

An index speeds up vector similarity search.
Milvus supports multiple ANN (Approximate Nearest Neighbor) index types:

Index

Use‑case

HNSW

high recall, fast search, graph‑based

IVF_FLAT / IVF_PQ

large datasets, trade‑off between speed and accuracy

DiskANN

extremely large datasets, minimal RAM usage

BIN_FLAT / BIN_IVF

binary vector search

Milvus documentation highlights its rich support for index types including HNSW, DiskANN, quantization, and binary indexes.
Indexes are built per segment, not per entire collection.

✅ 8. Metric Type

A similarity metric tells Milvus how to compare vectors.

Common metrics:

  • L2 (Euclidean): distance between vectors
  • IP (Inner Product): great for cosine similarity if vectors are normalized
  • COSINE
  • HAMMING (binary)
  • Jaccard (binary)

Milvus supports metric types for floating, binary, and sparse vectors.

✅ 9. Search (Top‑K / Range)

Two major search types:

⚫ Top‑K search

Find the K most similar vectors to the query vector.

⚫ Range search

Find all vectors within a distance threshold.

Milvus also supports hybrid filters:

  • search vectors AND filter metadata at the same time

Example:

Find top‑5 similar embeddings where category = "sports" and year > 2020

✅ 10. Loading / Unloading

Before you search, you must load a collection (or partition) into memory.

  • Load = bring vector index segments into RAM
  • Release = free RAM but keep data in storage

This is part of Milvus’s architecture to scale to billions of vectors.

✅ 11. Consistency Level

Milvus supports different read-after-write consistency levels:

  • Strong
  • Session
  • Bounded
  • Eventually

This determines how fresh the data must be in a search.
Useful in real‑time ingestion systems.
