Structural Patterns

Adapter • Bridge • Composite • Decorator • Facade • Flyweight • Proxy

Structural patterns deal with how classes and objects are composed to form larger structures. They use inheritance and composition to create flexible, efficient architectures.

🔌 Pattern 1: Adapter

Convert the interface of a class into another interface that clients expect.

Intent

Make two incompatible interfaces work together by placing a translator — the Adapter — between them. The client talks to the Adapter through the interface it expects, and the Adapter translates the call to the incompatible class.

The Problem It Solves

You want to use a third-party class but its interface does not match what your code expects. You cannot change either side. The Adapter sits in the middle and translates.

Structure

Client  ->  ITarget.Request()
                 |
           Adapter.Request()   <- wraps the Adaptee
                 |
           Adaptee.SpecificRequest()

C# Implementation

Step 1 — The target interface your client expects:

public interface IMediaPlayer
{
    void Play(string fileName);
}

Step 2 — The client that works with IMediaPlayer:

public class MediaApp
{
    private readonly IMediaPlayer _player;
    public MediaApp(IMediaPlayer player) { _player = player; }
    public void PlaySong(string file)    { _player.Play(file); }
}

Step 3 — The third-party class you cannot change (Adaptee):

// Third-party — cannot be modified
public class AdvancedAudioPlayer
{
    public void PlayAudio(string fileName)
        => Console.WriteLine($"Advanced player: {fileName}");
}

Step 4 — The Adapter (the translator):

public class AudioPlayerAdapter : IMediaPlayer
{
    private readonly AdvancedAudioPlayer _advanced;
    public AudioPlayerAdapter(AdvancedAudioPlayer advanced)
        => _advanced = advanced;
    // Translates Play() -> PlayAudio()
    public void Play(string fileName)
        => _advanced.PlayAudio(fileName);
}

Step 5 — Client code:

var advanced = new AdvancedAudioPlayer();
IMediaPlayer player = new AudioPlayerAdapter(advanced);
var app = new MediaApp(player);
app.PlaySong("song.mp3");
// MediaApp never changed. AdvancedAudioPlayer never changed.
// Only the Adapter is new code.

💡 Note: Object Adapter (uses composition) is the standard in C# because C# does not support multiple class inheritance. Class Adapter would require extending both Target and Adaptee simultaneously.

Interview Q&A

Q: What is the difference between Object Adapter and Class Adapter?

Object Adapter uses composition — wraps the adaptee as a field. Class Adapter uses inheritance — extends the adaptee. In C#, Object Adapter is standard because C# does not support multiple class inheritance.

Q: When would you use Adapter vs Facade?

Use Adapter when you need to match a specific existing interface your client already expects. Use Facade when you want to simplify a complex system — with no existing interface contract to match.

Q: Does Adapter change the behavior of the adaptee?

No. Adapter only translates the interface — it does not add, remove, or modify the underlying logic. If you need to add behavior, combine Adapter with Decorator.

When to Use / Avoid

✅ Use When

❌ Avoid When

Use an existing class but its interface does not match

You control both classes — just change one directly

Integrating third-party or legacy code you cannot modify

The interface difference is so large that adaptation is a full rewrite

Your system needs to work with multiple incompatible sources

Wrapping adds unnecessary complexity for a simple call

🌉 Pattern 2: Bridge

Decouple an abstraction from its implementation so the two can vary independently.

Intent

Separate a class into two independent hierarchies — abstraction and implementation — connected by a reference (the bridge). This prevents class explosion when two dimensions vary independently.

The Problem It Solves — Class Explosion

When two dimensions combine via inheritance, adding one new thing forces many new classes. Bridge replaces m x n subclasses with m + n classes.

❌ Without Pattern

// Inheritance — combines both dimensions
class AlertEmailNotification : Notification { }
class AlertSMSNotification   : Notification { }
class ReminderEmailNotification : Notification { }
class ReminderSMSNotification   : Notification { }
// Add Push channel -> 2 new classes
// Add Urgent type  -> 3 new classes
// Growth: m x n

✅ With Pattern

// Bridge — two independent hierarchies
// Notifications (abstraction side)
class AlertNotification    : Notification { }
class ReminderNotification : Notification { }
// Channels (implementation side)
class EmailSender : IMessageSender { }
class SMSSender   : IMessageSender { }
// Add Push -> 1 class. Add Urgent -> 1 class
// Growth: m + n

C# Implementation

Step 1 — Implementation interface (right side):

public interface IMessageSender
{
    void SendMessage(string message);
}
public class EmailSender : IMessageSender
{ public void SendMessage(string msg) => Console.WriteLine($"Email: {msg}"); }
public class SMSSender : IMessageSender
{ public void SendMessage(string msg) => Console.WriteLine($"SMS: {msg}"); }
public class PushSender : IMessageSender
{ public void SendMessage(string msg) => Console.WriteLine($"Push: {msg}"); }

Step 2 — Abstraction (left side) with the bridge reference:

public abstract class Notification
{
    // The bridge — Association to the implementation side
    protected readonly IMessageSender _sender;
    protected Notification(IMessageSender sender)
        => _sender = sender;
    public abstract void Notify(string message);
}
public class AlertNotification : Notification
{
    public AlertNotification(IMessageSender sender) : base(sender) { }
    public override void Notify(string message)
        => _sender.SendMessage($"ALERT: {message}");
}
public class ReminderNotification : Notification
{
    public ReminderNotification(IMessageSender sender) : base(sender) { }
    public override void Notify(string message)
        => _sender.SendMessage($"REMINDER: {message}");
}

Step 3 — Client code — mix and match freely:

Notification n1 = new AlertNotification(new EmailSender());
Notification n2 = new AlertNotification(new SMSSender());
Notification n3 = new AlertNotification(new PushSender());
Notification n4 = new ReminderNotification(new EmailSender());
n1.Notify("Server is down!");  // Email: ALERT: Server is down!
n2.Notify("Server is down!");  // SMS: ALERT: Server is down!
n3.Notify("Meeting at 3pm");   // Push: REMINDER: Meeting at 3pm

💡 Note: The book uses the word 'composition' to mean 'hold a reference instead of inheriting' — following the OOP principle of favoring composition over inheritance. The precise UML relation in Bridge is Association: the sender is injected, not owned.

Interview Q&A

Q: What does 'the bridge' literally refer to in code?

The bridge is the reference field — _sender inside Notification. It connects the two hierarchies at runtime. It is a single injected reference that allows any abstraction to work with any implementation.

Q: How is Bridge related to 'favor composition over inheritance'?

Bridge is a direct application of that principle. Instead of creating m x n subclasses, it uses a composed reference to connect two independent hierarchies — replacing class explosion with a clean m + n structure.

When to Use / Avoid

✅ Use When

❌ Avoid When

Two dimensions of a class vary independently

Only one dimension varies — plain inheritance is fine

You want to avoid class explosion from combining hierarchies

Abstraction and implementation are always paired together

You want to switch implementations at runtime

The system is small enough that the pattern adds unnecessary complexity

🌿 Pattern 3: Composite

Compose objects into tree structures and treat individual objects and compositions uniformly.

Intent

Compose objects into tree structures to represent part-whole hierarchies. The client treats a single object (Leaf) and a group of objects (Composite) through the same interface — with no type checking needed.

Structure

Component (common interface)

|-- Leaf -> end node, no children, does the work directly

+-- Composite -> has children List<Component>, delegates to them

Client calls the same method on both Leaf and Composite.
The call propagates down the tree automatically.

C# Implementation

Step 1 — Component interface:

public interface IOrderItem
{
    string Name     { get; }
    double GetPrice();
    void   Print(string indent = "");
}

Step 2 — Leaf (individual product):

public class Product : IOrderItem
{
    public string Name    { get; }
    private double _price;
    public Product(string name, double price)
    { Name = name; _price = price; }
    public double GetPrice() => _price;
    public void Print(string indent = "")
        => Console.WriteLine($"{indent}Product: {Name} = ${_price}");
}

Step 3 — Composite (box containing items):

public class Box : IOrderItem
{
    public string Name { get; }
    private readonly List<IOrderItem> _items = new();
    public Box(string name) { Name = name; }
    public void Add(IOrderItem item)    => _items.Add(item);
    public void Remove(IOrderItem item) => _items.Remove(item);
    // Delegates to all children — doesn't care if Leaf or Composite
    public double GetPrice() => _items.Sum(i => i.GetPrice());
    public void Print(string indent = "")
    {
        Console.WriteLine($"{indent}Box: {Name}");
        foreach (var item in _items)
            item.Print(indent + "  ");
    }
}

Step 4 — Client code:

var phone    = new Product("Phone",    699.99);
var charger  = new Product("Charger",   29.99);
var earbuds  = new Product("Earbuds",   49.99);
var giftCard = new Product("Gift Card", 50.00);
var accessoryBox = new Box("Accessory Box");
accessoryBox.Add(charger);
accessoryBox.Add(earbuds);
var orderBox = new Box("Order Box");
orderBox.Add(phone);
orderBox.Add(accessoryBox);  // box inside a box
orderBox.Add(giftCard);
Console.WriteLine($"Total: ${orderBox.GetPrice()}");
orderBox.Print();
// Total: $829.97
// Box: Order Box
//   Product: Phone = $699.99
//   Box: Accessory Box
//     Product: Charger = $29.99
//     Product: Earbuds = $49.99
//   Product: Gift Card = $50.00

💡 Note: The client called GetPrice() once on the root. The call propagated down the entire tree automatically — through boxes and products — without the client knowing or caring about the structure.

Interview Q&A

Q: What is the difference between a Leaf and a Composite?

A Leaf is an end node — it has no children and performs the operation directly. A Composite holds a collection of children and delegates the operation to them recursively. Both implement the same interface so the client cannot tell them apart.

Q: What UML relation exists between a Composite and its children?

Aggregation — the children are added from outside via Add() and can exist independently. The Composite holds a collection reference but does not own the lifecycle of its children.

When to Use / Avoid

✅ Use When

❌ Avoid When

You have a part-whole tree structure

Your structure is flat — no nesting needed

Clients should treat leaves and composites uniformly

Leaf and Composite are too different to share an interface

Operations need to propagate recursively through the tree

The tree is fixed and simple — the pattern adds unnecessary complexity

🎨 Pattern 4: Decorator

Attach additional behavior to an object dynamically without modifying the original class.

Intent

Wrap an object inside another object that adds behavior, while keeping the same interface. Decorators can be stacked in any combination — solving the class explosion problem that inheritance creates for behavior combinations.

C# Implementation

Step 1 — Component interface:

public interface ICoffee
{
    string GetDescription();
    double GetCost();
}

Step 2 — Concrete Component:

public class SimpleCoffee : ICoffee
{
    public string GetDescription() => "Simple Coffee";
    public double GetCost()        => 1.00;
}

Step 3 — Base Decorator:

public abstract class CoffeeDecorator : ICoffee
{
    private readonly ICoffee _coffee;
    protected CoffeeDecorator(ICoffee coffee)
        => _coffee = coffee;
    public virtual string GetDescription() => _coffee.GetDescription();
    public virtual double GetCost()        => _coffee.GetCost();
}

Step 4 — Concrete Decorators:

public class MilkDecorator : CoffeeDecorator
{
    public MilkDecorator(ICoffee coffee) : base(coffee) { }
    public override string GetDescription() => base.GetDescription() + ", Milk";
    public override double GetCost()        => base.GetCost() + 0.25;
}
public class SugarDecorator : CoffeeDecorator
{
    public SugarDecorator(ICoffee coffee) : base(coffee) { }
    public override string GetDescription() => base.GetDescription() + ", Sugar";
    public override double GetCost()        => base.GetCost() + 0.10;
}
public class VanillaDecorator : CoffeeDecorator
{
    public VanillaDecorator(ICoffee coffee) : base(coffee) { }
    public override string GetDescription() => base.GetDescription() + ", Vanilla";
    public override double GetCost()        => base.GetCost() + 0.50;
}

Step 5 — Client code:

ICoffee coffee = new SimpleCoffee();
coffee = new MilkDecorator(coffee);
coffee = new SugarDecorator(coffee);
coffee = new VanillaDecorator(coffee);
Console.WriteLine($"{coffee.GetDescription()} = ${coffee.GetCost()}");
// Simple Coffee, Milk, Sugar, Vanilla = $1.85
// Stack in any combination
ICoffee doubleSugar = new SugarDecorator(new SugarDecorator(new SimpleCoffee()));
// Simple Coffee, Sugar, Sugar = $1.20

💡 Note: Decorator and Proxy look structurally identical — both wrap one object behind the same interface. The difference is intent: Decorator adds behavior, Proxy controls access. This is a very common interview question.

Decorator vs Inheritance

Inheritance

Decorator

When decided

Compile time — fixed

Runtime — dynamic

Combinations

m x n subclasses

m + n classes

Reversible?

No

Yes — unwrap at runtime

Interview Q&A

Q: What is the difference between Decorator and inheritance?

Inheritance adds behavior at compile time and creates a fixed class hierarchy. Decorator adds behavior at runtime by wrapping objects dynamically — avoiding class explosion for behavior combinations.

Q: Why does the Base Decorator exist?

It holds the wrapped reference and provides default delegation to the wrapped object. Without it, every concrete decorator would duplicate that logic. It is optional for simple cases but important when you have many decorators.

When to Use / Avoid

✅ Use When

❌ Avoid When

Add behavior to objects at runtime

Number of decorators is fixed and small — subclassing is simpler

Many optional feature combinations are needed

Heavily stacked decorators make debugging hard to follow

You want to follow OCP — add behavior without modifying existing code

Client needs to know the specific concrete type

🏛️ Pattern 5: Facade

Provide a simplified interface to a complex subsystem.

Intent

Provide a single entry point that hides the complexity of a subsystem. The client talks to the Facade instead of coordinating many subsystem classes directly. Facade does not add new functionality — it simplifies access.

C# Implementation

Step 1 — Complex subsystem classes:

public class Amplifier
{
    public void TurnOn()           => Console.WriteLine("Amplifier: On");
    public void TurnOff()          => Console.WriteLine("Amplifier: Off");
    public void SetVolume(int vol) => Console.WriteLine($"Volume: {vol}");
}
public class DVDPlayer
{
    public void TurnOn()             => Console.WriteLine("DVD: On");
    public void TurnOff()            => Console.WriteLine("DVD: Off");
    public void SetDisk(string disk) => Console.WriteLine($"Disk: {disk}");
    public void Play()               => Console.WriteLine("DVD: Playing");
    public void Stop()               => Console.WriteLine("DVD: Stopped");
}
public class Projector
{
    public void TurnOn()             => Console.WriteLine("Projector: On");
    public void TurnOff()            => Console.WriteLine("Projector: Off");
    public void SetWideScreenMode()  => Console.WriteLine("Widescreen mode");
}
public class Lights
{
    public void Dim(int level) => Console.WriteLine($"Lights: {level}%");
    public void TurnOn()       => Console.WriteLine("Lights: Full");
}

Step 2 — The Facade:

public class HomeTheaterFacade
{
    private readonly Amplifier _amp;
    private readonly DVDPlayer _dvd;
    private readonly Projector _proj;
    private readonly Lights    _lights;
    public HomeTheaterFacade(Amplifier amp, DVDPlayer dvd,
                             Projector proj, Lights lights)
    { _amp = amp; _dvd = dvd; _proj = proj; _lights = lights; }
    public void WatchMovie(string movie)
    {
        _lights.Dim(30);
        _amp.TurnOn(); _amp.SetVolume(10);
        _proj.TurnOn(); _proj.SetWideScreenMode();
        _dvd.TurnOn(); _dvd.SetDisk(movie); _dvd.Play();
    }
    public void EndMovie()
    {
        _dvd.Stop(); _dvd.TurnOff();
        _proj.TurnOff(); _amp.TurnOff();
        _lights.TurnOn();
    }
}

Step 3 — Client code:

var facade = new HomeTheaterFacade(
    new Amplifier(), new DVDPlayer(),
    new Projector(), new Lights());
facade.WatchMovie("Inception");
// Lights: 30%, Amplifier: On, Volume: 10
// Projector: On, Widescreen mode
// DVD: On, Disk: Inception, Playing
facade.EndMovie();
// DVD: Stopped, DVD: Off, Projector: Off
// Amplifier: Off, Lights: Full

💡 Note: Facade does not lock clients out of the subsystem. A power user can still access Amplifier or DVDPlayer directly for fine-grained control. Facade is a convenience, not a restriction.

Interview Q&A

Q: What is the difference between Facade and Adapter?

Adapter makes two incompatible interfaces work together — it matches an existing expected interface. Facade creates a new simpler interface over a complex subsystem — there is no existing interface contract to match.

Q: What is the difference between Facade and Mediator?

Facade provides a simplified interface to a subsystem — subsystem classes do not know about the Facade. Mediator centralizes two-way communication between objects that are aware of the Mediator. Facade is one-directional simplification, Mediator is two-directional coordination.

When to Use / Avoid

✅ Use When

❌ Avoid When

A subsystem has grown too complex for clients to use directly

The subsystem is already simple — Facade adds no value

You want a simple interface for the most common use cases

All clients need fine-grained control — Facade would just be bypassed

Decouple clients from subsystem internals

Facade needs to expose too many methods — it becomes the subsystem

🪶 Pattern 6: Flyweight

Use sharing to support large numbers of fine-grained objects efficiently.

Intent

When you need millions of similar objects, instead of creating each from scratch, share the common state across all of them. Split object data into intrinsic (shared, immutable) and extrinsic (unique per instance) state.

Intrinsic vs Extrinsic State

Intrinsic State

Extrinsic State

Where stored

Inside the Flyweight

Outside — passed in at runtime

Shared?

Yes — same object reused

No — unique per instance

Mutable?

Never — must be immutable

Can change

Example

Tree name, color, texture

Tree position X, Y

C# Implementation

Step 1 — The Flyweight (shared intrinsic state):

public class TreeType
{
    public string Name    { get; }
    public string Color   { get; }
    public string Texture { get; }  // heavy shared data
    public TreeType(string name, string color, string texture)
    { Name = name; Color = color; Texture = texture; }
    // Extrinsic state passed in — not stored here
    public void Draw(int x, int y)
        => Console.WriteLine($"{Name}[{Color}] at ({x},{y})");
}

Step 2 — Flyweight Factory (the cache gatekeeper):

public class TreeTypeFactory
{
    private readonly Dictionary<string, TreeType> _cache = new();
    public TreeType GetTreeType(string name, string color, string texture)
    {
        string key = $"{name}_{color}_{texture}";
        if (!_cache.ContainsKey(key))
            _cache[key] = new TreeType(name, color, texture);
        return _cache[key];  // always returns shared instance
    }
}

Step 3 — Context (stores unique extrinsic state):

public class Tree
{
    public int X { get; }
    public int Y { get; }
    private readonly TreeType _treeType;  // reference, not a copy
    public Tree(int x, int y, TreeType treeType)
    { X = x; Y = y; _treeType = treeType; }
    public void Draw() => _treeType.Draw(X, Y);
}

Step 4 — Client code:

var factory = new TreeTypeFactory();
// 6 trees, only 2 unique types created
var trees = new List<Tree>
{
    new Tree(1,  5,  factory.GetTreeType("Oak",  "Green", "oak.png")),
    new Tree(10, 20, factory.GetTreeType("Oak",  "Green", "oak.png")),  // reused
    new Tree(30, 15, factory.GetTreeType("Oak",  "Green", "oak.png")),  // reused
    new Tree(5,  40, factory.GetTreeType("Pine", "Dark",  "pine.png")),
    new Tree(25, 60, factory.GetTreeType("Pine", "Dark",  "pine.png")), // reused
};
trees.ForEach(t => t.Draw());
// 5 trees drawn — only 2 TreeType objects exist in memory

💡 Note: Flyweights must be immutable because they are shared across many contexts simultaneously. If one context could modify the flyweight's state, it would affect all other contexts sharing the same object.

Interview Q&A

Q: What is the difference between Flyweight and Singleton?

Singleton guarantees one instance of a class for the entire application. Flyweight manages a pool of multiple shared instances — one per unique intrinsic state. Singleton is one object, Flyweight factory manages many shared objects.

Q: When does Flyweight not help?

When objects have very little shared state — most data is extrinsic and unique per instance. If everything differs between objects, there is nothing worth sharing and the pattern adds complexity for no gain.

When to Use / Avoid

✅ Use When

❌ Avoid When

Huge number of similar objects consuming too much memory

Small number of objects — pattern adds complexity for no gain

Most object state can be made extrinsic and passed in

Most state is unique per object — little to share

You can clearly separate intrinsic from extrinsic state

Objects need to carry their own fully independent mutable state

🔒 Pattern 7: Proxy

Provide a surrogate for another object to control access to it.

Intent

A Proxy sits between the client and the real object — intercepting calls and adding behavior like lazy loading, access control, logging, or caching. The client never knows it is talking to a proxy.

The Four Types of Proxy

Type

Problem Solved

Key Behavior

Virtual

Object is expensive to create

Creates real object only on first access

Protection

Object needs access control

Checks permissions before forwarding

Caching

Operation is expensive to repeat

Returns cached result on repeated calls

Remote

Object lives on another server

Handles network communication transparently

Virtual Proxy — Lazy Loading

public interface IImage { void Display(); }
public class HighResolutionImage : IImage
{
    private readonly string _filePath;
    public HighResolutionImage(string path)
    { _filePath = path; Console.WriteLine($"Loading: {path}"); }
    public void Display() => Console.WriteLine($"Displaying: {_filePath}");
}
public class ImageProxy : IImage
{
    private readonly string _filePath;
    private HighResolutionImage _realImage;  // null until needed
    public ImageProxy(string path) { _filePath = path; }
    public void Display()
    {
        if (_realImage == null)
            _realImage = new HighResolutionImage(_filePath);
        _realImage.Display();
    }
}
IImage image = new ImageProxy("photo.jpg");
// No loading yet
image.Display();  // Loading: photo.jpg -> Displaying: photo.jpg
image.Display();  // Displaying: photo.jpg (no reload)

Protection Proxy — Access Control

public interface IDocument { void View(); void Edit(); void Delete(); }
public class Document : IDocument
{
    public void View()   => Console.WriteLine("Viewing.");
    public void Edit()   => Console.WriteLine("Editing.");
    public void Delete() => Console.WriteLine("Deleting.");
}
public class DocumentProxy : IDocument
{
    private readonly Document _doc;
    private readonly string   _role;
    public DocumentProxy(string role) { _doc = new Document(); _role = role; }
    public void View() => _doc.View();
    public void Edit()
    {
        if (_role == "Editor" || _role == "Admin") _doc.Edit();
        else Console.WriteLine("Access denied: Edit requires Editor role.");
    }
    public void Delete()
    {
        if (_role == "Admin") _doc.Delete();
        else Console.WriteLine("Access denied: Delete requires Admin role.");
    }
}
IDocument viewer = new DocumentProxy("Viewer");
viewer.Edit();    // Access denied: Edit requires Editor role.
viewer.Delete();  // Access denied: Delete requires Admin role.

Caching Proxy

public interface IWeatherService { string GetWeather(string city); }
public class WeatherService : IWeatherService
{
    public string GetWeather(string city)
    {
        Console.WriteLine($"Calling API for {city}...");
        return $"Sunny, 28 in {city}";
    }
}
public class WeatherServiceProxy : IWeatherService
{
    private readonly IWeatherService _service;
    private readonly Dictionary<string,string> _cache = new();
    public WeatherServiceProxy(IWeatherService service)
        => _service = service;
    public string GetWeather(string city)
    {
        if (_cache.ContainsKey(city))
        { Console.WriteLine($"Cache hit: {city}"); return _cache[city]; }
        _cache[city] = _service.GetWeather(city);
        return _cache[city];
    }
}
IWeatherService svc = new WeatherServiceProxy(new WeatherService());
svc.GetWeather("Cairo");   // Calling API for Cairo...
svc.GetWeather("Cairo");   // Cache hit: Cairo
svc.GetWeather("London");  // Calling API for London...

💡 Note: Proxy and Decorator look structurally identical — both wrap one object behind the same interface. Decorator adds new behavior. Proxy controls access. This is the most common interview question about Proxy.

Interview Q&A

Q: What is the difference between Proxy and Facade?

Facade simplifies a complex subsystem — wraps multiple classes behind one simple interface. Proxy controls access to a single object — it has the same interface as the object it wraps. Facade reduces complexity, Proxy controls access.

Q: What is the difference between Virtual Proxy and Lazy initialization?

Lazy initialization is the technique — delay object creation until first use. Virtual Proxy is the pattern that applies this technique while maintaining the same interface as the real object. Lazy<T> in C# is essentially a built-in Virtual Proxy.

When to Use / Avoid

✅ Use When

❌ Avoid When

Object creation is expensive and should be deferred

The real object is cheap — lazy loading adds no value

You need access control without modifying the real object

Simple method guards inside the real class are sufficient

Results are expensive to compute and can be cached

The added indirection makes debugging unnecessarily harder
