SOLID Principles

Single Responsibility • Open/Closed • Liskov Substitution • Interface Segregation • Dependency Inversion

SOLID is a set of five design principles introduced by Robert C. Martin (Uncle Bob). Together they guide you toward code that is easier to maintain, extend, and test. Interviewers don't just ask what each letter stands for — they ask you to spot violations, refactor bad code, and explain trade-offs.

S — Single Responsibility Principle

"A class should have only one reason to change."

What It Means

A class should do one thing and do it well. If a class handles multiple concerns, a change to one concern risks breaking another. The 'reason to change' is the key phrase — count how many different stakeholders could ask you to change this class.

Violation vs Fix

❌ Violation

public class Invoice
{
    // Business logic
    public void CalculateTotal() { }
    // UI / formatting logic
    public void PrintInvoice() { }
    // Data access logic
    public void SaveToDatabase() { }
}
// 3 reasons to change:
// 1. Business rules change
// 2. Print format changes
// 3. DB schema changes

✅ Fixed

// Each class has ONE responsibility
public class Invoice
{
    public void CalculateTotal() { }
}
public class InvoicePrinter
{
    public void Print(Invoice i) { }
}
public class InvoiceRepository
{
    public void Save(Invoice i) { }
}
// Each has exactly one reason to change

Real-World Analogy

Think of a restaurant. The chef cooks, the waiter serves, the cashier handles payment. Each person has one responsibility. If the chef also handled billing, a change in tax law would disrupt the kitchen.

Interview Q&A

Q: How do you identify an SRP violation?

Count the number of 'reasons to change'. If different stakeholders — a business analyst, a DBA, a UI designer — could all ask you to change the same class, it has too many responsibilities.

Q: Does SRP mean every class should have exactly one method?

No. A class can have many methods as long as they all serve the same single responsibility. An Invoice class can have CalculateTotal(), ApplyDiscount(), and GetLineItems() — they all relate to invoice business logic.

Use / Violation Signals

✅ Use When

❌ Violation Signal

Class has one clear, focused purpose

Class name includes 'And' or 'Manager'

Different stakeholders each own one class

Class has imports from UI, DB, and business layers

Changing one feature does not risk breaking another

A change to print format breaks business logic

O — Open / Closed Principle

"A class should be open for extension, but closed for modification."

What It Means

You should be able to add new behavior to a system without changing existing, tested code. You extend by adding new classes — not by editing old ones. This protects stable code from being broken by new requirements.

Violation vs Fix

❌ Violation

public class DiscountCalculator
{
    public double Calculate(
        string customerType, double price)
    {
        if (customerType == "VIP")
            return price * 0.8;
        if (customerType == "Senior")
            return price * 0.9;
        // Adding new type = edit this class
        return price;
    }
}

✅ Fixed

public interface IDiscount
{
    double Apply(double price);
}
public class VIPDiscount : IDiscount
{ public double Apply(double p) => p * 0.8; }
public class SeniorDiscount : IDiscount
{ public double Apply(double p) => p * 0.9; }
// New type = new class only, zero edits
public class StudentDiscount : IDiscount
{ public double Apply(double p) => p * 0.85; }

Real-World Analogy

A power strip is open for extension — you plug in new devices without modifying the strip itself. You never rewire the strip just because you bought a new appliance. New devices (extensions) plug into the existing interface.

Interview Q&A

Q: How do you achieve OCP in practice?

Program to interfaces or abstract classes. When new behavior is needed, create a new class that implements the interface rather than modifying the existing implementation.

Q: Is it realistic to never modify existing code?

No — the goal is to minimize changes to stable, tested code. You use abstractions to isolate the parts most likely to change so that new requirements only require new classes, not edits to proven ones.

Use / Violation Signals

✅ Use When

❌ Violation Signal

New features are added by creating new classes

Every new requirement forces edits to existing classes

Existing classes are stable and rarely edited

A class has long if/else or switch chains on type strings

The system uses interfaces or abstract classes for varying behavior

Adding a new case breaks existing tests

L — Liskov Substitution Principle

"Subtypes must be substitutable for their base types without breaking the program."

What It Means

If class B extends class A, you must be able to replace A with B anywhere in the code and have everything still work correctly — no crashes, no wrong results, no surprises. The child must honor every promise the parent makes.

💡 Note: The Swap Test: take any variable typed as the parent, replace it with a child, and ask — did anything break? If yes, LSP is violated.

Example 1 — Rectangle / Square (Classic)

❌ Violation

public class Rectangle
{
    public virtual int Width  { get; set; }
    public virtual int Height { get; set; }
    public int Area() => Width * Height;
}
public class Square : Rectangle
{
    public override int Width
    { set { base.Width = base.Height = value; } }
    public override int Height
    { set { base.Width = base.Height = value; } }
}
void ResizeAndPrint(Rectangle r)
{
    r.Width = 4; r.Height = 5;
    Console.WriteLine(r.Area());
    // Expects 20, gets 25 with Square
}

✅ Fixed

// Don't force inheritance that breaks behavior
// Use a shared abstraction instead
public interface IShape { int Area(); }
public class Rectangle : IShape
{
    public int Width  { get; set; }
    public int Height { get; set; }
    public int Area() => Width * Height;
}
public class Square : IShape
{
    public int Side { get; set; }
    public int Area() => Side * Side;
}
// Neither breaks the other's contract

Example 2 — Bird / Penguin

❌ Violation

public class Bird
{
    public virtual void Fly()
        => Console.WriteLine("Flying!");
}
// Penguin is a bird but cannot fly
public class Penguin : Bird
{
    public override void Fly()
        => throw new NotSupportedException();
}
void MakeBirdFly(Bird bird)
{
    bird.Fly(); // Crashes with Penguin
}

✅ Fixed

public abstract class Bird
{
    public abstract void Eat();
}
public interface IFlyable { void Fly(); }
public class Eagle : Bird, IFlyable
{
    public override void Eat() { }
    public void Fly()
        => Console.WriteLine("Soaring!");
}
public class Penguin : Bird
{
    public override void Eat() { }
    // Never promised it could fly
}

Example 3 — Read-Only Repository

❌ Violation

public class Repository
{
    public virtual void Add(string item) { }
    public virtual void Delete(string item) { }
}
public class ReadOnlyRepository : Repository
{
    public override void Delete(string item)
        => throw new NotSupportedException();
        // Breaks the parent's contract
}

✅ Fixed

public interface IReadRepository
{
    string Get(int id);
}
public interface IWriteRepository
{
    void Add(string item);
    void Delete(string item);
}
public class FullRepository :
    IReadRepository, IWriteRepository { }
public class ReadOnlyRepository :
    IReadRepository { }  // Only promises what it delivers

The LSP Quick Test

// Before creating a subclass, ask yourself:

// 1. Can my subclass do everything the parent promises

// without throwing or returning wrong results?

// 2. Can I secretly replace the parent with my subclass

// anywhere in the code and have nobody notice?

// If the answer to either is NO ->
//   Don't use inheritance.
//   Use a separate interface or abstraction instead.

Interview Q&A

Q: What is the most common LSP violation pattern?

A subclass that overrides a method and throws NotImplementedException or NotSupportedException. This is a clear sign the subclass does not honor the parent contract.

Q: How does LSP relate to polymorphism?

LSP is what makes polymorphism safe. Polymorphism lets you treat objects of different types uniformly through a base type — LSP ensures that doing so never produces unexpected behavior.

Q: What is the swap test?

Replace any variable of the parent type with a child object and check if the program still behaves correctly. If something breaks, the child violates LSP.

Use / Violation Signals

✅ Use When

❌ Violation Signal

Child classes fully honor the parent's behavior contract

Subclass throws NotImplementedException or NotSupportedException

Substituting child for parent produces no surprises

Subclass weakens or ignores inherited method behavior

Inheritance reflects a true 'is-a' relationship in behavior

You need type checks like 'if (obj is Square)' before calling methods

I — Interface Segregation Principle

"No class should be forced to implement methods it does not use."

What It Means

Prefer many small, focused interfaces over one large fat interface. A class should only implement methods that are relevant to it. Forcing a class to implement irrelevant methods creates dead code and misleads callers.

Violation vs Fix

❌ Violation

public interface IWorker
{
    void Work();
    void Eat();
    void Sleep();
}
// Robot works but doesn't eat or sleep
public class Robot : IWorker
{
    public void Work()
        => Console.WriteLine("Robot working.");
    public void Eat()
        => throw new NotImplementedException();
    public void Sleep()
        => throw new NotImplementedException();
}

✅ Fixed

public interface IWorkable  { void Work(); }
public interface IEatable   { void Eat(); }
public interface ISleepable { void Sleep(); }
public class Human :
    IWorkable, IEatable, ISleepable
{
    public void Work()  { }
    public void Eat()   { }
    public void Sleep() { }
}
// Robot only implements what it needs
public class Robot : IWorkable
{
    public void Work()
        => Console.WriteLine("Robot working.");
}

Real-World Analogy

A TV remote has buttons for volume, channel, and power. An air conditioner remote has buttons for temperature, fan speed, and timer. You would never hand a TV remote to someone trying to control an air conditioner and tell them to 'just ignore the buttons that don't apply'.

Interview Q&A

Q: How small should an interface be?

An interface should represent one coherent capability or role. If you find yourself writing throw new NotImplementedException() in a class that implements an interface, the interface is too large and should be split.

Q: How does ISP relate to SRP?

SRP applies to classes — one reason to change. ISP applies to interfaces — one focused capability. Both push toward smaller, more focused units. Violating one often leads to violating the other.

Use / Violation Signals

✅ Use When

❌ Violation Signal

Interfaces represent one focused capability or role

Interface has many unrelated methods

Classes only implement methods they actually use

Classes implement methods with empty bodies or NotImplementedException

Adding a new class does not require implementing irrelevant methods

Changing one method in an interface forces changes in many unrelated classes

D — Dependency Inversion Principle

"High-level modules should not depend on low-level modules. Both should depend on abstractions."

What It Means

Your business logic (high-level) should not be directly tied to concrete implementations (low-level). Both should communicate through an interface. This makes it easy to swap implementations without touching business logic.

💡 Note: DIP is the principle. Dependency Injection (DI) is the technique used to achieve it. They are related but not the same thing.

Violation vs Fix

❌ Violation

public class OrderService
{
    // Directly depends on a concrete class
    private SqlOrderRepository _repo
        = new SqlOrderRepository();
    public void PlaceOrder(Order order)
        => _repo.Save(order);
}
// Switching to MongoDB means editing
// OrderService — a business class.
// That is wrong.

✅ Fixed

public interface IOrderRepository
{
    void Save(Order order);
}
public class SqlOrderRepository : IOrderRepository
{ public void Save(Order o) { /* SQL */ } }
public class MongoOrderRepository : IOrderRepository
{ public void Save(Order o) { /* Mongo */ } }
public class OrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo)
    { _repo = repo; }  // injected from outside
    public void PlaceOrder(Order order)
        => _repo.Save(order);
}

DIP in ASP.NET Core

// Program.cs — register the abstraction and its implementation
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
// OrderService never knows if it's SQL or Mongo
// The DI container injects the correct implementation at runtime
public class OrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo)
    {
        _repo = repo;
    }
    public void PlaceOrder(Order order) => _repo.Save(order);
}
// Switching to Mongo? Change one line in Program.cs:
// builder.Services.AddScoped<IOrderRepository, MongoOrderRepository>();
// OrderService code: zero changes.

💡 Note: In unit tests, you can inject a mock IOrderRepository without touching a real database. This is only possible because of DIP — OrderService depends on the abstraction, not the concrete class.

Interview Q&A

Q: What is the difference between DIP and Dependency Injection?

DIP is the principle — high-level modules should depend on abstractions. Dependency Injection is the technique where dependencies are provided from outside (via constructor, property, or method) rather than created inside the class. DI is the most common way to achieve DIP.

Q: What does 'inversion' mean in DIP?

Traditionally, high-level modules depended on low-level modules — the direction of dependency flowed downward. DIP inverts this: both levels depend on an abstraction. The flow of dependency is inverted to point toward the abstraction, not the concrete implementation.

Q: How does DIP make unit testing easier?

Because the class depends on an interface, you can inject a mock or fake implementation in tests. Without DIP, the class creates its own dependencies internally — making it impossible to isolate for testing.

Use / Violation Signals

✅ Use When

❌ Violation Signal

Business classes depend on interfaces, not concrete types

new ConcreteClass() appears inside business logic

Implementations are injected from outside via constructor

Business class imports from infrastructure or data access layers directly

Switching a database or service requires zero business logic changes

Unit testing the class requires a real database or external service

How SOLID Principles Connect

SRP -> Each class has one job

Makes classes smaller and focused

OCP -> Extend with new classes, not edits

Relies on SRP to keep existing classes stable

LSP -> Subtypes behave like their parent

Makes OCP safe — new subclasses won't break existing code

ISP -> Small focused interfaces

Makes LSP easier — classes only implement what they can honor

DIP -> Depend on abstractions

          Uses ISP interfaces as the abstractions to depend on
          Enables OCP — swap implementations without changing callers

The One-Sentence Interview Answer

S

Give every class one job so it has one reason to change.

O

Add new features by writing new code, not by editing existing code.

L

A child class must be a complete, trustworthy substitute for its parent.

I

Split large interfaces so classes only implement what they actually need.

D

Business logic should talk to interfaces, not to concrete implementations.
