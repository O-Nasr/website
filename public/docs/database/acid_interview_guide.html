ACID in Databases

Interview Preparation Guide

10-Year Senior Database Engineer Perspective

1. What is ACID?

ACID is a set of properties that guarantee database transactions are processed reliably.

A — Atomicity

C — Consistency

I — Isolation

D — Durability

2. Atomicity — All or Nothing

A transaction either fully completes or fully rolls back. There is no partial state.

Real-World Example

Bank transfer: Deduct 100 from Account A, Add 100 to Account B. If the system crashes mid-way, both operations are rolled back entirely.

How Databases Implement It

Write-Ahead Logging (WAL): Changes are written to a log before being applied. On crash, undo logs roll back incomplete transactions.

Key Terms

WAL (Write-Ahead Log): Changes logged to disk before applying.
Undo Logs: Used to reverse uncommitted changes on recovery.

3. Consistency — Valid State to Valid State

A transaction must move the database from one valid state to another, respecting all constraints, triggers, and business rules.

Example

Constraint: balance >= 0. If a transaction tries to set balance = -500, the database rejects and rolls back the transaction.

Who Is Responsible?

Both. The database enforces structural constraints. The developer enforces business logic correctness.

4. Isolation — Concurrency Control

Concurrent transactions must not see each other intermediate states. Isolation levels define how much interaction is allowed.

4.1 The 3 Core Anomalies

Dirty Read

Reading data that another transaction modified but has NOT committed yet. You may read something that gets rolled back later.

Non-Repeatable Read

Reading the same row twice in a transaction and getting different values because another transaction updated and committed that row in between.

Phantom Read

Running the same query twice and seeing a different set of rows because another transaction inserted or deleted matching rows. Key difference: Non-Repeatable = existing row changed. Phantom = row set changed.

4.3 How Isolation Is Implemented

Locking (Pessimistic)

Locks rows or ranges. Others must wait. Safe but reduces concurrency.

MVCC — Multi-Version Concurrency Control

Keeps multiple versions of rows. Each transaction sees a snapshot from when it started. Writers do not block readers. Used by PostgreSQL and MySQL InnoDB.

4.4 Serializable — Deep Dive

Highest isolation level. Result is equivalent to some sequential execution. Prevents all anomalies including write skew.

Write Skew

Two doctors both check if the other is on call, then both go off duty. No row conflicted directly, but the invariant broke. Only Serializable prevents this.

Implementation Strategies

1. Strict Two-Phase Locking (S2PL): Holds all locks until commit. Causes blocking and deadlocks.
2. Serializable Snapshot Isolation (SSI): Used by PostgreSQL. Tracks dependencies and aborts conflicting transactions.
3. Next-Key / Range Locking: Used by MySQL InnoDB. Locks rows and gaps to prevent phantom inserts.

5. Durability — Committed = Permanent

Once a transaction commits, it remains committed even if the system crashes immediately after.

How It Works

1. Changes are written to WAL on disk first.
2. fsync forces the log to physical disk, not just OS cache.
3. COMMIT is acknowledged to the client.
4. On restart after crash, the database replays the WAL to restore all committed transactions.

6. Interview Simulation — Q&A

Real interview simulation with senior-level feedback. Pattern: What it is, Why it matters, How it works technically.

Q1 — Explain ACID

Q: Can you explain what ACID means in databases?

A: ACID: Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (no interference between concurrent TX), Durability (committed survives crashes).
Feedback: Correct. Add a real-world bank transfer example and one sentence on why ACID matters.

Q2 — Why do isolation levels exist?

Q: What problem are isolation levels trying to solve?

A: To solve Dirty Read, Phantom Read, and Non-Repeatable Read anomalies.
Feedback: Correct anomalies but missing context. Lead with: multiple transactions run concurrently and without control they can interfere. Then mention the anomalies.

Q3 — Non-Repeatable vs Phantom Read

Q: What is the difference between Non-Repeatable Read and Phantom Read?

A: Non-Repeatable: same row updated by another TX. Phantom: new rows added or deleted.
Feedback: Correct but needs precision. Better: Non-Repeatable = existing row was modified. Phantom = set of rows matching a query changed.

Q4 — Which level prevents which anomaly?

Q: Which isolation level prevents Non-Repeatable Reads? Which prevents Phantom Reads?

A: Repeatable Read and Serializable prevent Non-Repeatable Reads. Serializable prevents Phantom Reads.
Feedback: Correct. Bonus: MySQL InnoDB Repeatable Read also prevents Phantoms using next-key locking.

Q5 — Crash mid-transaction

Q: A bank transfer crashes after deducting from A but before adding to B. Which ACID property?

A: Atomicity. The database uses undo logs to roll back the incomplete transaction.
Feedback: Correct. Add: DB writes to WAL before applying. On crash, undo logs roll back the incomplete transaction.

Q6 — Constraint violation

Q: Transaction tries to set balance = -200 with CHECK (balance >= 0). What happens?

A: Consistency. The database rejects the transaction.
Feedback: Correct. Complete it: DB rejects and rolls back because it violates the CHECK constraint, preserving consistent state.

Q7 — Durability after power failure

Q: Transaction commits. One second later power failure. Is the data still there?

A: Durability. Once committed, remains committed even after crashes.
Feedback: Correct. Add the mechanism: DB uses WAL with fsync before acknowledging commit. On restart, replays WAL.

Q8 — Choosing isolation level

Q: 1000 concurrent users, good performance, minor inconsistencies acceptable. Which isolation level?

A: Read Committed. Prevents dirty reads, allows non-repeatable reads, good balance of safety and performance.
Feedback: Strong. Add: Default in PostgreSQL for a reason — prevents dirty reads while avoiding long-term locks.

7. Quick Reference

Term

Short Definition

Atomicity

All operations succeed or all are rolled back. No partial state.

Consistency

Transaction moves DB from one valid state to another, respecting all constraints.

Isolation

Concurrent transactions do not see each other intermediate states.

Durability

Once committed, a transaction remains committed even after crashes.

Dirty Read

Reading uncommitted data from another transaction.

Non-Repeatable Read

Same row read twice returns different values because another TX updated it.

Phantom Read

Same query run twice returns different row set due to insert or delete by another TX.

WAL

Write-Ahead Log. Changes logged to disk before applying, enabling crash recovery.

MVCC

Multiple row versions so readers do not block writers.

SSI

Serializable Snapshot Isolation. Tracks dependencies and aborts conflicting transactions.
