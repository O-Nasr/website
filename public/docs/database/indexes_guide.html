Database Indexes

Complete Interview Preparation Guide

PostgreSQL & SQL Server — Senior Database Engineer Perspective

1. What is an Index?

An index is a separate data structure the database maintains alongside the table. It stores sorted column values with pointers back to the actual rows, enabling the database to find data without scanning every row.

The Problem Without an Index

Without an index, every query does a Full Table Scan — reading every row one by one until it finds matches. This is O(n).
SELECT * FROM users WHERE email = 'omar@example.com';

-- 10,000,000 rows → database checks ALL 10 million rows → SLOW

With an Index

A B-Tree index traverses the tree in O(log n) steps — jumping directly to matching rows. For 10 million rows, that is roughly 23 steps instead of 10 million.
Key Insight: O(n) without index vs O(log n) with B-Tree index. For 1 billion rows, that is the difference between 1 billion checks and about 30 steps.

1.1 The Read/Write Trade-off

Indexes are not free. Every index speeds up reads but slows down writes.

2. How B-Tree Index Works Internally

The default index type is a B-Tree (Balanced Tree). It has three levels: Root, Middle Levels, and Leaf Level.

Tree Traversal Example

To find customer_id = 30 in a B-Tree with root [50], left branch [25], right branch [75]:
Step 1: Start at root [50] — 30 < 50, go left.
Step 2: At [25] — 30 > 25, go right.
Step 3: Found [30] — follow pointer to actual row.
Result: 3 steps instead of scanning all rows. For 1,000,000 rows, a B-Tree finds the answer in about 20 steps.

2.1 Clustered Index — Leaf Level IS the Table

In a Clustered Index (SQL Server), the leaf level of the B-Tree does not store pointers — it stores the FULL ROW DATA. The index and the table are the same structure.

Why Physical Order Matters

In a clustered index, rows are physically stored on disk in key order. A range query on the clustered key reads rows that are physically next to each other on disk — no jumping between pages.
Without clustered index (heap): rows stored randomly. Range query must jump across many pages.
With clustered index: rows stored in sorted order. Range query reads consecutive pages. Much faster.
Why only ONE clustered index per table: Data can only be physically sorted one way on disk. You cannot have rows sorted by customer_id AND order_date simultaneously — just like a filing cabinet can only be sorted one way.

4.2 Clustered Index

Determines the physical order of data in the table. The table rows are stored in the order of the clustered index key. The leaf level of the B-Tree contains the actual row data.

-- SQL Server: Primary Key creates clustered index by default

CREATE CLUSTERED INDEX idx_orders_id ON orders(order_id);
-- PostgreSQL: one-time physical reorder (not automatically maintained)
CLUSTER orders USING idx_orders_customer_id;

4.3 Non-Clustered Index

A separate structure from the table. Contains the indexed column values plus a pointer (row locator) back to the actual row in the table. After finding the key in the index, it must do a Key Lookup to fetch remaining columns.
CREATE NONCLUSTERED INDEX idx_users_email ON users(email);  -- SQL Server
CREATE INDEX idx_users_email ON users(email);               -- PostgreSQL (all indexes are non-clustered)
Key Lookup: When a non-clustered index finds matching rows but the query needs columns not in the index, it must jump back to the table for each row. This is called a Key Lookup and is a performance warning sign in execution plans.

4.4 Covering Index

Includes all columns a query needs directly inside the index. Eliminates Key Lookups entirely, enabling an Index Only Scan — the fastest possible read.

-- Query needs: id, total, status — filtered by customer_id

SELECT id, total, status FROM orders WHERE customer_id = 42;
-- Regular index: finds customer_id, then Key Lookup for id/total/status (2 steps)
CREATE INDEX idx_orders_customer ON orders(customer_id);
-- Covering index: everything needed is inside the index (1 step)

CREATE INDEX idx_orders_covering

  ON orders(customer_id) INCLUDE (id, total, status);
Senior Insight: A covering index trades more storage for eliminating Key Lookups. On hot query paths — queries that run millions of times per day — this trade-off is almost always worth it.

4.5 Composite Index (Multi-Column Index)

An index on two or more columns. The Left-Most Prefix Rule applies: the query must include the leftmost column(s) of the index to use it.
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status);
-- Index: (customer_id, status)
WHERE customer_id = 5                        -- uses index (leading column present)
WHERE customer_id = 5 AND status = 'active'  -- uses index (both columns)
WHERE status = 'active' AND customer_id = 5  -- uses index (order in query does NOT matter)
WHERE status = 'active'                      -- does NOT use index (leading column missing)
Critical Rule: The order of columns in the WHERE clause does NOT matter. The optimizer reorders them automatically. What matters is the column order in the INDEX DEFINITION itself.

4.6 Unique Index

Enforces uniqueness on a column or combination of columns. Creating a PRIMARY KEY or UNIQUE constraint automatically creates a unique index behind the scenes.
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- This implicitly creates a unique index:

ALTER TABLE users ADD CONSTRAINT uq_email UNIQUE (email);

4.7 Hash Index

Uses a hash function instead of a tree. Lightning fast for equality lookups but cannot be used for ranges, sorting, or prefix searches. PostgreSQL only for disk-based tables.

-- PostgreSQL only

CREATE INDEX idx_users_email_hash ON users USING HASH (email);

4.8 Full-Text Index

Designed for searching inside text content — words, phrases, natural language. Regular LIKE with leading wildcards is extremely slow on large text. Full-Text index is the right tool.
-- SQL Server
CREATE FULLTEXT INDEX ON articles(content) KEY INDEX pk_articles;
SELECT * FROM articles WHERE CONTAINS(content, 'database optimization');
-- PostgreSQL
CREATE INDEX idx_articles_fts ON articles USING GIN(to_tsvector('english', content));
SELECT * FROM articles WHERE to_tsvector('english', content) @@ to_tsquery('database & optimization');

4.9 Partial Index (PostgreSQL) / Filtered Index (SQL Server)

An index built on a subset of rows based on a WHERE condition. Much smaller and faster to maintain than a full index when queries always target a specific subset.
-- Only index pending orders (50,000 out of 10,000,000 rows)
-- PostgreSQL
CREATE INDEX idx_orders_pending ON orders(customer_id) WHERE status = 'pending';
-- SQL Server
CREATE INDEX idx_orders_pending ON orders(customer_id) WHERE status = 'pending';
-- Uses the partial index (matches the filter condition)
SELECT * FROM orders WHERE status = 'pending' AND customer_id = 42;  -- uses index

-- Does NOT use the partial index

SELECT * FROM orders WHERE status = 'completed' AND customer_id = 42; -- no index

Bitmap Index Merge (PostgreSQL) / Index Intersection (SQL Server)

When multiple separate indexes exist and a query filters on both columns, the database can use BOTH indexes simultaneously and merge the results.

-- Two separate indexes:

CREATE INDEX idx_orders_customer ON orders(customer_id);
CREATE INDEX idx_orders_status ON orders(status);

-- Query uses both via Bitmap Index Merge:

SELECT * FROM orders WHERE customer_id = 5 AND status = 'pending';

-- PostgreSQL EXPLAIN shows:

-- BitmapAnd
--   Bitmap Index Scan on idx_orders_customer (customer_id = 5)
--   Bitmap Index Scan on idx_orders_status   (status = pending)

6. Fill Factor

When the database builds an index, it does not pack every page 100% full. Fill Factor controls how much empty space is left on each page for future inserts.
-- SQL Server: create index with 80% fill factor (leave 20% empty)
CREATE INDEX idx_orders_customer ON orders(customer_id) WITH (FILLFACTOR = 80);

8. Interview Q&A

Q1 — What is an index and what problem does it solve?

Q: What is an index and what problem does it solve?

A: An index is a separate data structure the database maintains alongside the table. Without an index, finding a row requires a full table scan — O(n) — reading every row one by one. With a B-Tree index, the database traverses the tree in O(log n) steps, jumping directly to matching rows. For 10 million rows that is the difference between 10 million comparisons and roughly 23 steps. The trade-off is that every index slows down INSERT, UPDATE, and DELETE because the index must be updated too.
Senior Tip: Mentioning O(n) vs O(log n) immediately sounds senior. Always add the write trade-off — interviewers expect it.

Q2 — Clustered vs Non-Clustered index?

Q: What is the difference between a Clustered and Non-Clustered index? How do SQL Server and PostgreSQL differ?

A: A clustered index determines the physical order of rows on disk. In SQL Server, the leaf level of the clustered index IS the table — it stores full row data, not just pointers. This means range queries are very fast because matching rows are physically adjacent on disk. There can only be ONE clustered index per table because data can only be sorted one way physically. A non-clustered index is a separate structure storing the key plus a pointer to the actual row. After finding the key, it must do a Key Lookup to fetch remaining columns. SQL Server creates a clustered index on the primary key by default. PostgreSQL has no true clustered index — all indexes are non-clustered heap-based structures.
Senior Tip: Always explain WHY only one clustered index is possible. The physical sorting reason is what shows deep understanding.

Q3 — What is a Key Lookup and how do you fix it?

Q: Your execution plan shows a Key Lookup. What does it mean and how do you fix it?

A: A Key Lookup happens when a non-clustered index finds matching rows, but the query needs columns that are not stored in the index. The database must jump back to the table for each matching row to fetch those columns. The fix is a covering index — using INCLUDE to add the needed columns directly into the index. This enables an Index Only Scan where the database never needs to touch the table at all.
Senior Tip: Always name the fix: covering index with INCLUDE. Show you know the before/after: Index Seek + Key Lookup becomes Index Only Scan.

Q4 — Left-Most Prefix Rule in Composite Indexes

Q: You have a composite index on (customer_id, status). Will WHERE status = 'pending' use the index?

A: No. The Left-Most Prefix Rule requires the query to include the leading column of the composite index. Since the index is on (customer_id, status) and the query only filters on status, the leading column customer_id is missing — the index is skipped and the database does a full table scan. Important note: the order of conditions in the WHERE clause does NOT matter — the optimizer reorders them automatically. What matters is the column order in the index definition. The fix is either a separate index on status, or redefine the composite index as (status, customer_id) if status is the more common filter.
Senior Tip: The key correction candidates often miss: query condition order does NOT matter, only index definition order matters.

Q5 — When would an index not be used?

Q: When would an index exist on a column but the optimizer still chooses not to use it?

A: Five main scenarios. First, a function applied to the indexed column — the index stores raw values, not function results. Second, a leading wildcard like LIKE '%Omar%' — the B-Tree has no starting point to seek from. Third, implicit type conversion — passing a VARCHAR to an INT column forces conversion on every row, breaking index usage. Fourth, low selectivity — on a column like gender with only 2 values, the optimizer decides a full scan is cheaper than following millions of index pointers back to the table. Fifth, OR conditions across different columns where merging two indexes costs more than a single table scan.
Senior Tip: Explaining WHY each one breaks the index — not just that it does — is what sounds senior. The optimizer reasoning is key.

Q6 — What is selectivity?

Q: What is index selectivity and why does it matter?

A: Selectivity measures how unique the values in a column are. High selectivity means many distinct values — the index can jump to a tiny fraction of rows, making it very useful. Low selectivity means few distinct values — the index leads to a large percentage of rows, making it nearly useless. For example, an index on email with 1 million distinct values is highly selective. An index on gender with 2 values is not. The optimizer uses statistics to estimate selectivity. If statistics are stale, the optimizer may make wrong decisions — which is why running ANALYZE in PostgreSQL and UPDATE STATISTICS in SQL Server matters.
Senior Tip: Connecting selectivity to statistics and the optimizer shows you understand the full picture, not just the definition.

Q7 — What are the downsides of too many indexes?

Q: What are the downsides of having too many indexes on a table?

A: Every index slows down INSERT, UPDATE, and DELETE because every write must update all indexes on the table. More indexes means more storage space. On write-heavy tables like event logs or transaction tables, having too many indexes can severely hurt throughput. Additionally, the query optimizer must evaluate more possible plans, which increases planning time. Senior engineers choose indexes carefully based on the actual query patterns — adding indexes where the read benefit clearly outweighs the write cost.
Senior Tip: This is a judgment question. Showing you understand the trade-off rather than just saying indexes are good demonstrates real engineering experience.
