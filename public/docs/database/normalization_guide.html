Interview Preparation Guide — Senior Database Engineer Perspective

1. What is Normalization?

Normalization is the process of structuring database tables to eliminate data redundancy and prevent data anomalies. It organizes columns and tables to ensure data dependencies make sense.

The 3 Anomalies Normalization Prevents

Anomaly

What Happens

Example

Update Anomaly

Same data stored in multiple rows — changing one misses others

Customer email stored in every order row

Deletion Anomaly

Deleting a row accidentally removes unrelated data

Deleting last order loses customer info

Insertion Anomaly

Cannot store data without another unrelated record

Cannot add a product without an order

Key Concept — Functional Dependency: Column A determines Column B (written A -> B). If you know A, you always know B. Example: customer_id -> customer_email

The 3 Rules of 1NF

1. Each column has a single atomic value.
2. Each column stores the same type of data.
3. Each row is unique (has a primary key).

3. Second Normal Form (2NF)

Rule: Must be in 1NF AND every non-key column must depend on the WHOLE primary key, not just part of it. Only applies when there is a composite primary key.

Key Concept — Partial Dependency

A partial dependency occurs when a non-key column depends on only PART of a composite primary key, not the whole key.

4. Third Normal Form (3NF)

Rule: Must be in 2NF AND no non-key column should depend on another non-key column. Eliminate transitive dependencies.

Key Concept — Transitive Dependency

A transitive dependency occurs when: A -> B -> C, where B is not a key. The non-key column C depends on another non-key column B instead of depending directly on the primary key.

5. BCNF — Boyce-Codd Normal Form (3.5NF)

Rule: Must be in 3NF AND for every functional dependency A -> B, A must be a superkey. Every determinant must be a candidate key.

Why BCNF Exists

3NF has one edge case it does not cover — when there are multiple overlapping candidate keys. BCNF is a stricter version of 3NF that handles this scenario.

Key Concept — Candidate Key vs Superkey

Candidate Key: A minimal set of columns that uniquely identifies every row.
Superkey: Any set of columns that uniquely identifies a row (can include extra columns).
BCNF Rule: Every column that determines another column must itself be a candidate key.

7. Real-World Engineering Tradeoff

Normalization is not always the right answer. Senior engineers choose based on use case.

Scenario

Recommendation

Reason

OLTP (banking, e-commerce)

Normalize to 3NF or BCNF

Data integrity is critical

OLAP (analytics, reporting)

Denormalize intentionally

Read speed matters more than write integrity

High-read APIs

Partial denormalization

Avoid expensive JOINs on hot paths

Senior Insight: Normalization is for write integrity. Denormalization is for read performance. Knowing when to apply each is what separates senior engineers.

8. Top Interview Questions & Answers

Master these 5 questions and you will handle any normalization topic in an interview.

Q1 — What is normalization and why do we use it?

Q: What is normalization and why do we use it?

A: Normalization is the process of organizing database tables to eliminate data redundancy and prevent anomalies. We use it to avoid update anomalies (same data in multiple rows), deletion anomalies (losing unrelated data when deleting a row), and insertion anomalies (unable to store data without unrelated records). It ensures each piece of data lives in exactly one place.
Senior Tip: Always mention the three anomalies by name. Interviewers love candidates who know the exact terminology.

Q2 — What is the difference between 2NF and 3NF?

Q: What is the difference between 2NF and 3NF?

A: 2NF eliminates partial dependencies — where a non-key column depends on only part of a composite primary key. 3NF eliminates transitive dependencies — where a non-key column depends on another non-key column instead of directly on the primary key. In short: 2NF is about the relationship between non-key columns and the primary key. 3NF is about relationships between non-key columns themselves.
Senior Tip: This is the most commonly confused question. The one-liner to remember: 2NF fixes partial deps, 3NF fixes transitive deps.

Q3 — What is a partial dependency? Give an example.

Q: What is a partial dependency? Give an example.

A: A partial dependency occurs when a non-key column depends on only part of a composite primary key. For example, in an order_products table with composite key (order_id, product_id), the column product_name depends only on product_id, not on the full key. This is a partial dependency and violates 2NF. The fix is to move product_name to a separate products table where product_id is the sole primary key.
Senior Tip: Always give a concrete example. Abstract definitions alone do not impress interviewers.

Q4 — What is a transitive dependency? Give an example.

Q: What is a transitive dependency? Give an example.

A: A transitive dependency occurs when a non-key column depends on another non-key column rather than directly on the primary key. For example, in an employees table: employee_id -> department_id -> department_name. The department_name depends on department_id which is itself a non-key column. This violates 3NF. The fix is to move department_id and department_name to a separate departments table.
Senior Tip: Use the arrow notation A -> B -> C when explaining. It shows you understand the concept precisely.

Q5 — When would you intentionally denormalize a database?

Q: When would you intentionally denormalize a database?

A: I would denormalize when read performance is more critical than write integrity. For example, in analytics and reporting systems (OLAP), running many JOINs across normalized tables is expensive. Storing pre-joined or aggregated data in a denormalized table makes reads much faster. Also, in high-traffic APIs where certain queries run millions of times per day, avoiding a JOIN can significantly reduce latency. The tradeoff is accepting some data redundancy in exchange for speed.
Senior Tip: This is a senior-level question. Showing you understand the tradeoff — not just the rule — is what gets you hired.

9. Quick Reference — Key Terms

Term

Definition

Normalization

Structuring tables to eliminate redundancy and prevent anomalies.

Functional Dependency

A -> B means knowing A always determines B. Example: customer_id -> email.

Atomic Value

A single indivisible value. No lists or multiple values in one cell.

Partial Dependency

Non-key column depends on only PART of a composite primary key. Violates 2NF.

Transitive Dependency

Non-key column depends on another non-key column (A->B->C). Violates 3NF.

Candidate Key

Minimal set of columns that uniquely identifies every row in a table.

Superkey

Any set of columns that uniquely identifies rows (may include extra columns).

Determinant

A column that determines another column in a functional dependency (the A in A->B).

Denormalization

Intentionally introducing redundancy to improve read performance. Used in OLAP systems.

Update Anomaly

Same data stored in multiple rows — updating one row misses others.

Deletion Anomaly

Deleting a row accidentally removes unrelated data.

Insertion Anomaly

Cannot store data without another unrelated record existing first.
