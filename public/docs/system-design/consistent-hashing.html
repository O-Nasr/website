Consistent Hashing

Here’s a short and clear explanation of Consistent Hashing:

✅ Definition

Consistent hashing is a distributed hashing technique that maps keys (like user IDs, cache keys) to servers or nodes in a way that minimizes data movement when nodes are added or removed.

🧠 What Problem Does It Solve?

In distributed systems (like caching or load balancing):

  • You often need to distribute data across multiple servers.
  • If you use a simple modulo-based hash (e.g., hash(key) % N):
    • Adding/removing a server changes N.
    • Almost all keys get remapped, causing massive data movement.

Consistent hashing solves this by:

  • Mapping keys and nodes onto a hash ring.
  • When a node is added or removed, only a small portion of keys move.
  • This makes scaling efficient and stable.

📌 Real-World Use Cases

  • Distributed caches (e.g., Memcached, Redis clusters)
  • Load balancers
  • Distributed databases (e.g., Cassandra, DynamoDB)
  • CDNs for routing requests to edge servers

Here’s a clear comparison between Modulo Hashing and Consistent Hashing:

✅ Modulo Hashing

  • Formula: hash(key) % N (where N = number of servers)
  • Problem: When you add or remove a server, N changes → almost all keys get remapped.
  • Impact: Huge data movement → expensive and slow during scaling.

✅ Consistent Hashing

  • Keys and servers are placed on a hash ring.
  • Each key maps to the next server clockwise on the ring.
  • When a server is added or removed:
    • Only keys near that server move.
    • Minimal data movement → efficient scaling.
