Chapter 6: Design a Key-Value Store

System Design Interview — Alex Xu

1. CAP Theorem

In a distributed system, you can only fully guarantee 2 out of 3 properties at the same time.

Property

Meaning

Example

Consistency (C)

Every read returns the most recent write

Bank balance

Availability (A)

Every request gets a response, even during failures

Facebook likes

Partition Tolerance (P)

System keeps working even if nodes can't communicate

Required in all distributed systems

Key Insight

Network partitions WILL happen in real systems. P is not optional. The real choice is: when a partition occurs, do you sacrifice C or A?

CP vs AP Systems

System Type

Behavior During Partition

Use Case

CP (Consistency + Partition)

Refuses to respond rather than return stale data

Bank transfers, financial systems

AP (Availability + Partition)

Responds with possibly stale data

Social feeds, location tracking, notifications

Interview Tip

Always name both the system type (AP/CP) AND the consistency model (Eventual/Strong). Example: 'Twitter uses an AP system with Eventual Consistency.'

2. Replication

Replication = copying data across multiple nodes so if one dies, others take over. Solves the Single Point of Failure (SPOF) problem.

Leader-Follower Replication

  • One Leader node handles all writes
  • Leader copies changes to Follower nodes
  • Reads can go to any Follower
  • If Leader dies, a Follower gets promoted

Pros

Cons

Simple to reason about

Leader is bottleneck for writes

Reads scale easily — add more followers

Leader failure requires election process

Clear data flow

Replication lag causes stale reads

Replication Lag

The delay between a write hitting the Leader and that write reaching Followers. During this window, reads from Followers return stale data.

Real World

A user posts a tweet and immediately refreshes — but doesn't see it for 2-3 seconds. That's replication lag in action.

3. Consistency Models

Model

Guarantee

Tradeoff

Use Case

Strong Consistency

Every read returns the most recent write

Higher latency, lower availability

Bank balance, payments

Eventual Consistency

Data will catch up — but reads may be temporarily stale

Low latency, high availability

Social feeds, likes, notifications

Connection to CAP

CP systems use Strong Consistency. AP systems use Eventual Consistency. They are locked together — you cannot have AP with Strong Consistency.

4. Quorum (W, R, N)

Used in leaderless replication. Defines how many nodes must confirm an operation.

Variable

Meaning

N

Total number of replicas

W

Number of nodes that must confirm a write

R

Number of nodes that must respond to a read

The Magic Rule

W + R > N → Strong Consistency

W + R ≤ N → Eventual Consistency

Configuration

Consistency

Use Case

N=3, W=2, R=2 (W+R=4 > 3)

Strong

Payments, DMs, critical data

N=3, W=1, R=1 (W+R=2 ≤ 3)

Eventual

Likes, location updates, counters

N=3, W=3, R=3 (W+R=6 > 3)

Strongest

Bank transactions — wait for all nodes

Real Systems Mix Both

Same system, different W/R per feature. Twitter DMs use high W+R. Twitter likes use low W+R. Design per use case, not per system.

5. Conflict Resolution

In leaderless replication, two nodes can accept different writes for the same key simultaneously. This is called a write conflict.

Last Write Wins (LWW)

  • Each write has a timestamp
  • The write with the latest timestamp wins
  • The other write is silently discarded

Pros

Cons

Simple to implement

Data loss — losing write is silent

Deterministic result

Clock skew makes timestamps unreliable

Never use for financial data or documents

Version Vectors

  • Track which version each node has seen
  • On conflict, the KV store keeps BOTH versions
  • The application decides how to merge

App Type

Merge Strategy

Google Docs

Merge both edits (CRDT)

Shopping cart (Amazon Dynamo)

Keep all items from both versions

User profile

Ask the user to resolve

Git

Auto-merge if possible, else ask developer

Interview Tip

For any system where data loss is unacceptable (documents, carts, profiles) — choose Version Vectors. LWW is only safe for cache entries and sessions.

Interview Answer Template

"Use consistent hashing to distribute keys across nodes. Replicate each key to N nodes for fault tolerance. Use quorum reads and writes (W, R, N) to tune between consistency and availability based on business needs. Handle write conflicts using Last Write Wins for simple cases or Version Vectors when data loss is unacceptable."
