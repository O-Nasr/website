SSTable & Bloom filter

Here’s a step-by-step deep dive into SSTables, real-world use cases, and comparisons with relational DBs, NoSQL, and Redis:

✅ What is SSTable?

SSTable stands for Sorted String Table.
It’s a persistent, immutable, sorted file format used in Log-Structured Merge (LSM) tree-based databases.

Key Characteristics

  • Immutable: Once written, never modified.
  • Sorted by key: Enables efficient range queries and merges.
  • Append-only writes: New updates create new SSTables.
  • Compaction: Periodically merges SSTables to reduce duplicates and reclaim space.

✅ How It Works

  1. Write Path:
    • Data first goes to Memtable (in-memory).
    • Also logged in Commit Log for durability.
    • When Memtable is full → flushed to disk as an SSTable.
  2. Read Path:
    • Check Memtable first.
    • If not found → search SSTables (using sparse index + Bloom filters).

✅ Why SSTables?

  • Optimized for write-heavy workloads.
  • Sequential disk writes → fast and efficient.
  • Great for huge datasets (billions of keys).

✅ Real-World Use Cases

  • Apache Cassandra: Stores column families as SSTables.
  • Google Bigtable: Original design uses SSTables.
  • ScyllaDB: Cassandra-compatible, uses SSTables.
  • LevelDB / RocksDB: Embedded key-value stores for caching and databases.
  • HBase: Hadoop-based NoSQL uses SSTables.

✅ Tools Using SSTables

  • Cassandra
  • Bigtable
  • ScyllaDB
  • RocksDB
  • LevelDB
  • HBase

✅ Comparison with Other Systems

Relational DB (e.g., MySQL, PostgreSQL)

  • Uses B-Trees for indexing.
  • Optimized for read-heavy workloads and complex queries.
  • Updates happen in place (random I/O).
  • SSTables → better for write-heavy, large-scale systems.

NoSQL DB (e.g., Cassandra, Bigtable)

  • Uses LSM Trees + SSTables.
  • Prioritizes fast writes, eventual consistency.
  • Great for time-series, logs, analytics.

Redis

  • In-memory key-value store.
  • Extremely fast for reads/writes.
  • Persistence via RDB snapshots or AOF logs, not SSTables.
  • Best for caching, not long-term storage.

===🡺Great question, Omar! Let’s break down why compaction is needed in SSTables and how it relates to read performance.

✅ Why Compaction Exists

  • SSTables are immutable.
  • Every update creates a new SSTable.
  • Over time, you can have hundreds or thousands of SSTables on disk.
  • Each SSTable may contain:
    • Duplicate keys (older versions).
    • Deleted keys (marked as tombstones).

✅ Impact on Reads

  • When you read a key:
    • The system must check multiple SSTables to find the latest version.
    • Uses Bloom filters and indexes, but still involves multiple disk lookups.
  • More SSTables = slower reads.

✅ What Compaction Does

  • Periodically merges SSTables into fewer, larger SSTables.
  • Removes:
    • Duplicate keys (keeps latest version).
    • Tombstones (deleted keys).
  • Result:
    • Fewer SSTables → faster reads.
    • Reclaims disk space.

✅ Types of Compaction

  • Size-Tiered: Merge smaller SSTables into bigger ones.
  • Leveled: Organize SSTables into levels; compaction keeps levels balanced.
  • Time-Window: Group SSTables by time for time-series data.

✅ Trade-Off

  • Pros:
    • Improves read performance.
    • Reduces storage overhead.
  • Cons:
    • Compaction is I/O heavy.
    • Can temporarily impact write performance.

✅ Real-World Example

  • Cassandra:
    • Without compaction: 100 SSTables → read checks all.
    • After compaction: 10 SSTables → read checks fewer files.
  • RocksDB:
    • Uses Leveled Compaction for predictable read latency.

===🡺Great question! Let’s break down Bloom filters in SSTables step by step:

✅ What is a Bloom Filter?

A Bloom filter is a space-efficient probabilistic data structure used to test whether an element might be in a set.

  • It can return:
    • True (element might be present)
    • False (element is definitely not present)
  • It never gives false negatives, but can give false positives.

✅ Why Bloom Filters in SSTables?

  • SSTables are immutable and stored on disk.
  • When reading a key:
    • Without Bloom filters → you might check every SSTable on disk.
    • With Bloom filters → you quickly skip SSTables that definitely don’t contain the key.
  • This reduces disk I/O and speeds up reads.

🧠 How It Works in SSTables:

  1. Each SSTable has a Bloom filter built for its keys.
  2. When a read request comes:
    • Check Memtable first.
    • Then check Bloom filters of SSTables:
      • If Bloom filter says “not present” → skip that SSTable.
      • If Bloom filter says “might be present” → check SSTable index and data.

✅ Real-World Example

  • Cassandra:
    • Uses Bloom filters for every SSTable.
    • Reduces read amplification when many SSTables exist.
  • RocksDB / LevelDB:
    • Same approach for fast key lookups.

✅ Benefits

  • Saves disk seeks.
  • Improves read performance in LSM-based systems.
  • Very memory-efficient (few bits per key).

✅ Trade-Off

  • False positives → you still check some SSTables unnecessarily.
  • But never false negatives, so correctness is guaranteed.

====🡺What is LSM? Great question! Let’s break down LSM step by step:

✅ What is LSM?

LSM stands for Log-Structured Merge Tree.
It’s a data structure used in modern databases (especially NoSQL) to handle high write throughput efficiently.

🧠 Core Idea

  • Instead of updating data in place (like B-Trees in relational DBs), LSM trees:
    • Write new data sequentially to disk (append-only).
    • Keep recent data in memory (Memtable).
    • Periodically flush Memtable to disk as immutable SSTables.
    • Use compaction to merge SSTables and remove old versions.

✅ Why LSM Trees?

  • Fast writes:
    • Sequential disk writes are much faster than random writes.
  • Great for large-scale systems:
    • Handles billions of keys efficiently.
  • Optimized for SSDs and HDDs:
    • Sequential writes reduce disk seek overhead.

✅ Read Path

  • Check Memtable first.
  • If not found → search SSTables using:
    • Sparse index
    • Bloom filters (to skip SSTables that don’t have the key)
  • Compaction reduces the number of SSTables → improves read speed.

✅ Real-World Use Cases

  • Cassandra
  • Google Bigtable
  • HBase
  • RocksDB
  • LevelDB
  • ScyllaDB

===🡺Great question, Omar! Let’s clear up the confusion between SSTable and LSM Tree because they are related but not the same thing.

✅ LSM Tree (Log-Structured Merge Tree)

  • What it is: A data structure used in modern databases to optimize write-heavy workloads.
  • Core principle:
    • Writes are append-only (fast sequential writes).
    • Data is first stored in memory (Memtable).
    • When Memtable is full → flushed to disk as SSTable.
    • Multiple SSTables exist → periodically compacted (merged).
  • Why LSM?
    • Avoids expensive random writes (common in B-Trees).
    • Great for SSD/HDD performance.
  • Used by: Cassandra, Bigtable, RocksDB, LevelDB, HBase.

✅ SSTable (Sorted String Table)

  • What it is: A file format used by LSM Trees to store data on disk.
  • Characteristics:
    • Immutable (never modified after creation).
    • Sorted by key → efficient range queries.
    • Contains:
    • Data blocks
    • Index blocks
    • Bloom filters (to speed up lookups)
  • Role:
    • SSTable is the physical representation of LSM Tree data on disk.

🧠 Relationship

  • LSM Tree = Algorithm + Structure
  • SSTable = Storage format used by LSM Tree
  • LSM Tree organizes data across Memtable + SSTables + Compaction.

✅ Real-World Example

  • Cassandra:
    • Writes go to Memtable → flushed as SSTable → compaction merges SSTables.
  • Bigtable:
    • Original Google design uses SSTables for persistent storage.
  • RocksDB:
    • Embedded key-value store using LSM + SSTables.
