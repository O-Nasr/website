Sticky Session

🔄 What Are Sticky Sessions?

Sticky sessions (also called session affinity) are a load balancing feature that ensures:
A user’s requests are always routed to the same backend server during a session.

🧠 Why Use Sticky Sessions?

In stateful systems, user session data (like login state, shopping cart, etc.) is stored in memory on the server.
If a user’s request goes to a different server, that data is lost.

Sticky sessions solve this by:

  • “Sticking” the user to the same server.
  • Ensuring consistent access to their session data.

🧱 How It Works in Classic Load Balancer (CLB)

In AWS Classic Load Balancer:

  • Sticky sessions are enabled using cookies.
  • The load balancer uses a cookie (e.g., AWSELB) to track which server handled the first request.
  • Future requests with that cookie are routed to the same server.

📌 Example Scenario

  1. User logs in → request goes to Server A.
  2. Server A stores session in memory.
  3. Load balancer sets a sticky cookie.
  4. Next request → load balancer sees cookie → routes to Server A again.

Without sticky sessions:

  • Request might go to Server B, which doesn’t have the session → user gets logged out or sees an error.

🧠 In Stateless Systems

  • You don’t need sticky sessions.
  • Session data is stored in a shared cache or database (e.g., Redis).
  • Any server can handle any request.

🔧 Best Practice

If possible, design your system to be stateless:

  • Use distributed session stores.
  • Avoid sticky sessions unless absolutely necessary.

🔁 Sticky Sessions in Context of Data Centers

When you have multiple data centers or multiple servers behind a load balancer, sticky sessions help ensure that:

A user’s requests are consistently routed to the same server (or region) during their session.
This is especially important in stateful systems, where session data (like login state, shopping cart, etc.) is stored in memory on the server.

📌 Why Mention Sticky Sessions in Data Center Design?

In a multi-region setup, sticky sessions can be used to:

  • Keep a user tied to a specific data center (e.g., Frankfurt).
  • Avoid session loss if the user’s requests are routed to a different region/server.

This is often done using:

  • Session cookies (e.g., AWSELB in AWS Classic Load Balancer)
  • IP hashing or custom routing logic

🔧 Example Scenario

  1. User in Italy connects to Frankfurt data center.
  2. Load balancer assigns them to Server A.
  3. A sticky session cookie is set.
  4. All future requests from that user go to Server A in Frankfurt, even if other servers are available.

🧠 Why It Matters

In stateful systems, sticky sessions help:

  • Maintain session continuity
  • Avoid re-authentication or data loss
  • Simplify session management

But in stateless systems, sticky sessions are not needed, because:

  • Session data is stored in a shared cache or database (e.g., Redis)
  • Any server can handle any request
