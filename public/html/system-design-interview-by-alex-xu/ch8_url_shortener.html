1 The Core Problem

A URL shortener converts a long URL into a short alias. When a user clicks the short URL, the server redirects them to the original long URL.

Long: https://www.example.com/some/very/long/path?with=query¶ms=here Short: https://bit.ly/zn9edcu

Simple on the surface — but at millions of URLs and billions of clicks, the design becomes interesting.

3 Why Hashing Alone Fails

Taking the first 7 characters of an MD5 hash seems simple — but has two critical problems.

ProblemWhat HappensImpact
Hash CollisionTwo different URLs produce same first 7 charsBoth map to same short code — one URL is lost
Duplicate URLsSame URL hashed twice gives same short codeMust check DB every single time — expensive
DB Check RequiredEvery write needs a DB read to detect collisionAt 1,160 writes/sec this adds significant latency
The Fix
Use Snowflake ID + Base62 encoding instead. Guaranteed unique IDs with zero collision checks needed.

5 301 vs 302 Redirect

When a user clicks a short URL, the server responds with a redirect. Two options — each with very different implications.

Interactive — Compare 301 vs 302 behavior
301 Permanent302 Temporary
Browser cachingCached forever — never hits server againNo caching — every click hits server
Server loadVery low after first visitHigher — every redirect goes through server
Analytics trackingCannot track repeat clicksEvery click is logged on server
Permission controlCannot revoke — browser ignores serverServer can validate access every click
Expiring linksCannot enforce expiry in browserServer enforces expiry on every request
Best forPublic static marketing linksPermission-based, expiring, or tracked links
Interview Answer
Use 302 for most URL shorteners — it enables analytics, permission enforcement, and expiry control. Use 301 only for static public links where reducing server load is the priority.
6 URL Expiry + Redis TTL

What is Redis TTL?

TTL = Time To Live. A countdown timer attached to a Redis key. When the timer hits zero — Redis auto-deletes the key. No manual cleanup needed.

// Store with TTL SET "zn9edcu" "{url, expires_at}" EX 86400 ← 24 hours SET "ab3Xk2" "{url, expires_at}" EX 604800 ← 7 days SET "pQ7mN1" "{url}" ← never expires // On expiry: Redis auto-deletes the key GET "zn9edcu" ← returns nil after expiry

Cache Invalidation on Revoke

If admin revokes a URL before it expires — don't wait for TTL. Delete the Redis key immediately.

Why Not Short TTL?
Setting a very short TTL means the revoked URL is still accessible for seconds or minutes. For permission-sensitive links this is unacceptable. Immediate cache deletion is the correct approach.

9 Interview Answer Template
"I'd use Snowflake ID converted to Base62 to generate collision-free 7-character short codes with no DB collision checks. Short URLs are stored in DB with optional expiry timestamps and cached in Redis with matching TTL. I'd use 302 redirects so every click hits the server — enabling analytics, permission checks, and expiry enforcement. On cache miss we query DB and repopulate Redis. On revocation we immediately delete the Redis key to prevent stale cache access. The system is read-heavy (10:1 ratio) so Redis absorbs most traffic with 80-90% cache hit rate."
