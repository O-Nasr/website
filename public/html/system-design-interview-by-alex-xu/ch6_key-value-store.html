CAP Theorem

In a distributed system, you can only fully guarantee 2 out of 3 properties at the same time.

CAP Properties

PropertyMeaningExample
Consistency (C)Every read returns the most recent writeBank balance
Availability (A)Every request gets a response, even during failuresFacebook likes
Partition Tolerance (P)System keeps working even if nodes can't communicateRequired in all distributed systems
💡Key Insight: Network partitions WILL happen in real systems. P is not optional. The real choice is: when a partition occurs, do you sacrifice C or A?

CP vs AP Systems

System TypeBehavior During PartitionUse Case
CP (Consistency + Partition)Refuses to respond rather than return stale dataBank transfers, financial systems
AP (Availability + Partition)Responds with possibly stale dataSocial feeds, location tracking, notifications
🎯Interview Tip: Always name both the system type (AP/CP) AND the consistency model (Eventual/Strong). Example: "Twitter uses an AP system with Eventual Consistency."
2

Replication

Copying data across multiple nodes so if one dies, others take over. Solves the Single Point of Failure (SPOF) problem.

Leader-Follower Replication

ProsCons
Simple to reason aboutLeader is bottleneck for writes
Reads scale easily — add more followersLeader failure requires election process
Clear data flowReplication lag causes stale reads

Replication Lag

The delay between a write hitting the Leader and that write reaching Followers. During this window, reads from Followers return stale data.

🌍Real World: A user posts a tweet and immediately refreshes — but doesn't see it for 2-3 seconds. That's replication lag in action.
3

Consistency Models

🔗Connection to CAP: CP systems use Strong Consistency. AP systems use Eventual Consistency. They are locked together — you cannot have AP with Strong Consistency.
4

Quorum (W, R, N)

Used in leaderless replication. Defines how many nodes must confirm an operation.

Variables

VariableMeaning
NTotal number of replicas
WNumber of nodes that must confirm a write
RNumber of nodes that must respond to a read
🎯Real Systems Mix Both: Same system, different W/R per feature. Twitter DMs use high W+R. Twitter likes use low W+R. Design per use case, not per system.
5

Conflict Resolution

In leaderless replication, two nodes can accept different writes for the same key simultaneously.

Last Write Wins (LWW)

ProsCons
Simple to implementData loss — losing write is silent
Deterministic resultClock skew makes timestamps unreliable
Never use for financial data or documents

Version Vectors — Merge Strategies

App TypeMerge Strategy
Google DocsMerge both edits (CRDT)
Shopping cart (Amazon Dynamo)Keep all items from both versions
User profileAsk the user to resolve
GitAuto-merge if possible, else ask developer
🎯Interview Tip: For any system where data loss is unacceptable (documents, carts, profiles) — choose Version Vectors. LWW is only safe for cache entries and sessions.

Interview Answer Template

"Use consistent hashing to distribute keys across nodes. Replicate each key to N nodes for fault tolerance. Use quorum reads and writes (W, R, N) to tune between consistency and availability based on business needs. Handle write conflicts using Last Write Wins for simple cases or Version Vectors when data loss is unacceptable."
