DDD Deep Dive

Repository Pattern with Aggregates & Domain Services vs Application Services

🗄️ The Core Rule

In DDD, a Repository is the only gateway between your domain model and your persistence layer. The critical constraint:

One Repository per Aggregate Root — not per Entity. If Order is your aggregate root and OrderLine is an entity inside it, there is NO IOrderLineRepository. You always load, save, and reason about the entire Order aggregate as a unit.

This is the persistence boundary. The aggregate root is the consistency boundary, and the repository is its persistence boundary.

🚫 Why Not a Repository Per Entity?

Challenge: Someone on your team creates an IOrderLineRepository that can update an order line without loading the parent Order. What breaks?

1. Invariant violations: The Order aggregate root enforces rules like "total must not exceed credit limit" or "max 50 line items." Modifying an OrderLine in isolation bypasses those checks entirely.

2. Concurrency corruption: Two requests could modify different OrderLines simultaneously. Without the aggregate root coordinating, you lose transactional consistency — you can't use a version stamp on the root to detect conflicts.

3. Event sourcing mismatch: Domain events are raised by the aggregate root. If you mutate a child entity directly, no event is emitted, so downstream projections and subscribers see stale data.

Bottom line: The aggregate root IS the transactional boundary. A repo per child entity blows a hole in that boundary.

📐 Repository Interface — Lives in the Domain Layer

The interface is defined in the Domain layer, because the domain dictates WHAT it needs. The concrete implementation lives in Infrastructure, because that's WHERE the persistence details go.

// Domain Layer — Domain/Repositories/IOrderRepository.cs
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(Guid orderId);
    Task<IReadOnlyList<Order>> GetByCustomerAsync(Guid customerId);
    Task AddAsync(Order order);
    void Remove(Order order);

    // NOTE: No "Update" method. EF tracks changes automatically.
    // Save happens via Unit of Work (IUnitOfWork) — not here.
}

// Optional: Generic base if you have many aggregates
public interface IRepository<T> where T : AggregateRoot
{
    Task<T?> GetByIdAsync(Guid id);
    Task AddAsync(T entity);
    void Remove(T entity);
}
// Infrastructure Layer — Persistence/Repositories/EfOrderRepository.cs
public class EfOrderRepository : IOrderRepository
{
    private readonly AppDbContext _db;

    public EfOrderRepository(AppDbContext db) => _db = db;

    public async Task<Order?> GetByIdAsync(Guid orderId)
    {
        // Always load the FULL aggregate — root + children
        return await _db.Orders
            .Include(o => o.Lines)        // child entities
            .Include(o => o.ShippingAddress) // value object (owned type)
            .FirstOrDefaultAsync(o => o.Id == orderId);
    }

    public async Task<IReadOnlyList<Order>> GetByCustomerAsync(Guid customerId)
    {
        return await _db.Orders
            .Include(o => o.Lines)
            .Where(o => o.CustomerId == customerId)
            .ToListAsync();
    }

    public async Task AddAsync(Order order)
    {
        await _db.Orders.AddAsync(order);
        // EF tracks all child entities automatically
    }

    public void Remove(Order order)
    {
        _db.Orders.Remove(order);
        // Cascade delete handles OrderLines via config
    }
}
// Program.cs or DI Extension
builder.Services.AddScoped<IOrderRepository, EfOrderRepository>();
builder.Services.AddScoped<IUnitOfWork, EfUnitOfWork>();

// ─── The Unit of Work ───
public interface IUnitOfWork
{
    Task<int> SaveChangesAsync(CancellationToken ct = default);
}

public class EfUnitOfWork : IUnitOfWork
{
    private readonly AppDbContext _db;
    public EfUnitOfWork(AppDbContext db) => _db = db;

    public Task<int> SaveChangesAsync(CancellationToken ct)
    {
        // This is where ALL tracked changes commit in one transaction
        return _db.SaveChangesAsync(ct);
    }
}

🔄 The Lifecycle: Load → Mutate → Save

Every aggregate operation follows the same three-step dance. This is the heartbeat of Repository + Aggregate persistence.

Controller / Handler
→
Application Service
↓
1. Load aggregate via repository
IOrderRepository
→
EF Core / DB
↓
2. Call behavior methods on the aggregate root
order.AddLine(...)
→
Invariants checked
↓
3. Persist via Unit of Work — single transaction
IUnitOfWork
→
SaveChangesAsync()
public class AddOrderLineHandler
    : IRequestHandler<AddOrderLineCommand>
{
    private readonly IOrderRepository _orders;
    private readonly IUnitOfWork _uow;

    public AddOrderLineHandler(
        IOrderRepository orders, IUnitOfWork uow)
    {
        _orders = orders;
        _uow = uow;
    }

    public async Task Handle(AddOrderLineCommand cmd, CancellationToken ct)
    {
        // 1. LOAD — full aggregate
        var order = await _orders.GetByIdAsync(cmd.OrderId)
            ?? throw new NotFoundException("Order", cmd.OrderId);

        // 2. MUTATE — domain logic lives on the aggregate
        order.AddLine(cmd.ProductId, cmd.Quantity, cmd.UnitPrice);
        // ↑ This enforces: max 50 lines, no duplicates, recalculates total

        // 3. SAVE — Unit of Work commits everything
        await _uow.SaveChangesAsync(ct);
    }
}
// ❌ ANTI-PATTERN: Anemic domain + fat service
public async Task Handle(AddOrderLineCommand cmd, CancellationToken ct)
{
    var order = await _orders.GetByIdAsync(cmd.OrderId);

    // ❌ Business logic in the SERVICE, not the domain
    if (order.Lines.Count >= 50)
        throw new DomainException("Too many lines");

    // ❌ Creating child entity directly — bypassing aggregate root
    var line = new OrderLine(cmd.ProductId, cmd.Quantity, cmd.UnitPrice);
    order.Lines.Add(line);

    // ❌ Manually recalculating — this belongs in Order.AddLine()
    order.Total = order.Lines.Sum(l => l.Quantity * l.UnitPrice);

    await _uow.SaveChangesAsync(ct);
}
Why this breaks DDD: The invariant checks and calculations leak into the application layer. If any other handler adds a line, it has to duplicate all this logic — or miss it entirely. The aggregate becomes an anemic data bag.

💎 Persisting Value Objects as Owned Types

Value Objects have no identity — they're defined entirely by their properties. EF Core maps them using OwnsOne / OwnsMany, which stores them in the same table as the parent (or a shadow table).

// Domain Layer — a Value Object
public class Address : ValueObject
{
    public string Street { get; }
    public string City { get; }
    public string ZipCode { get; }
    public string Country { get; }

    public Address(string street, string city,
               string zipCode, string country)
    {
        // Validation in constructor — always valid
        if (string.IsNullOrWhiteSpace(street))
            throw new DomainException("Street required");
        Street = street;
        City = city;
        ZipCode = zipCode;
        Country = country;
    }

    protected override IEnumerable<object> GetEqualityComponents()
    {
        yield return Street;
        yield return City;
        yield return ZipCode;
        yield return Country;
    }
}
// Infrastructure — EF Configuration
public class OrderConfiguration
    : IEntityTypeConfiguration<Order>
{
    public void Configure(EntityTypeBuilder<Order> builder)
    {
        builder.HasKey(o => o.Id);

        // Value Object → same table, prefixed columns
        builder.OwnsOne(o => o.ShippingAddress, sa =>
        {
            sa.Property(a => a.Street)
              .HasColumnName("ShippingStreet")
              .HasMaxLength(200);
            sa.Property(a => a.City)
              .HasColumnName("ShippingCity");
            sa.Property(a => a.ZipCode)
              .HasColumnName("ShippingZip");
            sa.Property(a => a.Country)
              .HasColumnName("ShippingCountry");
        });

        // Child entity collection → separate table
        builder.HasMany(o => o.Lines)
            .WithOne()
            .HasForeignKey("OrderId")
            .OnDelete(DeleteBehavior.Cascade);

        // Concurrency token on the aggregate root
        builder.Property<byte[]>("RowVersion")
            .IsRowVersion();
    }
}

🔒 Optimistic Concurrency — The Version Stamp

Because the entire aggregate is the transactional unit, you put the concurrency token on the aggregate root's table. If two requests load the same Order, mutate it differently, and try to save — the second one gets a DbUpdateConcurrencyException.

Request A
Request B
↓ both load Order v1
AddLine()
RemoveLine()
↓
A saves → v2 ✓
B saves → CONFLICT ✗
B's RowVersion doesn't match → DbUpdateConcurrencyException

Challenge: If you put the RowVersion on OrderLine instead of Order, what happens when Request A adds a line and Request B updates an existing line?

No conflict detected. They're touching different rows, so both writes succeed. But the aggregate-level invariant (e.g., "total must not exceed credit limit") might now be violated because neither request saw the other's change. The version stamp MUST be on the aggregate root to guard the entire boundary.

🧭 The Core Distinction

This is one of the most commonly confused concepts in DDD. The key question to ask yourself:

"Does this logic represent a business rule / domain concept, or does it orchestrate infrastructure and coordinate workflow?"
Business rule → Domain Service. Orchestration → Application Service.
API / ControllersHTTP concerns, model binding, routing
Application ServicesUse cases, orchestration, transactions, auth
Domain Services + Entities + Value ObjectsBusiness rules, invariants, domain logic
InfrastructureEF Core, message bus, email, file storage

Domain Layer Domain Service

When do you need one?

A Domain Service exists when business logic doesn't naturally belong to a single aggregate. The logic spans multiple aggregates or requires external domain knowledge (e.g., pricing rules, transfer policies) but is still purely a business concept — no infrastructure, no I/O.

  • Contains business rules and domain logic only
  • Operates on multiple aggregates or domain concepts
  • Has no dependency on infrastructure (no DbContext, no HttpClient)
  • Is stateless — takes inputs, returns outputs
  • Lives in the Domain layer alongside entities
  • Does NOT orchestrate workflows or transactions
  • Does NOT call repositories (it receives loaded aggregates)
// Domain Layer — Domain/Services/MoneyTransferService.cs
public class MoneyTransferService
{
    // Pure domain logic — no repos, no I/O, no DbContext
    public void Transfer(
        Account from,
        Account to,
        Money amount)
    {
        // Business rule: can't transfer to yourself
        if (from.Id == to.Id)
            throw new DomainException("Cannot transfer to same account");

        // Business rule: different currencies need conversion
        if (from.Currency != to.Currency)
            throw new DomainException("Cross-currency transfer not allowed here");

        // Delegate to each aggregate — THEY enforce their own invariants
        from.Debit(amount);   // throws if insufficient funds
        to.Credit(amount);
    }
}
Key observation: MoneyTransferService doesn't load accounts from the DB. It receives already-loaded aggregates. The "who loads and saves" job belongs to the Application Service above it. 
// Domain Layer — Domain/Services/PricingService.cs
public class PricingService
{
    // Business rules about how to calculate a price
    public Money CalculateOrderTotal(
        Order order,
        Customer customer,
        IReadOnlyList<DiscountPolicy> policies)
    {
        var subtotal = order.Lines
            .Sum(l => l.Quantity * l.UnitPrice);

        // Apply discount policies — pure domain logic
        var discount = policies
            .Where(p => p.IsApplicable(customer, order))
            .Select(p => p.CalculateDiscount(subtotal))
            .DefaultIfEmpty(Money.Zero)
            .Max();

        return subtotal - discount;
    }
}

Application Layer Application Service

The Orchestrator

An Application Service is a use-case coordinator. It doesn't contain business logic — it wires together the pieces: load aggregates, call domain services, dispatch events, commit transactions.

  • Orchestrates a complete use case / workflow
  • Calls repositories to load/save aggregates
  • Calls domain services for cross-aggregate logic
  • Manages transactions via Unit of Work
  • Handles cross-cutting: authorization, logging, event dispatch
  • Lives in the Application layer (above Domain)
  • Does NOT contain business rules or invariant checks
  • Does NOT directly manipulate entity state
// Application Layer — orchestrates the transfer use case
public class TransferMoneyHandler
    : IRequestHandler<TransferMoneyCommand>
{
    private readonly IAccountRepository _accounts;
    private readonly MoneyTransferService _transferService; // domain svc
    private readonly IUnitOfWork _uow;
    private readonly ILogger<TransferMoneyHandler> _logger;

    public async Task Handle(
        TransferMoneyCommand cmd, CancellationToken ct)
    {
        // 1. LOAD aggregates (Application concern)
        var from = await _accounts.GetByIdAsync(cmd.FromAccountId)
            ?? throw new NotFoundException("Account");
        var to = await _accounts.GetByIdAsync(cmd.ToAccountId)
            ?? throw new NotFoundException("Account");

        // 2. EXECUTE domain logic (delegated to Domain Service)
        _transferService.Transfer(from, to, cmd.Amount);

        // 3. SAVE (Application concern — transaction)
        await _uow.SaveChangesAsync(ct);

        // 4. SIDE EFFECTS (Application concern)
        _logger.LogInformation(
            "Transferred {Amount} from {From} to {To}",
            cmd.Amount, cmd.FromAccountId, cmd.ToAccountId);
    }
}
// Application Layer — PlaceOrder use case
public class PlaceOrderHandler
    : IRequestHandler<PlaceOrderCommand, Guid>
{
    private readonly IOrderRepository _orders;
    private readonly ICustomerRepository _customers;
    private readonly PricingService _pricing; // domain svc
    private readonly IUnitOfWork _uow;
    private readonly IMediator _mediator;

    public async Task<Guid> Handle(
        PlaceOrderCommand cmd, CancellationToken ct)
    {
        // Load
        var customer = await _customers.GetByIdAsync(cmd.CustomerId);

        // Create aggregate — factory method on the root
        var order = Order.Create(customer.Id, cmd.ShippingAddress);
        foreach (var item in cmd.Items)
            order.AddLine(item.ProductId, item.Qty, item.Price);

        // Domain service for cross-aggregate logic
        var total = _pricing.CalculateOrderTotal(
            order, customer, await GetPolicies());
        order.SetTotal(total);

        // Persist
        await _orders.AddAsync(order);
        await _uow.SaveChangesAsync(ct);

        // Dispatch domain events after commit
        foreach (var evt in order.DomainEvents)
            await _mediator.Publish(evt, ct);

        return order.Id;
    }
}

🤔 Decision Flowchart: Where Does This Logic Go?

Challenge: For each scenario, decide: Entity method, Domain Service, or Application Service?

Scenario 1

"Check if an order can be cancelled (must be in Pending status)"

Entity method — order.Cancel(). Single aggregate, single invariant. The Order knows its own status.

Scenario 2

"Transfer inventory from Warehouse A to Warehouse B with business rules about minimum stock levels"

Domain Service — Two aggregates (Warehouse A & B) involved. The transfer rule ("don't go below minimum") spans both. A StockTransferService receives both loaded warehouses.

Scenario 3

"Load the order, validate it, charge payment via Stripe, send confirmation email"

Application Service — This is workflow orchestration. It loads aggregates, calls a domain method for validation, then coordinates infrastructure (Stripe API, email service). The payment gateway and email sender are infrastructure concerns.

Scenario 4

"Determine if a Customer qualifies for Gold tier based on their total spend across all their orders"

Domain Service — The tier rule is pure business logic, but it needs data from both the Customer aggregate and multiple Order aggregates. A LoyaltyTierService takes the customer + order history and returns the tier. The Application Service loads the data and calls it.

⚡ Bonus: Infrastructure Service

Don't confuse Domain Services with Infrastructure Services. These are technical capabilities consumed by the application layer:

Infrastructure Service What It Does Layer
IEmailSender Sends emails via SMTP/SendGrid Infrastructure
IPaymentGateway Charges cards via Stripe/PayPal Infrastructure
IFileStorage Stores blobs in Azure/S3 Infrastructure
IMessageBus Publishes to RabbitMQ/Azure SB Infrastructure

The interface for an infrastructure service CAN live in the Application layer (since app services call them). The concrete implementation lives in Infrastructure. This follows the Dependency Inversion Principle — your inner layers define what they need; outer layers provide it.

🎯 Interview Quick-Fire Answers

"Why does the repository interface live in the Domain layer?"

Because the domain defines what it needs ("I need a way to get an Order by ID"), but it doesn't care about HOW that's implemented. This is the Dependency Inversion Principle — the inner layer owns the abstraction, the outer layer provides the implementation. If you put the interface in Infrastructure, your Domain would depend on Infrastructure — that inverts the dependency flow.

"When would you NOT use the Repository pattern?"

CQRS read side. Repositories are for the write model — loading aggregates to enforce invariants. For the read side, you often use thin queries (Dapper, raw SQL, read-optimized projections) that bypass the domain model entirely. No need to load a full aggregate graph just to display a list view. Also, in simple CRUD apps with no real business logic, a repository adds ceremony with no benefit — EF's DbContext already IS a Unit of Work + Repository.

"Can a Domain Service call a Repository?"

Purist DDD says no. A Domain Service should receive already-loaded aggregates from the Application Service. This keeps the Domain layer free of I/O and persistence concerns, making it trivially testable. However, in pragmatic implementations, some teams inject the repository interface into domain services for convenience. The trade-off is testability and purity vs. a simpler call chain. In an interview, state the purist view first, then acknowledge the pragmatic variation.
