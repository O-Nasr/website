01 Two Halves of DDD

DDD has two distinct halves that solve different problems. Most people learn one and miss the other:

Strategic DDD — the architect's view
How to split a system into services. Where to draw boundaries. How services communicate at boundaries.
Concepts: ubiquitous language, bounded context, subdomain types, context mapping, anti-corruption layer
Tactical DDD — the developer's view
How to structure code INSIDE a service. The building blocks of a rich domain model.
Concepts: entity, value object, aggregate, aggregate root, domain event, repository, domain service
Strategic DDD answers: "Where are the boundaries?"
  → You discover bounded contexts (Orders, Payments, Inventory)
  → Each bounded context becomes a microservice

Tactical DDD answers: "How do I build INSIDE each boundary?"
  → Inside Orders service: Order (aggregate), OrderLine (entity),
    Money (value object), OrderPlaced (domain event)

Strategic = draw the map. Tactical = build each city on the map.

02 Ubiquitous Language

The most important concept in all of DDD, yet the most overlooked. The problem: developers and business people use different words for the same thing.

Translation tax — every translation is a chance for a bug

Business says:          Developer writes:
"Customer"              → User (in the DB)
"Place an order"        → CreateOrderRequest (in the API)
"The order is active"   → Status = 2 (in the database)
"Discount"              → PriceModifier (in the code)

When business says "the customer's active order should get the loyalty discount," the developer mentally translates: "the User's Order where Status = 2 should get a PriceModifier of type Loyalty." Every translation is a chance for a bug.

// ❌ Developer language — business can't read this
public class User
{
    public int StatusCode { get; set; }  // what does 2 mean?
    public List<PriceModifier> Modifiers { get; set; }
}

// ✅ Ubiquitous language — business can read this
public class Customer
{
    public OrderStatus Status { get; set; }  // Active, Completed, Cancelled
    public LoyaltyDiscount? LoyaltyDiscount { get; set; }

    public bool HasActiveOrder =>
        Orders.Any(o => o.Status == OrderStatus.Active);
}
The rule: If a business person reads your class names and method names, they should understand what the code does without explanation. The code uses the exact same words as the business. No translation layer. No abbreviations. No "developer-speak."

03 Subdomain Types

There are three types of subdomain. Knowing which type determines how much effort you invest:

Core subdomain — your competitive advantage
What makes your business UNIQUE. Complex, changes frequently, gives market edge.
Examples: Amazon's recommendation engine. Uber's surge pricing. Netflix's personalization.
Build: In-house, best engineers, full DDD tactical patterns. Never outsource.
Supporting subdomain — necessary but not differentiating
Your business needs it, but it's not what makes you special. Some custom logic, no competitive advantage.
Examples: Inventory management for e-commerce. Employee scheduling for a restaurant chain.
Build: In-house but simpler. CRUD is fine. Junior/mid developers. Don't over-engineer.
Generic subdomain — solved problems, buy don't build
Every company needs this, nothing unique about your version. A solved problem.
Examples: Authentication (Auth0). Email (SendGrid). Payments (Stripe). Storage (S3).
Build: DON'T. Buy it or use SaaS. Every hour on this is an hour NOT on your core domain.

Build or buy?Build in-houseBuild simpleBuy / SaaS

The interview answer: "I apply full DDD tactical patterns only to core subdomains — that's where complexity lives and where business differentiation matters. Supporting subdomains get simple CRUD. Generic subdomains get bought, not built."

04 Bounded Context — Deeper

You know bounded contexts from the microservices roadmap. Let's go deeper with what lives inside one:

The word "Order" means different things in different contexts:

Orders context
Order = items, quantity, status, customer
Payments context
Order = amount, payment method, transaction ID
Shipping context
Order = weight, dimensions, address, carrier
Analytics context
Order = revenue, conversion data, funnel step

Each context owns its definition. They never share internal domain objects. Communication at boundaries is through events or API contracts with translation.

One bounded context contains multiple aggregates:

Orders bounded context (= one microservice)
├── Aggregate: Order    (root: Order, entities: OrderLine)
├── Aggregate: Cart     (root: Cart, entities: CartItem)
└── Aggregate: Discount (root: Discount, entities: DiscountRule)

Each aggregate has its own repository and its own transaction.
They reference each other by ID and communicate via domain events.
Domain: e-commerce
├── Bounded Context: Orders (= Orders microservice)
│   ├── Order aggregate
│   ├── Cart aggregate
│   └── Discount aggregate
├── Bounded Context: Catalogue (= Catalogue microservice)
│   ├── Product aggregate
│   ├── Category aggregate
│   └── Review aggregate
├── Bounded Context: Identity (= Identity microservice)
│   ├── Customer aggregate
│   └── Address aggregate
└── Bounded Context: Payments (= Payments microservice)
    ├── Payment aggregate
    └── Refund aggregate

05 Context Mapping Patterns

How do bounded contexts relate to each other? Context mapping defines the relationship patterns:

Partnership — equals, co-evolve

Two teams work together and evolve their APIs jointly. Neither dominates.

Example: Orders team and Payments team build checkout together, agree on shared events.

When: Two internal teams with strong dependency and good communication. Rare — requires high trust.

Customer-Supplier — upstream serves downstream

The upstream (supplier) provides an API. The downstream (customer) consumes it. Upstream considers downstream's needs but makes final decisions.

Example: Inventory (upstream) provides stock data. Orders (downstream) consumes it.

When: Most internal service-to-service relationships. The most common pattern.

Conformist — just use their model

Downstream has no influence over upstream. Must conform to whatever is provided.

Example: Your service integrates with Stripe's API. You can't ask Stripe to change their data model.

When: Integrating with external services, third-party APIs, legacy systems that won't change.

Anti-corruption layer (ACL) — protect your model

Like conformist, upstream won't change. But instead of conforming, you build a translation layer that converts the external model into YOUR clean model.

Example: Legacy ERP exports CUST_NR and ORD_DT. ACL translates into clean Customer and OrderDate.

When: External model is so different that conforming would pollute your domain. The most important defensive pattern in DDD.

Shared kernel — shared code between contexts

Two contexts share a small piece of code (types, events, contracts). Changes require agreement from both teams.

Example: Orders and Payments share Money value object and OrderId type in a shared NuGet package.

When: Use sparingly. Keep as small as possible. Shared events are the most common shared kernel.

06 Anti-Corruption Layer — Full C# Implementation

The ACL translates ugly external models into your clean domain. The most important defensive pattern:

// The legacy ERP response — you didn't design this, it's horrible
public class ErpOrderResponse
{
    public string CUST_NR { get; set; }
    public string ORD_DT { get; set; }     // "20260328" — not even ISO format
    public decimal TOT_AMT { get; set; }
    public int STAT_CD { get; set; }        // 1=active, 2=complete, 99=cancelled
    public List<ErpLineItem> LN_ITMS { get; set; }
}
// Your clean domain model — ubiquitous language
public class Order
{
    public CustomerId CustomerId { get; }
    public DateTime OrderDate { get; }
    public Money TotalAmount { get; }
    public OrderStatus Status { get; }
    public IReadOnlyList<OrderLine> Lines { get; }
}
// The Anti-Corruption Layer — translates between the two
public class ErpOrderTranslator
{
    public Order Translate(ErpOrderResponse erp)
    {
        return new Order(
            customerId: new CustomerId(erp.CUST_NR),
            orderDate: DateTime.ParseExact(erp.ORD_DT, "yyyyMMdd", null),
            totalAmount: Money.FromDecimal(erp.TOT_AMT, Currency.USD),
            status: MapStatus(erp.STAT_CD),
            lines: erp.LN_ITMS.Select(TranslateLine).ToList()
        );
    }

    private OrderStatus MapStatus(int code) => code switch
    {
        1  => OrderStatus.Active,
        2  => OrderStatus.Completed,
        99 => OrderStatus.Cancelled,
        _  => throw new DomainException($"Unknown ERP status: {code}")
    };
}
// Your service uses the translator — never touches ERP model directly
public class OrderService
{
    private readonly IErpClient _erp;
    private readonly ErpOrderTranslator _translator;

    public async Task<Order> GetOrderAsync(string orderId)
    {
        ErpOrderResponse ugly = await _erp.FetchOrderAsync(orderId);
        Order clean = _translator.Translate(ugly);  // ACL does the work
        return clean;  // your domain never sees CUST_NR or STAT_CD
    }
}
ERP changes field names? Update one translator. Your domain model is untouched. The mess is isolated in the ACL.

07 Interview Q&A

Q: What is ubiquitous language?
The code uses the exact same words as the business. Class names, method names, enums — all match how the business describes the domain. No translation between "business language" and "developer language." Every translation is a chance for a bug.
Q: What are the three subdomain types?
Core = your competitive advantage (build in-house, best engineers, full DDD). Supporting = necessary but not differentiating (build simple, CRUD). Generic = solved problem (buy/SaaS — auth, email, payments). Apply DDD patterns only to core subdomains.
Q: What is a bounded context?
An explicit boundary within which a domain model applies. The same word ("Order") means different things in different contexts — Orders context (items, status), Payments context (amount, transaction), Shipping context (weight, address). Each context owns its definition. They never share internal models.
Q: What is context mapping?
The patterns that describe how bounded contexts relate to each other. Partnership (co-evolve), Customer-Supplier (upstream serves downstream), Conformist (accept external model), Anti-corruption layer (translate external to internal), Shared kernel (small shared code).
Q: What is an anti-corruption layer?
A translation layer that protects your clean domain model from ugly external systems. Instead of conforming to a legacy ERP's CUST_NR and STAT_CD fields, the ACL translates them into Customer and OrderStatus. External mess stays in the translator; your domain stays clean.
Q: When do you NOT use DDD?
When the domain is simple. CRUD apps, data pipelines, supporting subdomains with no complex business rules. DDD adds overhead — only apply it to core subdomains where complex business logic and differentiation justify the investment.
