The Homeless Logic Problem

In DDD, the Entity is king. Business logic lives on the Entity: Order.AddItem(), Order.Cancel(), Customer.Deactivate(). The entity owns its own rules. That's the default home for business logic.

But here's the problem: some business logic doesn't belong to any single entity.

When Logic Is Homeless

This question needs the Order's total (from the Order aggregate) AND the Customer's credit limit (from the Customer aggregate). Neither entity can see the other.

The Answer: Domain Service

A Domain Service receives both aggregates as method parameters and makes the business decision using both. It doesn't go fetch them from a database. It doesn't save anything. Someone else hands them in, and the service just thinks.

That "someone else" who does the fetching and saving — that's the Application Service. Let's look at that next.

The Brain vs Hands Mental Model

The simplest way to remember the difference:

Domain Service = The Brain

You hand it ingredients, it does something smart with them, and it hands you back a result. It doesn't go shopping for the ingredients itself — someone else does that.

Think of a head chef. You put the raw chicken and spices on the counter, they cook something brilliant. They never walk to the grocery store. They never wash dishes. They just cook.

Application Service = The Hands

It fetches things, hands them to the brain, saves the result, and handles side effects. Zero business decisions.

Think of a kitchen manager. They go to the store (repository), buy ingredients (load aggregates), put them on the counter for the chef (call domain service), clean up after (save changes), and serve the plate (return DTO).

The "Would a Business Expert Care?" Test

For every step in a use case, ask: "Would a banker, a product manager — a domain expert — care about this step?" If yes, it's business logic (Domain layer). If no, it's plumbing (Application Service).

Use Case: "Place an order with credit check"
1 Load the Customer from the database — Would a banker care? No. plumbing
2 Build the Order with its items — Would a banker care? Yes. business
3 Can this customer afford it? — Would a banker care? Yes. business
4 Save the Order to the database — Would a banker care? No. plumbing
5 Send confirmation email — Would a banker care? No. plumbing
Bottom line: Plumbing = Application Service (the hands). Business decisions = Domain layer (the brain). Steps 1, 4, 5 are hands work. Steps 2, 3 are brain work.

Three Layers of Logic Ownership

There isn't just a two-way split. Business logic itself has two homes — the Entity and the Domain Service — and they do different things. Here's the full hierarchy:

Application Service zero business rules Loads, calls, saves, notifies — just orchestration
Domain Service cross-aggregate rules Coordinates multiple entities for a business decision
Entity / Aggregate Root single-aggregate rules Owns invariants within its own boundary
Infrastructure Repositories, email senders, message queues

How the Domain Service Delegates to Entities

The Domain Service doesn't do everything — it coordinates the entities, and each entity still owns its own single-aggregate rules. Consider the money transfer:

// Domain Service — coordinates two aggregates
public class MoneyTransferService
{
    public void Transfer(
        Account from,
        Account to,
        Money amount)
    {
        // Business rule: check frozen status (cross-aggregate concern)
        if (from.IsFrozen || to.IsFrozen)
            throw new AccountFrozenException();

        // Delegate to each entity — they own their own invariants
        from.Debit(amount);   // Account internally checks balance >= 0
        to.Credit(amount);     // Account internally updates balance
    }
}
Key insight: from.Debit(amount) internally checks "do I have enough balance?" and throws if not. The Domain Service doesn't duplicate that check — it just orchestrates the sequence: verify not frozen, debit one, credit the other. Each entity still protects its own invariant.

The Hierarchy of Logic Ownership

Who Owns What — Quick Reference

"Can this order be cancelled?" → Entity — single aggregate, order.Cancel() checks its own status.

"Is the total price correct given this customer's tier?" → Domain Service — needs Order + Customer data.

"Debit one account, credit another, check both aren't frozen" → Domain Service — coordinates two entities.

"Load the order, cancel it, save, send email" → Application Service — just orchestration verbs.

"Log the HTTP request and return a response" → Controller — presentation layer, not even business-adjacent.

The Three Nevers of a Domain Service

A Domain Service is a pure brain. It thinks, but it never reaches out into the outside world. All three of these rules follow from that principle:

Never #1: Access the Database

No repositories, no DbContext, no SQL. If the Domain Service needs data, it receives it as parameters (aggregates passed in by the Application Service) or through a Domain-layer interface like IEmailUniquenessChecker (covered in Tab 5). It never directly reaches into the database.

Never #2: Call SaveChangesAsync()

The Domain Service can mutate the objects it receives (like calling account.Debit(amount)), but it never persists anything. The Application Service decides when to commit — this is what makes the Unit of Work pattern possible. Bundling multiple changes into one transaction only works if no one saves prematurely.

Never #3: Trigger Side Effects

No sending emails, no publishing to message queues, no calling external HTTP APIs directly. If placing an order should send a confirmation email, that happens in the Application Service after the Domain Service finishes its business logic. The Domain Service's world ends at the boundary of the domain objects it was given.

One sentence to remember: The Domain Service never reaches out. It only works with what's placed on its desk. Loading is reaching out. Saving is reaching out. Sending an email is reaching out.

Complete Responsibility Split

Domain Service (brain)

→ Cross-aggregate business rules

→ Coordinate entity methods

→ Validate + throw if invalid

→ Call domain-layer interfaces (e.g., IEmailUniquenessChecker)

Application Service (hands)

→ Load from repositories

→ Call SaveChangesAsync()

→ Send emails / notifications

→ Map DTOs to domain objects

The Interface Trick — When Domain Needs DB Access

Why It Matters

The Scenario

"Before creating a new Customer, check that no other customer already has this email address."

This is clearly a business rule — a domain expert cares about it. But to answer it, you need to query the database to check if any existing customer has this email. The Domain Service can't touch the database. So what do you do?

The tension: The rule belongs in the Domain layer (it's a business rule). But answering it requires database access (which the Domain layer can't do). How do we resolve this?

Wrong Approach: Application Service Does the Check

You might think: "The Application Service can query the repository and pass the result to the Domain Service." Like this:

// Application Service — doing the check itself
var existingCustomer = await _customerRepo.GetByEmailAsync(email);
if (existingCustomer != null)
    throw new EmailAlreadyTakenException();

This has two problems:

Problem 1: Business Logic Leaks Into the Application Service

That if (existingCustomer != null) is a business decision. The Application Service decided that != null means "email is taken" and that this should throw an exception. That's brain work sitting in the hands layer. The Application Service is now interpreting a query result — and interpreting data to make a business decision is the Domain layer's job.

Problem 2: Repository Gets Shaped Around a Business Rule

Putting GetByEmailAsync on the Repository shapes the Repository's interface around a specific business rule rather than around Aggregate retrieval. The Repository's job is to be a clean, collection-like abstraction: "give me this Order," "save this Customer." It starts accumulating query methods that only exist to serve individual business checks — GetByEmail, GetByPhone, GetByTaxId — and slowly becomes a general-purpose query layer.

Right Approach: Domain-Layer Interface

The Domain layer defines an interface — a contract that says "I need the ability to check if an email is taken." The Domain Service depends on that interface. Infrastructure implements it.

// ── Domain Layer ──

// The interface lives HERE, in the Domain
public interface IEmailUniquenessChecker
{
    Task<bool> IsEmailTakenAsync(string email);
}

// The Domain Service depends on the interface
public class CustomerRegistrationService
{
    private readonly IEmailUniquenessChecker _checker;

    public CustomerRegistrationService(
        IEmailUniquenessChecker checker) => _checker = checker;

    public async Task<Customer> RegisterAsync(
        string name, string email)
    {
        // The ENTIRE business rule lives here in the Domain
        if (await _checker.IsEmailTakenAsync(email))
            throw new EmailAlreadyTakenException(email);

        return new Customer(name, email);
    }
}
// ── Infrastructure Layer ──

// The implementation lives HERE, in Infrastructure
public class EfEmailUniquenessChecker : IEmailUniquenessChecker
{
    private readonly AppDbContext _db;

    public EfEmailUniquenessChecker(AppDbContext db) => _db = db;

    public async Task<bool> IsEmailTakenAsync(string email)
    {
        return await _db.Customers
            .AnyAsync(c => c.Email == email);
    }
}
Same pattern as Repository: IOrderRepository lives in Domain, OrderRepository lives in Infrastructure. The IEmailUniquenessChecker is just another instance of Dependency Inversion — the Domain defines what it needs, Infrastructure conforms to it.

Why This Matters — The Deeper Reason

Even if you remove the if statement and just pass data to the Domain Service, there's still a subtle problem. Imagine:

// Application Service — no if, just passing data
var existingCustomer = await _customerRepo.GetByEmailAsync(email);
_registrationService.Register(newCustomerData, existingCustomer);

The Application Service knows that it needs to search by email, and it knows why — because uniqueness matters. It's performing the first half of a business rule (gathering evidence for a uniqueness check) and letting the Domain Service perform the second half (making the judgment). The business rule has been split across two layers.

The split-rule problem: Half the logic ("search by email because uniqueness matters") lives in the Application Service. The other half ("if taken, reject") lives in the Domain Service. This makes the rule harder to understand, test, and maintain — because you can't see the whole business rule in one place.

With the IEmailUniquenessChecker approach, the entire business rule — the "what to query," the "how to interpret it," and the "what to do about it" — lives as a single cohesive unit inside the Domain layer. The Application Service doesn't even know that a uniqueness check is happening. It just calls registrationService.RegisterAsync(name, email) and the Domain Service handles everything internally through its interface.

The principle: A business rule should be visible in one place. If you have to read two layers to understand "how does email uniqueness work?", the rule has been fragmented. The Domain-layer interface pattern keeps it cohesive.

Domain Layer — Entities, Domain Service, Interfaces

// ── Aggregate Root ──
public class Order : AggregateRoot
{
    private readonly List<OrderItem> _items = new();
    public IReadOnlyList<OrderItem> Items => _items;
    public CustomerId CustomerId { get; }
    public OrderStatus Status { get; private set; }

    public void AddItem(ProductId productId, int qty, Money unitPrice)
    {
        // Entity-level invariant — single aggregate rule
        if (_items.Count >= 20)
            throw new DomainException("Max 20 items per order.");
        _items.Add(new OrderItem(productId, qty, unitPrice));
    }

    public Money GetSubtotal()
        => _items.Aggregate(Money.Zero, (sum, i) => sum + i.LineTotal);
}

// ── Domain Service — cross-aggregate business rule ──
public class OrderPricingService
{
    public Money CalculateFinalPrice(
        Order order,
        Customer customer,
        IReadOnlyList<DiscountPolicy> policies)
    {
        // Needs Order + Customer + Policies = cross-aggregate
        var total = order.GetSubtotal();
        foreach (var p in policies)
            total = p.Apply(total, customer.Tier);
        return total;
    }
}

// ── Repository Interfaces — live in Domain ──
public interface IOrderRepository : IRepository<Order> { }
public interface ICustomerRepository : IRepository<Customer> { }
public interface IDiscountPolicyRepository : IRepository<DiscountPolicy>
{
    Task<IReadOnlyList<DiscountPolicy>> GetActiveAsync(CancellationToken ct);
}

Application Layer — The Conductor

public class PlaceOrderHandler
{
    private readonly IOrderRepository _orders;
    private readonly ICustomerRepository _customers;
    private readonly IDiscountPolicyRepository _policies;
    private readonly OrderPricingService _pricing;
    private readonly IUnitOfWork _uow;
    private readonly INotificationService _notifications;

    public async Task<PlaceOrderResult> HandleAsync(PlaceOrderCommand cmd)
    {
        // 1. LOAD — orchestration (hands)
        var customer = await _customers.GetByIdAsync(cmd.CustomerId);
        var policies = await _policies.GetActiveAsync(default);

        // 2. BUILD AGGREGATE — entity domain logic (brain)
        var order = new Order(customer.Id);
        foreach (var item in cmd.Items)
            order.AddItem(item.ProductId, item.Quantity, item.UnitPrice);

        // 3. DOMAIN SERVICE — cross-aggregate rule (brain)
        var finalPrice = _pricing.CalculateFinalPrice(
            order, customer, policies);

        // 4. PERSIST — orchestration (hands)
        await _orders.AddAsync(order);
        await _uow.SaveChangesAsync();

        // 5. SIDE EFFECT — orchestration (hands)
        await _notifications.SendAsync(
            customer.Id,
            $"Order placed. Total: {finalPrice}");

        // 6. RETURN DTO — orchestration (hands)
        return new PlaceOrderResult(order.Id, finalPrice);
    }
}
Audit the handler: Steps 1, 4, 5, 6 are pure orchestration (load, save, notify, map). Steps 2 and 3 are delegated to Domain objects. The Application Service contains zero business decisions — it's a conductor, not a musician.

DI Registration

// Program.cs — wire everything together

// Repositories (Infrastructure implements, Domain defines interface)
builder.Services.AddScoped<IOrderRepository, OrderRepository>();
builder.Services.AddScoped<ICustomerRepository, CustomerRepository>();
builder.Services.AddScoped<IDiscountPolicyRepository, DiscountPolicyRepository>();

// Unit of Work (DbContext already implements it)
builder.Services.AddScoped<IUnitOfWork>(sp =>
    sp.GetRequiredService<AppDbContext>());

// Domain Service (no interface needed — it's pure logic)
builder.Services.AddScoped<OrderPricingService>();

// Domain-layer interface implementations
builder.Services.AddScoped<IEmailUniquenessChecker, EfEmailUniquenessChecker>();

// Application Service / Handler
builder.Services.AddScoped<PlaceOrderHandler>();
Notice: The Domain Service (OrderPricingService) is registered without an interface. It's pure logic with no I/O — you can test it directly, no mocking needed. Only things that touch infrastructure (repositories, uniqueness checkers) need the interface-to-implementation split.

Q&A Deep Dive

Key questions and detailed answers from the teaching session. Click each to expand.

Why can't the Domain Service just receive IOrderRepository and load aggregates itself?

If a MoneyTransferService received IAccountRepository in its constructor and called await _accounts.GetByIdAsync(...) internally, it would be doing I/O — reaching into the database. The moment a Domain Service talks to a database (even through a repository interface), it stops being pure business logic.

This creates three problems. First, it becomes harder to unit test — you now need to mock the repository. Second, its responsibility blurs with the Application Service's job (loading is orchestration, not business logic). Third, the Domain Service can no longer be understood as a pure function — it has hidden dependencies on the state of the database.

The correct approach: the Application Service loads both aggregates first, then passes them as parameters to the Domain Service's method. The Domain Service never knows repositories exist.

For uniqueness checks — why not have the Application Service search by email and pass the result to the Domain Service?

This approach has two problems:

Problem 1 — Split business rule: Even without an if statement, the Application Service knows it needs to search by email and knows why — because uniqueness matters. It's performing the first half of a business rule (gathering evidence for a uniqueness check) and letting the Domain Service perform the second half (making the judgment). The business rule is now fragmented across two layers, making it harder to understand, test, and maintain.

Problem 2 — Repository gets shaped by business rules: Putting GetByEmailAsync on the Repository shapes the Repository's interface around a specific business rule rather than around Aggregate retrieval. The Repository's job is a clean collection-like abstraction: "give me this Order," "save this Customer." Adding query methods for every business check (GetByEmail, GetByPhone, GetByTaxId) slowly turns it into a general-purpose query layer, which is not what it's for.

The correct solution: Define an IEmailUniquenessChecker interface in the Domain layer. The Domain Service depends on it and owns the entire rule. Infrastructure implements it with EF Core. The Application Service never even knows a uniqueness check happens.

Is "transferring money" an Application Service task or a Domain Service task?

It's a Domain Service task. The word "transfer" sounds operational (like plumbing), but the act of moving money — debit one account, credit another, verify neither is frozen — is the core business operation. A banker would absolutely care about these rules.

Use the "would a business expert care?" test. A banker cares deeply about "you can't transfer from a frozen account" and "debit one, credit the other." A banker does NOT care about "load the account from row 47 in the database and commit a transaction." The transfer logic is brain work. The loading and saving is hands work.

The Application Service's role is: load both accounts from the repository, call transferService.Transfer(from, to, amount), then call SaveChangesAsync(). Zero business decisions.

Does a Domain Service need an interface?

Usually no. A Domain Service is pure business logic with no I/O. You can test it directly by creating an instance and calling its methods with real domain objects — no mocking needed. Interfaces are for abstracting away infrastructure concerns (databases, HTTP, file systems).

You would create an interface for a Domain Service only if you needed to swap between multiple strategies at runtime (e.g., different pricing algorithms). But in most cases, you just register the concrete class directly: services.AddScoped<OrderPricingService>().

What's the difference between a Domain Service and a Domain Event handler?

A Domain Service is called synchronously by the Application Service as part of a use case. It receives aggregates, applies a cross-aggregate business rule, and returns immediately. The caller controls when it runs.

A Domain Event handler is triggered reactively — it responds to something that already happened (e.g., OrderPlacedEvent). It's decoupled from the original use case. The original code doesn't call it directly; it just raises an event, and the handler reacts.

They're complementary patterns, not competing ones. A Domain Service makes decisions during a use case. A Domain Event handler reacts after a decision has been made.

Can an Application Service contain business logic if it's "simple enough"?

No. The temptation is always "it's just one if-statement, I'll put it here." But this is how Anemic Domain Models are born — business logic slowly migrates into Application Services because each individual rule seems "too simple" for a Domain Service. Before you know it, your domain objects are just data bags and all the intelligence lives in fat Application Services.

Even a single if (balance < amount) belongs on the Entity or Domain Service. The Application Service should contain only orchestration verbs: load, call, save, notify, map. If you can't describe a step with one of those verbs, it doesn't belong there.
