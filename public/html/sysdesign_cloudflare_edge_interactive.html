Core Problem

Cloudflare needed a rate limiter that could scale to millions of domains, handle billions of requests/day, operate at the edge of their global network, and be fast and resilient under DDoS attacks.

Key Challenges: Traffic is distributed across many servers and data centers (PoPs). Centralized counting is too slow due to network latency. Central services are harder to keep online during attacks.
Results: Billions of requests/day handled. Mitigated attacks with 400K req/sec to a single domain. Achieved 0.003% error rate across 400 million requests.
2

What is "The Edge of the Network"?

Geographically distributed servers closest to end users — part of a CDN or edge computing infrastructure. Instead of sending every request to a central data center, handle it as close to the user as possible.

Design Solutions

Each Point of Presence (PoP) maintains its own counters. Reduces latency and improves fault tolerance — no single point of failure.

Twemproxy clusters shard counters across servers. Consistent hashing ensures minimal disruption when resizing. Only two numbers per counter needed — minimal memory footprint.

AlgorithmHow It WorksTrade-off
Fixed WindowCount requests in fixed time windowsSimple but allows spikes at boundaries
Sliding WindowWeighted average of previous + current windowSmooth, accurate, low overhead
Leaky BucketProcesses requests at constant rateAccurate but harder with Memcached

Counting done asynchronously to avoid slowing legitimate requests. Once threshold exceeded, mitigation triggered and cached locally. Servers don't re-check the counter for every request — just enforce the cached mitigation.

Test Yourself

Q1: Why does Cloudflare count at the edge instead of centrally?

Centralized counting adds network round-trip latency to every request and creates a single point of failure. During DDoS attacks, a central service is harder to keep online. Edge counting is local, fast, and fault-tolerant.

Q2: Does the edge server only do rate limiting?

Edge servers are multi-purpose: caching, TLS termination, rate limiting, bot detection, firewall rules, and even running custom code (Workers). They only contact the origin when necessary.

Q3: Why does the sliding window algorithm work better than fixed window for rate limiting?

Fixed windows allow burst spikes at boundaries (e.g., 100 requests at end of window 1 + 100 at start of window 2 = 200 in a short period). Sliding window approximation weights the previous window's count to smooth this, with only two counters needed — minimal overhead.
