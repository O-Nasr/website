What Are Generics?

Generics let you write type-safe reusable code without duplicating the same logic for each type. The type parameter T is a placeholder replaced with the actual type at compile time.

⚠️ T is just a convention. You can name it anything: TItem, TEntity, TKey, TValue. Descriptive names are preferred for clarity.

The Problem → The Solution

NumberBox (int)
StringBox (string)

→
Box<T>

Three duplicated classes collapse into one generic class

C#// Must repeat for every type 😱
public class NumberBox
{
    private int _value;
    public void Set(int value) { _value = value; }
    public int Get() { return _value; }
}

public class StringBox
{
    private string _value;
    public void Set(string value) { _value = value; }
    public string Get() { return _value; }
}
C#// One class handles ALL types
public class Box<T>
{
    private T _value;
    public void Set(T value) { _value = value; }
    public T Get() { return _value; }
}

// Usage — T replaced at compile time
Box<int>    intBox    = new Box<int>();
Box<string> stringBox = new Box<string>();
Box<Order>  orderBox  = new Box<Order>();

intBox.Set(42);
Console.WriteLine(intBox.Get()); // 42

Generic Methods

Individual methods can be generic without making the whole class generic. The compiler can often infer T from the argument automatically.

C#public class Utilities
{
    // Generic method — works with any type
    public T GetFirst<T>(List<T> items)
    {
        if (items == null || items.Count == 0)
            throw new ArgumentException("List is empty");
        return items[0];
    }
}

var utils = new Utilities();
int first = utils.GetFirst(new List<int> { 1, 2, 3 });        // infers T = int
string name = utils.GetFirst(new List<string> { "Ali", "Omar" }); // infers T = string

Generic Constraints

Without constraints, T is completely unknown — you can't call any methods on it. Constraints restrict what types T can be.

C#public class Repository<T> where T : class
{
    public void Add(T item)
    {
        if (item == null)  // ✅ can check null — T is reference type
            throw new ArgumentNullException();
    }
}

var repo = new Repository<Order>();  // ✅ Order is a class
var repo = new Repository<int>();    // ❌ compile error — int is value type
C#public class ValueWrapper<T> where T : struct
{
    private T _value;
    public T Value => _value;
}

var w = new ValueWrapper<int>(42);     // ✅ int is value type
var w = new ValueWrapper<string>("");  // ❌ string is reference type
C#public class Factory<T> where T : new()
{
    public T Create() => new T(); // ✅ guaranteed constructor exists
}

var f = new Factory<Order>();    // ✅ has parameterless ctor
var f = new Factory<Invoice>();  // ❌ only has Invoice(int id)
C#public class AnimalShelter<T> where T : Animal
{
    public void Add(T animal)
    {
        animal.Feed();  // ✅ T is guaranteed to be Animal
    }
}

var s = new AnimalShelter<Dog>(); // ✅ Dog inherits Animal
var s = new AnimalShelter<Car>(); // ❌ Car doesn't inherit Animal
C#public class Sorter<T> where T : IComparable<T>
{
    public T FindMax(List<T> items)
    {
        T max = items[0];
        foreach (T item in items)
            if (item.CompareTo(max) > 0) max = item;
        return max;
    }
}
C#public class Repository<T>
    where T : class,    // must be reference type
              IEntity,  // must implement IEntity
              new()    // must have parameterless constructor
{
    public T CreateAndSave()
    {
        T entity = new T();          // ✅ new()
        entity.Id = Guid.NewGuid(); // ✅ IEntity
        Save(entity);
        return entity;
    }
}

Generic Repository Pattern

The most common real-world use of generics in backend development — write CRUD once, reuse for every entity.

Inheritance Chain

C#// Generic repository interface
public interface IRepository<T> where T : class
{
    T GetById(int id);
    IEnumerable<T> GetAll();
    void Add(T entity);
    void Update(T entity);
    void Delete(int id);
}

// Base implementation — write once
public class Repository<T> : IRepository<T> where T : class
{
    protected readonly DbSet<T> _dbSet;
    public T GetById(int id) => _dbSet.Find(id);
    public IEnumerable<T> GetAll() => _dbSet.ToList();
    public void Add(T entity) { _dbSet.Add(entity); }
    // ... Update, Delete
}

// Specific repos — just inherit, add entity-specific methods
public class OrderRepository : Repository<Order>
{
    public IEnumerable<Order> GetPendingOrders()
        => _dbSet.Where(o => o.Status == "Pending").ToList();
}

Covariance & Contravariance

Why are generics invariant by default? Even though Dog inherits Animal, List<Dog> can't be assigned to List<Animal> — because you could then .Add(new Cat()) to what's really a Dog list!

Interactive Variance Explorer

Select a variance type to see what's allowed:

// Covariance: out keyword — T only comes OUT (return values) interface IEnumerable<out T> { T GetEnumerator(); } IEnumerable<Animal> animals = new List<Dog>(); // ✅ Dog → Animal OK // Safe because you can only READ items out // Dog IS an Animal — returning Dog as Animal is fine
// Contravariance: in keyword — T only goes IN (parameters) interface IProcessor<in T> { void Process(T item); } IProcessor<Dog> proc = new AnimalProcessor(); // ✅ Animal → Dog OK // Safe because an AnimalProcessor can handle ANY Animal // Dog IS an Animal — so it can definitely process a Dog
// Invariance: no keyword — T goes both in AND out class List<T> { void Add(T item); T Get(); } List<Animal> animals = new List<Dog>(); // ❌ COMPILE ERROR // Dangerous: could .Add(new Cat()) into a Dog list! // Must use EXACT type match

default(T) Keyword

Returns the default value for any type — very useful in generic code:

C#// default values by type:
default(int)      // 0
default(bool)     // false
default(double)   // 0.0
default(string)   // null
default(Order)    // null

Test Yourself

Q1: What is the main advantage of generics over using object?

Using object requires boxing/unboxing for value types (performance cost) and loses type safety — errors only surface at runtime. Generics check the type at compile time and avoid boxing entirely.

Q2: What does where T : new() allow you to do?

The new() constraint guarantees T has a parameterless constructor, so new T() is valid. Without it, the compiler doesn't know if T can be instantiated.

Q3: Why can't List<Dog> be assigned to List<Animal>?

List<T> is invariant because it both produces AND consumes T. If this were allowed, you could .Add(new Cat()) — the compiler accepts Cat as Animal, but the underlying list is Dogs only. Type safety breaks.

Q4: Which keyword enables covariance?

out = covariance = T only comes OUT as return values = safe to use more derived types. in = contravariance = T only goes IN as parameters.

Interview Answer Bank

Using object requires boxing/unboxing for value types which hurts performance and loses type safety at compile time. Generics are type-safe at compile time — the type is checked when you write the code, not at runtime. They also avoid boxing completely because the actual type is used directly.
Generic constraints restrict what types T can be. Without constraints T is completely unknown — you cannot call any methods on it. Constraints guarantee what capabilities T has. For example where T : IComparable<T> lets you call CompareTo(), and where T : new() lets you create instances.
Covariance uses out — a more derived type where base is expected. Safe because T only flows out as a return. IEnumerable<T> is covariant so List<Dog> → IEnumerable<Animal>. Contravariance uses in — a less derived type where derived is expected. Safe because T only flows in as a parameter.
The most common real-world use is the generic repository pattern — a single base repository handles all CRUD for any entity type. Eliminates code duplication. Specific repos just inherit and add entity-specific methods.
