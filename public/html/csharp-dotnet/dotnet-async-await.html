The Problem — Why We Need Async

Synchronous code blocks the thread while waiting. In web/UI apps this means the UI freezes or the server can't handle other requests.
❌ Synchronous — blocks thread
public string GetDataSync()
{
    Thread.Sleep(3000); // thread blocked — doing nothing
    return "Data";
}
✅ Asynchronous — releases thread
public async Task<string> GetDataAsync()
{
    await Task.Delay(3000); // thread released — free
    return "Data";
}

Thread vs Task

Thread — Old Way (~1MB each)
Thread thread = new Thread(() =>
{
    Thread.Sleep(3000);
    Console.WriteLine("Done");
});
thread.Start();
Task — Modern Way (Thread Pool)
Task<string> task = Task.Run(() =>
{
    Thread.Sleep(2000);
    return "Data from DB";
});

string result = await task;
Console.WriteLine(result);

async and await — How They Work

async marks the method. await is the real magic — releases the thread while waiting.
How await works — step by step
public async Task<string> GetDataAsync()
{
    Console.WriteLine("1 - Before await, Thread: "
        + Thread.CurrentThread.ManagedThreadId);

    string data = await FetchFromDBAsync(); // ← thread released here

    Console.WriteLine("3 - After await, Thread: "
        + Thread.CurrentThread.ManagedThreadId);
    return data;
}

// Step 1 → method runs until await
// Step 2 → thread released back to thread pool
// Step 3 → FetchFromDBAsync() runs
// Step 4 → when complete, method resumes from where it left off
4

Return Types

Three return types
// Task — no return value
public async Task SaveDataAsync()
{
    await File.WriteAllTextAsync("file.txt", "data");
}

// Task<T> — returns a value
public async Task<string> GetDataAsync()
{
    return await File.ReadAllTextAsync("file.txt");
}

// void — ONLY for event handlers — avoid everywhere else
public async void Button_Click(object sender, EventArgs e)
{
    await DoSomethingAsync();
}
⚠️Never use async void except for event handlers. Exceptions from async void cannot be caught by the caller and will crash the application silently.

Common Mistakes

Mistake 1 — Deadlock with .Result or .Wait()

In environments with SynchronizationContext (WinForms, WPF, classic ASP.NET) — calling .Result blocks the thread and causes a deadlock because the async method tries to resume on the same blocked thread.

❌ Deadlock
public string GetData()
{
    return GetDataAsync().Result;
    // blocks thread — deadlock! 💀
}
✅ Fix — await all the way
public async Task<string> GetData()
{
    return await GetDataAsync();
    // no deadlock
}

Mistake 2 — Forgetting await

❌ Fire and forget
public async Task SaveAsync()
{
    SaveToDBAsync(); // no await!
    Console.WriteLine("Saved");
    // prints even if save failed
}
✅ Correct
public async Task SaveAsync()
{
    await SaveToDBAsync();
    Console.WriteLine("Saved");
    // prints only after save
}

Mistake 3 — Sequential Instead of Parallel

❌ Sequential — slow
foreach (int id in ids)
{
    await GetUserAsync(id);
    // one by one — slow
}
✅ Parallel — fast 🚀
var tasks = ids
    .Select(id => GetUserAsync(id))
    .ToList();
await Task.WhenAll(tasks);
// all parallel — fast 🚀
6

Task.WhenAll vs Task.WhenAny

WhenAll vs WhenAny
// Task.WhenAll — wait for ALL to complete
Task<string> task1 = GetFromDB();
Task<string> task2 = GetFromAPI();
Task<string> task3 = GetFromCache();

string[] results = await Task.WhenAll(task1, task2, task3);
// All three run in parallel — total time = slowest one

// Task.WhenAny — continue when FIRST completes
Task<string> winner = await Task.WhenAny(task1, task2);
string result = await winner; // fastest server wins
7

ConfigureAwait

ConfigureAwait usage
// Default — resumes on original thread (needed for UI updates)
string data = await GetDataAsync();

// ConfigureAwait(false) — resumes on any thread pool thread
// Use in library/service code — slightly more performant
string data = await GetDataAsync().ConfigureAwait(false);
📌Rule: UI apps → don't use ConfigureAwait(false) when updating UI. Library code → always use ConfigureAwait(false).

Interview Questions

▶What is the difference between async/await and Thread?
A Thread creates an OS-level thread consuming ~1MB of memory and requires manual management. async/await uses the Thread Pool through the Task abstraction — threads are reused instead of created fresh each time. The key difference is that await releases the thread while waiting for an operation to complete, whereas Thread.Sleep blocks the thread entirely. This makes async/await far more efficient for I/O-bound operations.
▶Why does .Result cause a deadlock?
Deadlock occurs in environments with a SynchronizationContext like WinForms, WPF, or classic ASP.NET. When await captures the context it promises to resume on that specific thread. But if that thread is blocked by .Result waiting for the async method to finish, neither can proceed — the async method needs the thread to resume and the thread needs the async method to finish. Console apps and ASP.NET Core have no SynchronizationContext so they can resume on any thread pool thread — no deadlock.
▶What is the difference between Task.WhenAll and Task.WhenAny?
Task.WhenAll waits for all tasks to complete and is used for parallel execution where you need all results. The total time equals the slowest task instead of the sum of all tasks. Task.WhenAny completes as soon as the first task finishes — useful for redundancy scenarios like querying multiple servers and using whichever responds first.
