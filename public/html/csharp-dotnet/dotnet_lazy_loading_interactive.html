Lazy Loading

Don't load data until you actually need it

What Is It Advantages Disadvantages N+1 Simulator Implementation Lazy<T> When to Use Quiz
1

What is Lazy Loading?

Lazy Loading is a design pattern where you delay loading of an object until the exact moment you need it. Instead of loading everything upfront, child data loads on demand.

Restaurant analogy: You don't cook every dish when a customer walks in. You wait until they actually order. That's lazy loading — prepare only what's asked for. 
C#Customer o = new Customer();        // Orders NOT loaded yet
Console.WriteLine(o.CustomerName);  // Orders still NOT loaded

foreach (Order o1 in o.Orders)      // Orders load HERE — on demand
{
    Console.WriteLine(o1.OrderNumber);
}

Object Created

Customer loaded — Orders are NOT loaded

Access Name

CustomerName returned — Orders still NOT loaded

Access Orders

NOW the database query fires and Orders load

Access Orders Again

Returns cached result — no second query

2

Advantages

Only essential data loads at launch. Heavy or optional data waits until needed.

C#public Customer()
{
    _CustomerName = "Ali";  // loads immediately — lightweight
    _Orders = new Lazy<List<Order>>(() => LoadOrders()); // NOT fired yet
}

E-commerce app: loads customer profile on login. 500 order records only load when they click "My Orders".

Objects that are never accessed are never loaded into memory. User opens an article, reads it, and leaves — only that one article was ever in memory, not all articles and all comments.

C#// Without Lazy — DB query fires EVERY time
_Orders = LoadOrders(); // always hits DB even if never used

// With Lazy — query only fires when accessed
_Orders = new Lazy<List<Order>>(() => LoadOrders());

100 Customer objects but only 10 accessed Orders = 90 saved DB calls.

Less memory + fewer queries = more concurrent users on the same resources.

3

Disadvantages

C#// Without Lazy — simple
public List<Order> Orders { get; set; }

// With Lazy — extra null check everywhere
public List<Order> Orders
{
    get
    {
        if (_Orders == null)
            _Orders = LoadOrders();
        return _Orders;
    }
}

First time a lazy property is accessed there's an extra check: "Is this loaded? No? Load it now." Tiny delay — negligible in most cases, matters in performance-critical paths.

C#var customers = db.Customers.ToList(); // 1 query — loads 100 customers

foreach (var customer in customers)
{
    // Each access fires a SEPARATE DB query!
    Console.WriteLine(customer.Orders.Count);
}

// Result: 1 + 100 = 101 total queries 😱
⚠️ The N+1 problem is one of the most common performance killers in .NET apps. Always watch for lazy loading inside loops.

Data loads at unpredictable times. A NullReferenceException or connection error may appear deep inside a loop, far from where the object was created.

Manual Implementation

Two changes: remove loading from constructor, add null check in getter.

C#public class Customer
{
    private List<Order> _Orders = null;

    public Customer()
    {
        _CustomerName = "Ali";
        _Orders = LoadOrders(); // ❌ Always loads — even if never needed
    }
}
C#public class Customer
{
    private List<Order> _Orders = null;

    public Customer()
    {
        _CustomerName = "Ali";
        // ✅ Nothing loaded here
    }

    public List<Order> Orders
    {
        get
        {
            if (_Orders == null)        // First access?
                _Orders = LoadOrders(); // Load and cache
            return _Orders;            // Cached every time after
        }
    }
}
6

Lazy<T> — Built-in Solution

.NET provides Lazy<T> which handles the null-check automatically and adds thread safety for free.

C#public class Customer
{
    private Lazy<List<Order>> _Orders = null;
    private string _CustomerName;

    public string CustomerName => _CustomerName;

    public List<Order> Orders => _Orders.Value; // clean — no null check

    public Customer()
    {
        _CustomerName = "Ali";
        _Orders = new Lazy<List<Order>>(() => LoadOrders()); // attached, not fired
    }
}

⚠️ Lazy<T> is thread safe by default. If two threads access the property simultaneously, it will NOT load twice. The manual null-check pattern does NOT protect you from this.

8

Test Yourself

Q1: What is the N+1 problem?

The N+1 problem: 1 query loads the list, then each of the N items triggers its own lazy-load query = N+1 total. Fix: use eager loading (.Include() in EF) when you know you'll need child data.

Q2: What advantage does Lazy<T> have over manual null-check?

Lazy<T> handles the null check automatically AND is thread safe — if two threads hit .Value simultaneously, the factory only runs once. Manual null-check can cause double-loading in multithreaded code.

Q3: When should you prefer Eager Loading over Lazy Loading?

When you know you'll iterate through items AND need their children (e.g. displaying a table of customers with order counts), eager loading avoids the N+1 problem by fetching everything in one query.

Q4: What does Lazy Loading delay?

Lazy loading delays loading of child/related data. The parent object is created normally — but its related objects (like Orders on a Customer) aren't loaded until someone actually accesses that property.
