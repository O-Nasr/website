⚠️ Variables always live on the stack. What they point to lives on the heap. The stack holds the reference (pointer), the heap holds the actual object data.
C#void MyMethod()
{
    int x = 5;              // x → stack (actual value)
    string name = "Omar";   // reference → stack, "Omar" → heap
    Person p = new Person(); // reference → stack, object → heap
} // method returns → stack frame popped instantly
  // heap objects stay until GC collects them

GC Generations

The GC uses a generational system based on one key observation: most objects die young.

Managed Heap Layout

LOH
Gen 0 — Frequent, cheap
Gen 1 — Occasional

All new objects start here. GC collects Gen 0 most frequently — fast and cheap. Most objects never leave Gen 0.

C#for (int i = 0; i < 1000; i++)
{
    var temp = new TempObject(); // Gen 0
    DoWork(temp);
} // temp becomes garbage → collected in Gen 0 ✅ cheap and fast

Objects that survived one Gen 0 collection. Acts as buffer between Gen 0 and Gen 2. Objects with finalizers get promoted here — they survive an extra GC cycle.

Survived Gen 0 AND Gen 1. Collected least frequently — expensive full GC that may pause the app. Singletons, caches, static data end up here.

public static readonly AppCache _instance = new AppCache(); // Gen 2
public static readonly string ConnStr = "...";              // Gen 2

Large Object Heap (LOH)

Objects larger than 85,000 bytes (~83KB) go directly to the LOH — skipping Gen 0 entirely.

Normal Heap (SOH)Large Object Heap (LOH)
GC compaction✅ Moves objects together❌ Objects stay in place
Fragmentation✅ None — compacted❌ Can fragment over time
When collectedGen 0, 1, or 2Only during Gen 2 (full GC)
App pauseShortLonger — full GC required
⚠️ LOH Fragmentation: Free memory exists but not as a contiguous block. 130KB free total but no single 120KB block → allocation fails! The silent memory killer.
C#// ❌ LOH every iteration → fragmentation
for (int i = 0; i < 1000; i++)
{
    var buffer = new byte[100000]; // LOH every time 😱
    ProcessData(buffer);
}
C#// ✅ ArrayPool — borrow and return — zero GC pressure
for (int i = 0; i < 1000; i++)
{
    byte[] buffer = ArrayPool<byte>.Shared.Rent(100000);
    try { ProcessData(buffer); }
    finally { ArrayPool<byte>.Shared.Return(buffer); } // no GC ✅
}

Boxing & Unboxing

Boxing converts a value type to a reference type — creating a new heap object. One is negligible; repeated boxing in loops causes massive GC pressure.

C#int x = 42;            // x on stack — no heap
object boxed = x;      // BOXING: allocate heap object, copy 42 into it

// ❌ 1 million heap allocations
for (int i = 0; i < 1000000; i++)
    object boxed = i;  // new heap object every iteration 😱

// ✅ Use generics to avoid boxing
List<int> typed = new List<int>();  // no boxing ✅
ArrayList untyped = new ArrayList();
untyped.Add(42);                       // boxing! int → object 😱

WeakReference — Let GC Decide

A strong reference prevents collection. A WeakReference tells GC: keep it if memory allows, collect when needed.

C#var cache = new Dictionary<string, object>();
cache["key"] = data;  // strong reference
data = null;            // you think data is gone
                         // BUT cache["key"] still references it → GC can't collect 😱
C#var weakRef = new WeakReference<LargeObject>(data);
data = null; // strong ref gone — only weak remains
             // GC CAN collect when memory needed ✅

// Always check before using!
if (weakRef.TryGetTarget(out LargeObject obj))
    obj.DoWork();  // still alive ✅
else
    obj = new LargeObject(); // collected — recreate

Common Memory Leak Patterns

🔥 Event Handlers

Publisher holds ref to subscriber. Even if subscriber is "done" — GC can't collect it. Fix: always -= handler

🔥 Static Collections

Static list grows every request, never cleaned. After 1M requests = 1M strings in memory → crash. Fix: limit size or use MemoryCache

🔥 Undisposed Objects

SqlConnection opened but never disposed — connection stays open, pool exhausts. Fix: always using(...)

C#// ❌ Subscriber held alive by publisher
publisher.OnEvent += HandleEvent;
// Publisher now holds reference → GC cannot collect Subscriber 😱

// ✅ Always unsubscribe when done
publisher.OnEvent -= HandleEvent;

Test Yourself

Q1: Where does the reference to a heap object live?

Variables (including references/pointers) always live on the stack. What they point to — the actual object data — lives on the heap.

Q2: Why does the GC use generations?

The generational hypothesis: most objects die young. By collecting the small Gen 0 area frequently and cheaply, the GC avoids expensive full collections most of the time.

Q3: What happens to objects > 85KB?

Objects > 85KB skip Gen 0 entirely and land in the LOH. The LOH is never compacted by default, which causes fragmentation. It's only collected during expensive Gen 2 full GC. Use ArrayPool to avoid LOH pressure.

Q4: Why do static collections cause memory leaks?

GC traces from roots through reference chains. Static fields are always reachable (they live for the app's lifetime), so everything they hold a reference to also stays alive forever — the GC can never collect those objects.

Q5: What does a WeakReference allow?

A WeakReference doesn't prevent GC from collecting. Unlike a strong reference that keeps an object alive, a WeakReference says "collect this if you need the memory." You must always call TryGetTarget to check if it's still alive before using it.

Interview Answer Bank

The stack stores value types and references — cleaned automatically and instantly when a method returns. The heap stores reference type objects — managed by the GC nondeterministically. Variables always live on the stack; what they point to lives on the heap. The stack is fast and automatic; the heap is flexible but requires GC overhead.
GC uses three generations: Gen 0 holds new short-lived objects (collected frequently and cheaply), Gen 1 holds survivors of Gen 0, Gen 2 holds long-lived objects (collected rarely in expensive full GC that may pause the app). The generational approach improves performance by focusing most collection on the small Gen 0 area where most garbage is found.
Objects > 85KB go to the LOH instead of Gen 0. The LOH is not compacted by default — objects stay in place after collection, causing fragmentation where free memory exists but not as a contiguous block. Only collected during Gen 2 full GC. Fix: use ArrayPool to rent and return large buffers.
GC collects an object only when zero references point to it — directly or through any chain. If anything anywhere holds a reference, the object stays alive. This is why static collections leak — the static field lives forever and everything it references lives forever too.
A WeakReference holds a reference without preventing GC collection. Unlike strong references, GC can collect the object when memory is needed. Always call TryGetTarget before using. Useful for caching — cache objects but let GC reclaim them under memory pressure.
