Func vs Expression — The Fundamental Difference

Both look identical as lambdas — but internally they are completely different.
Same lambda, different types
// Func — compiled code — executable
Func<int, bool> func = x => x > 5;

// Expression — data structure — inspectable and translatable
Expression<Func<int, bool>> expr = x => x > 5;

Comparison

Func<T, bool>Expression<Func<T, bool>>
What it isCompiled executable codeCode stored as a data tree
Can execute?✅ Directly✅ After .Compile()
Can inspect?❌ Binary code✅ Full tree
Translate to SQL?❌ No✅ EF does this
Used withIEnumerable — in memoryIQueryable — database
🧠Analogy: Func = a cooked meal (ready to eat, can't see the recipe). Expression = the written recipe (you can read it, modify it, translate it to any language).
2

What is an Expression Tree?

A tree data structure that represents code as data — every piece is an inspectable node.
Inspecting the tree
Expression<Func<int, bool>> expr = x => x > 5;

Console.WriteLine(expr.Body);     // (x > 5)
Console.WriteLine(expr.NodeType); // Lambda

BinaryExpression body = (BinaryExpression)expr.Body;
Console.WriteLine(body.NodeType); // GreaterThan
Console.WriteLine(body.Left);     // x
Console.WriteLine(body.Right);    // 5
3

Why This Matters — EF SQL Translation

EF walks the expression tree and translates each node to SQL.
EF reads the tree → builds SQL
var employees = dbContext.Employees
    .Where(e => e.Salary > 5000)
    .ToList();

// EF receives expression tree:
//       GreaterThan (>)
//      /               \
// Property(Salary)   Constant(5000)
//
// Walks tree → translates each node:
//   Property(Salary) → "Salary"
//   GreaterThan      → ">"
//   Constant(5000)   → "5000"
//
// Builds: WHERE Salary > 5000
// Only matching rows returned ✅
⚠️IQueryable vs IEnumerable: IQueryable.Where takes Expression → translates to SQL (efficient). IEnumerable.Where takes Func → runs in C# memory after ALL data is loaded (slow for large tables).
4

Func vs Expression — Side by Side

❌ Func — filters in memory
Func<Employee, bool> func =
    e => e.Salary > 5000;
var result = employees.Where(func);
// SQL: SELECT * FROM Employees
// ← ALL 1M rows transferred 😱
// Then filtered in C# memory
✅ Expression — filters in DB
Expression<Func<Employee, bool>> expr =
    e => e.Salary > 5000;
var result = dbContext.Employees.Where(expr);
// SQL: SELECT * FROM Employees
//      WHERE Salary > 5000
// ← only 10 rows transferred ✅

Building Expressions Dynamically

Build queries at runtime when filter conditions aren't known at compile time.
Dynamic filter — runtime field name
public IQueryable<Employee> Filter(string fieldName, int value)
{
    var param      = Expression.Parameter(typeof(Employee), "e");
    var property   = Expression.Property(param, fieldName); // dynamic!
    var constant   = Expression.Constant(value);
    var comparison = Expression.GreaterThan(property, constant);
    var lambda     = Expression.Lambda<Func<Employee, bool>>(comparison, param);

    return dbContext.Employees.Where(lambda); // → SQL!
}

Filter("Salary", 5000); // WHERE Salary > 5000
Filter("Age", 30);      // WHERE Age > 30
// Impossible with hardcoded LINQ ✅

Generic Repository Pattern

Repository with Expression — all SQL
public interface IRepository<T> where T : class
{
    IQueryable<T> Find(Expression<Func<T, bool>> predicate);
}

public class Repository<T> : IRepository<T> where T : class
{
    private readonly DbSet<T> _dbSet;
    public Repository(AppDbContext ctx) { _dbSet = ctx.Set<T>(); }
    public IQueryable<T> Find(Expression<Func<T, bool>> predicate)
        => _dbSet.Where(predicate); // → SQL ✅
}

// Usage — all translated to SQL
var highEarners  = repo.Find(e => e.Salary > 5000);
var seniors      = repo.Find(e => e.Age > 40 && e.Department == "IT");
var recentOrders = repo.Find(o => o.CreatedAt > DateTime.Today.AddDays(-7));

Interview Questions

▶What is the difference between Func and Expression?
Func is compiled to executable code — callable but not inspectable. Expression stores the lambda as a data tree that can be inspected, translated to SQL, and built dynamically. Expression is what allows EF to translate LINQ to SQL instead of executing in C# memory.
▶Why does IQueryable use Expression instead of Func?
IQueryable needs to translate filters to SQL. Only possible with Expression trees because the code is data that can be inspected. If it used Func, all data would be loaded into memory first — exactly what IEnumerable does.
▶When would you build an Expression tree manually?
When filter conditions aren't known at compile time — e.g. a search where the user selects which fields to filter. You build the expression tree dynamically at runtime and pass it to IQueryable.Where() — still translated to SQL automatically.
▶IQueryable vs IEnumerable — what's the real difference?
IEnumerable.Where takes Func — predicate runs in C# memory after ALL data is loaded. IQueryable.Where takes Expression — predicate is translated to SQL WHERE clause, filtering happens in the database. Dramatically more efficient for large datasets.
