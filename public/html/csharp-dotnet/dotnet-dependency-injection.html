The Problem — Tight Coupling

When a class creates its own dependencies, changing or testing becomes impossible.
❌ Tight coupling
public class OrderService
{
    private EmailService _email = new EmailService();
    public void PlaceOrder(string num)
    {
        _email.SendEmail(num);
    }
}
// Can't swap, can't mock, can't test
✅ Loose coupling (DI)
public class OrderService
{
    private readonly INotificationService _svc;
    public OrderService(INotificationService svc)
    { _svc = svc; }
    public void PlaceOrder(string num)
    { _svc.Send(num); }
}
// Swap Email/SMS/Mock — zero changes

Three Types of DI

1. Constructor Injection — Most Common
public class OrderService
{
    private readonly INotificationService _notification;
    private readonly ILogger _logger;

    // All required dependencies declared upfront
    public OrderService(INotificationService notification, ILogger logger)
    {
        _notification = notification;
        _logger = logger;
    }
}
2. Property Injection — Optional Dependencies
public class OrderService
{
    private readonly INotificationService _notification;
    public ILogger Logger { get; set; } // optional — can be null

    public OrderService(INotificationService notification)
    { _notification = notification; }

    public void PlaceOrder(string num)
    {
        Logger?.Log("Placing: " + num); // null-safe
        _notification.Send(num);
    }
}
3. Method Injection — Per-Method
public class OrderService
{
    public void PlaceOrder(string num, INotificationService svc)
    {
        svc.Send(num); // per-call, not stored
    }
}

Captive Dependency Problem

⚠️A Singleton holding a Scoped dependency traps it forever — the Scoped service never gets disposed per request. ASP.NET Core throws InvalidOperationException at runtime.
❌ Singleton holding Scoped — CAPTIVE!
public class MySingletonService
{
    private readonly IOrderService _orderService; // Scoped!
    public MySingletonService(IOrderService orderService)
    { _orderService = orderService; } // captured forever 😱
}
// ASP.NET Core throws:
// InvalidOperationException: Cannot consume scoped service
// 'IOrderService' from singleton 'MySingletonService'

Lifetime Rules

Singleton → can use → Singleton only. Scoped → can use → Singleton + Scoped. Transient → can use → Singleton + Scoped + Transient.

6

DI Makes Testing Easy

Unit test with mock — no real services
public class MockNotificationService : INotificationService
{
    public List<string> SentMessages = new();
    public void Send(string msg) { SentMessages.Add(msg); }
}

[Test]
public void PlaceOrder_ShouldSendNotification()
{
    var mock = new MockNotificationService();
    var service = new OrderService(mock); // inject mock

    service.PlaceOrder("ORD-001");

    Assert.AreEqual(1, mock.SentMessages.Count);
    Assert.AreEqual("ORD-001", mock.SentMessages[0]);
}

Interview Questions

▶What is Dependency Injection and why do we use it?
Dependency Injection is a design pattern where a class receives its dependencies from outside rather than creating them itself. This promotes loose coupling — the class depends on abstractions not implementations, making it easy to swap implementations and test in isolation using mocks without touching production code.
▶What are the three service lifetimes in ASP.NET Core?
Transient creates a new instance on every injection — best for lightweight stateless services. Scoped creates one instance per HTTP request shared across that request — best for DbContext and Unit of Work. Singleton creates one instance for the entire application lifetime — best for Logger, Cache, and Configuration. Singletons must be thread-safe since they are shared across all requests.
▶What is the captive dependency problem?
A captive dependency occurs when a Singleton service holds a reference to a Scoped service, preventing the Scoped service from being properly disposed after each request. The Scoped service becomes trapped in the Singleton forever. ASP.NET Core detects this and throws an InvalidOperationException at runtime.
