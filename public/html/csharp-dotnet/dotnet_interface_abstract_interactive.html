Interface vs Abstract Class

Contract for unrelated vs partial blueprint for related

Abstract Class — Partial Blueprint

Provides a partial implementation that related classes inherit from. Can have both implemented methods (shared code) and abstract methods (forced override).

abstract Shape
↓ inherits
Circle
Rectangle
Triangle
C#abstract class Shape
{
    public string Color { get; set; }   // shared property — all shapes have color

    public void DisplayColor()           // implemented — shared by all shapes
    {
        Console.WriteLine("Color: " + Color);
    }

    public abstract double GetArea();    // abstract — no body, child MUST override
}

class Circle : Shape
{
    public double Radius { get; set; }
    public override double GetArea() => Math.PI * Radius * Radius;
}

class Rectangle : Shape
{
    public double Width { get; set; }
    public double Height { get; set; }
    public override double GetArea() => Width * Height;
}
⚠️ Abstract methods have NO body — child MUST override. Regular methods in abstract class HAVE a body — child CAN override but doesn't have to.
2

Interface — Full Blueprint (Contract)

A pure contract that defines what a class must do without specifying how. Any unrelated class can implement it.

INotificationService
↓ implements
EmailNotification
SmsNotification
PushNotification
C#interface INotificationService
{
    void Send(string message);   // contract — no implementation
    bool IsAvailable();          // contract — no implementation
}

class EmailNotification : INotificationService
{
    public void Send(string message)
        => Console.WriteLine("Email: " + message);
    public bool IsAvailable() => true;
}

class SmsNotification : INotificationService
{
    public void Send(string message)
        => Console.WriteLine("SMS: " + message);
    public bool IsAvailable() => CheckSmsGateway();
}
⚠️ Since C# 8.0, interfaces CAN have default implementations. However in interviews, interfaces are still primarily described as contracts. Mention C# 8.0 as a bonus point.

Interactive Decision Guide

Answer the questions to find out which one to use for your scenario:

Do the classes share common code or state?
Do you need multiple inheritance?
→ Use Abstract Class

Your classes are related and share common code/state. Abstract class lets you provide base implementations and shared fields. Examples: Shape → Circle, Rectangle · Vehicle → Car, Truck · Repository<T> base

→ Use Interface

Your classes are unrelated but need to follow the same contract. Interface provides pure abstraction that any class can implement. Examples: IDisposable · INotificationService · IComparable

→ Consider Both!

If classes share code AND need multiple inheritance, use an abstract base class for the shared code plus interfaces for additional contracts. This is a common pattern in real-world code.

Test Yourself

Q1: How many abstract classes can a C# class inherit from?

C# only allows single class inheritance. A class can inherit from exactly one abstract class but can implement multiple interfaces.

Q2: SqlConnection and FileStream both implement IDisposable. Why is IDisposable an interface, not an abstract class?

SqlConnection and FileStream share no common code or state. They're completely unrelated classes that both need cleanup behavior. An interface is perfect for this — it defines a contract without forcing inheritance from a common base.

Q3: Can interfaces have default implementations in C#?

Since C# 8.0, interfaces can have default implementations. This was mainly added for versioning — adding new methods without breaking existing implementations. But interfaces are still primarily used as pure contracts.

Q4: Can an abstract class have fields and constructors?

Abstract classes can have everything a normal class has: fields, constructors, access modifiers, implemented methods, and properties. The only addition is the ability to declare abstract members that children must override.

Interview Answer Bank

An abstract class provides a partial implementation that related classes inherit from — it can have implemented methods, constructors, fields, and access modifiers. An interface is a pure contract that defines what a class must do without how, and any unrelated class can implement it. The key practical difference: a class can only inherit from one abstract class but can implement multiple interfaces.
Use interface when completely unrelated classes need to follow the same contract, or when you need multiple inheritance. Use abstract class when related classes share common code and state. Example: IDisposable is an interface because SqlConnection and FileStream share no common code — they just both need cleanup behavior.
Since C# 8.0, interfaces can have default implementations. However interfaces are still primarily used as pure contracts. Default interface implementations are mainly for versioning — adding new methods without breaking existing implementations.
