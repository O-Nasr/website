Managed vs Unmanaged Code

Managed (GC handles)

string, int, List<T>, Dictionary, any C# class, SqlConnection object

Unmanaged (YOU handle)

File handles, DB connections (sockets), native memory (IntPtr), COM objects, GDI handles

⚠️ The GC only frees managed memory. It does NOT free unmanaged resources. This is one of the most important concepts in .NET internals.

IDisposable & the using Statement

GC is nondeterministic — runs when it decides. For scarce resources we need deterministic cleanup → that's Dispose().

C#using (SqlConnection con = new SqlConnection(connStr))
{
    // do your work here
} // Dispose() called automatically — even if exception thrown ✅
C#SqlConnection con = new SqlConnection(connStr);
try
{
    // do work
}
finally
{
    con.Dispose(); // Always dispose in finally
}
GC manages memory. IDisposable manages resources. Two completely different things.
4

Finalizer (~ClassName) — Safety Net

A finalizer is called by GC when collecting an object. It's a safety net if the developer forgets Dispose(). But it's expensive.

C#~MyClass()
{
    // Called by GC automatically
    // You don't control WHEN this runs
}
Why finalizers are expensive: Object moves to Finalization Queue → survives one full GC cycle (promoted to Gen 1/2) → finalizer thread processes it → only then eligible for collection. Objects with finalizers live longer in memory.

5

Full Dispose + Finalizer Pattern

Use ONLY when your class directly owns unmanaged resources (like IntPtr):

C#class MyUnmanagedClass : IDisposable
{
    private IntPtr _nativeMemory;
    private bool _disposed = false;

    public MyUnmanagedClass()
    {
        _nativeMemory = Marshal.AllocHGlobal(100);
    }

    ~MyUnmanagedClass() { Dispose(false); } // safety net

    protected virtual void Dispose(bool disposing)
    {
        if (!_disposed)
        {
            if (disposing) { // free managed resources }
            if (_nativeMemory != IntPtr.Zero)
            {
                Marshal.FreeHGlobal(_nativeMemory); // free unmanaged
                _nativeMemory = IntPtr.Zero;
            }
            _disposed = true;
        }
    }

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this); // skip finalizer — already cleaned ✅
    }
}

SafeHandle — Modern Approach

ApproachNeed Your Own Finalizer?Risk Level
Using IntPtr directlyYESHigh — easy to leak
Using SafeHandleNOLow — SafeHandle handles it
Using FileStream / SqlConnectionNONone — already managed
If your class uses SafeHandle (or classes that use it internally like FileStream), you do NOT need your own finalizer. SafeHandle implements a critical finalizer internally. Just implement IDisposable and call Dispose() on the SafeHandle.
Quick Rules:
• GC handles managed memory — you handle unmanaged resources
• Always use using(...) for IDisposable objects
• Add a finalizer ONLY if your class owns raw IntPtr
• Call GC.SuppressFinalize(this) inside Dispose()
• Setting a reference to null does NOT close the resource
8

Test Yourself

Q1: Why doesn't the GC clean unmanaged resources?

The GC only tracks objects inside the managed heap under CLR control. File handles, sockets, native memory are allocated by the OS — the GC has no knowledge of them. We must use IDisposable and Dispose() to release them.

Q2: Why implement BOTH Dispose() and a Finalizer?

Dispose() gives deterministic, immediate cleanup. The finalizer is a safety net for when Dispose() isn't called. Inside Dispose(), we call GC.SuppressFinalize(this) to skip the expensive finalization when it's not needed.

Q3: What happens if you set SqlConnection con = null?

Setting to null only removes the reference — it does NOT close the connection. The GC is nondeterministic and may run seconds or minutes later. The underlying OS connection is only released when Dispose() is called explicitly.

Q4: When should you add a finalizer to a class?

Only when your class directly owns native memory (IntPtr). If you use SafeHandle, FileStream, SqlConnection, or other IDisposable wrappers — they already handle their own cleanup. Adding unnecessary finalizers makes objects live longer in memory.
