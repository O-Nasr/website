Why String is Immutable

Every time you modify or concatenate a string, a brand new object is created in memory. The old one becomes garbage.

C#string result = "";
result = result + "A"; // NEW string "A" at loc #2 — loc #1 abandoned
result = result + "B"; // NEW string "AB" at loc #3 — loc #2 abandoned
result = result + "C"; // NEW string "ABC" at loc #4 — loc #3 abandoned
In a loop of 10,000 iterations: 10,000 new allocations + 10,000 abandoned strings for GC. Massive memory pressure + slow performance.
2

StringBuilder — The Solution

StringBuilder maintains a single mutable buffer and appends directly — no new allocations on every change.

C#StringBuilder sb = new StringBuilder();

for (int i = 0; i < 10000; i++)
    sb.Append(i.ToString()); // modifies SAME buffer — no new allocation

string result = sb.ToString(); // ONE string created at the end

Common StringBuilder Methods

C#StringBuilder sb = new StringBuilder();

sb.Append("Hello");           // add text at the end
sb.AppendLine("Hello");       // add text + new line
sb.Insert(0, "Start: ");      // insert at position
sb.Replace("Hello", "Hi");    // find and replace
sb.Remove(0, 5);              // remove characters
sb.Clear();                   // clear everything

string result = sb.ToString(); // convert when done
⚠️ Rule of thumb: If you are concatenating strings more than 3-4 times — especially inside a loop — always use StringBuilder.

String Interning — Bonus Point

.NET has a String Intern Pool. Identical string literals share the same memory location.

C#string a = "Hello";
string b = "Hello";
Console.WriteLine(object.ReferenceEquals(a, b)); // true — same location!

// Dynamically built strings are NOT interned
string c = new string(new char[]{'H','e','l','l','o'});
Console.WriteLine(object.ReferenceEquals(a, c)); // false — different

Test Yourself

Q1: Why is string concatenation in a loop bad?

Strings are immutable — every + creates a brand new object and abandons the old one. In a 10k loop, that's 10,000 allocations and 10,000 objects for GC to collect.

Q2: How many string objects does StringBuilder create for 10,000 appends?

StringBuilder works with a mutable internal buffer. All 10,000 appends modify that same buffer. Only one string is created at the end when you call .ToString().

Q3: Is string thread-safe? Why?

Immutability makes strings inherently thread-safe — since no thread can modify a string object, there's no race condition possible. StringBuilder is NOT thread-safe because its buffer is mutable.

Q4: Do two string literals "Hello" share the same memory?

.NET's String Intern Pool ensures identical string literals point to the same memory location. However, dynamically built strings (e.g., new string(...)) are NOT interned automatically.

StringBuilder maintains a single mutable buffer and appends directly without creating new objects. Use it whenever concatenating multiple times — especially in loops. Can be up to 100x faster because it creates one allocation instead of thousands.
Once a string is created it cannot be changed. Any operation that appears to modify it actually creates a brand new object. This makes string thread-safe by design but also makes repeated modification expensive.
