Catching — Most Specific First

try/catch/finally — correct order
public void ProcessOrder(string orderId)
{
    try
    {
        var order = GetOrder(orderId);
        SaveOrder(order);
    }
    catch (OrderNotFoundException ex)  // most specific first
    {
        Console.WriteLine("Order not found: " + ex.OrderId);
    }
    catch (DatabaseException ex)       // more general
    {
        Console.WriteLine("DB error: " + ex.Message);
    }
    catch (Exception ex)               // catch-all — last resort
    {
        Console.WriteLine("Unexpected: " + ex.Message);
        throw; // rethrow after logging
    }
    finally
    {
        // ALWAYS runs — exception, return, or normal flow
        Console.WriteLine("Processing complete");
    }
}

throw vs throw ex

⚠️throw preserves the original stack trace. throw ex RESETS it — you lose where the error originally happened. Always use throw.
❌ throw ex — loses stack trace
catch (Exception ex)
{
    throw ex; // stack trace starts HERE
    // original location LOST 😱
}
✅ throw — preserves stack trace
catch (Exception ex)
{
    Logger.Log(ex);
    throw; // original stack trace preserved ✅
}

Finally Block

ALWAYS runs — even with return, exceptions, or break inside try.
finally runs even with return
public string Test()
{
    try
    {
        return "from try"; // return is here
    }
    finally
    {
        Console.WriteLine("finally runs!"); // STILL runs!
    }
}
// Output: "finally runs!" then returns "from try"
// Only skipped on: Environment.FailFast() or power failure

Custom Exceptions

Custom exception with context properties
public class OrderNotFoundException : Exception
{
    public string OrderId { get; }
    public DateTime AttemptedAt { get; }

    public OrderNotFoundException(string orderId)
        : base($"Order '{orderId}' was not found")
    {
        OrderId = orderId;
        AttemptedAt = DateTime.UtcNow;
    }

    // Always include InnerException constructor
    public OrderNotFoundException(string orderId, Exception inner)
        : base($"Order '{orderId}' was not found", inner)
    {
        OrderId = orderId;
        AttemptedAt = DateTime.UtcNow;
    }
}

Global Exception Handler — ASP.NET Core

Middleware — catches all unhandled exceptions
public class GlobalExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger _logger;

    public GlobalExceptionMiddleware(RequestDelegate next, ILogger logger)
    { _next = next; _logger = logger; }

    public async Task Invoke(HttpContext context)
    {
        try { await _next(context); }
        catch (OrderNotFoundException ex)
        {
            _logger.LogWarning(ex, "Order not found");
            await HandleException(context, 404, ex.Message);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Unexpected error");
            await HandleException(context, 500, "An unexpected error occurred");
        }
    }
}

// Register in Program.cs
app.UseMiddleware<GlobalExceptionMiddleware>();

Interview Questions

▶What is the difference between throw and throw ex?
throw preserves the original stack trace showing exactly where the exception first occurred. throw ex resets the stack trace making it appear the exception originated at the catch block. Always use throw when rethrowing.
▶When does the finally block run?
The finally block always runs regardless of whether an exception was thrown, a return statement was hit, or code completed normally. The only exceptions: Environment.FailFast() or machine power failure. Used for cleanup — disposing connections, closing files.
▶When should you catch vs let it bubble up?
Catch when you can recover, need to translate exception types, need to log at this layer, or return specific HTTP responses. Let bubble up when you can't do anything useful, a higher layer handles it better, or the global handler catches it. Never catch just to rethrow without adding value.
