IComparable vs IComparer

Internal default sort vs external flexible sort strategies

Summary IComparable IComparer Return Values Live Sort Demo LINQ Quiz
1

Quick Summary

INTERNAL IComparable

The object knows how to sort itself. Sorting logic lives inside the class via CompareTo().

One fixed default sort — like the object judging itself.

vs

EXTERNAL IComparer

A separate class that knows how to sort from outside via Compare().

Multiple flexible strategies — like hiring an external judge.

⚠️ IComparable = one fixed default sort. IComparer = multiple flexible sort strategies. Think of IComparable as the object judging itself, and IComparer as hiring an external judge.
2

IComparable — Internal Default Sorting

The class implements sorting logic inside itself using CompareTo(). One default sort criteria.

C#class Student : IComparable<Student>
{
    public string Name { get; set; }
    public int Age { get; set; }

    // Sorting logic lives INSIDE the class
    public int CompareTo(Student other)
    {
        return this.Age.CompareTo(other.Age); // fixed default — sort by Age
    }
}

students.Sort(); // uses CompareTo() internally — sorts by Age
The Problem: What if sometimes you want to sort by Name and other times by Age? With IComparable you are stuck — the logic is hardcoded inside the class. students.Sort() ALWAYS sorts by Age.
3

IComparer — External Flexible Sorting

Create separate classes for each sort criteria. Logic lives outside the Student class. Zero modifications to original class.

C#// Sort by Name — external class
class SortByName : IComparer<Student>
{
    public int Compare(Student x, Student y)
        => string.Compare(x.Name, y.Name);
}

// Sort by Age — external class
class SortByAge : IComparer<Student>
{
    public int Compare(Student x, Student y)
        => x.Age.CompareTo(y.Age);
}

// Sort any way you want — zero changes to Student class
students.Sort(new SortByName()); // alphabetical by name
students.Sort(new SortByAge());  // ascending by age
⚠️ Open/Closed Principle — IComparer keeps the class closed for modification but open for extension. Mentioning this in interviews instantly shows mid-level thinking.

Modern Alternative — LINQ OrderBy

C#// Instead of IComparer classes — use LINQ
var byName = students.OrderBy(s => s.Name).ToList();
var byAge  = students.OrderBy(s => s.Age).ToList();
var byAgeThenName = students
    .OrderBy(s => s.Age)
    .ThenBy(s => s.Name)
    .ToList();

Test Yourself

Q1: Where does the sorting logic live in IComparable?

IComparable means the class implements CompareTo() internally — the object knows how to sort itself. This is the fixed default sort.

Q2: Which SOLID principle does IComparer follow?

IComparer keeps the original class closed for modification but open for extension — you can add new sort strategies without changing the class.

Q3: What does CompareTo() return when the current object is LESS than the other?

Negative = less than, Zero = equal, Positive = greater than. This convention is shared by both CompareTo() and Compare().

Interview Answer Bank

IComparable defines a default sort implementation inside the class using CompareTo(). Limited to one fixed criteria — sorting logic is coupled to the class. IComparer separates sorting into external classes using Compare(), allowing multiple strategies without modifying the original class. IComparer follows the Open/Closed principle.
Use IComparer when you need multiple sort strategies — e.g. employees by name in one context and salary in another. Use IComparable when there's one obvious natural default sort that will never change — e.g. dates chronologically. In modern code, LINQ OrderBy is often preferred for readability.
Negative = current object is less than the other. Zero = equal. Positive = greater than. This convention is shared by both IComparable.CompareTo() and IComparer.Compare(). The Sort method uses these return values to determine ordering.
