01

The Problem Without a Gateway

Without a gateway, every client talks directly to every service. The mobile app needs to know 8 service URLs. Auth is implemented in every service. CORS is configured everywhere.

Without API Gateway
Every client knows every service URL. Auth logic duplicated in every service. CORS configured everywhere. Move a service = update all clients. No aggregation across services.
Tight coupling
With API Gateway
One front door. Clients talk to one URL. Gateway routes, authenticates, rate-limits, and aggregates. Services stay completely hidden behind it.
Single entry point

Responsibility 1 — Routing with YARP

The gateway maps URL patterns to services. A request for /api/orders/123 goes to Orders service. The client never knows the internal address. In .NET, YARP (Yet Another Reverse Proxy) is Microsoft's production-grade gateway library.

bash — Install
dotnet add package Yarp.ReverseProxy
JSON — appsettings.json — declarative routing
{
  "ReverseProxy": {
    "Routes": {
      "orders-route": {
        "ClusterId": "orders-cluster",
        "Match": { "Path": "/api/orders/{**catch-all}" }
      },
      "users-route": {
        "ClusterId": "users-cluster",
        "Match": { "Path": "/api/users/{**catch-all}" }
      },
      "payments-route": {
        "ClusterId": "payments-cluster",
        "Match": { "Path": "/api/payments/{**catch-all}" }
      }
    },
    "Clusters": {
      "orders-cluster": {
        "Destinations": {
          "primary": { "Address": "http://orders-service:8001/" }
        }
      },
      "payments-cluster": {
        "Destinations": {
          "primary": { "Address": "http://payments-service:8002/" }
        }
      }
    }
  }
}
C# — Program.cs — wire it up
builder.Services.AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

app.MapReverseProxy();

// That's it. YARP reads config, listens on port 80,
// forwards traffic to the right service.
// Change an internal address = update one config line, zero clients affected.

Responsibility 2 — Authentication

The gateway verifies JWTs once before forwarding. Services downstream trust that if a request arrived through the gateway, it's already authenticated — they don't parse tokens.

C# — Gateway — JWT verification + header injection
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://your-identity-server";
        options.Audience  = "api-gateway";
    });

app.UseAuthentication();
app.UseAuthorization();

app.MapReverseProxy(proxyPipeline =>
{
    proxyPipeline.Use(async (context, next) =>
    {
        // Stamp the verified userId into a header
        // Downstream services read this — no JWT parsing needed
        if (context.User.Identity?.IsAuthenticated == true)
        {
            var userId = context.User.FindFirst("sub")?.Value;
            context.Request.Headers["X-User-Id"] = userId;
        }
        await next();
    });
});
C# — Orders service — trusts the header, never parses JWT
[HttpPost]
public async Task<IActionResult> PlaceOrder()
{
    // Gateway already verified the token — just read the header
    var userId = Request.Headers["X-User-Id"].ToString();
    // ... use userId directly — no auth library needed in this service
}
✅
The gateway is the trust boundary. Inside the network, services communicate freely without re-authenticating. This is called the "internal network trust" model — every service trusts headers that came from the gateway.

Responsibility 3 — Rate Limiting

Protect services from being overwhelmed. The gateway enforces limits per client before requests even reach your services.

C# — Rate limiting policies
builder.Services.AddRateLimiter(options =>
{
    // Standard: 100 requests per minute per IP
    options.AddFixedWindowLimiter("standard", opt =>
    {
        opt.PermitLimit = 100;
        opt.Window      = TimeSpan.FromMinutes(1);
        opt.QueueLimit  = 10;
    });

    // Premium: 1000/min for authenticated premium clients
    options.AddFixedWindowLimiter("premium", opt =>
    {
        opt.PermitLimit = 1000;
        opt.Window      = TimeSpan.FromMinutes(1);
    });

    // Return 429 Too Many Requests when limit hit
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
});

app.UseRateLimiter();

Responsibility 4 — Request Aggregation

A mobile app's home screen needs data from Users, Orders, and Inventory simultaneously. Without aggregation: 3 round trips. With aggregation: the gateway fans out in parallel and returns one merged response.

C# — Aggregation endpoint in gateway
[HttpGet("/api/dashboard")]
public async Task<IActionResult> GetDashboard()
{
    var userId = Request.Headers["X-User-Id"].ToString();

    // Fan out — all 3 calls fire simultaneously
    var profileTask   = _usersClient.GetProfileAsync(userId);
    var ordersTask    = _ordersClient.GetRecentAsync(userId);
    var inventoryTask = _inventoryClient.GetAlertsAsync();

    await Task.WhenAll(profileTask, ordersTask, inventoryTask);

    // Merge into one response — client gets everything in one round trip
    return Ok(new
    {
        Profile   = profileTask.Result,
        Orders    = ordersTask.Result,
        Inventory = inventoryTask.Result
    });
}

The BFF Pattern — Backend for Frontend

Different clients (mobile, web, partner API) have different data needs. Instead of one gateway trying to serve everyone, you create a dedicated gateway per client type.

Mobile BFF
iOS / Android
Compact payloads. Offline sync support. Minimal data transfer. Owned by the mobile team.
Web BFF
React / Vue app
Rich data aggregations. Server-side auth. Session management. Owned by the web team.
Partner BFF
Third party API
Versioned, heavily throttled, well-documented public API. Owned by the platform team.
💡
Each BFF is owned by the frontend team that uses it. The mobile team owns the Mobile BFF — they control what data they get without negotiating with backend teams. No conflicting requirements from multiple clients in one gateway.

Gateway vs Backend — What Lives Where

The gateway handles infrastructure concerns. Business logic stays in services. This distinction came up in our conversation — you correctly identified that aggregation and BFF logic belong in the application layer, not the pure proxy.

Lives in the dedicated gateway (YARP/Nginx/Kong)
Routing — URL pattern → service mapping
SSL termination — HTTPS at the edge
JWT verification — auth happens once here
Rate limiting — per IP or per client
CORS — configured in one place
Access logging — all requests audited
Lives in ASP.NET Core BFF (application layer)
Request aggregation — fan-out + merge
Response transformation — reshape for client
BFF logic — client-specific data shaping
Business rules — context-aware decisions
Session management — stateful operations
💡
In practice most teams run both: a dedicated reverse proxy (YARP or Kong) for pure infrastructure routing, and an ASP.NET Core BFF behind it for the aggregation logic. Two layers, two concerns.

Interview Q&A

What is an API Gateway and why do microservices need one?
A single entry point for all client requests. It handles cross-cutting concerns — routing, auth, rate limiting, SSL termination — in one place instead of duplicating them in every service. Without it, clients are coupled to service addresses, and every service must independently implement auth and rate limiting.
What is the BFF pattern and when do you use it?
Backend for Frontend — a dedicated gateway per client type (mobile, web, partner). Use it when different clients have meaningfully different data requirements. A mobile app needs compact payloads; a web app needs rich aggregations; a partner API needs versioning. One gateway serving all three becomes a bottleneck of conflicting requirements. Each BFF is owned by the frontend team that consumes it.
What's the difference between an API Gateway and a load balancer?
A load balancer distributes traffic across multiple instances of the same service — purely availability and throughput. A gateway operates at the application layer: it understands routes, reads JWTs, applies business logic, transforms requests, and aggregates responses across multiple different services. You typically use both — the gateway does smart routing, load balancers handle instance distribution behind it.
What are the risks of an API Gateway?
Two main ones. First, it becomes a single point of failure — if it goes down, everything goes down. Mitigation: run multiple gateway instances behind a load balancer. Second, it becomes a bottleneck or "God service" if too much business logic is put inside it. A gateway should handle cross-cutting concerns only — not order processing or database calls.
Why does the gateway inject X-User-Id instead of passing the JWT to services?
The gateway is the trust boundary — it verifies the JWT once and stamps the verified identity into a simple header. Downstream services trust this header without needing JWT libraries, public key configs, or auth complexity. It decouples auth from business services. If the auth mechanism changes (e.g. switch from JWT to OAuth2), only the gateway changes — services are unaffected.
