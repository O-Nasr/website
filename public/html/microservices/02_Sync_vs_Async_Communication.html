01

The Core Problem

In a monolith, one module calls another with a direct method call — no address, no network, no failure modes. In microservices, that same call crosses a network. You now must make a conscious choice: how should these services talk?

Synchronous
Caller sends a request and waits for the response before continuing. Like a phone call — you can't hang up until the other person answers.
Asynchronous
Caller fires a message and moves on immediately. Like sending an email — you don't wait for a reply to continue your work.
02

Synchronous Communication

The caller blocks until the response arrives. Two protocols dominate.

03

REST in Depth

REST uses HTTP verbs (GET, POST, PUT, DELETE) over HTTP/1.1. It's human-readable, universally supported, and trivially debuggable with a browser or curl.

C# — REST with HttpClient
public class OrdersService
{
    private readonly HttpClient _http;
    public OrdersService(HttpClient http) => _http = http;

    public async Task<PaymentResult> ChargeAsync(Order order)
    {
        // Serialize to JSON, send over HTTP, wait for response
        var response = await _http.PostAsJsonAsync(
            "http://payments-service/api/payments",
            new { OrderId = order.Id, Amount = order.Total, Currency = "USD" }
        );
        response.EnsureSuccessStatusCode();
        return await response.Content.ReadFromJsonAsync<PaymentResult>();
    }
}

// On the wire:
// POST /api/payments HTTP/1.1
// Content-Type: application/json
// { "orderId": "ord-123", "amount": 59.99, "currency": "USD" }
// ← 200 { "paymentId": "pay-456", "status": "authorised" }
04

gRPC in Depth

gRPC uses HTTP/2 and serializes with Protocol Buffers (binary). 5–10× faster and smaller than REST. Tradeoff: harder to debug, requires tooling, not browser-friendly. You define a schema-first contract in a .proto file.

Protobuf — payments.proto (shared contract)
syntax = "proto3";

service PaymentsService {
  rpc ChargeOrder (ChargeRequest) returns (ChargeResponse);
}

message ChargeRequest {
  string order_id  = 1;
  double amount    = 2;
  string currency  = 3;
}

message ChargeResponse {
  string payment_id = 1;
  string status     = 2;
}
C# — gRPC client (.NET has first-class support)
public class OrdersService
{
    private readonly PaymentsService.PaymentsServiceClient _client;

    public async Task<ChargeResponse> ChargeAsync(Order order)
    {
        // Strongly typed — compiler catches contract mismatches
        // Binary serialization — 5–10x faster than JSON
        return await _client.ChargeOrderAsync(new ChargeRequest
        {
            OrderId  = order.Id,
            Amount   = order.Total,
            Currency = "USD"
        });
    }
}
💡
Contract enforcement: If Payments changes its .proto and breaks the contract, the compiler tells you before it hits production. This is the key advantage over REST where contract changes are discovered at runtime.
05

REST vs gRPC — When to Use Which

Practical rule: REST for anything external-facing or browser-accessible. gRPC for high-throughput internal service-to-service calls where you control both ends.

DimensionREST (HTTP/JSON)gRPC (HTTP/2 + Protobuf)
SpeedModerate — text serializationFast — binary, 5–10× smaller
ContractImplicit — OpenAPI optionalExplicit — .proto enforced
Browser supportYes — nativeNo — needs grpc-web proxy
DebuggabilityEasy — curl, PostmanHard — needs tooling
StreamingNo — request/response onlyYes — bi-directional streams
VersioningURL versioning /v1/, /v2/Field numbers in .proto
Best forPublic APIs, external clientsInternal service-to-service
06

Asynchronous Communication

The caller publishes a message and moves on immediately. It doesn't wait. It doesn't even know if anyone received the message. The consumer processes it in their own time. This is the backbone of truly independent microservices.

Orders publishes one OrderPlaced event. Payments, Inventory, and Notifications each receive it independently. Orders doesn't know or care how many consumers exist — loose coupling in action.

07

MassTransit — C# Async Messaging

MassTransit is the standard .NET library for async messaging, supporting RabbitMQ, Azure Service Bus, and Kafka.

C# — Publisher (Orders service)
// Shared event contract — plain C# record
public record OrderPlaced(
    string OrderId, string CustomerId,
    decimal Total, string Currency, DateTimeOffset PlacedAt);

public class OrderService
{
    private readonly IPublishEndpoint _bus;
    public OrderService(IPublishEndpoint bus) => _bus = bus;

    public async Task PlaceOrderAsync(Cart cart)
    {
        var order = await SaveOrderAsync(cart);

        // Publish event — fire and forget, no waiting
        await _bus.Publish(new OrderPlaced(
            order.Id, order.CustomerId, order.Total,
            "USD", DateTimeOffset.UtcNow));

        // Return immediately — Payments/Inventory process independently
    }
}
C# — Consumer (Payments service)
public class OrderPlacedConsumer : IConsumer<OrderPlaced>
{
    private readonly IPaymentGateway _gateway;
    public OrderPlacedConsumer(IPaymentGateway gateway) => _gateway = gateway;

    public async Task Consume(ConsumeContext<OrderPlaced> context)
    {
        var order = context.Message;
        // Processes in its own time, independently of the publisher
        await _gateway.ChargeAsync(order.OrderId, order.Total, order.Currency);
    }
}

// Register in Program.cs
builder.Services.AddMassTransit(x =>
{
    x.AddConsumer<OrderPlacedConsumer>();
    x.UsingRabbitMq((ctx, cfg) =>
    {
        cfg.Host("rabbitmq://localhost");
        cfg.ConfigureEndpoints(ctx);
    });
});

The Decision Framework

One question decides it: does the caller need the result right now to complete this request?

09

The Sync Chain Danger

The most common mistake: defaulting to synchronous HTTP for everything. A chain of sync calls is fragile — one slow service blocks everyone upstream.

10

Real "Place Order" Flow — Hybrid Architecture

Real systems use both sync and async. Steps that need an immediate answer (stock reservation, payment) are synchronous. Side effects that don't affect the response (email, analytics, loyalty points) are asynchronous.

Kafka vs RabbitMQ vs Pulsar

FeatureRabbitMQKafkaPulsar
ModelWork queue — message deleted after ACKAppend-only log — events retainedUnified queue + streaming
Consumer modelPush — broker delivers to consumerPull — consumer reads at own offsetCursor-based subscription
Multiple consumersCompeting — each msg goes to ONEIndependent groups — all read same logMultiple subscription types
Event replayNo — deleted after ACKYes — retained for days/weeksYes — BookKeeper storage
StorageIn brokerIn broker (partitioned)Separate (BookKeeper)
Best forTask queues, work distributionEvent sourcing, high-throughput, auditMulti-tenancy, geo-replication

The one-line summary: RabbitMQ — message disappears after one consumer reads it. Kafka — event stays in the log, every consumer group reads independently at its own pace. Pulsar — separates compute (broker) from storage (BookKeeper), scale each independently.

12

Interview Q&A

When would you use gRPC over REST?
Internal high-frequency service-to-service calls where you control both ends — e.g. an Orders service calling Inventory 10,000 times/sec. You get binary serialization, streaming, and strong typing from the .proto contract. REST is better for external APIs, public-facing endpoints, or anything a browser talks to directly.
What's the problem with chaining synchronous calls?
Latency adds up — total response time is the sum of every hop. More critically, availability compounds: if each service is 99.9% available and you chain 5 synchronously, overall availability is 99.9%^5 ≈ 99.5%. A downstream failure blocks everyone upstream. Fix: make anything that doesn't need an immediate result asynchronous.
What guarantee does a message queue give you?
At-least-once delivery — a message will be delivered at least once, but may be delivered more than once on retries. This means consumers must be idempotent — processing the same message twice produces the same result as once. Exactly-once is possible (Kafka has it) but expensive. Design for at-least-once.
What is eventual consistency and when does it apply?
When you go async, different services are temporarily out of sync — the email might be sent 200ms after the order is confirmed, inventory count might lag by a second. The system will eventually converge to the correct state, but not immediately. Acceptable for non-financial side effects. Not acceptable for "has this payment been charged" — that must be synchronous.
What is the difference between Kafka and RabbitMQ?
RabbitMQ is a message queue — each message is consumed by one instance and deleted after acknowledgement. Use for task distribution (one worker handles one job). Kafka is an event log — events are written to an append-only log and never deleted until retention expires. Multiple consumer groups each read the full log independently at their own offset. Use for event sourcing, audit logs, and when multiple systems need the same events.
