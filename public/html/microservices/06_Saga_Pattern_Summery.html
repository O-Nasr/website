Saga Pattern

Choreography vs orchestration, compensating transactions, idempotency, crash recovery. Full C# code.

01 The Problem — No Shared Transactions

In a monolith: BEGIN TRANSACTION wraps everything. In microservices: each service has its own DB. If Payment fails after Orders committed — order exists, stock reduced, but money never charged.

2PC (Two-Phase Commit) exists but doesn't scale: blocking, slow, tightly coupled.

02 Saga = Local Transactions + Compensations

All steps succeed

Create order (PENDING)
→
Reserve stock
→
Charge payment
→
CONFIRMED

Payment fails — compensations run backward

Create order
→
Reserve stock
→
Payment FAILED
Undo: release stock
←
Undo: cancel order

System back to consistent state. No money charged, stock restored, order cancelled.

A rollback undoes uncommitted changes automatically. A compensation is a new committed transaction that reverses the previous one — you write the logic.

Step 2 does stock -= 5 → compensation does stock += 5
Step 3 charged $100 → compensation calls Stripe's refund API

03 Choreography vs Orchestration

Choreography — no boss, just events

Orders creates order → publishes OrderCreated
Inventory hears OrderCreated → reserves stock → publishes StockReserved
Payments hears StockReserved → charges card → publishes PaymentCompleted
Orders hears PaymentCompleted → status = CONFIRMED

No service calls another directly. They publish/subscribe via message queue. Problem at scale: no visibility of full workflow, hard to debug.

Orchestration — one boss controls everything

public class OrderSagaOrchestrator
{
    public async Task Handle(OrderCreated reply)
    {
        saga.CurrentStep = "ReserveStock";
        await _db.SaveChangesAsync();   // save state before each step
        await _bus.SendCommandAsync(new ReserveStockCommand(saga.Id));
    }
    
    public async Task Handle(PaymentFailed reply)
    {
        saga.Status = "COMPENSATING";
        await _bus.SendCommandAsync(new ReleaseStockCommand(saga.Id));
        await _bus.SendCommandAsync(new CancelOrderCommand(saga.Id));
    }
}

Entire workflow in one class. New developer reads it in 2 minutes. Crash recovery: saves state to DB, another instance resumes.

04 Idempotency

public async Task Handle(ChargePaymentCommand cmd)
{
    var existing = await _db.Payments
        .FirstOrDefaultAsync(p => p.SagaId == cmd.SagaId);
    if (existing != null)
    {
        await _bus.PublishAsync(new PaymentCompleted(cmd.SagaId));
        return; // already processed — safe to re-publish
    }
    // First time — actually charge
    var result = await _stripe.ChargeAsync(cmd.Amount);
    // ... save and publish
}

SagaId acts as idempotency key. Same pattern as notification dedup from Chapter 10.

05 Interview Q&A

Q: Does a saga guarantee ACID?
No. ACD but not Isolation. Between steps, other transactions see intermediate states. Handle with status fields (PENDING → CONFIRMED).
Q: How do you handle orchestrator crashing?
Saves state to DB before every step. Multiple instances. On restart, query for IN_PROGRESS sagas and resume. Commands must be idempotent.
