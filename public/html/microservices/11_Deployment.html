01 The Problem — Deploying Without Downtime

In a monolith, you deploy one thing. Scary, but simple. In microservices, you deploy 20+ services independently, each with its own schedule. The question: how do you update a running service without users noticing?

Three strategies exist, each with different tradeoffs. All three achieve zero downtime — they differ in rollback speed, resource cost, and blast radius.

StrategyOne-linerRollback
RollingReplace pods one by one (K8s default)Slow — re-roll all pods
Blue-GreenTwo full environments, switch traffic instantlyInstant — switch LB back
CanaryGradual traffic shift: 5% → 25% → 50% → 100%Instant — route back to v1

02 Strategy 1 — Rolling Deploy (K8s Default)

This is what happens when you run kubectl set image — which you've done before with k9s. K8s replaces pods one at a time, waiting for each new pod to pass its readiness probe before continuing.

Before deploy: 3 pods running v1

Pod 1 — orders:v1
Pod 2 — orders:v1
Pod 3 — orders:v1

All traffic goes to v1. You run: kubectl set image deploy/orders orders=orders:v2

Step 1: K8s creates one v2 pod, terminates one v1 pod

Pod 1 — orders:v2 (new)
Pod 2 — orders:v1
Pod 3 — orders:v1

Traffic splits: ~33% goes to v2, ~67% still on v1. K8s waits for v2 pod to pass readiness probe before continuing.

Step 2: second v2 pod replaces another v1

Pod 1 — orders:v2
Pod 2 — orders:v2 (new)
Pod 3 — orders:v1

Traffic: ~67% v2, ~33% v1.

Step 3: last v1 pod replaced

Pod 1 — orders:v2
Pod 2 — orders:v2
Pod 3 — orders:v2 (new)

All traffic on v2. Deploy complete.

Rolling deploy summary:
K8s replaces pods one at a time (configurable via maxSurge / maxUnavailable).
Each new pod must pass readiness probe before K8s continues.
During transition: BOTH v1 and v2 serve traffic simultaneously.
Rollback: kubectl rollout undo deploy/orders (re-does rolling update back to v1).

03 K8s Deployment YAML

The deployment config that controls rolling updates — these are the fields interviewers expect you to know:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: orders-service
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # create 1 extra pod during update
      maxUnavailable: 0   # never have fewer than 3 running pods
  template:
    spec:
      containers:
        - name: orders
          image: orders:v2
          readinessProbe:       # K8s waits for this before routing traffic
            httpGet:
              path: /health/ready
              port: 5001
            initialDelaySeconds: 5
            periodSeconds: 10
maxSurge: 1 = create 1 extra pod during update (temporarily 4 pods). maxUnavailable: 0 = never have fewer than 3 healthy pods. Together: at all times, at least 3 pods are serving traffic.

Blue never stopped running. Rollback is one LB config change — takes seconds, not minutes. This is blue-green's killer feature.

05 Strategy 3 — Canary Deploy

Named after the canary in a coal mine — send a small group in first, see if they survive. Deploy the new version alongside the old and gradually shift traffic.

Stage 1: send 5% of traffic to v2

Traffic
v1 — 95%
5%

Only 5% of users see v2. Monitor error rates, latency, and metrics for 15–30 minutes. If anything looks wrong, only 5% of users were affected — tiny blast radius.

Stage 2: metrics look good — increase to 25%

Traffic
v1 — 75%
v2 — 25%

Error rate on v2 is the same as v1. Latency is normal. Monitor for another 15 minutes. Confidence growing.

Stage 3: increase to 50%

Traffic
v1 — 50%
v2 — 50%

Half your users are on v2 now. If a subtle bug only appears under load, this is where you'd catch it.

Stage 4: full rollout — 100% on v2

Traffic
v2 — 100%

All traffic on v2. v1 pods can be terminated. Deploy complete. At no point did you risk all users on untested code.

Bug detected at 25%? Immediately route all traffic back to v1

Traffic
v1 — 100% (rollback)
Only 25% of users experienced the bug. Rollback is instant — just shift traffic back. v1 was still running. This is why canary is the safest strategy for critical services.

06 CI/CD for Microservices

In a monolith: one repo, one pipeline, one deploy. In microservices: each service has its own pipeline. A change to the Orders service deploys only the Orders service — nothing else.

Repository structure:
├── orders-service/
│   ├── src/
│   ├── Dockerfile
│   ├── k8s/deployment.yaml
│   └── .github/workflows/deploy.yml    ← its own pipeline
│
├── payments-service/
│   ├── src/
│   ├── Dockerfile
│   ├── k8s/deployment.yaml
│   └── .github/workflows/deploy.yml    ← its own pipeline
│
└── inventory-service/
    ├── src/
    ├── Dockerfile
    ├── k8s/deployment.yaml
    └── .github/workflows/deploy.yml    ← its own pipeline

A typical CI/CD pipeline per service (GitHub Actions):

# .github/workflows/deploy.yml
name: Deploy Orders Service
on:
  push:
    branches: [main]
    paths: ['orders-service/**']   # only trigger when this service changes

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 1. Run tests
      - run: dotnet test orders-service/tests/

      # 2. Build Docker image
      - run: docker build -t orders-service:${{ github.sha }} orders-service/

      # 3. Push to container registry
      - run: |
          docker tag orders-service:${{ github.sha }} registry.io/orders:${{ github.sha }}
          docker push registry.io/orders:${{ github.sha }}

      # 4. Deploy to K8s (rolling update)
      - run: |
          kubectl set image deployment/orders-service \
            orders=registry.io/orders:${{ github.sha }}

      # 5. Wait for rollout to complete
      - run: kubectl rollout status deployment/orders-service --timeout=300s
The key: paths: ['orders-service/**'] ensures a change to Payments doesn't trigger the Orders pipeline. Each team deploys independently.

08 Interview Q&A

Q: What is a rolling deployment?
Kubernetes replaces pods one at a time with the new version. Each new pod must pass its readiness probe before K8s terminates the next old pod. During the transition, both versions serve traffic simultaneously. It's the K8s default, costs no extra resources, but rollback is slow because it has to re-roll all pods.
Q: What is blue-green deployment?
You run two identical environments — Blue (current live) and Green (new version). Green is deployed and tested independently while Blue serves all traffic. When ready, you switch the load balancer from Blue to Green instantly. Rollback is instant — switch back to Blue, which never stopped running. The tradeoff is 2x resource cost during deployment.
Q: What is canary deployment?
You deploy the new version alongside the old and gradually shift traffic: first 5%, then 25%, then 50%, then 100%. At each stage you monitor error rates and latency. If anything goes wrong, you route all traffic back to the old version — only the small percentage of users on the canary were affected. It's the safest strategy but requires traffic splitting infrastructure (Istio, Nginx, or a feature flag).
Q: How does CI/CD change with microservices?
Each service has its own independent pipeline triggered only by changes to its own codebase. The pipeline runs tests, builds a Docker image tagged with the commit SHA, pushes to a container registry, and deploys to Kubernetes. This allows teams to deploy their services independently — a change to Orders doesn't trigger or affect the Payments pipeline.
Q: Which deployment strategy would you choose for a payment processing service?
Blue-green or canary — never a simple rolling deploy for payments. With blue-green, I can fully test the new version in the Green environment with synthetic transactions before switching any real traffic. With canary, I can route 1% of payment traffic to the new version and monitor success rates before expanding. Rolling deploy is risky because both versions serve traffic during transition, and a subtle bug in payment processing could affect real money before you catch it.
