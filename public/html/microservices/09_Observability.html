01 The Debugging Nightmare

User reports: "My order failed." The request passed through Gateway → Orders → Inventory → Payments → Stripe. Error happened somewhere in this chain. Where?

With Console.WriteLine: SSH into Gateway server → read log. SSH into Orders server → read log. SSH into Payments server → read log. SSH into Inventory server → read log. Try to match timestamps manually.

Problems: Logs are on different servers. Clocks are slightly out of sync. 1000s of requests per second — which log line belongs to THIS user? Containers restart → logs are GONE.
MonolithMicroservices
1 server, 1 log file10 servers, 10 log streams
Stack trace shows the full chainStack trace ends at HTTP call
Debugger works end-to-endCan't attach debugger to 10 services
In-process calls = no networkNetwork adds latency + failures
grep "error" app.log → Done.Need: centralized logs, tracing, metrics, correlation IDs

03 Pillar 1 — Structured Logging with Serilog

Console.WriteLine("Order failed for user") is useless. Which order? Which user? Which service? When?

Structured logging gives you searchable, filterable JSON:

// Serilog structured logging — properties become searchable fields
Log.Error("Order failed {@OrderId} for {@UserId} — reason: {@Reason}",
    order.Id, userId, "insufficient_stock");

This outputs structured JSON:

{
  "timestamp": "2026-03-26T14:32:01.123Z",
  "level": "Error",
  "message": "Order failed ord-123 for user-abc — reason: insufficient_stock",
  "properties": {
    "OrderId": "ord-123",
    "UserId": "user-abc",
    "Reason": "insufficient_stock",
    "ServiceName": "orders-service",
    "TraceId": "abc123def456"
  }
}

Now you can search and filter: "show me all errors for user-abc in the last hour across all services."

But logs from 10 services are on 10 different servers. Send them all to one central place:

// Program.cs — send logs to Seq (or ELK, Datadog, etc.)
Log.Logger = new LoggerConfiguration()
    .Enrich.WithProperty("ServiceName", "orders-service")
    .Enrich.FromLogContext()
    .WriteTo.Console()
    .WriteTo.Seq("http://seq-server:5341")   // centralized log server
    .CreateLogger();

04 The Correlation ID — Key to Everything

1000 requests per second, 10 services each = 10,000 log lines per second. How do you find the logs for one specific user request?

The answer: a Correlation ID — one unique ID, generated at the entry point, passed through every service call.

User → Gateway              generates: X-Correlation-Id: abc-123
       Gateway → Orders     forwards:  X-Correlation-Id: abc-123
       Orders → Inventory   forwards:  X-Correlation-Id: abc-123
       Orders → Payments    forwards:  X-Correlation-Id: abc-123

Search for abc-123 → you see the entire journey of that one request across all services, in order.

// Middleware — extracts or creates the correlation ID
public class CorrelationIdMiddleware
{
    private readonly RequestDelegate _next;

    public async Task InvokeAsync(HttpContext context)
    {
        // Check if caller already passed a correlation ID
        var correlationId = context.Request.Headers["X-Correlation-Id"]
                                .FirstOrDefault()
                            ?? Guid.NewGuid().ToString();

        // Add to response headers (so caller can see it)
        context.Response.Headers["X-Correlation-Id"] = correlationId;

        // Push into Serilog context — every log in this request gets it
        using (LogContext.PushProperty("CorrelationId", correlationId))
        {
            await _next(context);
        }
    }
}

// Register in Program.cs
app.UseMiddleware<CorrelationIdMiddleware>();
// DelegatingHandler — forwards correlation ID on outgoing HTTP calls
public class CorrelationIdHandler : DelegatingHandler
{
    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        // Get current correlation ID from Activity context
        var correlationId = Activity.Current?.Id
                            ?? Guid.NewGuid().ToString();
        request.Headers.Add("X-Correlation-Id", correlationId);

        return await base.SendAsync(request, ct);
    }
}

// Register on HttpClient
builder.Services.AddHttpClient("Inventory")
    .AddHttpMessageHandler<CorrelationIdHandler>();

06 Your Question: "Trace and Metric Feel the Same"

This was a great question. Here's why they're fundamentally different:

This is ONE specific request from ONE user. Shows the full journey: Gateway → Orders → Inventory → Payments with exact timing for each hop.

You debug: "Why was USER X's order slow? Ah — Inventory DB was slow at 200ms."

This is ALL requests aggregated over time. No individual request details — just numbers and trends.

Last hour: 45,000 requests, error rate 2.3%, avg latency 120ms, P99 latency 890ms, CPU 72%.

You alert: "Error rate spiked from 0.1% to 2.3% — something is broken!"

Step 1 — Metrics alert you
→
Step 2 — Traces pinpoint the cause
→
Step 3 — Logs give the details

Step 1: "Error rate on Payments service jumped to 5%" (Grafana dashboard)

Step 2: Open a failing trace → see Stripe API returning 503 after 5s timeout (Jaeger)

Step 3: Search correlation ID → see exact Stripe error message and payload (Seq/ELK)

07 Pillar 3 — Metrics (Prometheus + Grafana)

Metrics are numeric measurements aggregated over time. You don't look at individual data points — you look at trends:

// Custom business metrics
private static readonly Counter<long> _ordersPlaced =
    Meter.CreateCounter<long>("orders.placed", "count");

private static readonly Histogram<double> _orderLatency =
    Meter.CreateHistogram<double>("orders.latency", "ms");

public async Task PlaceOrderAsync(CreateOrderRequest req)
{
    var sw = Stopwatch.StartNew();

    // ... business logic ...

    sw.Stop();
    _ordersPlaced.Add(1, new("status", "confirmed"));
    _orderLatency.Record(sw.ElapsedMilliseconds);
}

Prometheus scrapes the /metrics endpoint every 15 seconds. Grafana visualizes it as dashboards: request rate, error rate, P99 latency, CPU/memory.

08 Full C# — OpenTelemetry Setup

// Program.cs — one setup for both tracing AND metrics
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            .SetResourceBuilder(ResourceBuilder.CreateDefault()
                .AddService("orders-service"))
            .AddAspNetCoreInstrumentation()         // auto-trace incoming HTTP
            .AddHttpClientInstrumentation()          // auto-trace outgoing HTTP
            .AddEntityFrameworkCoreInstrumentation()  // auto-trace EF queries
            .AddJaegerExporter(o =>                  // send traces to Jaeger
            {
                o.Endpoint = new Uri("http://jaeger:14268/api/traces");
            });
    })
    .WithMetrics(metrics =>
    {
        metrics
            .AddAspNetCoreInstrumentation()   // request count, duration
            .AddHttpClientInstrumentation()    // outgoing call metrics
            .AddPrometheusExporter();           // expose /metrics endpoint
    });
That's it. OpenTelemetry automatically creates spans for every incoming request, every outgoing HTTP call, and every EF Core query. The correlation ID is propagated through HTTP headers automatically.

09 Custom Spans + Custom Business Metrics

// For business logic that isn't auto-instrumented
private static readonly ActivitySource _tracer = new("OrdersService");

public async Task PlaceOrderAsync(CreateOrderRequest req)
{
    using var span = _tracer.StartActivity("PlaceOrder");
    span?.SetTag("order.userId", req.UserId);
    span?.SetTag("order.itemCount", req.Items.Count);

10 Your Question: "Does OpenTelemetry Replace Jaeger?"

No. OpenTelemetry does NOT replace Jaeger, Prometheus, or Seq. It replaces the client libraries you use in your code — not the backends.

Your code → Jaeger client SDK    → Jaeger backend (traces)
Your code → Prometheus client SDK → Prometheus backend (metrics)
Your code → Serilog + Seq sink   → Seq backend (logs)

3 different libraries in your code.
Want to switch from Jaeger to Zipkin? Rewrite all your tracing code.
Your code → OpenTelemetry SDK → Jaeger backend    (traces)
                              → Prometheus backend (metrics)
                              → Seq backend        (logs)

1 library in your code.
Want to switch from Jaeger to Zipkin? Change one config line.
Your code stays exactly the same.

OpenTelemetry is the universal adapter between your code and whatever backend you choose. Jaeger, Prometheus, Seq, Datadog — they all still exist and do the actual storage, visualization, and alerting.

// Switch from Jaeger to Zipkin — only this line changes
.AddJaegerExporter()    // ← before
.AddZipkinExporter()    // ← after

// Your instrumentation code? Zero changes.
using var span = _tracer.StartActivity("PlaceOrder");
// This line works the same regardless of backend
The real power: Your team instruments the code once. Then the ops team picks whatever backend the company uses — and they can switch backends without any developer touching any service code.

11 The Debugging Workflow — How All 3 Connect

Metrics alert you
→
Traces pinpoint the cause
→
Logs give the details

Think of it as zooming from wide to narrow:

Metrics = the satellite view. "Something is wrong somewhere." (Error rate spiked, P99 latency doubled.)

Traces = the street view. "The problem is in the Payments → Stripe call." (Open a failing trace, see the 5s timeout.)

Logs = the security camera. "Here's exactly what happened." (Search correlation ID, see the Stripe 503 error body.)

The Correlation ID is the thread that connects all three. A metric alert leads you to traces for that time window. A trace gives you the correlation ID. The correlation ID finds you every log line from every service involved in that one request.

12 Interview Q&A

Q: What is observability in microservices?
Observability is the ability to understand what's happening inside your distributed system from the outside. It has 3 pillars: logs (what happened — discrete events with context), metrics (how the system is performing — aggregated numbers over time), and distributed traces (where time was spent — a request's journey across all services with timing). Together they let you debug, monitor, and optimize a system of many independent services.
Q: What is a Correlation ID and why is it critical?
A Correlation ID is a unique identifier generated at the entry point (API Gateway) and propagated through every service in the request chain via HTTP headers. Every log line includes it. To debug a user's failed request, you search for that one ID and see the complete journey across all services in chronological order. Without it, matching log lines across 10 services at 1000 requests/second is impossible.
Q: What is distributed tracing?
A trace represents the complete journey of a single request through all services. It's made of spans — each span is one operation (an HTTP call, a DB query, a method). Spans are nested: the Gateway span contains the Orders span, which contains the Inventory span and the Payment span. A waterfall visualization shows the timing of each span, instantly revealing which service or dependency is the bottleneck.
Q: What is OpenTelemetry?
OpenTelemetry is a vendor-neutral open standard for generating, collecting, and exporting telemetry data — traces, metrics, and logs. You instrument your code once with the OTel SDK, then export to any backend (Jaeger, Prometheus, Datadog) by changing configuration. It replaces the client libraries (Jaeger SDK, Prometheus client), NOT the backends themselves. Prevents vendor lock-in and is now the industry standard.
Q: How would you set up observability for a new microservices project?
Add OpenTelemetry SDK to every service. Enable auto-instrumentation for ASP.NET Core (incoming requests), HttpClient (outgoing calls), and Entity Framework (DB queries). Export traces to Jaeger, metrics to Prometheus + Grafana, and logs to a centralized store like Seq or ELK. Use correlation IDs propagated via HTTP headers. Add custom business metrics for domain-specific monitoring. Set up alerts on error rate spikes and P99 latency thresholds.
