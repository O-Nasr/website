Phase 1 — Foundation
TOPIC 01

Monolith vs Microservices

The Core Tradeoff

Microservices are not better than monoliths. They're a tradeoff: you gain independence and scale, you pay with complexity and distributed systems problems. Every interview answer must reflect this balance.

Monolith

One deployable unit. All modules share the same process, memory, and database. Deploy everything together. Simple to develop, test, and debug. A single stack trace shows the full chain.

Microservices

Many small, independent services. Each owns its data, deploys separately, communicates over a network. Complex but independently scalable. Stack traces end at HTTP boundaries.

When to Break the Monolith

Don't break it preemptively. Break when you hit concrete pain:

  • Deployment bottleneck — one team's change blocks everyone else's release. 3 teams want to deploy but must coordinate because it's one binary.
  • Scaling mismatch — search gets 100× more traffic than admin, but you scale both together because they're the same process.
  • Technology divergence — ML team needs Python, core team uses C#, mobile backend needs Node. One monolith locks you to one stack.
  • Team ownership — at 200 engineers, nobody knows who owns what. Services give clear boundaries: the Orders team deploys the Orders service.
Martin Fowler's MonolithFirst Pattern
Start with a well-structured monolith (modular monolith). Migrate to services when you hit concrete pain. A distributed monolith — services that must deploy together, share a database, or can't function independently — is the worst of both worlds.

Decomposition Strategy 1: By Business Domain (DDD)

The most important strategy. Split by business capability, not by technical layer.

The technical split gives you a distributed monolith — every feature change still touches all 3 services. The business domain split means "add a discount to orders" lives entirely inside the Orders service.

What is DDD?

Domain-Driven Design structures software around how the business actually works, not how the database is structured. Three key concepts:

  • Domain — the problem your business solves (e-commerce)
  • Subdomain — a distinct slice of that domain (Orders, Payments, Inventory, Shipping)
  • Bounded Context — the explicit boundary where a domain model applies

The Bounded Context — Core Concept

The word "Order" means something different in different parts of the business:

Each service owns its bounded context and doesn't share its internal model. Services communicate by translating at the boundary via events or API contracts — never by sharing internal domain objects or databases.

How to Decide Service Boundaries

Follow three signals:

  • Team ownership — who changes the code together?
  • Change frequency — what changes at the same rate?
  • Data affinity — what data is tightly coupled?

If two modules are always deployed together, changed by the same team, and share all their tables — they're probably one service. If they change independently, scale differently, or need different tech — they're ready to split.

Entity-Based vs Capability-Based Cut

❌  Entity-based cut               ✅  Capability-based cut
─────────────────────              ──────────────────────────
UserService                        Identity service
  → manages User table             → auth, sessions, tokens

ProductService                     Catalogue service
  → manages Product table          → search, browse, pricing

OrderService                       Ordering service
  → manages Order table            → cart, checkout, order lifecycle

The entity-based cut sounds clean but is a trap. Every feature that crosses entities requires network calls between services. The capability-based cut keeps related operations together.

Decomposition Strategy 2: By Subdomain (Fine-Grained)

When a bounded context grows too large — 8 subdomains, 3 teams, 200K lines of code — it's time to split further. For example, the Orders context might split into:

  • Cart Service — add, remove, pricing (high read traffic)
  • Checkout Service — validate, confirm (transactional, complex)
  • Order Lifecycle Service — track, cancel, modify (long-running)
  • Returns Service — refund, restock (different team, different regulations)

The "Right Size" Question

A service should do one business capability end to end. Not one function, not one table — one capability.

Too big:   OrdersAndPaymentsAndInventoryService   ← just a monolith
Just right: OrdersService (create, read, cancel, history)
Too small: CreateOrderService + CancelOrderService  ← chatty, fragile

Data Sharing Between Services

When two services need the same data, they each keep their own copy:

  • Orders stores product name and price at time of order
  • Inventory stores stock levels and warehouse location
  • They sync via events — ProductPriceUpdated published by Catalogue, consumed by whoever cares

Data duplication is intentional — it buys independence.

Q: "Would you start a new project with microservices?"
No, usually not. Start with a well-structured modular monolith. Migrate when you hit concrete pain: scaling mismatch, deployment bottleneck, or team growth. This is Martin Fowler's MonolithFirst pattern.
Q: "What is a bounded context?"
A DDD term — the explicit boundary within which a domain model applies. Inside, terms are consistent. "Order" means one thing inside Orders service and something different inside Payments. Each service owns its bounded context and doesn't share its internal model.
Q: "What's the biggest challenge in microservices?"
Data consistency. In a monolith, one transaction wraps everything. In microservices, you handle partial failures, eventual consistency, and compensating transactions manually (Saga pattern).
Q: "What's a distributed monolith?"
A system that looks like microservices but behaves like a monolith — services must deploy together, share a database, or one can't function without the other. All the operational complexity, none of the independence benefits.
Q: "What happens when two services need the same data?"
They each keep their own copy. The Orders service stores what it needs about products (name, price at time of order). Sync via events — ProductPriceUpdated published by Catalogue, consumed by whoever cares. Duplication is intentional — it buys independence.

TOPIC 02

Inter-Service Communication — Sync vs Async

The Core Problem

In a monolith, calling another module is just a function call — no network, no failure modes. In microservices, that same call crosses a network — it can be slow, fail, time out, or the other service might be down. You must make a conscious choice: how should these services talk?

Part 1 — Synchronous: REST vs gRPC

REST uses HTTP verbs over plain HTTP/1.1. Human-readable JSON, universally supported, debuggable with curl or Postman.

// Orders service calling Payments via REST
public class OrdersService
{
    private readonly HttpClient _http;

    public async Task<PaymentResult> ChargeAsync(Order order)
    {
        var response = await _http.PostAsJsonAsync(
            "http://payments-service/api/payments",
            new { OrderId = order.Id, Amount = order.Total, Currency = "USD" }
        );
        response.EnsureSuccessStatusCode();
        return await response.Content.ReadFromJsonAsync<PaymentResult>();
    }
}

gRPC uses HTTP/2 and Protocol Buffers (binary). 5–10× faster, strict contract via .proto file, compiler catches contract mismatches.

// payments.proto — shared contract
syntax = "proto3";
service PaymentsService {
  rpc ChargeOrder (ChargeRequest) returns (ChargeResponse);
}
message ChargeRequest {
  string order_id  = 1;
  double amount    = 2;
  string currency  = 3;
}
// Orders service calling Payments via gRPC in C#
public async Task<ChargeResponse> ChargeAsync(Order order)
{
    return await _client.ChargeOrderAsync(new ChargeRequest
    {
        OrderId  = order.Id,
        Amount   = order.Total,
        Currency = "USD"
    });
}

Part 2 — Asynchronous: Message Queues

The caller publishes a message and moves on immediately. The consumer processes it in their own time. This is the backbone of truly independent microservices.

Kafka vs RabbitMQ

RabbitMQ — message queue. Messages are pushed to consumers and deleted after acknowledged. Great for task distribution and work queues.

Kafka — event log. Events are written to an append-only log and consumers read at their own pace. Events are retained (days/weeks/forever). Multiple consumer groups can each read the full history independently.

RabbitMQ:  Orders → [Queue] → Payments reads it → message gone
Kafka:     Orders → [Log]   → Payments reads it at offset 1042
                             → Analytics reads it at offset 1042
                             → Inventory reads it at offset 1042
                             → All three independently, event preserved

The Hybrid Pattern — Real-World Architecture

Real systems use both. The rule: sync when you need the answer NOW, async when you don't.

1. Client → API Gateway        [sync REST]     validate JWT
2. Gateway → Orders service    [sync REST]     create order
3. Orders → Inventory          [sync gRPC]     reserve stock (need confirmation NOW)
4. Orders → Payments           [sync gRPC]     charge card (need success/fail NOW)
5. Orders → Message Queue      [async event]   publish OrderPlaced
6. Queue → Notifications       [async]         send email (don't need to wait)
7. Queue → Analytics           [async]         log revenue event
8. Queue → Loyalty service     [async]         award points

Steps 3-4 must be synchronous — you can't tell the customer "order placed" unless you've confirmed stock and payment. Steps 6-8 don't affect the customer's experience right now, so they go async.

The Chaining Problem
If you chain 5 synchronous calls: total latency = sum of all hops. Availability compounds: 5 services at 99.9% each = 99.9%⁵ ≈ 99.5% overall. A downstream failure blocks everyone upstream.
Q: When would you use gRPC over REST?
Internal high-frequency service-to-service calls where you control both ends. You get binary serialization, streaming, and strong typing from the .proto contract. REST is better for external APIs, public-facing endpoints, or anything a browser talks to directly.
Q: What guarantees does a message queue give you?
At-least-once delivery — a message will be delivered at least once, but may be duplicated on retries. Consumers must be idempotent — processing the same message twice produces the same result. Exactly-once is theoretically possible (Kafka) but expensive. Design for at-least-once.
Q: What is eventual consistency?
When you go async, different services will be temporarily out of sync. The email might be sent 200ms after the order is confirmed. The inventory count might lag. The system converges to the correct state, but not immediately. Acceptable for side effects — not for "has this payment been charged."

API Gateway + BFF Pattern

The Problem Without a Gateway

Without a gateway, every client talks directly to every service. The mobile app needs to know 8 service URLs. Auth is implemented in every service. CORS is configured everywhere. With a gateway: one front door, one URL, routing + auth + rate limiting in one place.

Gateway Responsibilities

ResponsibilityWhat It DoesWhy at the Gateway
RoutingMaps /api/orders → Orders serviceClients know one URL, not 20
AuthenticationVerifies JWT once before forwardingServices don't re-verify tokens
Rate Limiting100 req/min per IPProtects all services from abuse
SSL TerminationHTTPS → HTTP internallyServices don't manage certificates
AggregationCombine 3 service responses into 1Mobile gets one call instead of three

Authentication at the Gateway

// Gateway verifies JWT, stamps userId into header
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://your-identity-server";
        options.Audience  = "api-gateway";
    });

// YARP pipeline — inject X-User-Id header
app.MapReverseProxy(proxyPipeline =>
{
    proxyPipeline.Use(async (context, next) =>
    {
        if (context.User.Identity?.IsAuthenticated == true)
        {
            var userId = context.User.FindFirst("sub")?.Value;
            context.Request.Headers["X-User-Id"] = userId;
        }
        await next();
    });
});
// Orders service — trusts the header, never touches a JWT

{

    // ... use userId directly
}

Rate Limiting

builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("standard", opt =>
    {
        opt.PermitLimit = 100;
        opt.Window = TimeSpan.FromMinutes(1);
        opt.QueueLimit = 10;
    });
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;

app.UseRateLimiter();

Request Aggregation

A mobile app's home screen needs data from Users, Orders, and Inventory simultaneously. Without aggregation: 3 separate API calls. With aggregation at the gateway: 1 call, the gateway fans out internally and combines the responses.

The BFF Pattern (Backend for Frontend)

Different clients need different data shapes. Instead of one gateway serving all clients, create one dedicated gateway per client type:

  • Mobile BFF — compact payloads, fewer fields, optimized for bandwidth
  • Web BFF — rich aggregations, multiple data sources, detailed views
  • Partner API BFF — strict versioning, heavy throttling, webhooks

Each BFF is typically owned by the frontend team that consumes it.

Two-Layer Architecture

In practice, most teams run a dedicated reverse proxy (YARP or Kong) as the pure infrastructure gateway, then an ASP.NET Core BFF behind it for aggregation logic. Two layers, two concerns.

Q: What's the difference between an API Gateway and a load balancer?
A load balancer distributes traffic across instances of the same service — purely availability and throughput. A gateway operates at the application layer: understands routes, reads JWTs, applies business logic, transforms requests, aggregates across different services. You often have both — gateway does smart routing, LBs handle instance distribution behind it.
Q: What are the risks of an API Gateway?
Single point of failure — mitigate by running multiple instances behind a load balancer. Second risk: becoming a "God service" if too much business logic is added. A gateway should only do cross-cutting concerns — routing, auth, rate limiting. Never database calls or order processing logic.

Circuit Breaker, Retry & Timeout

The Cascading Failure Problem

Service A calls Service B. B is slow or dead. Without protection: threads in A pile up waiting → A runs out of resources → A dies → everything upstream dies too. One dead service takes down the whole system.

The 3 States of a Circuit Breaker

CLOSED ──(failure rate > 50% over 10 calls)──→ OPEN
                                                  │
                                          (wait 30 seconds)
                                                  │
CLOSED ←──(probe succeeds)── HALF-OPEN ──(probe fails)──→ OPEN
StateBehaviorTransition
ClosedAll calls go through. Failures counted in a sliding window.→ Open when failure rate exceeds threshold (e.g., 50% of last 10 calls)
OpenAll calls fail immediately — no network call made. Returns fallback.→ Half-Open after recovery timeout (e.g., 30 seconds)
Half-OpenONE probe request is allowed through to test.→ Closed if probe succeeds. → Open if probe fails.

Timeout Pattern

Every outgoing call needs a timeout. Without it, a hung service means your thread waits forever.

// Bad — no timeout, thread hangs indefinitely
var response = await _http.GetAsync("http://payments/api/charge");

// Good — fail after 3 seconds
var cts = new CancellationTokenSource(TimeSpan.FromSeconds(3));
var response = await _http.GetAsync("http://payments/api/charge", cts.Token);

Retry with Exponential Backoff + Jitter

Transient failures (network blip, temporary overload) often resolve themselves. Retry with increasing delays:

Attempt 1: immediate
Attempt 2: wait 200ms + random(0-50ms)
Attempt 3: wait 400ms + random(0-50ms)
Attempt 4: wait 800ms + random(0-50ms)

Why jitter? Without it, all instances retry at identical intervals — a synchronized storm that hits the recovering service at the same moment. Jitter spreads the load across time.

What NOT to Retry
400 Bad Request — your payload is wrong, retrying won't fix it.
401 Unauthorized — your token is invalid.
404 Not Found — the resource doesn't exist.

Only retry: 503 Service Unavailable, 429 Too Many Requests, 504 Gateway Timeout, network-level failures.

Polly — Full Implementation in .NET

// Install: Microsoft.Extensions.Http.Resilience

builder.Services
    .AddHttpClient<IPaymentsClient, PaymentsClient>()
    .AddResilienceHandler("payments-pipeline", builder =>
    {
        // Layer 1 — Timeout: fail any single call after 3 seconds
        builder.AddTimeout(TimeSpan.FromSeconds(3));

        // Layer 2 — Retry: up to 3 retries with exponential backoff + jitter
        builder.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            Delay             = TimeSpan.FromMilliseconds(200),
            BackoffType       = DelayBackoffType.Exponential,
            UseJitter         = true,
            ShouldHandle      = args => ValueTask.FromResult(
                args.Outcome.Exception is HttpRequestException ||
                args.Outcome.Result?.StatusCode
                    is HttpStatusCode.ServiceUnavailable
                    or HttpStatusCode.TooManyRequests
            )
        });

        // Layer 3 — Circuit Breaker: open after 50% failure rate
        builder.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            FailureRatio          = 0.5,
            SamplingDuration      = TimeSpan.FromSeconds(30),
            MinimumThroughput     = 10,
            BreakDuration         = TimeSpan.FromSeconds(30),
            ShouldHandle          = args => ValueTask.FromResult(
                args.Outcome.Exception is HttpRequestException ||
                args.Outcome.Result?.StatusCode
                    is HttpStatusCode.ServiceUnavailable
                    or HttpStatusCode.InternalServerError
            )
        });
    });
Execution Order: Circuit Breaker → Retry → Timeout
Reading inside-out: the circuit breaker decides if a call is even attempted, the retry decides how many times, and the timeout limits each individual attempt.

Circuit breaker open? → fail immediately, no network call
Attempt 1 with 3s timeout → fails
Retry: wait 200ms + jitter
Attempt 2 with 3s timeout → fails
Retry: wait 400ms + jitter
Attempt 3 with 3s timeout → fails
Failure recorded → circuit breaker checks failure rate

Fallback — What to Return When Everything Fails

public async Task<PaymentResult> ChargeAsync(PaymentRequest request)
{
    try
    {
        return await _paymentsClient.ChargeAsync(request);
    }
    catch (BrokenCircuitException)
    {
        // Circuit is open — Payments is known to be down
        return PaymentResult.ServiceUnavailable(
            "Payment processing is temporarily unavailable."
        );
    }
    catch (TimeoutRejectedException)
    {
        return PaymentResult.TimedOut();
    }
}

Bulkhead Pattern

Isolate thread pools per downstream dependency so one slow service can't consume all available threads. Named after ship bulkheads — watertight compartments that contain flooding. One compartment fills, the others survive.

Q: What is the difference between a circuit breaker and a timeout?
A timeout limits how long you wait for a single call. A circuit breaker tracks the history of calls over time and stops making calls entirely when the service is clearly unhealthy — even before the timeout would trigger. They work together: timeouts generate the failures that the circuit breaker counts.
Q: Why does retry need jitter?
Without jitter, all service instances retry at identical intervals after a failure spike — a synchronized storm that hits the recovering service at the same moment. Jitter (random ±50ms) spreads the retry load across time, giving the service room to recover.

Service Discovery

The Problem — Why Hardcoded URLs Break

In a monolith, you call another module in-process. In microservices, you need Service B's address. Hardcoding http://192.168.1.45:5001 in config breaks in three ways:

  • Scaling — you spin up 3 Inventory instances, but config only knows one IP. Two instances sit idle.
  • Crashes — Inventory crashes, gets a new IP. Config still points to the dead address. All requests fail.
  • Deploys — new deployment gets a different IP. Nobody told the callers. Requests fail.

Two Discovery Models

AspectClient-Side DiscoveryServer-Side Discovery
Who picks the instance?The caller queries registry, picks oneLoad balancer / DNS picks for you
How it worksClient gets full list of healthy IPs, applies round-robinClient hits one stable DNS name, LB routes to healthy pod
ExamplesNetflix Ribbon + Eureka, gRPC built-inKubernetes Services, AWS ALB
ProNo extra network hopSimple for developers — just use a DNS name
ConEvery service needs discovery logic built inExtra hop through load balancer

How Kubernetes Does It (Server-Side)

Your code just says http://inventory-service:5001/api. Kubernetes handles everything:

  • CoreDNS resolves "inventory-service" to a virtual ClusterIP (e.g., 10.96.0.15)
  • kube-proxy maintains iptables rules routing from ClusterIP to healthy pod IPs
  • Endpoints API auto-registers pods when they start
  • Readiness probes remove unhealthy pods from the routing pool
// In C# — nothing special! Just use the service name
builder.Services.AddHttpClient("Inventory", client =>
{
    client.BaseAddress = new Uri("http://inventory-service:5001");
});

// Kubernetes DNS resolves "inventory-service" automatically

Without Kubernetes — Consul

On bare metal, VMs, or Docker Compose, use a dedicated registry. The service discovers its own address dynamically at startup (not hardcoded):

public async Task StartAsync(CancellationToken ct)
{
    var address = Environment.GetEnvironmentVariable("HOST_IP")
                  ?? GetLocalIPAddress();
    var port = int.Parse(Environment.GetEnvironmentVariable("PORT") ?? "5001");

    await _consul.Agent.ServiceRegister(new AgentServiceRegistration
    {
        ID = $"inventory-{Guid.NewGuid()}",
        Name = "inventory-service",
        Address = address,       // dynamic!
        Port = port,             // from env!
        Check = new AgentServiceCheck
        {
            HTTP = $"http://{address}:{port}/health",
            Interval = TimeSpan.FromSeconds(10),
            DeregisterCriticalServiceAfter = TimeSpan.FromSeconds(30)
        }
    });
}

    .AddDbContextCheck<AppDbContext>()

Liveness /healthIs the process alive?K8s restarts the pod

A service starting up and loading cache? It's alive (liveness = healthy) but not ready (readiness = unhealthy). K8s keeps it running but doesn't send traffic until the cache is warm.

Q: How does Kubernetes service discovery work?
When you create a K8s Service, it gets a stable DNS name and virtual IP. CoreDNS resolves the service name to that IP. kube-proxy maintains iptables rules routing from virtual IP to healthy pod IPs. Pods auto-register via Endpoints API. Health is monitored through liveness and readiness probes.
Q: What's the difference between liveness and readiness probes?
Liveness checks if the process is alive — failure triggers a restart. Readiness checks if it can handle traffic — failure removes it from the load balancer but keeps it running. A service loading its cache is alive but not ready.

Saga Pattern

The Problem — Why Normal Transactions Break

In a monolith: BEGIN TRANSACTION wraps all steps — create order, reduce stock, charge payment. Either all commit or all rollback. ACID.

In microservices: each service has its own database. There is no single transaction that spans all three. If Payment fails after Orders committed, you're stuck — the order exists, stock was reduced, but money was never charged.

Why Not 2PC (Two-Phase Commit)?

  • Blocking — all services lock rows and wait for coordinator. If coordinator crashes, everyone stays locked forever.
  • Performance — every service waits on the slowest. 50ms flow becomes 500ms.
  • Coupling — all services must be available simultaneously.

What is a Saga?

A sequence of local transactions, where each service does its own work and publishes an event. If any step fails, compensating transactions run backward to undo previous steps.

Compensation ≠ Rollback
A rollback undoes uncommitted changes automatically. A compensation is a new committed transaction that semantically reverses the previous one — you write the logic yourself.

Step 2 does stock -= 5 → compensation does stock += 5
Step 3 charged $100 to Stripe → compensation calls Stripe's refund API

Happy Path vs Failure Path

Happy path: Create order (PENDING) → Reserve stock → Charge payment → Confirm order. All succeed → system consistent.

Failure path: Create order → Reserve stock → Payment FAILS → Compensation: release stock (stock += quantity) → Cancel order (status = CANCELLED). System returns to clean state.

Choreography — No Boss, Just Events

Each service publishes events and reacts to other services' events. No central controller:

Orders creates order → publishes OrderCreated
    ↓
Inventory hears OrderCreated → reserves stock → publishes StockReserved
    ↓
Payments hears StockReserved → charges card → publishes PaymentCompleted
    ↓
Orders hears PaymentCompleted → status = CONFIRMED
// Events (shared contracts)
public record OrderCreated(Guid OrderId, string ProductId, int Qty);
public record StockReserved(Guid OrderId);
public record PaymentCompleted(Guid OrderId);
public record PaymentFailed(Guid OrderId, string Reason);

// Inventory Service — reacts to OrderCreated
public class InventoryService
{
    public async Task Handle(OrderCreated evt)
    {
        var stock = await _db.Stock.FindAsync(evt.ProductId);
        if (stock.Quantity >= evt.Qty)
        {
            stock.Quantity -= evt.Qty;
            await _db.SaveChangesAsync();
            await _bus.PublishAsync(new StockReserved(evt.OrderId));
        }
        else
        {
            await _bus.PublishAsync(new PaymentFailed(evt.OrderId, "Out of stock"));
        }
    }
}

Choreography Problems at Scale

  • No visibility — you can't look at one place and see "this saga is 60% done"
  • Cyclic dependencies — 10 services publishing events to each other can create event loops
  • Hard to add steps — inserting a fraud check means modifying multiple services' subscriptions

Orchestration — One Boss Controls Everything

A Saga Orchestrator knows the entire workflow and tells each service what to do. Hub-and-spoke model.

public class OrderSagaOrchestrator
{
    public async Task StartSaga(CreateOrderRequest req)
    {
        var saga = new OrderSaga
        {
            Id = Guid.NewGuid(),
            CurrentStep = "CreateOrder",
            Status = "IN_PROGRESS"
        };
        _db.Sagas.Add(saga);
        await _db.SaveChangesAsync();
        await _bus.SendCommandAsync(new CreateOrderCommand(saga.Id, req));
    }

    public async Task Handle(OrderCreated reply)
    {
        var saga = await _db.Sagas.FindAsync(reply.SagaId);
        saga.CurrentStep = "ReserveStock";
        await _db.SaveChangesAsync();
        await _bus.SendCommandAsync(new ReserveStockCommand(saga.Id, ...));
    }

    public async Task Handle(StockReserved reply)
    {
        var saga = await _db.Sagas.FindAsync(reply.SagaId);
        saga.CurrentStep = "ChargePayment";
        await _db.SaveChangesAsync();
        await _bus.SendCommandAsync(new ChargePaymentCommand(saga.Id, ...));
    }

    public async Task Handle(PaymentCompleted reply)
    {
        var saga = await _db.Sagas.FindAsync(reply.SagaId);
        saga.Status = "COMPLETED";
        await _db.SaveChangesAsync();
        await _bus.SendCommandAsync(new ConfirmOrderCommand(saga.Id));
    }

    // COMPENSATION — payment failed
    public async Task Handle(PaymentFailed reply)
    {
        var saga = await _db.Sagas.FindAsync(reply.SagaId);
        saga.Status = "COMPENSATING";
        await _db.SaveChangesAsync();
        await _bus.SendCommandAsync(new ReleaseStockCommand(saga.Id));
        await _bus.SendCommandAsync(new CancelOrderCommand(saga.Id));
        saga.Status = "FAILED";
        await _db.SaveChangesAsync();
    }
}

Crash Recovery — Durable State Machine

The orchestrator saves CurrentStep and Status to DB before every command. If it crashes, another instance queries for IN_PROGRESS sagas and resumes exactly where it left off.

Idempotency — Preventing Double-Processing

public async Task Handle(ChargePaymentCommand cmd)
{
    // Already processed? Just re-publish the reply
    var existing = await _db.Payments
        .FirstOrDefaultAsync(p => p.SagaId == cmd.SagaId);
    if (existing != null)
    {
        await _bus.PublishAsync(new PaymentCompleted(cmd.SagaId));
        return;
    }
    // First time — actually charge
    var result = await _stripe.ChargeAsync(cmd.Amount);
    _db.Payments.Add(new Payment { SagaId = cmd.SagaId, ... });
    await _db.SaveChangesAsync();
    await _bus.PublishAsync(new PaymentCompleted(cmd.SagaId));
}
Q: Does a saga guarantee ACID?
No. A saga guarantees ACD (Atomicity via compensations, Consistency eventually, Durability via local commits) but not Isolation. Between steps, other transactions can see intermediate states. Handle with status fields (PENDING → CONFIRMED).
Q: How do you handle the orchestrator crashing?
The orchestrator persists saga state to DB before every step. Run multiple instances for availability. On restart, any instance queries for IN_PROGRESS sagas and resumes. Commands must be idempotent so retries are safe.
Q: What's the difference between a compensating transaction and a rollback?
A rollback undoes uncommitted changes automatically. A compensation is a new committed transaction that semantically reverses a previous one — you write the logic yourself (e.g., calling Stripe's refund API, not "un-charging").

TOPIC 07

Database per Service + CQRS

The Rule: Database per Service

Each service owns its database privately. No other service can access it directly — not through SQL, not through a shared connection string, not through a back door. If Service A needs data from Service B, it must ask via API or consume an event.

Why Shared Database Breaks Microservices

  • Tight coupling — Inventory adds a column to Products → Orders service SQL breaks → forced to deploy together
  • No independent scaling — search queries hammer the DB → Orders writes slow down → one bottleneck for everyone
  • No technology freedom — search needs Elasticsearch, Inventory needs a document store, but everyone is stuck on one PostgreSQL
A shared database makes your microservices a distributed monolith
You get all the operational complexity of microservices with none of the independence benefits.

Polyglot Persistence — Pick the Right Tool

ServiceWorkloadBest Fit
OrdersTransactions, relations, ACIDPostgreSQL / SQL Server
InventoryFast reads, flexible schemaMongoDB
SearchFull-text search, faceted queriesElasticsearch
SessionsKey-value lookups, TTLRedis
AnalyticsAppend-only events, time-seriesClickHouse / Cassandra

Data Sharing via Events (Not Direct DB Access)

Wrong: Orders queries Catalogue's DB directly with SELECT name FROM products. If Catalogue renames the column → Orders breaks at runtime.

Right: Catalogue publishes ProductUpdated events. Orders consumes them and stores a local copy (ProductCache table) with only the fields it needs. Zero coupling to Catalogue — data is eventually consistent.

CQRS — Command Query Responsibility Segregation

The deeper problem: reads and writes want different data shapes.

WRITE: UPDATE Products SET Price = 29.99 WHERE Id = 'prod-123'
  → 1 table, 1 row. Needs normalization, ACID. Rare operation.

READ:  SELECT p.Name, p.Price, AVG(r.Rating), COUNT(r.Id),
              i.StockLevel, c.Name as CategoryName
       FROM Products p JOIN Reviews r ... JOIN Inventory i ...
  → 4 JOINs, aggregations, 100K times/second. Must be FAST.

CQRS splits your application into two separate models:

The Full CQRS Flow in C#

Command Handler (write side):

public class UpdateProductPriceHandler
{
    private readonly WriteDbContext _db;
    private readonly IEventBus _bus;

    public async Task Handle(UpdateProductPriceCommand cmd)
    {
        var product = await _db.Products.FindAsync(cmd.ProductId);
        if (cmd.NewPrice <= 0) throw new ValidationException("Price must be positive");

        product.Price = cmd.NewPrice;
        await _db.SaveChangesAsync();

        // Publish event — read side picks this up
        await _bus.PublishAsync(new ProductPriceUpdated(
            cmd.ProductId, cmd.NewPrice, product.Name));
    }
}

Query Handler (read side):

public class GetProductDetailsHandler
{
    private readonly ReadDbContext _readDb;

    public async Task<ProductDetailsDto> Handle(GetProductDetailsQuery query)
    {
        // Single read, no JOINs — everything pre-joined in the view
        return await _readDb.ProductViews
            .FirstOrDefaultAsync(p => p.ProductId == query.ProductId);
    }
}

The Read Model — Denormalized View:

public class ProductView  // in the Read DB
{
    public string ProductId { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public string CategoryName { get; set; }    // pre-joined from Catalogue!
    public double AverageRating { get; set; }   // pre-calculated from Reviews!
    public int ReviewCount { get; set; }        // pre-calculated!
    public int StockLevel { get; set; }         // from Inventory events!
}

Event Handler — bridges write to read:

public class ProductReadModelUpdater
{
    public async Task Handle(ProductPriceUpdated evt)
    {
        var view = await _readDb.ProductViews.FindAsync(evt.ProductId);
        view.Price = evt.NewPrice;
        await _readDb.SaveChangesAsync();
    }

    // Also handles events from OTHER services
    public async Task Handle(ReviewPosted evt)
    {
        var view = await _readDb.ProductViews.FindAsync(evt.ProductId);
        view.ReviewCount++;
        view.AverageRating = evt.NewAverageRating;
        await _readDb.SaveChangesAsync();
    }

    public async Task Handle(StockLevelChanged evt)
    {
        var view = await _readDb.ProductViews.FindAsync(evt.ProductId);
        view.StockLevel = evt.NewLevel;
        await _readDb.SaveChangesAsync();
    }
}

The read model handler consumes events from multiple services and builds one flat, pre-joined view. The user's query hits this single table — no JOINs, blazing fast.

When NOT to Use CQRS
Simple CRUD with one entity. Read/write patterns are similar. Small scale. Team is small. Most services don't need CQRS — reserve it for read-heavy, cross-service aggregation scenarios (product pages, dashboards, search results, news feeds).
Q: What's the tradeoff of CQRS?
Eventual consistency — the read model lags behind writes (milliseconds to seconds). Increased complexity: two databases, event handlers, sync logic. Only worth it when read/write patterns are significantly different or when aggregating data from multiple services.
Q: How does CQRS connect to the Saga pattern?
Sagas handle distributed writes (coordinating transactions across services). CQRS handles distributed reads (building query-optimized views from events published by multiple services). They often work together — a Saga orchestrates the write flow, and the events feed CQRS read models.

TOPIC 08

Event Sourcing

The Problem

Traditional databases store current state only. Product price is $29.99. What was it yesterday? Last month? Who changed it and why? You don't know — the previous state was overwritten. Like editing a Word document without Track Changes.

Core Idea: Store What HAPPENED, Not What Things ARE

Event 1: ProductCreated  { id: "prod-123", name: "Gaming Mouse", price: 19.99 }
Event 2: PriceChanged    { id: "prod-123", newPrice: 24.99, reason: "demand" }
Event 3: PriceChanged    { id: "prod-123", newPrice: 29.99, reason: "new model" }
Event 4: ReviewPosted    { id: "prod-123", rating: 5, userId: "user-abc" }

Current state = replay all events: $19.99 → $24.99 → $29.99 = $29.99

This is exactly how Git works — Git doesn't store "the current file." It stores every commit (event). Current file = replay all commits.

The Apply Pattern in C#

public class Product
{
    public string Id { get; private set; }
    public decimal Price { get; private set; }
    public bool IsActive { get; private set; }
    private readonly List<object> _pendingEvents = new();

    // Rebuild from history
    public Product(IEnumerable<object> eventHistory)
    {
        foreach (var evt in eventHistory) Apply(evt);
    }

    // Business operations produce events, NOT state changes
    public void ChangePrice(decimal newPrice, string reason)
    {
        if (!IsActive) throw new InvalidOperationException("Inactive product");
        if (newPrice <= 0) throw new ValidationException("Must be positive");
        var evt = new PriceChanged(Id, Price, newPrice, reason, DateTime.UtcNow);
        Apply(evt);
        _pendingEvents.Add(evt);
    }

    // The ONLY place state changes
    private void Apply(object evt)
    {
        switch (evt)
        {
            case ProductCreated e:
                Id = e.ProductId; Price = e.Price; IsActive = true; break;
            case PriceChanged e:
                Price = e.NewPrice; break;
            case ProductDeactivated:
                IsActive = false; break;
        }
    }
}

Using It Together

public async Task ChangePriceAsync(string productId, decimal newPrice, string reason)
{
    // 1. Load all events for this product
    var events = await _store.GetEventsAsync($"product-{productId}");
    // 2. Rebuild current state by replaying events
    var product = new Product(events);
    // 3. Execute business logic (produces new events)
    product.ChangePrice(newPrice, reason);
    // 4. Save new events to the store
    await _store.SaveEventsAsync($"product-{productId}", product.GetPendingEvents());
    // 5. Publish events — CQRS read model handlers pick these up
    foreach (var evt in product.GetPendingEvents())
        await _bus.PublishAsync(evt);
}

Projections — Multiple Views from Same Events

One PriceChanged event feeds multiple projections simultaneously:

  • Product detail projection — updates the product page view
  • Price history projection — adds a row to the price timeline for analytics
  • Revenue forecast projection — recalculates projected revenue

Need a new projection six months later? Replay all events from the store → new projection is fully populated with historical data.

Snapshots — Performance Optimization

If a product has 10,000 events, replaying all is slow. A snapshot captures state at a specific version. To load: read snapshot at event 9,900, replay only the last 100.

Event Sourcing + CQRS Together

Event Store (source of truth, append-only)
    → publishes events to Event Bus (Kafka)
    → Projection Handlers build denormalized Read DB views
    
Read DB corrupted? Delete it, replay events → fully restored
Need new field in view? Replay events → populated with all history
Add new projection? Replay events → ready from day one
When NOT to Use Event Sourcing
Simple CRUD. Team doesn't need replay power. Events would be trivial (just "FieldUpdated"). Schema evolution of events is hard to manage. Most services should use a regular database + CQRS. Reserve event sourcing for domains where history is a business requirement — finance, medical records, audit-heavy systems.
Q: What's the hardest part of Event Sourcing?
Event schema evolution. Once stored, you can't change an event's structure — it's an immutable fact. Need a new field? Create PriceChangedV2 and handle both versions in your Apply method. Over time, managing multiple event versions adds complexity.
Q: How does Event Sourcing relate to CQRS?
They're independent but pair naturally. Event sourcing provides the write side (events appended to store). Those events feed projection handlers that build read models. The killer benefit: if you need a new read model or the existing one is corrupted, replay all events to rebuild it.

Observability

The Problem

In a monolith: one server, one log file, grep "error" app.log, done. In microservices: 10 servers, 10 log streams. A user reports "my order failed" — which service caused it? Where in the chain did it break?

With Console.WriteLine: SSH into each server, read each log, try to match timestamps manually. 1000s of requests per second — which log line belongs to THIS user? Containers restart — logs are GONE.

The Correlation ID

A unique ID generated at the API Gateway, propagated through every service via X-Correlation-Id header. Every log line includes it. Search for one ID → see the entire request journey across all services in chronological order.

// Middleware — extract or create correlation ID
public class CorrelationIdMiddleware
{
    public async Task InvokeAsync(HttpContext context)
    {
        var id = context.Request.Headers["X-Correlation-Id"].FirstOrDefault()
                 ?? Guid.NewGuid().ToString();
        context.Response.Headers["X-Correlation-Id"] = id;

        using (LogContext.PushProperty("CorrelationId", id))
            await _next(context);
    }
}

// When calling another service — forward it
request.Headers.Add("X-Correlation-Id", correlationId);

Q: What is a Correlation ID and why is it critical?
A unique ID generated at the entry point and propagated through every service via HTTP headers. Every log line includes it. To debug a failed request, search for that one ID → see the complete journey across all services in order. Without it, matching logs across 10 services at 1000 req/sec is impossible.
Q: What is OpenTelemetry?
A vendor-neutral standard for telemetry. You instrument once with OTel SDK, export to any backend (Jaeger, Prometheus, Datadog) by changing config. It replaces client libraries, not backends themselves. Prevents vendor lock-in — the industry standard supported by all major cloud providers.

Security Across Services

Part 1 — External: JWT Authentication

In microservices, session-based auth doesn't work — which service stores the session? Solution: stateless tokens. The user carries their identity in every request as a JWT.

A JWT has 3 parts: Header (algorithm), Payload (claims — userId, role, expiry), Signature (proves it wasn't tampered). The Identity Server signs with its PRIVATE key. Any service verifies using the PUBLIC key — no network call needed.

Auth Flow

User logs in
→
Identity Server returns JWT
→
JWT sent with every request
→
Gateway verifies locally
→
Forwards to services
// Gateway — verify JWT (no call to Identity Server!)
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://identity-server.example.com";
        options.Audience = "api-gateway";
        // SDK auto-fetches + caches public key from .well-known/openid-configuration
    });

JWT Propagation — Two Strategies

Strategy A: Forward Full JWTStrategy B: Extract Claims at Gateway
Each service re-verifies the signatureGateway stamps X-User-Id, X-User-Role headers
More secure — every service validatesSimpler — services just read headers
Adds verification overhead per serviceRequires trusted internal network

Part 2 — Internal: mTLS (Mutual TLS)

Regular TLS: client verifies server (one-way). mTLS: both sides verify each other (two-way). A rogue service without a valid certificate is rejected. All traffic is encrypted — sniffing captures gibberish.

Without mTLS: attacker gets into network → deploys fake service → calls Payments directly → steals money. Internal traffic is unencrypted → attacker sniffs credit card data.

With mTLS: fake service has no valid cert → Payments rejects. All traffic encrypted → sniffing captures nothing.

Part 3 — Authorization (RBAC vs Policy-Based)

// RBAC — simple, covers 80% of cases
[Authorize(Roles = "admin")]
[HttpDelete("/api/products/{id}")]
public async Task<IActionResult> DeleteProduct(string id) { ... }

// Policy-based — flexible, handles complex rules
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("CanDeleteProducts", policy =>
        policy.RequireRole("admin")
              .RequireClaim("department", "catalog-team"));
});

Part 4 — Service Mesh (Istio, Linkerd)

A sidecar proxy (Envoy) deployed alongside each service handles mTLS, certificate rotation, and authorization policies. Your app sends plain HTTP to localhost — the proxy handles all security. Zero code changes.

Zero-Trust Principle
Never trust, always verify — even inside your own network. Every call is authenticated and authorized regardless of origin. mTLS for service identity, JWT at every boundary, least-privilege policies, encrypted traffic everywhere. If an attacker compromises one service, zero-trust limits the blast radius.
Q: How do you handle authentication in microservices?
Users authenticate against an Identity Server and receive a JWT. The API Gateway verifies the JWT signature using the cached public key — no network call. Internally, either the full JWT is propagated or the Gateway extracts claims and forwards them as trusted headers.
Q: What is mTLS and why do microservices need it?
Mutual TLS — both client and server present certificates. Ensures only legitimate services communicate. Also encrypts all internal traffic. Foundation of zero-trust: don't trust the network, verify every caller.

TOPIC 11

Deployment & Containers

The Problem

In a monolith: one deploy. In microservices: 20+ services deployed independently. How do you update a running service without users noticing?

Strategy 1 — Rolling Deploy (Kubernetes Default)

K8s replaces pods one at a time. Each new pod must pass its readiness probe before K8s terminates the next old pod. During transition, both v1 and v2 serve traffic.

Before:  [v1] [v1] [v1]
Step 1:  [v2] [v1] [v1]   ← new pod created, old pod terminated
Step 2:  [v2] [v2] [v1]   ← next pod replaced
Step 3:  [v2] [v2] [v2]   ← deploy complete
# K8s deployment config
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1        # create 1 extra pod during update
    maxUnavailable: 0   # never fewer than 3 running pods

Rollback: kubectl rollout undo deploy/orders (slow — re-rolls all pods).

Strategy 2 — Blue-Green Deploy

Two full environments: Blue (current live) and Green (new version). Green is deployed and tested while Blue serves traffic. When ready, switch the load balancer from Blue to Green instantly.

Before:  Users → LB → Blue (v1) [LIVE]     Green (empty)
Deploy:  Users → LB → Blue (v1) [LIVE]     Green (v2) [testing]
Switch:  Users → LB → Green (v2) [LIVE]    Blue (v1) [standby]
Rollback: Users → LB → Blue (v1) [LIVE]    Green (v2) [investigate]

Pro: instant rollback (Blue never stopped). Con: 2× resource cost during deployment.

Strategy 3 — Canary Deploy

Deploy new version alongside old. Gradually shift traffic: 5% → 25% → 50% → 100%. Monitor error rates at each stage. If anything breaks, route all traffic back — only the small % on the canary was affected.

Stage 1:  [████████████████████v1 95%████████████████████][v2 5%]
Stage 2:  [███████████████v1 75%███████████████][████v2 25%████]
Stage 3:  [██████████v1 50%██████████][██████████v2 50%██████████]
Stage 4:  [████████████████████████v2 100%████████████████████████]

Pro: smallest blast radius, safest. Con: requires traffic splitting infrastructure (Istio, Nginx, feature flag).

CI/CD for Microservices

Each service has its own independent pipeline, triggered only by changes to its codebase:

# GitHub Actions — triggers only for this service
name: Deploy Orders Service

  push:
    branches: [main]

Q: Which strategy for a payment processing service?
Blue-green or canary — never simple rolling deploy. With blue-green, fully test the new version with synthetic transactions before switching. With canary, route 1% of payment traffic first and monitor success rates. Rolling deploy is risky because both versions serve traffic — a subtle bug in payments could affect real money before you catch it.
Q: How does CI/CD change with microservices?
Each service has its own independent pipeline triggered only by changes to its own codebase (via path filters). Pipeline: test → build Docker image (tagged with commit SHA) → push to registry → deploy to K8s. Teams deploy independently — a change to Orders doesn't trigger or affect Payments.

Common Interview Traps

Trap: "Microservices are always better than monoliths"
Wrong. They're a tradeoff. Start monolith, migrate when you hit concrete pain. A distributed monolith is the worst of both worlds.
Trap: "Use a distributed transaction (2PC) across services"
2PC blocks all participants, doesn't scale, creates tight coupling. Use the Saga pattern — local transactions + compensations.
Trap: "Services can share a database for simplicity"
That's a distributed monolith. Schema changes in one service break all others. Each service must own its data privately.
Trap: "Retry everything on failure"
Only retry transient errors (503, 429, timeouts). Never retry 400 or 401. Always use exponential backoff + jitter to prevent thundering herd.
Trap: "Internal services trust each other"
Zero-trust: never trust, always verify. mTLS between services, encrypt all traffic, least-privilege authorization even internally.
Trap: "Console.WriteLine is good enough for logging"
In microservices, logs are on 10 different servers, containers restart (logs vanish), 1000s of requests per second. You need structured logging (Serilog), centralized collection (ELK/Seq), and correlation IDs to trace one request across all services.
