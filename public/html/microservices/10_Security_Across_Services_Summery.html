02 Part 1 — External Security: JWT

Why not sessions?

In a monolith, you store user sessions server-side (cookies + session store). In microservices: which service stores the session? They all need it. The answer: stateless tokens — the user carries their identity with every request as a JWT.

JWT Anatomy — 3 Parts

Click each part to see what's inside:

eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyLTQ1NiIsInJvbGUiOiJhZG1pbiJ9.SflKxwRJSMeKKF2QT4fw...
Part 1 — Header (algorithm + type)
{ "alg": "RS256", // signing algorithm "typ": "JWT" }
Part 2 — Payload (claims = the actual data)
{ "sub": "user-456", // subject (who) "role": "admin", // authorization claim "email": "omar@example.com", "iat": 1711452000, // issued at "exp": 1711455600 // expires (1 hour) }
Part 3 — Signature (proves it wasn't tampered)
RS256( base64(header) + "." + base64(payload), PRIVATE_KEY // only the Identity Server has this )
The key insight: The Identity Server signs with its PRIVATE key. Any service can verify the signature using the PUBLIC key — without calling the Identity Server. That's why it's stateless — no central session store needed. The payload is NOT encrypted — anyone can decode it with base64. The signature only proves it wasn't modified. Never put passwords or credit cards in a JWT.

Stateless verification! Gateway decodes JWT, verifies signature using Identity Server's PUBLIC key (cached locally). No network call to Identity Server needed.

// Gateway — verify JWT (auto-fetches + caches public key)
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://identity-server.example.com";
        options.Audience = "api-gateway";
        // SDK fetches from .well-known/openid-configuration
    });

Step 3: JWT propagated to internal services

Two strategies for passing identity internally:

Strategy A: Forward full JWT
Each service re-verifies the signature
More secure — every service validates
Adds verification overhead per service
Strategy B: Extract claims at Gateway
Gateway stamps X-User-Id, X-User-Role headers
Simpler — services just read headers
Requires trusted internal network
// Gateway stamps userId into header
if (context.User.Identity?.IsAuthenticated == true)
{
    var userId = context.User.FindFirst("sub")?.Value;
    context.Request.Headers["X-User-Id"] = userId;
}

// Orders service — trusts the header, never touches a JWT
[HttpPost]
public async Task<IActionResult> PlaceOrder()
{
    var userId = Request.Headers["X-User-Id"].ToString();
    // ... use userId directly
}

Gateway verifies JWT once. Internal calls carry X-User-Id header. Identity Server only involved at login — not on every request.

The Identity Server's public key is cached at the Gateway on startup (fetched from .well-known/openid-configuration). So JWT verification is a local CPU operation — no network call. This is why it scales to millions of requests.

03 Part 2 — Internal Security: mTLS

You know regular TLS — when you visit https://google.com, your browser verifies Google's certificate. That's one-way. mTLS is two-way.

Regular TLS — client verifies server only

Browser (client)
Identity: unknown
Anyone can be a client
Google (server)
Identity: verified by certificate
"I am who I say I am"

Server proves who it is. Client is anonymous. Used for: HTTPS websites, public APIs.

mTLS — both sides verify each other

Orders service
Identity: VERIFIED
"Show YOUR certificate"
Payments service
Identity: VERIFIED
"Show YOUR certificate too"

Both services prove identity before any data is exchanged. A rogue service without a valid certificate is rejected. All traffic is encrypted.

The attacker scenario

Without mTLS
Attacker gets into network
→ Deploys fake service
→ Calls Payments directly → steals money
→ Sniffs unencrypted traffic → reads credit cards
With mTLS
Fake service has no valid cert
→ Payments REJECTS the connection
→ All traffic encrypted
→ Sniffing captures only gibberish
This is the foundation of zero-trust: don't trust the network, verify every caller. Even if an attacker compromises one service, mTLS limits the blast radius.

04 Part 3 — Authorization

Authentication = "who are you?" Authorization = "what are you allowed to do?"

Role-Based Access Control — simple, covers 80% of cases

// Check role from JWT claims
[Authorize(Roles = "admin")]
[HttpDelete("/api/products/{id}")]
public async Task<IActionResult> DeleteProduct(string id)
{
    // Only users with role=admin in their JWT can reach this
    await _productService.DeleteAsync(id);
    return NoContent();
}

[Authorize(Roles = "admin,manager")]
[HttpPut("/api/products/{id}/price")]
public async Task<IActionResult> UpdatePrice(string id, UpdatePriceRequest req)
{
    // admin OR manager can update prices
    await _productService.ChangePriceAsync(id, req.NewPrice);
    return Ok();
}

Policy-Based — flexible, handles complex rules

// Program.cs — define custom policies
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("CanDeleteProducts", policy =>
        policy.RequireRole("admin")
              .RequireClaim("department", "catalog-team"));

    options.AddPolicy("CanViewReports", policy =>
        policy.RequireAssertion(context =>
        {
            var user = context.User;
            var role = user.FindFirst("role")?.Value;
            var tier = user.FindFirst("subscription_tier")?.Value;

            // Admin always can, or premium users in analytics dept
            return role == "admin" ||
                   (tier == "premium" &&
                    user.HasClaim("department", "analytics"));
        }));
});

// Usage
[Authorize(Policy = "CanDeleteProducts")]
[HttpDelete("/api/products/{id}")]
public async Task<IActionResult> DeleteProduct(string id) { ... }

05 Part 4 — Service Mesh

Managing mTLS certificates, authorization policies, and traffic encryption across 20 services is a lot of work in code. A service mesh handles all of it automatically.

Sidecar proxy pattern

A sidecar proxy (Envoy) is deployed alongside each service inside the same pod. All network traffic goes through the proxy — your app doesn't even know it's there.

Orders pod
Orders app (your code)
↕ plain HTTP (localhost)
Envoy proxy (sidecar)
mTLS
encrypted
Payments pod
Payments app (your code)
↕ plain HTTP (localhost)
Envoy proxy (sidecar)

What the mesh handles (zero code changes)

  • mTLS certificates — auto-generated, auto-rotated. You never touch a cert file.
  • Authorization policies — "Orders CAN call Payments, Search service CANNOT"
  • Traffic encryption — every byte encrypted between proxies
  • Observability — traces and metrics collected automatically
  • Traffic management — retries, timeouts, circuit breaking at the proxy level

Your code vs what the proxy does

Your application codeEnvoy sidecar proxy
Sends plain HTTP to localhostIntercepts, encrypts with mTLS
No certificate managementAuto-generates + rotates certs
No auth policy codeEnforces "who can call whom" rules
Business logic onlyAll networking concerns

Istio and Linkerd are the two most popular service meshes. In interviews, knowing they exist and what they do is enough.

06 Part 5 — Secrets Management

API keys, DB passwords, Stripe secrets, JWT signing keys — where do you store them?

Never do these

// ❌ NEVER — hardcoded in code
var stripeKey = "sk_live_abc123";

// ❌ NEVER — in appsettings.json committed to Git
{
  "Stripe": {
    "SecretKey": "sk_live_abc123"
  }
}

Anyone with access to the Git repo sees your production secrets. Bots scan GitHub for these patterns and exploit them within minutes.

07 Zero-Trust — Tying It All Together

Traditional: hard shell, soft interior

Firewall protects the perimeter
Everything INSIDE the network is trusted
Services talk to each other freely — no auth, no encryption
Attacker gets past firewall → has access to EVERYTHING

Zero-trust: verify everything, everywhere

Every call authenticated and authorized
mTLS between all services — identity verified on every call
All traffic encrypted — even internal
Least-privilege policies — Orders can call Payments, but Search cannot
Attacker compromises one service → blast radius is LIMITED
Zero-trust in one sentence: Never trust, always verify — even inside your own network. Implemented through JWT at the gateway, mTLS between services, least-privilege authorization policies, encrypted traffic everywhere, and secrets in a vault — not in code.

08 Interview Q&A

Q: How do you handle authentication in microservices?
Users authenticate against an Identity Server (Keycloak, Auth0) and receive a JWT. The JWT is sent with every request. The API Gateway verifies the signature using the Identity Server's cached public key — no network call. Internally, either the full JWT is propagated, or the Gateway extracts claims and forwards them as trusted headers (X-User-Id, X-User-Role).
Q: Why is JWT verification stateless?
The Identity Server signs with a private key. The Gateway verifies using the public key, which is cached locally. No session store, no database lookup, no call to the Identity Server on each request. Just a CPU operation to verify the signature. This is why it scales to millions of requests.
Q: What is mTLS and why do microservices need it?
Mutual TLS — both client and server present certificates. Ensures only legitimate services can communicate — a rogue service without a valid cert is rejected. Also encrypts all internal traffic, preventing eavesdropping. Foundation of zero-trust: don't trust the network, verify every caller.
Q: What is the difference between authentication and authorization?
Authentication = "who are you?" (verify identity via JWT). Authorization = "what can you do?" (check permissions via roles or policies). A user can be authenticated (valid JWT) but not authorized (doesn't have the admin role to delete products).
Q: What is a service mesh?
An infrastructure layer (Istio, Linkerd) that handles service-to-service security, traffic management, and observability without changing application code. Deploys a sidecar proxy (Envoy) alongside each service. The proxy handles mTLS, cert rotation, auth policies, and encryption. Your app sends plain HTTP to localhost — the proxy handles everything.
Q: Where should secrets be stored?
Never in source code or config files committed to Git. In production: dedicated secrets manager (Azure Key Vault, AWS Secrets Manager, HashiCorp Vault). For K8s: K8s Secrets with encryption at rest. The application reads secrets at startup from the manager, never from hardcoded strings.
Q: What is zero-trust?
Assumes no implicit trust based on network location. Every request — even between internal services — must be authenticated and authorized. Implemented through mTLS (service identity), JWT at every boundary, least-privilege policies, encrypted traffic everywhere. If an attacker compromises one service, zero-trust limits the blast radius.
Q: JWT payload is not encrypted — isn't that a security risk?
The payload is base64-encoded, not encrypted — anyone can read it. But the signature proves it wasn't tampered with. This is by design: the token carries non-sensitive claims (userId, role, expiry). Never put passwords, credit cards, or sensitive data in a JWT. If you need encrypted tokens, use JWE (JSON Web Encryption) — but this is rare.
