01 The Cascading Failure Problem

Service A calls Service B. B is dead. Without protection: threads in A pile up → A dies → everything upstream dies. One dead service takes down the whole system.

Orders (healthy)
→
Payments (healthy)
→
Stripe (DOWN)

Payments waits for Stripe → threads pile up → Orders waits for Payments → Users see 500 errors

02 The 3 States

CLOSED — normal operation
All calls go through. Failures counted in a sliding window.
Transitions to OPEN when failure rate exceeds threshold (e.g., 50% of last 10 calls).
OPEN — blocking all calls
All calls fail immediately — NO network call made. Returns fallback response.
Transitions to HALF-OPEN after recovery timeout (e.g., 30 seconds).
HALF-OPEN — testing recovery
ONE probe request allowed through to test.
If probe succeeds → back to CLOSED. If probe fails → back to OPEN.
CLOSED ──(failure rate > 50%)──→ OPEN
                                   │
                           (wait 30 seconds)
                                   │
CLOSED ←──(probe succeeds)── HALF-OPEN ──(probe fails)──→ OPEN

03 Retry + Timeout + Circuit Breaker in Polly

Timeout — limit each individual call

// Bad — no timeout, thread hangs indefinitely
var response = await _http.GetAsync("http://payments/api/charge");

// Good — fail after 3 seconds
var cts = new CancellationTokenSource(TimeSpan.FromSeconds(3));
var response = await _http.GetAsync("http://payments/api/charge", cts.Token);

Retry with exponential backoff + jitter

Attempt 1: immediate
Attempt 2: wait 200ms + random(0-50ms)
Attempt 3: wait 400ms + random(0-50ms)
Attempt 4: wait 800ms + random(0-50ms)

Why jitter? Without it, all instances retry at identical intervals — a synchronized storm that overwhelms the recovering service. Jitter spreads the load.

builder.Services
    .AddHttpClient<IPaymentsClient, PaymentsClient>()
    .AddResilienceHandler("payments-pipeline", builder =>
    {
        builder.AddTimeout(TimeSpan.FromSeconds(3));

        builder.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            Delay = TimeSpan.FromMilliseconds(200),
            BackoffType = DelayBackoffType.Exponential,
            UseJitter = true,
            ShouldHandle = args => ValueTask.FromResult(
                args.Outcome.Result?.StatusCode
                    is HttpStatusCode.ServiceUnavailable
                    or HttpStatusCode.TooManyRequests)
        });

        builder.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            FailureRatio = 0.5,
            SamplingDuration = TimeSpan.FromSeconds(30),
            MinimumThroughput = 10,
            BreakDuration = TimeSpan.FromSeconds(30)
        });
    });
Order: Circuit Breaker → Retry → Timeout

Circuit breaker open? → fail immediately, no network call
Attempt 1 with 3s timeout → fails
Retry: wait 200ms + jitter
Attempt 2 with 3s timeout → fails
Retry: wait 400ms + jitter
Attempt 3 with 3s timeout → fails
Failure recorded → circuit breaker checks rate
Never retry these:
400 Bad Request — payload is wrong, retrying won't fix it
401 Unauthorized — token is invalid
404 Not Found — resource doesn't exist

Only retry: 503 Service Unavailable, 429 Too Many Requests, 504 Gateway Timeout, network failures

04 Fallback + Bulkhead

public async Task<PaymentResult> ChargeAsync(PaymentRequest request)
{
    try { return await _paymentsClient.ChargeAsync(request); }
    catch (BrokenCircuitException)
    {
        return PaymentResult.ServiceUnavailable(
            "Payment processing is temporarily unavailable.");
    }
}

Bulkhead: isolate thread pools per dependency. One slow service can't consume all threads. Named after ship compartments — one floods, others survive.

05 Interview Q&A

Q: Circuit breaker vs timeout?
Timeout limits one call. Circuit breaker tracks history and stops all calls when service is clearly unhealthy. They work together: timeouts generate the failures the circuit breaker counts.
Q: Why does retry need jitter?
Without jitter, all instances retry at identical intervals — a synchronized storm. Jitter (random ±50ms) spreads retry load across time, giving the service room to recover.
