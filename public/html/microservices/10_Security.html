02 Part 1 — JWT Deep Dive

In a monolith, you store sessions server-side (cookies + session store). In microservices that doesn't work — which service stores the session? The answer: stateless tokens. The user carries their identity in every request as a JWT.

A JWT is a self-contained, signed token with 3 parts separated by dots:

eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyLTQ1NiIsInJvbGUiOiJhZG1pbiJ9.SflKxwRJSMeKKF2QT4fw...
Part 1 — Header (algorithm + type)
{ "alg": "RS256", // signing algorithm "typ": "JWT" }
Part 2 — Payload (claims = the actual data)
{ "sub": "user-456", // subject (who) "role": "admin", // authorization claim "email": "omar@example.com", "iat": 1711452000, // issued at "exp": 1711455600 // expires (1 hour) }
Part 3 — Signature (proves it wasn't tampered with)
RS256( base64(header) + "." + base64(payload), PRIVATE_KEY // only the Identity Server has this )
Key insight: The Identity Server signs with its PRIVATE key. Any service can verify the signature using the PUBLIC key — without calling the Identity Server. That's why it's stateless — no central session store needed.
Important: The payload is NOT encrypted — anyone can decode it with base64. The signature only proves it wasn't modified. Never put passwords or credit card numbers inside a JWT.

04 JWT Verification in C#

// Gateway — verify JWT on every request
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://identity-server.example.com";
        options.Audience = "api-gateway";
        // SDK automatically fetches public key from
        // /.well-known/openid-configuration and caches it
    });

app.UseAuthentication();
app.UseAuthorization();

app.MapReverseProxy(proxyPipeline =>
{
    proxyPipeline.Use(async (context, next) =>
    {
        // Gateway stamps the verified userId into a header
        // Downstream services read this — no token parsing needed
        if (context.User.Identity?.IsAuthenticated == true)
        {
            var userId = context.User.FindFirst("sub")?.Value;
            context.Request.Headers["X-User-Id"] = userId;
        }
        await next();
    });
});
// Orders service — trusts the header, never touches a JWT
[HttpPost]
public async Task<IActionResult> PlaceOrder()
{
    // Gateway already verified the token — just read the header
    var userId = Request.Headers["X-User-Id"].ToString();
    // ... use userId directly
}
The gateway is the trust boundary. Inside the network, services communicate freely without re-authenticating. The Gateway verifies once, stamps X-User-Id, and downstream services trust that header.

05 Part 2 — mTLS (Mutual TLS)

You know regular TLS — when you visit https://google.com, your browser verifies Google's certificate. That's one-way TLS. mTLS is two-way: the server also verifies the client.

Browser (client)              Google (server)
identity: unknown             identity: verified

    → "Show me your certificate" →
    ← "Here it is, signed by a trusted CA" ←

Server proves who it is. Client is anonymous.
Used for: HTTPS websites, public APIs
Orders service                Payments service
identity: verified            identity: verified

    → "Show YOUR certificate" →
    ← "Show YOUR certificate too" ←

Both services prove identity before any data is exchanged.
A rogue service without a valid certificate is REJECTED.
Used for: service-to-service calls inside microservices
Without mTLS — attacker gets into the network:
Deploys a fake service → calls Payments directly → steals money.
Internal traffic is unencrypted → attacker sniffs credit card data.
With mTLS — attacker is blocked:
Fake service has no valid certificate → Payments rejects the connection.
All traffic is encrypted → sniffing captures only gibberish.

This is the foundation of zero-trust: don't trust the network, verify every caller.

06 Part 3 — Authorization (Who Can Do What)

Authentication says "who are you?" Authorization says "what are you allowed to do?"

// === Role-Based Access Control (RBAC) — simple, common ===

[Authorize(Roles = "admin")]
[HttpDelete("/api/products/{id}")]
public async Task<IActionResult> DeleteProduct(string id)
{
    // Only users with role=admin in their JWT can reach this
    await _productService.DeleteAsync(id);
    return NoContent();
}

[Authorize(Roles = "admin,manager")]
[HttpPut("/api/products/{id}/price")]
public async Task<IActionResult> UpdatePrice(string id, UpdatePriceRequest req)
{
    // admin OR manager can update prices
    await _productService.ChangePriceAsync(id, req.NewPrice);
    return Ok();
}
// === Policy-Based Access Control — flexible, powerful ===

// Program.cs — define policies
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("CanDeleteProducts", policy =>
        policy.RequireRole("admin")
              .RequireClaim("department", "catalog-team"));

    options.AddPolicy("CanViewReports", policy =>
        policy.RequireAssertion(context =>
        {
            var user = context.User;
            var role = user.FindFirst("role")?.Value;
            var tier = user.FindFirst("subscription_tier")?.Value;

            // Admin always can, or premium users in analytics dept
            return role == "admin" ||
                   (tier == "premium" &&
                    user.HasClaim("department", "analytics"));
        }));
});

// Usage
[Authorize(Policy = "CanDeleteProducts")]
[HttpDelete("/api/products/{id}")]
public async Task<IActionResult> DeleteProduct(string id) { ... }
RBACPolicy-Based
Check role: admin, manager, userCheck any combination of claims
"Only admins can delete""Only premium users in analytics dept"
Simple, covers 80% of casesFlexible, handles complex rules

07 Part 4 — Service Mesh (Security as Infrastructure)

Managing mTLS certificates, authorization policies, and traffic encryption across 20 services is a lot of work if you do it in code. A service mesh handles all of it automatically.

┌─────────────────────────┐       ┌─────────────────────────┐
│  Orders pod             │       │  Payments pod           │
│                         │       │                         │
│  ┌───────────────┐      │       │      ┌───────────────┐  │
│  │  Orders app   │      │       │      │ Payments app  │  │
│  └───────┬───────┘      │       │      └───────┬───────┘  │
│          │ plain HTTP    │       │    plain HTTP │          │
│  ┌───────▼───────┐      │       │      ┌───────▼───────┐  │
│  │  Envoy proxy  │ ═══════mTLS encrypted═══════▶ Envoy proxy  │  │
│  │  (sidecar)    │      │       │      │  (sidecar)    │  │
│  └───────────────┘      │       │      └───────────────┘  │
└─────────────────────────┘       └─────────────────────────┘
What the mesh handles (zero code changes):
• mTLS certificates — auto-generated, auto-rotated
• Authorization policies — "Orders CAN call Payments, Search CANNOT"
• Traffic encryption — every byte encrypted between proxies
• Your app code sends plain HTTP → proxy handles all security

Istio and Linkerd are the two most popular service meshes. In interviews, knowing they exist and what they do is enough — you don't need to know how to configure them.

08 Part 5 — Secrets Management

API keys, DB passwords, Stripe secrets, JWT signing keys — where do you store them?

ApproachSecurityWhen to use
Hardcoded in codeTerribleNever
appsettings.json in GitTerribleNever
Environment variablesOK for devDocker Compose, simple deploys
K8s SecretsGoodKubernetes environments
Azure Key Vault / HashiCorp VaultBestProduction
// ❌ NEVER — hardcoded in code
var stripeKey = "sk_live_abc123";

// ❌ NEVER — in appsettings.json committed to Git
{ "Stripe": { "SecretKey": "sk_live_abc123" } }

// ⚠️ OK for dev only — environment variables
var stripeKey = Environment.GetEnvironmentVariable("STRIPE_SECRET_KEY");
// ✅ BEST — dedicated secrets manager
// Azure Key Vault
builder.Configuration.AddAzureKeyVault(
    new Uri("https://my-vault.vault.azure.net/"),
    new DefaultAzureCredential());

// Then access like any config — no secret in code or Git
var stripeKey = builder.Configuration["StripeSecretKey"];
The application reads secrets at startup from the secrets manager. Never from environment variables in production. The vault handles access control, rotation, and audit logging.

09 Zero-Trust Principle — Tying It All Together

Everything in this topic connects to one philosophy:

Never trust, always verify. Even inside your own network.
Traditional securityZero-trust
Hard shell (firewall), soft interiorEvery call authenticated + authorized
Once past the firewall → trustedNo implicit trust based on location
Attacker past firewall → full accessAttacker compromises one service → blast radius limited

How all 5 parts implement zero-trust:

Part 1 (JWT)     → Users prove identity with signed tokens
Part 2 (mTLS)    → Services prove identity with certificates
Part 3 (AuthZ)   → Every endpoint checks permissions
Part 4 (Mesh)    → Infrastructure enforces policies automatically
Part 5 (Secrets) → Credentials never exposed in code or config

10 Interview Q&A

Q: How do you handle authentication in microservices?
Users authenticate against an Identity Server (Keycloak, Auth0) and receive a JWT. The JWT is sent with every request in the Authorization header. The API Gateway verifies the JWT signature using the Identity Server's cached public key — no network call needed. Internally, either the full JWT is propagated to downstream services, or the Gateway extracts claims and forwards them as trusted headers (X-User-Id, X-User-Role).
Q: What is mTLS and why do microservices need it?
Mutual TLS means both the client and server present certificates to verify each other's identity. In microservices, it ensures that only legitimate services can communicate — a rogue service without a valid certificate is rejected. It also encrypts all internal traffic, preventing eavesdropping even if an attacker gains network access. This is the foundation of zero-trust architecture.
Q: What is a service mesh?
A service mesh (Istio, Linkerd) is an infrastructure layer that handles service-to-service security, traffic management, and observability without changing application code. It deploys a sidecar proxy (Envoy) alongside each service. The proxy handles mTLS, certificate rotation, authorization policies, and traffic encryption. Your application sends plain HTTP to localhost — the proxy handles everything.
Q: Where should secrets be stored?
Never in source code or config files committed to Git. In production, use a dedicated secrets manager — Azure Key Vault, AWS Secrets Manager, or HashiCorp Vault. For Kubernetes, K8s Secrets are acceptable with encryption at rest enabled. The application reads secrets at startup from the secrets manager, never from environment variables in production.
Q: What is zero-trust?
Zero-trust assumes no implicit trust based on network location. Every request — even between internal services — must be authenticated and authorized. This is implemented through mTLS for service identity, JWT validation at every boundary, least-privilege authorization policies, and encrypted traffic everywhere. If an attacker compromises one service, zero-trust limits the blast radius.
