Microservices · Topic 02

Sync vs Async Communication

REST, gRPC, Kafka, RabbitMQ, the hybrid pattern. Interactive diagrams and C# code from our conversation.

01 Two Fundamental Approaches

In a monolith, calling another module is just a function call — no network. In microservices, that call crosses a network. You must choose: how should these services talk?

Synchronous — caller waits for response

Orders
→ request →
Payments
→ response →
Orders continues

Like a phone call. Caller blocks until response arrives. Protocols: REST (HTTP/JSON), gRPC (HTTP/2 + Protobuf).

Asynchronous — caller fires and forgets

Orders
→ event →
Message Queue
→ later →
Notifications

Like sending an email. Orders publishes and moves on immediately. Consumer processes when ready. Tools: RabbitMQ, Kafka.

The rule: Sync when you need the answer NOW (stock check, payment charge). Async when you don't (send email, log analytics, award points). Most things don't need an immediate answer.

02 REST vs gRPC

REST — HTTP/1.1 + JSON. Human-readable, universally supported.

public async Task<PaymentResult> ChargeAsync(Order order)
{
    var response = await _http.PostAsJsonAsync(
        "http://payments-service/api/payments",
        new { OrderId = order.Id, Amount = order.Total, Currency = "USD" }
    );
    response.EnsureSuccessStatusCode();
    return await response.Content.ReadFromJsonAsync<PaymentResult>();
}

gRPC — HTTP/2 + Protobuf (binary). 5–10× faster, strict contract.

// payments.proto — shared contract
syntax = "proto3";
service PaymentsService {
  rpc ChargeOrder (ChargeRequest) returns (ChargeResponse);
}
message ChargeRequest {
  string order_id = 1;
  double amount = 2;
  string currency = 3;
}
// C# client — strongly typed, compiler catches contract mismatches
return await _client.ChargeOrderAsync(new ChargeRequest
{
    OrderId = order.Id, Amount = order.Total, Currency = "USD"
});

03 Kafka vs RabbitMQ

RabbitMQ — message queue

Messages pushed to consumers and deleted after acknowledged. Great for task distribution and work queues.

Orders → [Queue] → Payments reads it → message GONE

Kafka — event log

Events written to an append-only log. Consumers read at their own pace. Events retained (days/weeks/forever). Multiple consumer groups read independently.

Orders → [Log] → Payments reads at offset 1042
                → Analytics reads at offset 1042
                → Inventory reads at offset 1042
                → All three independently, event PRESERVED

04 The Hybrid Pattern (Real-World)

1. Client → Gateway         [sync REST]    validate JWT
2. Gateway → Orders         [sync REST]    create order
3. Orders → Inventory       [sync gRPC]    reserve stock (need answer NOW)
4. Orders → Payments        [sync gRPC]    charge card (need success/fail NOW)
5. Orders → Message Queue   [async event]  publish OrderPlaced
6. Queue → Notifications    [async]        send email (don't need to wait)
7. Queue → Analytics        [async]        log revenue event
8. Queue → Loyalty          [async]        award points

Steps 3–4 must be synchronous. Steps 6–8 don't affect the user's experience — async.

The chaining problem: 5 sync calls chained: total latency = sum of all hops. Availability compounds: 5 services at 99.9% = 99.5%. A downstream failure blocks everyone upstream.

05 Interview Q&A

Q: When would you use gRPC over REST?
Internal high-frequency service-to-service calls where you control both ends. Binary serialization, streaming, strong typing from .proto. REST is better for external APIs or anything a browser talks to.
Q: What guarantees does a message queue give?
At-least-once delivery — may be duplicated on retries. Consumers must be idempotent. Exactly-once is possible (Kafka) but expensive. Design for at-least-once.
Q: What is eventual consistency?
When you go async, services are temporarily out of sync. Email sent 200ms after order confirmed. Inventory count lags by a second. System converges to correct state, but not immediately. Acceptable for side effects — not for "has payment been charged."
