Event Sourcing

Store events not state. Apply pattern, projections, snapshots, pairing with CQRS.

01 Core Idea

Traditional: overwrite current state

Create: $19.99
→
Update: $24.99
→
Update: $29.99

Database row: Price = $29.99. $19.99 and $24.99 are gone forever.

Event sourcing: append events, never overwrite

Event 1: ProductCreated  { price: 19.99 }
Event 2: PriceChanged    { newPrice: 24.99, reason: "demand" }
Event 3: PriceChanged    { newPrice: 29.99, reason: "new model" }

Current price = replay: 19.99 → 24.99 → 29.99 = $29.99
All history preserved. Like Git — every commit, not just latest file.
Rebuild read model = replay all events

Read DB corrupted? Delete it, replay events → fully restored.
Need new field in view? Replay events → populated with all history.
Add new projection? Replay events → ready from day one.

02 The Apply Pattern (C#)

public class Product
{
    public string Id { get; private set; }
    public decimal Price { get; private set; }

    // Rebuild from history
    public Product(IEnumerable<object> history)
    {
        foreach (var evt in history) Apply(evt);
    }

    // Business operations produce events, NOT state changes
    public void ChangePrice(decimal newPrice, string reason)
    {
        if (newPrice <= 0) throw new ValidationException();
        var evt = new PriceChanged(Id, Price, newPrice, reason);
        Apply(evt);
        _pendingEvents.Add(evt);
    }

    // ONLY place state changes
    private void Apply(object evt) => _ = evt switch
    {
        ProductCreated e => (Id, Price) = (e.ProductId, e.Price),
        PriceChanged e   => Price = e.NewPrice,
        _ => default
    };
}

03 Projections + Snapshots

Projections: One PriceChanged event feeds: product detail page, price history chart, revenue forecast — different views from same events.

Snapshots: If 10,000 events, replaying all is slow. Snapshot captures state at version 9,900. Load snapshot → replay only last 100.

04 Interview Q&A

Q: Hardest part of Event Sourcing?
Event schema evolution. Once stored, can't change structure. Need new field? Create PriceChangedV2. Managing multiple event versions adds complexity over time.
Q: How does it relate to CQRS?
Independent but pair naturally. Event sourcing provides the write side (events appended). Those events feed projection handlers that build read models. Key benefit: read DB corrupted → replay all events to rebuild.
