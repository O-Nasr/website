01 The Problem

In a monolith: BEGIN TRANSACTION wraps all steps — create order, reduce stock, charge payment. Either all commit or all rollback. ACID.

In microservices: each service has its own database. There is no single transaction that spans all three. If Payment fails after Orders committed, you're stuck — the order exists, stock was reduced, but money was never charged.

Orders service
Orders DB → order created ✓
Inventory service
Inventory DB → stock reduced ✓
Payment service
Payments DB → CHARGE FAILED ✗

Order exists, stock is reduced, but money was never charged. System is inconsistent.

Each service has its own database. There is no BEGIN TRANSACTION that spans all three. Each service commits independently. If step 3 fails after steps 1 and 2 committed — those commits are permanent. You can't "roll back" a committed transaction in another service's database.

02 Why Not 2PC?

Two-Phase Commit (2PC) exists — a coordinator asks all services to "prepare," then tells them all to "commit." But it doesn't work for microservices:

  • Blocking — all services lock rows and wait for the coordinator. If coordinator crashes, everyone stays locked forever.
  • Performance — every service waits on the slowest. A 50ms flow becomes 500ms.
  • Coupling — all services must be available simultaneously. One down = all stuck.
2PC creates tight coupling between services — the exact thing microservices are supposed to eliminate. This is why Sagas exist.

03 What is a Saga?

A sequence of local transactions, where each service does its own work and publishes an event. If any step fails, compensating transactions run backward to undo previous steps.

A rollback undoes uncommitted changes automatically — the database does it for you.
A compensation is a NEW committed transaction that semantically reverses the previous one — you write the logic yourself.

The original transaction already committed. It's gone. You can't undo it at the DB level. You must write a new transaction that achieves the reverse effect.
StepOriginal actionCompensation
Create orderINSERT order (status=PENDING)UPDATE order SET status=CANCELLED
Reserve stockstock -= 5stock += 5 (ReleaseStock)
Charge paymentCall Stripe charge APICall Stripe refund API

Notice: the Stripe compensation isn't "undo the charge" — it's "call the refund endpoint." A completely new API call.

05 Choreography — No Boss, Just Events

Each service publishes events and reacts to other services' events. No central coordinator:

Orders creates order → publishes OrderCreated
    ↓
Inventory hears OrderCreated → reserves stock → publishes StockReserved
    ↓
Payments hears StockReserved → charges card → publishes PaymentCompleted
    ↓
Orders hears PaymentCompleted → status = CONFIRMED

--- If payment fails: ---
Payments publishes PaymentFailed
    ↓
Inventory hears PaymentFailed → releases stock (qty += 5)
    ↓
Orders hears PaymentFailed → status = CANCELLED
// Events (shared contracts)
public record OrderCreated(Guid OrderId, string ProductId, int Qty);
public record StockReserved(Guid OrderId);
public record PaymentCompleted(Guid OrderId);
public record PaymentFailed(Guid OrderId, string Reason);

// Inventory Service — reacts to OrderCreated
public class InventoryHandler
{
    public async Task Handle(OrderCreated evt)
    {
        var stock = await _db.Stock.FindAsync(evt.ProductId);
        if (stock.Quantity >= evt.Qty)
        {
            stock.Quantity -= evt.Qty;
            await _db.SaveChangesAsync();
            await _bus.PublishAsync(new StockReserved(evt.OrderId));
        }
        else
        {
            await _bus.PublishAsync(new PaymentFailed(evt.OrderId, "Out of stock"));
        }
    }
}
  • No visibility — you can't look at one place and see "this saga is 60% done." Workflow is scattered across 4 services.
  • Cyclic dependencies — 10 services publishing events to each other can create event loops.
  • Hard to add steps — inserting a fraud check means modifying multiple services' subscriptions.
  • Hard to debug — a saga fails, and you must trace events across 4 separate log streams.
Choreography works for simple sagas (2–4 steps). Beyond that, use orchestration.

06 Orchestration — One Boss Controls Everything

A Saga Orchestrator knows the entire workflow and tells each service what to do. Hub-and-spoke model. Services never talk to each other — only to the orchestrator.

Orchestrator sends: CreateOrderCommand    → Orders service
  ← receives:       OrderCreated reply

Orchestrator sends: ReserveStockCommand   → Inventory service
  ← receives:       StockReserved reply

Orchestrator sends: ChargePaymentCommand  → Payments service
  ← receives:       PaymentCompleted reply (or PaymentFailed)

Orchestrator sends: ConfirmOrderCommand   → Orders service

The entire workflow is in one class. A new developer reads OrderSagaOrchestrator and understands the complete order flow in 2 minutes.

1. CreateOrder
→
2. ReserveStock
→
3. ChargePayment
→
4. ConfirmOrder

At each step, the orchestrator: (1) saves its state to DB, (2) sends a command, (3) waits for a reply, (4) advances to the next step.

Payment fails at step 3:

1. CreateOrder ✓
→
2. ReserveStock ✓
→
3. ChargePayment ✗

Orchestrator receives PaymentFailed → transitions to COMPENSATING state:

ReleaseStock
→
CancelOrder
→
Saga FAILED

Both compensations are new forward writes — not DB rollbacks. Orchestrator runs them in reverse order of the original steps.

07 Full C# Orchestrator Code

public class OrderSagaOrchestrator
{
    private readonly IEventBus _bus;
    private readonly SagaDbContext _db;

    public async Task StartSaga(CreateOrderRequest req)
    {
        // Save saga state — so we can recover if orchestrator crashes
        var saga = new OrderSaga
        {
            Id = Guid.NewGuid(),
            CurrentStep = "CreateOrder",
            Status = "IN_PROGRESS"
        };
        _db.Sagas.Add(saga);
        await _db.SaveChangesAsync();

        // Step 1: tell Orders to create the order
        await _bus.SendCommandAsync(new CreateOrderCommand(saga.Id, req));
    }
}
public async Task Handle(OrderCreated reply)
{
    var saga = await _db.Sagas.FindAsync(reply.SagaId);
    saga.CurrentStep = "ReserveStock";
    await _db.SaveChangesAsync();   // save state BEFORE sending command
    await _bus.SendCommandAsync(new ReserveStockCommand(saga.Id, reply.ProductId, reply.Qty));
}

public async Task Handle(StockReserved reply)
{
    var saga = await _db.Sagas.FindAsync(reply.SagaId);
    saga.CurrentStep = "ChargePayment";
    await _db.SaveChangesAsync();
    await _bus.SendCommandAsync(new ChargePaymentCommand(saga.Id, reply.Amount));
}

public async Task Handle(PaymentCompleted reply)
{
    var saga = await _db.Sagas.FindAsync(reply.SagaId);
    saga.CurrentStep = "Completed";
    saga.Status = "COMPLETED";
    await _db.SaveChangesAsync();
    await _bus.SendCommandAsync(new ConfirmOrderCommand(saga.Id));
}
// COMPENSATION — payment failed
public async Task Handle(PaymentFailed reply)
{
    var saga = await _db.Sagas.FindAsync(reply.SagaId);
    saga.Status = "COMPENSATING";
    await _db.SaveChangesAsync();

    // Run compensations backward
    await _bus.SendCommandAsync(new ReleaseStockCommand(saga.Id));
    await _bus.SendCommandAsync(new CancelOrderCommand(saga.Id));

    saga.Status = "FAILED";
    await _db.SaveChangesAsync();
}

Key insight: the orchestrator saves CurrentStep and Status to DB before every command. This is what enables crash recovery.

public class OrderSagaOrchestrator
{
    public async Task StartSaga(CreateOrderRequest req)      { /* save + CreateOrderCommand */ }
    public async Task Handle(OrderCreated reply)             { /* save + ReserveStockCommand */ }
    public async Task Handle(StockReserved reply)            { /* save + ChargePaymentCommand */ }
    public async Task Handle(PaymentCompleted reply)         { /* save + ConfirmOrderCommand */ }
    public async Task Handle(PaymentFailed reply)            { /* save + ReleaseStock + CancelOrder */ }
}

08 MassTransit State Machine

In production, you use MassTransit's built-in saga support instead of writing the orchestrator manually:

public class OrderSagaStateMachine : MassTransitStateMachine<OrderSagaState>
{
    public State OrderCreated { get; private set; }
    public State StockReserved { get; private set; }
    public State Completed { get; private set; }
    public State Failed { get; private set; }

    public Event<PlaceOrderCommand> PlaceOrder { get; private set; }
    public Event<StockReservedEvent> StockReserved_ { get; private set; }
    public Event<PaymentFailedEvent> PaymentFailed { get; private set; }
    public Event<PaymentSucceededEvent> PaymentSucceeded { get; private set; }

    public OrderSagaStateMachine()
    {
        CorrelateBy<PlaceOrderCommand>(x => x.Message.OrderId, x => x.OrderId);
        CorrelateBy<StockReservedEvent>(x => x.Message.OrderId, x => x.OrderId);

        Initially(
            When(PlaceOrder)
                .Then(ctx => { ctx.Saga.OrderId = ctx.Message.OrderId; })
                .Publish(ctx => new ReserveStockCommand(ctx.Saga.OrderId))
                .TransitionTo(OrderCreated)
        );

        During(OrderCreated,
            When(StockReserved_)
                .Publish(ctx => new ChargeCardCommand(ctx.Saga.OrderId, ctx.Saga.Total))
                .TransitionTo(StockReserved)
        );

        During(StockReserved,
            When(PaymentSucceeded)
                .Publish(ctx => new ConfirmOrderCommand(ctx.Saga.OrderId))
                .TransitionTo(Completed)
                .Finalize(),

            // COMPENSATION PATH
            When(PaymentFailed)
                .Publish(ctx => new ReleaseStockCommand(ctx.Saga.OrderId))
                .Publish(ctx => new CancelOrderCommand(ctx.Saga.OrderId))
                .TransitionTo(Failed)
                .Finalize()
        );
    }
}
// Program.cs — register with EF Core persistence
builder.Services.AddMassTransit(x =>
{
    x.AddSagaStateMachine<OrderSagaStateMachine, OrderSagaState>()
     .EntityFrameworkRepository(r =>
     {
         r.ExistingDbContext<AppDbContext>();
         r.UseSqlServer();    // state persisted in DB
     });

    x.UsingRabbitMq((ctx, cfg) =>
    {
        cfg.Host("rabbitmq://localhost");
        cfg.ConfigureEndpoints(ctx);
    });
});

The state machine persists its current state (OrderCreated, StockReserved, etc.) to the database after every transition. If the process crashes, it reads the persisted state and continues.

09 Crash Recovery — Durable State Machine

The orchestrator crashes after step 2 but before step 3:

Step 1: CreateOrder  ✓  (committed)
Step 2: ReserveStock ✓  (committed)
--- CRASH ---
Step 3: ChargePayment   (never sent)

The saga is stuck. Order exists, stock is reserved, but nothing charges the card. Without recovery, this order stays in limbo forever.

The orchestrator saved its state to DB before the crash:

Saga #abc → CurrentStep: "ChargePayment", Status: "IN_PROGRESS"

When the orchestrator restarts, it queries the DB: "give me all sagas that are IN_PROGRESS" — and picks up exactly where it left off. This is a durable state machine.

The key: state was saved before sending the command. If the command was already sent before the crash, the receiving service must be idempotent — processing the same command twice is safe.

Run 2+ orchestrator instances behind a load balancer. If one dies, the others keep processing. The DB is the shared source of truth — any instance can pick up any saga.

Subtlety: two instances must not pick up the same saga simultaneously. MassTransit handles this via competing consumers — each message from the queue is delivered to only one consumer instance.

10 Idempotency — Preventing Double-Processing

Orchestrator sends ChargePaymentCommand. Payment charges $100. The reply message gets lost (network blip). Orchestrator never gets PaymentCompleted, so it retries the command.

Customer is charged $200. The charge happened twice.

This happens because messages in distributed systems can be delivered more than once — due to retries, network issues, or orchestrator crashes and replays.

public class PaymentService
{
    public async Task Handle(ChargePaymentCommand cmd)
    {
        // Idempotency check — did I already process this saga step?
        var existing = await _db.Payments
            .FirstOrDefaultAsync(p => p.SagaId == cmd.SagaId);

        if (existing != null)
        {
            // Already charged — just re-publish the reply
            await _bus.PublishAsync(new PaymentCompleted(cmd.SagaId));
            return;  // no double charge!
        }

        // First time — actually charge
        var result = await _stripe.ChargeAsync(cmd.Amount);
        _db.Payments.Add(new Payment
        {
            SagaId = cmd.SagaId,
            StripeId = result.Id,
            Amount = cmd.Amount
        });
        await _db.SaveChangesAsync();

        await _bus.PublishAsync(new PaymentCompleted(cmd.SagaId));
    }
}

The SagaId acts as an idempotency key. Before doing any work, the service checks: "have I already processed a command with this SagaId?" If yes → skip the work, re-publish the reply. If no → do the work.

This is the same pattern you learned in Chapter 10 (notification dedup with event_id in Redis) — same concept, different context.

Rule: every saga step and every compensation MUST be idempotent. Safe to execute more than once with the same result. This is non-negotiable in distributed systems.

11 Eventual Consistency

During a saga's execution, the system is temporarily inconsistent:

Step 1 done: order exists (PENDING), stock not yet reserved
Step 2 done: order exists, stock reserved, payment not yet charged
Step 3 done: order CONFIRMED, stock reserved, payment charged ← CONSISTENT

Between steps, other transactions can see intermediate states. This is expected. The saga will eventually reach either a fully committed or fully compensated state.

Application must handle this: don't show "order confirmed" until the saga completes. Use status fields: PENDING → CONFIRMED or PENDING → CANCELLED.

A saga guarantees ACD but not I:
Atomicity — via compensations (all or nothing, eventually)
Consistency — eventual, not immediate
Durability — each local commit is durable
Isolation — ❌ NO. Intermediate states are visible to other transactions.

12 Interview Q&A

Q: What is a Saga and why do microservices need it?
A saga is a sequence of local transactions — one per service — coordinated to behave like a single distributed operation. Microservices can't use database transactions across service boundaries, so when a multi-step operation fails partway through, sagas use compensating transactions to undo completed steps and restore consistency.
Q: What's the difference between choreography and orchestration?
Choreography: no central coordinator — each service publishes events and reacts to others' events. Loose coupling but hard to debug and add steps. Orchestration: a central state machine sends commands to each service and waits for responses. Easier to reason about, one class shows the full workflow, but the orchestrator is a single point of failure.
Q: What is a compensating transaction?
A new forward database write that semantically reverses a previous step. It is NOT a database rollback — the original transaction already committed. ReleaseStock (qty += 1) compensates for ReserveStock (qty -= 1). Stripe refund API compensates for Stripe charge API.
Q: Why must every saga step be idempotent?
Messages can be delivered more than once — retries, network issues, orchestrator crashes and replays. If a step isn't idempotent, processing the same message twice corrupts data (charging a card twice). Every step checks: "have I already processed this SagaId?" If yes, skip and re-publish the reply.
Q: Does a saga guarantee ACID?
No. ACD but not Isolation. Between steps, other transactions can see intermediate states — stock reserved but payment not yet charged. Handle with status fields (PENDING → CONFIRMED). The saga eventually reaches a fully committed or fully compensated state.
Q: How do you handle the orchestrator crashing?
The orchestrator persists saga state to DB before every command. Run multiple instances. On restart, query for IN_PROGRESS sagas and resume. Commands must be idempotent so retries are safe. MassTransit handles this with EntityFramework persistence and competing consumers on the message queue.
Q: Why not use 2PC instead of Sagas?
2PC blocks all participants while waiting for coordinator, doesn't scale, and creates tight coupling — all services must be available simultaneously. Sagas use local transactions per service, are non-blocking, and handle partial failures through compensation. 2PC violates microservices independence principles.
