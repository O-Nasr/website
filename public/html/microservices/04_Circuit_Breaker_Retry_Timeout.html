01

The Problem — Cascading Failure

Without these patterns, one slow service can bring down the entire system. Threads pile up waiting for a dead dependency, memory fills, and the failure spreads upstream until everything is down.

⚡
The chain reaction: Database slows → Payments threads fill waiting → Inventory threads fill waiting for Payments → Orders saturates → Client gets 100% errors. One dependency took down four healthy services.
02

Pattern 1 — Timeout

The simplest defence. Never wait forever. If a service doesn't respond within N milliseconds, cancel and fail fast. This prevents threads from being held indefinitely.

C#
// ❌ Without timeout — thread blocks forever if Payments hangs
var result = await _http.PostAsJsonAsync("/api/payments", request);

// ✅ With timeout — fail after 3 seconds
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(3));
try
{
    var result = await _http.PostAsJsonAsync("/api/payments", request, cts.Token);
}
catch (OperationCanceledException)
{
    return PaymentResult.TimedOut(); // fail fast, free the thread
}
✅
Setting the right timeout: 99th percentile response time × 2 under normal load. Too short → false failures on legitimate slow calls. Too long → threads still pile up.
03

Pattern 2 — Retry with Exponential Backoff

A failure might be transient — a brief network blip, a service restarting. Retry automatically, but wait progressively longer between attempts so you don't hammer a struggling service.

C# — Manual retry (conceptual)
int maxAttempts = 3;
int delayMs     = 200; // start at 200ms

for (int attempt = 1; attempt <= maxAttempts; attempt++)
{
    try { return await _paymentsClient.ChargeAsync(request); }
    catch (HttpRequestException ex) when (IsTransient(ex))
    {
        if (attempt == maxAttempts) throw; // give up

        // 200ms → 400ms → 800ms  +  jitter
        int jitter = Random.Shared.Next(0, 50);
        await Task.Delay(delayMs + jitter);
        delayMs *= 2; // exponential
    }
}

// ✅ What to retry — transient errors only
private bool IsTransient(HttpRequestException ex) =>
    ex.StatusCode is
        HttpStatusCode.ServiceUnavailable  // 503 ✓
     or HttpStatusCode.TooManyRequests    // 429 ✓
     or HttpStatusCode.GatewayTimeout;   // 504 ✓
     // 400 Bad Request  → NOT transient ✗ (your payload is wrong)
     // 401 Unauthorized → NOT transient ✗ (your token is invalid)
⚠️
Jitter is required. Without it, all service instances retry at exactly the same millisecond after a failure spike — a synchronized storm that can take a recovering service straight back down. Random ±50ms spreads the load across time.
04

Pattern 3 — Circuit Breaker

The most important pattern. When a downstream service keeps failing, stop calling it entirely — give it time to recover. The circuit breaker has three states.

Closed
Normal operation. All requests pass through. The breaker counts failures in a sliding window silently.
Transitions OPEN when failure ratio exceeds threshold (e.g. 50% of last 10 calls)
Open
All requests fail immediately — no network call made at all. Returns a cached response or error instantly.
After BreakDuration (e.g. 30s) transitions to HALF-OPEN
Half-Open
Lets ONE probe request through to test if the service recovered.
Probe success → CLOSED. Probe fail → OPEN again for another BreakDuration.
05

The Two Types of Timeout

This is the most commonly missed concept. In Polly there are two separate timeouts with completely different jobs. Confusing them is the root of most misunderstandings about how exceptions fire.

Attempt Timeout
Per-attempt timeout
Limits how long ONE single HTTP call can take. If attempt 1 hangs for more than e.g. 2s, kill that attempt only. Retry can still fire and try again with a fresh 2s budget.
Total Timeout
Whole-operation timeout
Limits the TOTAL time for ALL attempts combined. When this fires, everything stops — no more retries, no more attempts, game over. Throws TimeoutRejectedException to your catch block.
⚠️
Both throw TimeoutRejectedException — but they have very different meanings. Attempt timeout = "this one call was too slow, retry it." Total timeout = "we've spent too long overall, give up entirely."
06

Polly Pipeline — Layers & Exception Routing

The pipeline is registered in this order, but each layer wraps the next. Reading from outside in:

Exception Routing — Which Catch Block Fires When

ExceptionCircuit stateNetwork call made?Retry fires?What triggered it
BrokenCircuitException OPEN No — blocked No — not in ShouldHandle Previous calls crossed the failure threshold
TimeoutRejectedException CLOSED Yes — calls were made No — total budget gone Total 10s budget ran out across all retries
HttpRequestException CLOSED Yes — all retries made Yes — 3× before giving up All 3 retry attempts failed (fast failures)
💡
TimeoutRejectedException fires when circuit is CLOSED — the service is being called, just too slowly. The circuit hasn't opened yet because failures haven't accumulated to the threshold. It represents "currently slow", not "known broken".
C# — The three catch blocks with precise annotations
public async Task<PaymentResult> ChargeAsync(PaymentRequest request)
{
    try
    {
        // This single await triggers the ENTIRE Polly pipeline:
        // CB checks state → attempt 1 (2s budget) → retry wait → attempt 2 → ...
        // Your code only resumes when Polly is completely done
        return await _paymentsClient.ChargeAsync(request);
    }
    catch (BrokenCircuitException)
    {
        // Fires when: circuit OPEN — previous requests already broke it
        // No network call was made. Retry did NOT fire (not in ShouldHandle)
        return PaymentResult.ServiceUnavailable();
    }
    catch (TimeoutRejectedException)
    {
        // Fires when: total 10s budget expired (circuit was CLOSED)
        // Calls WERE made — just too slow. Retry cannot help, budget is gone.
        return PaymentResult.TimedOut();
    }
    catch (HttpRequestException)
    {
        // Fires when: all 3 retry attempts exhausted, each failed fast
        // By now 3 network calls have already been made by Polly
        return PaymentResult.Failed();
    }
}
07

What the Circuit Breaker Counts

Critical distinction — the CB counts each individual attempt that passes through it, not each retry group.

⚠️
The count that matters: CB's threshold is based on a failure ratio over a sampling window (e.g. 50% of last 20 data points). 3 retry attempts from one logical request contribute 3 separate data points to that window — not 1.

Full C# Implementation — Polly v8

bash — Install
dotnet add package Microsoft.Extensions.Http.Resilience
C# — Program.cs — full pipeline registration
builder.Services
    .AddHttpClient<IPaymentsClient, PaymentsClient>()
    .AddResilienceHandler("payments-pipeline", pipeline =>
    {
        // ─── LAYER 1: Total Timeout (outermost) ──────────────────────
        // Kills EVERYTHING after 10s — no more retries allowed
        // Throws: TimeoutRejectedException → your catch
        pipeline.AddTimeout(TimeSpan.FromSeconds(10));

        // ─── LAYER 2: Retry ──────────────────────────────────────────
        // Loops inner layers up to 3×. Gives up when max hit OR total timeout fires.
        pipeline.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            Delay       = TimeSpan.FromMilliseconds(300),
            BackoffType = DelayBackoffType.Exponential, // 300→600→1200ms
            UseJitter   = true,
            ShouldHandle = args => ValueTask.FromResult(
                args.Outcome.Exception is HttpRequestException ||
                args.Outcome.Exception is TimeoutRejectedException || // ← attempt timeout
                args.Outcome.Result?.StatusCode is
                    HttpStatusCode.ServiceUnavailable or
                    HttpStatusCode.TooManyRequests
                // BrokenCircuitException is NOT here → retry skips it
            )
        });

        // ─── LAYER 3: Attempt Timeout ────────────────────────────────
        // Kills ONE slow attempt after 2s → retry catches and loops again
        // Throws: TimeoutRejectedException (caught by Retry above)
        pipeline.AddTimeout(TimeSpan.FromSeconds(2));

        // ─── LAYER 4: Circuit Breaker (innermost) ────────────────────
        // Always watching. When OPEN → throws BrokenCircuitException
        // Retry does NOT catch BrokenCircuitException
        pipeline.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            FailureRatio      = 0.5,                      // 50% threshold
            SamplingDuration  = TimeSpan.FromSeconds(30), // sliding window
            MinimumThroughput = 8,                       // need ≥8 calls first
            BreakDuration     = TimeSpan.FromSeconds(30)  // stay open 30s
        });
    });
C# — Service usage with all catch blocks
public async Task<PaymentResult> ChargeAsync(PaymentRequest request)
{
    try
    {
        // One await — Polly handles all retries internally
        // Your code runs exactly ONCE. Retry loop is inside Polly.
        return await _paymentsClient.ChargeAsync(request);
    }
    catch (BrokenCircuitException)
    {
        // When: circuit OPEN from previous failures
        // No network call made. Instant fail.
        _logger.LogWarning("Payments circuit open — failing fast");
        return PaymentResult.ServiceUnavailable(
            "Payment service temporarily unavailable. Your cart is saved.");
    }
    catch (TimeoutRejectedException)
    {
        // When: total 10s budget expired (circuit was CLOSED)
        // Calls were made — just too slow for too long
        _logger.LogWarning("Payments timed out — total budget exceeded");
        return PaymentResult.TimedOut();
    }
    catch (HttpRequestException ex)
    {
        // When: all 3 retries exhausted within the time budget
        // 3 actual network calls were made before reaching here
        _logger.LogError(ex, "Payments failed after all retries");
        return PaymentResult.Failed();
    }
}

Interview Q&A

What is a circuit breaker and what problem does it solve?
A circuit breaker monitors failure rates to a downstream service and stops making calls when failures exceed a threshold — instead failing immediately with a cached or degraded response. It solves cascading failure: without it, threads pile up waiting for a dead service, exhausting resources and taking down healthy services too.
What are the three states and what triggers each transition?
Closed → Open: failure ratio exceeds threshold over the sampling window. Open → Half-Open: BreakDuration elapses. Half-Open → Closed: probe request succeeds. Half-Open → Open: probe request fails (back to Open for another BreakDuration).
When retry reaches max attempts, does it move to the circuit breaker?
No. When retry hits max attempts it simply throws the exception upward and stops. It does NOT "move to" the circuit breaker. The CB is not a next step — it has been watching silently the entire time, counting every individual attempt that passed through it. They are independent layers.
What's the difference between attempt timeout and total timeout?
Attempt timeout kills ONE slow call after e.g. 2s — retry can still fire and try again with a fresh budget. Total timeout kills the ENTIRE operation (all retries combined) after e.g. 10s — retry cannot help. Both throw TimeoutRejectedException but mean very different things.
Why does TimeoutRejectedException fire when the circuit is CLOSED?
Because it fires when the total budget runs out while the service is still being called (just slowly). The circuit is CLOSED because failures haven't accumulated to the threshold yet. It represents "currently slow", not "known broken." BrokenCircuitException represents "known broken from previous requests."
Does one retry group (3 attempts) count as 1 or 3 failures in the circuit breaker?
3 failures. The circuit breaker counts each individual attempt that passes through it — not each retry group. So 3 retry attempts = 3 data points in the CB's failure ratio window. This matters because the threshold is based on a ratio (e.g. 50% of the last 20 calls).
Why is retry's jitter required?
Without jitter, all service instances retry at identical intervals after a failure spike — a synchronized storm that hits the recovering service at the same millisecond. Random ±50ms spreads retry load across time, giving the service room to recover instead of being immediately overwhelmed again.
What should you NOT retry?
Non-transient errors: 400 Bad Request (your payload is wrong — retrying won't fix it), 401 Unauthorized (your token is invalid), 404 Not Found. Only retry errors indicating temporary unavailability: 503 Service Unavailable, 429 Too Many Requests, 504 Gateway Timeout, and network-level failures.

The Three Rules — Memorize These

Rule 1
Retry max attempts hit → throws exception upward and stops. Does NOT loop again. Does NOT "move to" circuit breaker. CB was watching the whole time.
Rule 2
Circuit Breaker is NOT a next step → it is always running, always counting. When it opens, it affects the NEXT request — not the current one that triggered the opening.
Rule 3
Two timeouts — two jobs → Attempt Timeout kills one slow attempt (retry can still fire). Total Timeout kills everything (retry cannot fire anymore).
