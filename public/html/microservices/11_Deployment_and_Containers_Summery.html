01 The Problem

In a monolith, you deploy one thing. In microservices, you deploy 20+ services independently. How do you update a running service without users noticing?

Three requirements: zero downtime (users never see errors), safe rollback (revert quickly if buggy), and independence (deploying Orders doesn't require deploying Payments).

Rolling deploy
Replace pods one by one. Old and new run together during transition. K8s default. Zero extra cost. Slowest rollback.
Blue-green deploy
Two full environments. Deploy and test the new one privately. Switch all traffic instantly. Instant rollback. 2× resource cost.
Canary deploy
Send small % of traffic to new version first. Monitor. Gradually increase. Smallest blast radius. Most complex to set up.

Rolling deploy — summary

K8s replaces pods one at a time (configurable via maxSurge / maxUnavailable).
Each new pod must pass readiness probe before K8s continues.
During transition: both v1 and v2 serve traffic simultaneously.
Rollback: kubectl rollout undo deploy/orders (re-does rolling update back to v1 — slow).

Your new version must be backward compatible — some users hit v1 and some hit v2 during the transition.
Q: What happens if the readiness probe fails on the new pod?
K8s stops the rollout. The new pod is never added to the Service endpoints — it never receives traffic. K8s keeps trying. Remaining v1 pods continue serving normally. You either fix the image or run kubectl rollout undo.

04 Canary Deploy

Named after the canary in a coal mine — send a small group first, see if they survive. Click through each stage:

Stage 1: send 5% of traffic to v2

v1 — 95%
5%

Only 5% of users see v2. Monitor error rates, latency, and metrics for 15–30 minutes. If anything looks wrong, only 5% were affected — tiny blast radius.

Stage 2: metrics look good — increase to 25%

v1 — 75%
v2 — 25%

Error rate on v2 matches v1. Latency is normal. Monitor another 15 minutes. Confidence growing.

Stage 3: increase to 50%

v1 — 50%
v2 — 50%

Half your users on v2. If a subtle bug only appears under load, this is where you'd catch it.

Stage 4: full rollout — 100% on v2

v2 — 100%

All traffic on v2. v1 pods terminated. Deploy complete. At no point did you risk all users on untested code.

Bug detected at 25%? Route all traffic back to v1

v1 — 100% (rollback)

Only 25% of users experienced the bug. Rollback is instant — shift traffic back. v1 was still running. This is why canary is the safest strategy.

Implementation options: Istio (VirtualService traffic splitting), Nginx Ingress (canary annotations), Argo Rollouts (K8s-native canary with automatic analysis), or feature flags (route specific user segments to v2 at the application level).

06 CI/CD for Microservices

The fundamental change

Monolith: one repo, one pipeline, one deploy. Microservices: each service has its own independent pipeline.

Repository structure

├── orders-service/
│   ├── src/
│   ├── tests/
│   ├── Dockerfile
│   ├── k8s/deployment.yaml
│   └── .github/workflows/deploy.yml    ← its own pipeline
│
├── payments-service/
│   ├── src/
│   ├── Dockerfile
│   └── .github/workflows/deploy.yml    ← its own pipeline
│
└── inventory-service/
    ├── src/
    ├── Dockerfile
    └── .github/workflows/deploy.yml    ← its own pipeline

Full pipeline (GitHub Actions)

name: Deploy Orders Service
on:
  push:
    branches: [main]
    paths: ['orders-service/**']   # ONLY trigger for THIS service

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 1. Run tests — gate the deploy
      - run: dotnet test orders-service/tests/

      # 2. Build Docker image — tag with commit SHA
      - run: docker build -t orders:${{ github.sha }} orders-service/

      # 3. Push to container registry
      - run: |
          docker tag orders:${{ github.sha }} registry.io/orders:${{ github.sha }}
          docker push registry.io/orders:${{ github.sha }}

      # 4. Deploy to K8s (triggers rolling update)
      - run: |
          kubectl set image deployment/orders-service \
            orders=registry.io/orders:${{ github.sha }}

      # 5. Wait for rollout (fail pipeline if stuck)
      - run: kubectl rollout status deployment/orders-service --timeout=300s

The Dockerfile (multi-stage build)

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY src/ .
RUN dotnet publish -c Release -o /app

FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=build /app .
EXPOSE 5001
ENTRYPOINT ["dotnet", "OrdersService.dll"]
Why multi-stage? SDK image is ~700MB (compilers, tools). Runtime image is ~80MB. Build in the big image, copy only compiled output to the small one. Smaller images = faster pulls = faster deploys = smaller attack surface.
Why tag with commit SHA? With :latest, you can't tell which code is running. With :abc123def, you git checkout that exact commit. It also prevents K8s from pulling a different image — :latest changes every build, :abc123def is immutable.

07 Kubernetes Deployment YAML Deep Dive

The full annotated config that controls everything — probes, strategy, resources, secrets:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: orders-service
spec:
  replicas: 3

  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # create at most 1 EXTRA pod during update
      maxUnavailable: 0   # never fewer than 3 running pods

  template:
    spec:
      containers:
        - name: orders
          image: registry.io/orders:abc123def

          readinessProbe:              # "Can this pod handle traffic?"
            httpGet:
              path: /health/ready
              port: 5001
            initialDelaySeconds: 5
            periodSeconds: 10
            failureThreshold: 3

          livenessProbe:               # "Is the process alive?"
            httpGet:
              path: /health
              port: 5001
            initialDelaySeconds: 15
            periodSeconds: 20
            failureThreshold: 3

          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"

          env:
            - name: ASPNETCORE_ENVIRONMENT
              value: "Production"
            - name: ConnectionStrings__OrdersDb
              valueFrom:
                secretKeyRef:
                  name: orders-secrets
                  key: db-connection

08 Interview Q&A

Q: What is a rolling deployment?
K8s replaces pods one at a time. Each new pod must pass readiness probe before K8s terminates the next old pod. Both versions serve during transition. K8s default, zero extra cost, but rollback is slow (re-rolls all pods). Configured via maxSurge and maxUnavailable.
Q: What is blue-green deployment?
Two identical environments — Blue (live) and Green (new). Green is deployed and tested while Blue serves traffic. Switch LB from Blue to Green instantly. Rollback: switch back. Blue never stopped. Tradeoff: 2× resources.
Q: What is canary deployment?
Deploy new version alongside old, shift traffic gradually: 5%→25%→50%→100%. Monitor at each stage. If anything breaks, route back — only the small % was affected. Safest strategy. Requires traffic splitting (Istio, Nginx, Argo Rollouts).
Q: Which strategy for a payment processing service?
Blue-green or canary — never simple rolling. Blue-green: fully test with synthetic transactions before switching. Canary: route 1% first, monitor success rates. Rolling is risky because both versions serve during transition — a subtle bug could affect real money.
Q: How does CI/CD change with microservices?
Each service has its own pipeline triggered only by its own code changes (path filter). Pipeline: test → build Docker image (tag: commit SHA) → push to registry → deploy to K8s → verify rollout. Teams deploy independently.
Q: What happens during a rolling deploy if the new version has a bug?
If readiness probe fails: K8s pauses rollout, bad pod never receives traffic, old pods continue normally. If probe passes but bug is logical: some users experience it during transition. Run kubectl rollout undo — slow because it re-rolls everything. This is why blue-green and canary offer instant rollback.
Q: What is maxSurge vs maxUnavailable?
maxSurge = how many extra pods above desired count during update. maxUnavailable = how many can be missing. maxSurge=1, maxUnavailable=0: always ≥3 pods, briefly 4 — safest. maxSurge=0, maxUnavailable=1: kill first, create after — saves resources but reduced capacity briefly.
Q: Why tag images with commit SHA?
Traceability. :latest — you don't know which code is running. :abc123def — git checkout that exact commit. Also immutable: :latest changes every build, SHA never changes.
Q: What is a multi-stage Docker build?
Build in SDK image (~700MB), copy only compiled output to runtime image (~80MB). Final image has no source code, no build tools. Smaller = faster deploys = smaller attack surface.
