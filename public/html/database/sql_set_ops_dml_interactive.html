INSERT

SQL-- Single row
INSERT INTO customers (name, email) VALUES ('Omar', 'omar@ex.com');

-- Multiple rows
INSERT INTO customers (name, email) VALUES
    ('Ahmed', 'ahmed@ex.com'),
    ('Sara',  'sara@ex.com');

-- INSERT from SELECT
INSERT INTO archive (name, email)
SELECT name, email FROM customers WHERE status = 'inactive';
SQL-- PostgreSQL: RETURNING
INSERT INTO customers (name) VALUES ('Omar') RETURNING customer_id, name;

-- SQL Server: OUTPUT
INSERT INTO customers (name) OUTPUT INSERTED.customer_id VALUES ('Omar');
Get auto-generated IDs without a separate SELECT — very useful in application code.

UPDATE & DELETE

SQL-- UPDATE with safety
UPDATE customers SET email = 'new@ex.com' WHERE customer_id = 5;

-- DELETE with condition
DELETE FROM orders WHERE status = 'cancelled' AND order_date < '2022-01-01';
⚠️ Without WHERE: UPDATE updates EVERY row. DELETE deletes EVERY row. Always write WHERE first, then SET. Run SELECT with the same WHERE to verify first!

MERGE — Upsert

INSERT + UPDATE + DELETE in one statement. If row exists → update. If not → insert.

SQLMERGE INTO customers AS target
USING new_customers AS source
    ON target.customer_id = source.customer_id

WHEN MATCHED THEN                  -- exists in both → UPDATE
    UPDATE SET target.name = source.name

WHEN NOT MATCHED BY TARGET THEN    -- only in source → INSERT
    INSERT (customer_id, name) VALUES (source.customer_id, source.name)

WHEN NOT MATCHED BY SOURCE THEN    -- only in target → DELETE
    DELETE;
SQLINSERT INTO customers (customer_id, name, email)
VALUES (5, 'Omar', 'omar@ex.com')
ON CONFLICT (customer_id)
DO UPDATE SET
    name  = EXCLUDED.name,
    email = EXCLUDED.email;
-- EXCLUDED = the row that failed to insert
INSERT ON CONFLICT is the most common PostgreSQL upsert pattern. EXCLUDED refers to the conflicting row — a key detail for interviews.

Test Yourself

Q1: UNION vs UNION ALL — which is faster and why?

UNION ALL simply appends rows — no sorting or hashing needed. UNION must find and remove duplicates, which requires an expensive sort or hash operation. Default to UNION ALL unless deduplication is a business requirement.

Q2: Is A EXCEPT B the same as B EXCEPT A?

EXCEPT is directional: it returns rows from the first query not in the second. Swapping the queries gives completely different results. UNION and INTERSECT are commutative; EXCEPT is not.

Q3: DELETE vs TRUNCATE — which fires triggers?

DELETE removes rows one by one, logging each and firing DELETE triggers. TRUNCATE deallocates data pages — much faster, but no triggers fire and identity resets. Both can be rolled back inside a transaction.

Q4: What does EXCLUDED refer to in PostgreSQL's INSERT ON CONFLICT?

EXCLUDED is a pseudo-table representing the row that would have been inserted if no conflict existed. You use it in DO UPDATE SET to copy values from the attempted insert: name = EXCLUDED.name.

Q5: What happens if you run UPDATE customers SET status = 'inactive' without WHERE?

Without WHERE, UPDATE affects EVERY row. Best practice: write WHERE first, run SELECT with the same WHERE to verify, then add SET. Use transactions so you can ROLLBACK if needed.

UNION removes duplicates (expensive sort/hash). UNION ALL keeps all rows (just appends — faster). Default to UNION ALL unless deduplication is a business requirement. Both require same column count and compatible types.
EXCEPT is directional: A EXCEPT B = rows in A not in B. B EXCEPT A = completely different result. Use case: orders from 2023 EXCEPT 2024 = lost customers. Data reconciliation: records in system A EXCEPT system B = what's missing.
DELETE: row-by-row, logs each, fires triggers, WHERE supported, identity continues. TRUNCATE: deallocates pages, much faster, no triggers, no WHERE, resets identity. Both rollback in transactions. Use TRUNCATE to empty a table; DELETE when you need to filter or triggers must fire.
Single statement combining INSERT + UPDATE + DELETE. WHEN MATCHED → update. WHEN NOT MATCHED BY TARGET → insert. WHEN NOT MATCHED BY SOURCE → delete. Used in ETL pipelines, data warehouse loading, inventory sync. PostgreSQL alternative: INSERT ON CONFLICT (EXCLUDED = the conflicting row).
