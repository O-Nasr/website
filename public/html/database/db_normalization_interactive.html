What is Normalization?

Structuring tables to eliminate redundancy and prevent anomalies. Each piece of data lives in exactly one place.

AnomalyWhat HappensExample
UpdateSame data in multiple rows — update one, miss othersCustomer email in every order row
DeletionDeleting a row accidentally removes unrelated dataDeleting last order loses customer info
InsertionCan't store data without another unrelated recordCan't add product without an order
Functional Dependency: A → B means knowing A always determines B. Example: customer_id → customer_email

Real-World Tradeoff

ScenarioRecommendationReason
OLTP (banking, e-commerce)Normalize to 3NF/BCNFData integrity is critical
OLAP (analytics, reporting)DenormalizeRead speed > write integrity
High-read APIsPartial denormalizationAvoid expensive JOINs on hot paths
Senior Insight: Normalization is for write integrity. Denormalization is for read performance. Knowing when to apply each is what separates senior engineers.
5

Test Yourself

Q1: What is a partial dependency?

Partial dependency = non-key depends on part of a composite PK (violates 2NF). The second option describes a transitive dependency (violates 3NF). This is the most commonly confused distinction.

Q2: employee_id → dept_id → dept_name. Which NF is violated?

dept_name depends on dept_id, which is a non-key column. This chain A→B→C where B is not a key is a transitive dependency — violates 3NF.

Q3: When would you intentionally denormalize?

Denormalize for read-heavy scenarios: analytics/reporting (OLAP), high-traffic APIs where JOINs are expensive. Accept some redundancy for speed. This is a senior-level engineering judgment call.

Normalization Quiz

★

Interview Answer Bank

Organizing tables to eliminate redundancy and prevent update anomalies (same data in multiple rows), deletion anomalies (losing unrelated data), and insertion anomalies (can't store data without unrelated records). Each piece of data lives in exactly one place.
2NF eliminates partial dependencies — non-key depends on part of composite PK. 3NF eliminates transitive dependencies — non-key depends on another non-key. 2NF is about the key relationship; 3NF is about non-key-to-non-key relationships.
When read performance > write integrity. Analytics/OLAP systems, high-traffic APIs where JOINs are expensive. Store pre-joined or aggregated data. Accept redundancy for speed. The tradeoff understanding is what separates senior engineers.
