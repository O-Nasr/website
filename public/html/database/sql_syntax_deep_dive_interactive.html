HAVING — WHERE vs HAVING

WHEREHAVING
StepStep 2 — before GROUP BYStep 4 — after GROUP BY
FiltersIndividual rowsGroups
Aggregates❌ NOT allowed — groups don't exist yet✅ Allowed — groups already exist
Use forstatus, date, raw valuesCOUNT > 5, SUM > 1000
SQLSELECT customer_id, COUNT(*) AS total_orders
FROM orders
WHERE status = 'completed'       -- Step 2: keep only completed rows
GROUP BY customer_id           -- Step 3: group by customer
HAVING COUNT(*) > 3            -- Step 4: keep groups with > 3
ORDER BY total_orders DESC;    -- Step 6: sort highest first
⚠️ Alias Trap: HAVING total_orders > 3 — fails in SQL Server! SELECT (Step 5) hasn't run yet at HAVING (Step 4). Safe rule: always use full expression HAVING COUNT(*) > 3.

ORDER BY, TOP & LIMIT

ORDER BY runs at Step 6 — last. CAN use aliases because SELECT already ran.

SQL-- SQL Server pagination
SELECT TOP 10 * FROM orders ORDER BY total DESC;

-- PostgreSQL pagination
SELECT * FROM orders ORDER BY total DESC LIMIT 10 OFFSET 20; -- page 3

-- SQL Server equivalent
SELECT * FROM orders ORDER BY total DESC
OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY;

Complete Query — All Clauses

SQLSELECT                              -- Step 5
    customer_id,
    COUNT(*)     AS order_count,
    SUM(total)   AS total_spent,
    AVG(total)   AS avg_order_value,
    MAX(total)   AS largest_order
FROM orders                         -- Step 1
WHERE                               -- Step 2
    status = 'completed'
    AND created_at >= '2024-01-01'
GROUP BY customer_id                -- Step 3
HAVING                              -- Step 4
    COUNT(*) > 3
    AND SUM(total) > 1000
ORDER BY total_spent DESC           -- Step 6
LIMIT 10;
In plain English: From orders, take only completed orders from 2024. Group by customer. Keep customers with more than 3 orders AND total spending above 1000. Show their stats sorted by highest spender first. Return top 10.
7

Test Yourself

Q1: Why can't you use COUNT(*) in a WHERE clause?

WHERE (Step 2) runs before GROUP BY (Step 3). At Step 2, rows haven't been grouped yet — COUNT has nothing to aggregate. HAVING (Step 4) runs after groups exist, so aggregates work there.

Q2: WHERE email = NULL — how many rows does this return?

NULL means "unknown." Comparing anything with = NULL returns NULL, not TRUE. In a WHERE clause, NULL is treated as false — so zero rows match. Always use IS NULL.

Q3: 10 rows, 3 have NULL email. What is COUNT(*) vs COUNT(email)?

COUNT(*) counts all rows regardless of NULLs. COUNT(column) only counts rows where that column is NOT NULL. This is a classic interview trap.

Q4: Can you use a SELECT alias in ORDER BY?

ORDER BY (Step 6) runs after SELECT (Step 5), so aliases are already defined and safe to use. This is the opposite of HAVING (Step 4), which runs before SELECT — making aliases unreliable there.

Q5: Walk through the SQL execution order.

FROM (1) → WHERE (2) → GROUP BY (3) → HAVING (4) → SELECT (5) → ORDER BY (6). This order explains everything: why COUNT fails in WHERE, why aliases fail in HAVING (SQL Server), and why aliases work in ORDER BY.
