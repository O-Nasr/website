What is an Index?

A separate data structure storing sorted column values with pointers to rows. Enables O(log n) lookups instead of O(n) full table scans.

For 10 million rows: without index = 10,000,000 checks. With B-Tree = ~23 steps. For 1 billion rows: ~30 steps.
⚠️ Read/Write trade-off: Every index speeds up reads but slows down writes (INSERT, UPDATE, DELETE must update all indexes). Never add indexes blindly.

3

Index Types

Includes all columns the query needs directly inside the index → Index Only Scan (fastest possible read).

SQL-- Regular: finds customer_id, then Key Lookup for other cols
CREATE INDEX idx ON orders(customer_id);

-- Covering: everything needed is in the index
CREATE INDEX idx ON orders(customer_id) INCLUDE (id, total, status);
SQLCREATE INDEX idx ON orders(customer_id, status);

-- ✅ Uses index (leading column present)
WHERE customer_id = 5
WHERE customer_id = 5 AND status = 'active'
WHERE status = 'active' AND customer_id = 5  -- optimizer reorders!

-- ❌ Does NOT use index (leading column missing)
WHERE status = 'active'
WHERE clause order does NOT matter — optimizer reorders. What matters is the column order in the index definition.
SQL-- Only index pending orders (50K out of 10M rows)
CREATE INDEX idx ON orders(customer_id) WHERE status = 'pending';
-- 200x smaller than a full index!

Test Yourself

Q1: Why can there only be ONE clustered index per table?

A clustered index determines the physical row order on disk. You can't sort rows by customer_id AND order_date simultaneously — just like a filing cabinet can only be organized one way.

Q2: Index on (customer_id, status) — does WHERE status = 'active' use it?

The Left-Most Prefix Rule: the query must include the leading column(s) of the composite index. Since customer_id is missing, the index is skipped.

Q3: What is a Key Lookup and how do you fix it?

A Key Lookup means the non-clustered index found matching rows, but the query needs columns not in the index. The fix is a covering index with INCLUDE to store all needed columns, enabling an Index Only Scan.

Q4: Why does WHERE YEAR(created_at) = 2024 not use the index?

The index stores raw created_at values. Applying YEAR() means the DB must evaluate the function on every row — defeating the index. Fix: use range conditions like created_at >= '2024-01-01' AND created_at < '2025-01-01'.
