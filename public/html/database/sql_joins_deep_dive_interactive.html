CROSS JOIN & SELF JOIN

Every row from left × every row from right. No ON needed. 3 sizes × 4 colors = 12 product variants.

SQLSELECT sizes.name, colors.name
FROM sizes CROSS JOIN colors;
-- S-Red, S-Blue, S-Black, S-White, M-Red, M-Blue...
⚠️ Danger: 1000 × 1000 = 1,000,000 rows. A forgotten ON condition accidentally creates a CROSS JOIN — most common cause of queries that never finish.

Same table aliased twice. Used when a column references another row in the same table (e.g., manager_id → employee_id).

SQLSELECT
    emp.name        AS employee,
    mgr.name        AS manager
FROM employees AS emp
LEFT JOIN employees AS mgr
    ON emp.manager_id = mgr.employee_id;
-- Ahmed | NULL   ← CEO, no manager
-- Sara  | Ahmed
-- Omar  | Ahmed
-- Mona  | Sara
Why LEFT JOIN not INNER? Ahmed has no manager (NULL). INNER JOIN would drop the CEO from results. LEFT JOIN keeps all employees.
4

ON vs WHERE — The Senior-Level Distinction

SQL-- ❌ WRONG: Omar disappears (LEFT JOIN → INNER JOIN silently)
SELECT c.name, o.total
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.total > 100;  -- NULL > 100 = NULL = false → Omar removed!

-- ✅ CORRECT: Move filter to ON — preserves LEFT JOIN behavior
SELECT c.name, o.total
FROM customers c
LEFT JOIN orders o
    ON c.id = o.customer_id
    AND o.total > 100;  -- filters which orders match, not which customers
Key Rule: WHERE on the right table filters the final result (removes NULLs). ON filters which rows get matched (keeps all left rows). Moving the filter to ON preserves LEFT JOIN behavior.
5

Test Yourself

Q1: INNER JOIN vs LEFT JOIN — what happens to Omar (no orders)?

INNER JOIN only returns matched rows — Omar has no orders, so no match, so excluded. LEFT JOIN returns ALL left rows — Omar stays with NULL for all right-side (order) columns.

Q2: You LEFT JOIN to get all customers, but customers with no orders are missing. Why?

After LEFT JOIN, customers with no orders have NULL for order columns. WHERE orders.total > 100 evaluates NULL > 100 = NULL = false, removing those rows. Fix: move the filter into the ON condition.

Q3: Why is RIGHT JOIN almost never used?

FROM customers RIGHT JOIN orders is identical to FROM orders LEFT JOIN customers. LEFT JOIN is more natural (left to right reading), so developers default to it and just swap tables.

Q4: What does a CROSS JOIN of 1000 rows × 1000 rows produce?

CROSS JOIN returns every possible combination: m × n rows. This is why a forgotten ON condition (which creates an accidental CROSS JOIN) is one of the most common causes of queries that never finish.

Q5: In a SELF JOIN for employee-manager, why use LEFT JOIN instead of INNER JOIN?

The top-level employee (CEO) has NULL for manager_id. INNER JOIN requires a match on both sides — no match means the CEO disappears. LEFT JOIN keeps all employees, showing NULL for manager name when none exists.

Interview Answer Bank

INNER returns only matched rows — non-matching excluded. LEFT returns all left rows + matched right rows, NULL if no match. Use INNER when you need rows with relationships in both tables. Use LEFT when all rows from one side must appear regardless of matches.
WHERE on the right table removes NULL rows — silently converting LEFT JOIN to INNER JOIN. NULL > 100 = NULL = false → row removed. Fix: move the filter into the ON condition, which filters which rows get matched without affecting which left rows appear.
ON filters during the join (which rows match). WHERE filters after the join (final result). For INNER JOIN the difference is minimal. For LEFT JOIN it's critical — WHERE removes NULLs, ON preserves all left rows. This is the senior-level distinction most candidates miss.
Cartesian product — every combination of rows. Real use: generating product variants (3 sizes × 4 colors = 12 SKUs). Danger: forgotten ON condition creates accidental CROSS JOIN. On large tables → billions of rows, query never finishes.
Table joined to itself with two aliases. Used when a column references another row in the same table — employee-manager hierarchies, category-parent trees. Use LEFT JOIN so top-level items (CEO, root category) with NULL parent still appear.
