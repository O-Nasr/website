Committed = permanent, even after crashes.

Bank transfer: Deduct $100 from A, add $100 to B. If crash mid-way, both operations roll back. Implemented via Write-Ahead Logging (WAL) — changes logged to disk before applying. Undo logs reverse uncommitted changes on recovery.

Constraint: balance >= 0. If a TX tries to set balance = -500, the DB rejects and rolls back. Both the DB (structural constraints) and the developer (business logic) are responsible.

Isolation levels define how much interaction is allowed between concurrent TXs. Implemented via Locking (pessimistic) or MVCC (keeps multiple row versions — writers don't block readers). See deep dive below.

How: Changes written to WAL on disk first → fsync forces to physical disk → COMMIT acknowledged → On crash, DB replays WAL to restore all committed TXs.

Isolation — Deep Dive

The 3 core anomalies:

AnomalyWhat Happens
Dirty ReadReading data another TX modified but NOT committed yet. May be rolled back.
Non-Repeatable ReadSame row read twice → different values (another TX updated & committed in between).
Phantom ReadSame query twice → different ROW SET (another TX inserted/deleted matching rows).
Key distinction: Non-Repeatable = existing row changed. Phantom = row set changed (new/deleted rows).

Test Yourself

Q1: Bank transfer crashes after deducting A but before adding to B. Which property handles this?

Atomicity = all or nothing. The DB uses WAL and undo logs to roll back the incomplete deduction. Neither account is modified — as if the transaction never happened.

Q2: What's the difference between Non-Repeatable Read and Phantom Read?

Non-Repeatable: you read the same row twice and get different values because another TX updated it. Phantom: you run the same query twice and get a different set of rows because another TX inserted or deleted matching rows.

Q3: 1000 concurrent users, good performance, minor inconsistencies OK. Which isolation level?

Read Committed prevents dirty reads while avoiding long-term locks. It's PostgreSQL's default for a reason — good balance. Serializable would be too expensive for 1000 concurrent users.

Q4: TX commits, then power failure 1 second later. Is the data still there?

Durability: the DB writes to WAL and calls fsync (forces to physical disk) BEFORE acknowledging COMMIT. On restart, it replays the WAL to restore all committed transactions.

Interview Answer Bank

Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (no interference between concurrent TXs), Durability (committed survives crashes). Bank transfer example: deduct A + add B must both succeed or both roll back. ACID ensures reliable transaction processing.
Multiple transactions run concurrently and without control they can interfere — dirty reads (reading uncommitted data), non-repeatable reads (same row changed), phantom reads (row set changed). Isolation levels balance safety vs performance.
Changes written to WAL (Write-Ahead Log) on disk first → fsync forces to physical disk → COMMIT acknowledged to client. On crash, DB replays WAL to restore all committed transactions. This is why committed = permanent.
