The Problem Window Functions Solve

GROUP BY collapses rows — you lose individual details. Window functions compute across rows without collapsing them.

SQL-- GROUP BY: 2 rows, order details GONE
SELECT customer_id, MAX(total) FROM orders GROUP BY customer_id;

-- Window function: ALL 4 rows, each knows its customer max
SELECT order_id, customer_id, total,
    MAX(total) OVER (PARTITION BY customer_id) AS customer_max
FROM orders;
Key Insight: Window functions perform calculations across rows WITHOUT collapsing them. Every row stays. This is the fundamental difference from GROUP BY.

⚠️ Top N Per Group: Use ROW_NUMBER, not RANK. If two orders tie, RANK gives both rank=1 → more than N rows. ROW_NUMBER always picks exactly N. Must wrap in CTE/subquery — can't filter window functions in WHERE.
SQL-- Top 1 order per customer (classic interview question)
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY customer_id ORDER BY total DESC
    ) AS rn FROM orders
)
SELECT * FROM ranked WHERE rn = 1;

LAG & LEAD — Access Other Rows

Access previous/next row values without a self join:

order_idtotalLAG(total,1)change
101500NULLNULL
102200500-300
103300200+100
104150300-150

LAG = previous row (NULL for first). LEAD = next row (NULL for last). Real use: month-over-month growth, daily comparisons.

SQL-- NTILE: divide into quartiles
NTILE(4) OVER (ORDER BY total_spent DESC) AS quartile
-- quartile 1 = top 25%, quartile 4 = bottom 25%

-- FIRST_VALUE: first order total per customer
FIRST_VALUE(total) OVER (
    PARTITION BY customer_id ORDER BY order_id
) AS first_order

ROWS BETWEEN — Frame Definition

Controls exactly which rows are included in the calculation window:

FrameMeaningUse Case
UNBOUNDED PRECEDING AND CURRENT ROWFirst row → currentRunning total
2 PRECEDING AND CURRENT ROWLast 3 rowsMoving average
CURRENT ROW AND UNBOUNDED FOLLOWINGCurrent → last rowReverse running
UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWINGAll rowsPartition total
SQL-- 7-day moving average
SELECT sale_date, daily_total,
    AVG(daily_total) OVER (
        ORDER BY sale_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS moving_avg_7d
FROM daily_sales;
⚠️ Window functions run after WHERE/GROUP BY/HAVING but before final SELECT. Can't use in WHERE — must wrap in CTE/subquery first.

Test Yourself

Q1: ROW_NUMBER vs RANK — two rows tied at 500. What happens?

ROW_NUMBER never ties — assigns unique sequential numbers. RANK gives tied values the same rank but skips the next. DENSE_RANK gives same rank but doesn't skip: 1, 1, 2.

Q2: Top 1 per customer — why ROW_NUMBER not RANK?

RANK returns ALL tied rows at rank 1 — so you might get 2 or 3 rows per customer. ROW_NUMBER arbitrarily picks one even for ties, guaranteeing exactly 1 row per partition when filtered with WHERE rn = 1.

Q3: What does ORDER BY inside OVER do to SUM()?

Without ORDER BY in OVER: SUM returns the same total for every row in the partition. With ORDER BY: SUM becomes cumulative — each row shows the running sum up to that point. The ORDER BY creates the "running" behavior.

Q4: Why can't you use a window function in WHERE?

Execution order: FROM → WHERE → GROUP BY → HAVING → Window Functions → SELECT → ORDER BY. Window functions run after WHERE, so at WHERE time the window result doesn't exist. Wrap in CTE/subquery first.

Q5: What is LAG(total, 1) used for?

LAG(column, n) looks back n rows. Real use: month-over-month revenue (current - LAG(revenue,1) = change). Without LAG, you'd need a complex self join with offset conditions.

Interview Answer Bank

Window functions compute across rows WITHOUT collapsing them — every row stays. GROUP BY reduces to one row per group, losing individual details. With GROUP BY you get max total per customer but lose order_id. With OVER(PARTITION BY) every order row stays and also shows the customer max.
All assign positions but differ on ties. ROW_NUMBER = 1,2,3,4 (always unique). RANK = 1,1,3,4 (ties share, gaps after). DENSE_RANK = 1,1,2,3 (ties share, no gaps). Use ROW_NUMBER for exactly-one-per-group. RANK/DENSE_RANK when ties should be visible.
SUM(total) OVER (PARTITION BY customer_id ORDER BY order_id). The ORDER BY inside OVER changes SUM from partition total to cumulative. Without ORDER BY = same total for all rows. With = running sum up to current row.
LAG(column, n) returns value from n rows back. Use: month-over-month revenue growth — current month minus LAG(revenue, 1). Also daily metrics vs yesterday. Without LAG you'd need a complex self join. LAG is cleaner and faster.
Execution order: WHERE runs before window functions. At WHERE time, the window result doesn't exist. Fix: wrap in CTE or subquery. Window function runs in inner query, outer WHERE filters the result.
