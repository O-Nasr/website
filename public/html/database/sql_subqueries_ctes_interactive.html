Scalar Subquery — Returns Exactly One Value

SQL-- Orders above the average total
SELECT order_id, total
FROM orders
WHERE total > (
    SELECT AVG(total) FROM orders  -- returns one number e.g. 450.00
);
⚠️ If a scalar subquery returns more than one row → ERROR. Always ensure it can only return one value.

Derived Table — Virtual Table in FROM

SQL-- Average orders per customer
SELECT AVG(order_count) AS avg_orders
FROM (
    SELECT customer_id, COUNT(*) AS order_count
    FROM orders GROUP BY customer_id
) AS customer_totals;  -- alias is REQUIRED
⚠️ Derived tables MUST have an alias. Forgetting it causes an error in every database.

Correlated Subquery — Runs Per Row (Slow!)

SQL-- Customers whose total spending > 1000
SELECT name FROM customers c
WHERE (
    SELECT SUM(total) FROM orders o
    WHERE o.customer_id = c.customer_id  -- references OUTER query
) > 1000;
-- 1M customers → inner query runs 1M times! 😱
⚠️ Performance: Runs once per outer row. Rewrite as JOIN + GROUP BY for large tables.

EXISTS — Check If Any Row Exists

SQL-- Customers who have at least one order
SELECT name FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o      -- SELECT 1: value doesn't matter
    WHERE o.customer_id = c.customer_id
);

-- NOT EXISTS: customers with NO orders
WHERE NOT EXISTS ( ... );
EXISTS stops at the first match — fast! SELECT 1 is convention showing only existence matters, not the actual data.

CTE — Common Table Expression

A named temporary result set using WITH. Breaks complex queries into readable steps. Only exists for the single query it belongs to.

Same logic — two ways to write it:

SQL-- Deeply nested — hard to read and debug
SELECT name FROM customers
WHERE customer_id IN (
    SELECT customer_id FROM orders
    WHERE total > (
        SELECT AVG(total) FROM orders
    )
);
SQL-- Clean, named steps — reads like a recipe
WITH avg_total AS (
    SELECT AVG(total) AS avg FROM orders
),
high_value_orders AS (
    SELECT customer_id FROM orders
    WHERE total > (SELECT avg FROM avg_total)
)
SELECT name FROM customers
WHERE customer_id IN (SELECT customer_id FROM high_value_orders);
SQL-- Multiple CTEs — each builds on the previous
WITH
completed_orders AS (
    SELECT customer_id, total FROM orders WHERE status = 'completed'
),
customer_totals AS (
    SELECT customer_id, SUM(total) AS total_spent
    FROM completed_orders  -- references first CTE
    GROUP BY customer_id
),
top_customers AS (
    SELECT customer_id FROM customer_totals
    WHERE total_spent > 1000  -- references second CTE
)
SELECT name FROM customers
WHERE customer_id IN (SELECT customer_id FROM top_customers);

Recursive CTE — Hierarchies & Trees

A CTE that calls itself. Base case + recursive case joined with UNION ALL.

SQL-- Employee hierarchy traversal (PostgreSQL)
WITH RECURSIVE employee_tree AS (
    -- Base case: CEO (no manager)
    SELECT employee_id, name, manager_id, 0 AS level
    FROM employees WHERE manager_id IS NULL

    UNION ALL

    -- Recursive case: find direct reports
    SELECT e.employee_id, e.name, e.manager_id, et.level + 1
    FROM employees e
    INNER JOIN employee_tree et ON e.manager_id = et.employee_id
)
SELECT name, level FROM employee_tree ORDER BY level;

-- Ahmed │ 0  ← CEO
-- Sara  │ 1  ← reports to Ahmed
-- Omar  │ 1  ← reports to Ahmed
-- Mona  │ 2  ← reports to Sara

5

Test Yourself

Q1: Why is a correlated subquery slow?

A correlated subquery depends on the outer row, so it can't run independently — it must execute for every single outer row. Fix: rewrite as JOIN + GROUP BY for a single-pass approach.

Q2: WHERE customer_id NOT IN (SELECT customer_id FROM orders) — orders has a NULL customer_id. What happens?

Any comparison with NULL is unknown. NOT IN can't determine if the outer value is "not in" a list containing NULL — so every row evaluates to unknown and nothing is returned. Fix: use NOT EXISTS which handles NULLs correctly.

Q3: When would you use a CTE over a subquery?

Performance is typically the same — the optimizer treats them similarly. CTEs win on readability (named steps), reusability (reference the same CTE multiple times), and recursion support (subqueries can't recurse).

Q4: What are the two required parts of a recursive CTE?

The base case provides starting rows (e.g., CEO with no manager). The recursive case joins back to the CTE to find the next level (direct reports). UNION ALL combines them. It keeps running until no new rows are found.

Q5: Why does EXISTS use SELECT 1?

EXISTS returns TRUE/FALSE based on whether any rows exist — it doesn't care about the values returned. SELECT 1, SELECT *, or SELECT 'hello' all behave identically. The convention makes intent clear.
