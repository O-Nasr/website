VIEW — Saved SELECT Query

SQLCREATE VIEW active_customers AS
SELECT customer_id, name, email
FROM customers WHERE status = 'active';

-- Use like a table
SELECT * FROM active_customers WHERE city = 'Cairo';

Benefits: Simplify complex JOINs, security (expose only specific columns), reusability, abstraction.
Limitations: No data stored (runs fresh every time), no parameters, no plan caching.

Materialized View — Cached on Disk

SQL-- PostgreSQL only
CREATE MATERIALIZED VIEW monthly_sales AS
SELECT DATE_TRUNC('month', order_date) AS month, SUM(total)
FROM orders GROUP BY 1;

-- Must refresh manually!
REFRESH MATERIALIZED VIEW monthly_sales;

Fast reads (data pre-stored), but stale until refreshed. SQL Server equivalent: Indexed View.

Stored Procedure — Reusable Program

SQL-- SQL Server
CREATE PROCEDURE GetCustomerOrders
    @customer_id INT, @min_total DECIMAL
AS BEGIN
    SELECT order_id, total FROM orders
    WHERE customer_id = @customer_id AND total >= @min_total;
END;

EXEC GetCustomerOrders @customer_id = 5, @min_total = 100;

Plan cached on first call → reused every call. Can contain logic, loops, transactions, INSERT/UPDATE/DELETE.

Scalar UDF — Returns One Value

SQLCREATE FUNCTION GetFullName(@first VARCHAR(50), @last VARCHAR(50))
RETURNS VARCHAR(100) AS BEGIN
    RETURN @first + ' ' + @last;
END;

-- Used INLINE in SELECT
SELECT dbo.GetFullName(first_name, last_name) FROM customers;
⚠️ Black box to optimizer! Runs row-by-row. 1M rows = 1M individual calls. Can severely hurt performance.

Table-Valued UDF — Returns a Table

SQLCREATE FUNCTION GetOrdersByCustomer(@customer_id INT)
RETURNS TABLE AS RETURN (
    SELECT order_id, total, status FROM orders
    WHERE customer_id = @customer_id
);

-- Use like a view WITH parameters!
SELECT * FROM GetOrdersByCustomer(5) WHERE status = 'completed';

Inline table-valued functions — optimizer can see inside. Much better than scalar UDFs for performance.

Parameter Sniffing

The plan cached for the first parameter values may be wrong for different values:

SQL-- First call: customer_id=5 returns 10 rows → plan optimized for small result
EXEC GetCustomerOrders @customer_id = 5;

-- Second call: customer_id=1 returns 1,000,000 rows → reuses BAD plan!
EXEC GetCustomerOrders @customer_id = 1;

-- Fix 1: RECOMPILE every call
CREATE PROCEDURE ... WITH RECOMPILE AS ...

-- Fix 2: RECOMPILE one specific call
EXEC GetCustomerOrders @customer_id = 1 WITH RECOMPILE;

-- Fix 3: local variable hides parameter from optimizer
DECLARE @id INT = @customer_id
SELECT * FROM orders WHERE customer_id = @id;
⚠️ Parameter sniffing is the counter-point to plan caching. First call's plan may be terrible for later calls with different data. Always mention this trade-off in interviews.

6

Test Yourself

Q1: Can you use a stored procedure in a SELECT statement?

Stored procedures cannot be used inline in SELECT, WHERE, or JOIN. If you need inline logic in a query, use a scalar UDF (returns one value) or inline table-valued function (returns a table).

Q2: What is the main performance benefit of stored procedures?

On first call, the optimizer builds and caches the plan. Every subsequent call reuses it — skipping the expensive optimization step entirely. The risk: parameter sniffing can make the cached plan wrong for different inputs.

Q3: Why can scalar UDFs hurt performance in SQL Server?

Scalar UDFs prevent the optimizer from generating efficient plans. They execute once per row — no batching, no parallelism. Fix: inline the logic, use inline table-valued functions, or SQL Server 2019+ scalar UDF inlining.

Q4: View vs Materialized View — key trade-off?

Views run the query fresh every time (always current, can be slow). Materialized views store data on disk (fast reads, but stale until you run REFRESH MATERIALIZED VIEW). The staleness trade-off is the key operational cost.

Q5: What is parameter sniffing?

First call with customer_id=5 (10 rows) → plan optimized for small result. Second call with customer_id=1 (1M rows) → reuses that small-result plan → terrible performance. Fix: RECOMPILE hint or local variable technique.
