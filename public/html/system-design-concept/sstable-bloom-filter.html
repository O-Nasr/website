1

LSM Tree (Log-Structured Merge Tree)

The data structure behind write-optimized databases — append-only, never update in place.

Core Idea

Instead of updating data in place (like B-Trees in relational DBs), LSM trees write new data sequentially to disk (append-only), keep recent data in memory (Memtable), periodically flush to disk as SSTables, and use compaction to merge and clean up. Sequential writes are much faster than random writes — that's the key insight.

🧠Relationship: LSM Tree = Algorithm + Structure. SSTable = Storage format used by LSM Tree. LSM organizes data across Memtable + SSTables + Compaction.
2

SSTable (Sorted String Table)

A persistent, immutable, sorted file format — the physical representation of LSM Tree data on disk.

Why SSTables?

Optimized for write-heavy workloads. Sequential disk writes are fast and efficient. Great for huge datasets (billions of keys). Used by Cassandra, Bigtable, ScyllaDB, RocksDB, LevelDB, HBase.

3

Write Path

Data → Commit Log + Memtable → flush to SSTable when full.
4

Read Path

Check Memtable first → then search SSTables using Bloom Filters + sparse index.
5

Bloom Filter

A space-efficient probabilistic data structure — "is this key definitely NOT here?"

Benefits in SSTables

Each SSTable has its own Bloom filter. On a read, the system checks each SSTable's Bloom filter first. If it says "not present" → skip that file entirely. This avoids unnecessary disk seeks and dramatically speeds up reads when there are many SSTables. Very memory-efficient: just a few bits per key.

6

Compaction

Periodically merge SSTables to remove duplicates, tombstones, and speed up reads.
