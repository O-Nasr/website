What Are Sticky Sessions?

Also called session affinity — ensuring a user always hits the same server.

Definition

Sticky sessions (session affinity) are a load balancing feature that ensures a user's requests are always routed to the same backend server during a session. In stateful systems, session data (login state, shopping cart, etc.) is stored in memory on the server. If a request goes to a different server, that data is lost.

🧠Why it exists: In stateful systems, user session data lives in the server's memory. Sticky sessions "stick" the user to one server so they consistently access their session data.
2

How Sticky Sessions Work

Cookie-based tracking: the load balancer sets a cookie to remember which server handled the user.
3

Without Sticky Sessions

When a request hits a different server, session data is lost — user gets logged out.
4

The Better Way: Stateless Design

Store session data in a shared cache (Redis) — any server can handle any request.

Stateful vs Stateless Comparison

Sticky Sessions in Multi-DC Context

In multi-region setups, sticky sessions can tie a user to a specific data center.

Mechanisms for Sticky Sessions

MethodHow It Works

IP hashingHash the client IP to consistently route to the same server
Custom routing logicApplication-level routing based on user ID or session token

Pros & Cons

✅ Pros

  • Maintains session state — useful for stateful apps
  • Avoids session loss — prevents user disruption
  • Easy to implement — just enable cookie-based stickiness

❌ Cons

  • Breaks stateless design — ties user to a specific server
  • Poor scalability — one server may get overloaded
  • Harder failover — if server crashes, session is lost

Best Practice

🏆If possible, design your system to be stateless. Use distributed session stores (Redis). Avoid sticky sessions unless absolutely necessary. Stateless = scalable, fault-tolerant, and easy to operate.

Interview Questions

▶What are sticky sessions and when would you use them?
Sticky sessions (session affinity) route a user's requests to the same backend server using cookies or IP hashing. Use them in stateful systems where session data is stored in server memory — but prefer stateless design with a shared session store (Redis) when possible.
▶What happens if the sticky server crashes?
The session data is lost — the user may be logged out, lose their shopping cart, or see an error. This is the main weakness of sticky sessions. With a shared session store (Redis), any other server can pick up seamlessly.
▶Why is stateless design preferred over sticky sessions?
Stateless design stores session data in a shared cache (Redis), so any server can handle any request. This gives better scalability (no hot servers), better fault tolerance (server crash = no session loss), and simpler load balancing (pure round-robin works).
