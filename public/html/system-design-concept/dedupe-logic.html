Dedupe Logic & Idempotency

Why distributed systems can't guarantee exactly-once delivery — and how event ID deduplication makes at-least-once delivery safe.

1

Why Duplicates Happen

Worker crashes after sending but before acknowledging — queue re-delivers the same message.
🔥Why exactly-once is impossible: You cannot distinguish between "message was processed but ACK lost" and "message was never processed." Both look identical from the sender's perspective. This is the Two Generals Problem.
2

Dedupe Logic (Event ID Check)

At-least-once delivery + idempotency = effectively-once processing.

Three Related Concepts

ConceptDefinition
DeduplicationDetect and discard duplicate messages using event IDs
IdempotencyRunning an operation multiple times produces the same result as running it once
Exactly-once deliveryA theoretical guarantee — almost impossible in pure distributed systems
🧠Your design is: At-least-once delivery + dedupe logic = effectively-once processing. That's the correct mental model.

Redis SETNX Pattern

Atomic, fast, with automatic TTL expiration — the perfect dedupe store.
Redis — Dedupe Command
// Atomic dedupe check + store in one command
SET event_id value NX EX 86400

// NX = only set if NOT exists (dedupe check)
// EX 86400 = expire after 24 hours (TTL)

// If command succeeds → new event → PROCESS
// If command fails   → duplicate → DISCARD

Notification System Architecture (Figure 10-14)

The full notification pipeline — where dedupe fits, what templates are, and where event IDs live.

What is a Notification Template?

A reusable message blueprint with placeholders. Example: Template ORDER_SHIPPED → "Hi {{first_name}}, your order {{order_id}} has shipped." At runtime, the worker fetches the template, fills placeholders, and sends the final message. Benefits: centralized control, easy A/B testing, localization (English/Arabic).

Where Does Event ID Live?

Store in the Notification Log DB — not in the user/device DB. Dedupe belongs to delivery tracking (event_id, status, timestamp, channel), not user state (device tokens, preferences, settings). In high-scale systems: fast-path dedupe in Redis + durable record in Notification Log DB.

What If Redis Is Down?

The answer depends on the business — notifications vs payments have opposite priorities.

Three Options When Dedupe Is Unavailable

OptionBehaviorPrioritizesRisk
A) RejectDrop all notificationsConsistencyUsers miss messages
B) Send anywaySkip dedupe, risk duplicatesAvailabilityDuplicate notifications
C) Block/waitRetry until Redis returnsCorrectnessCascading failures, thread exhaustion

Decision by System Type

System TypeBest ChoiceWhy
NotificationsB) Send anywayDuplicate notification is annoying. Missing notification can be critical. Blocking can crash the pipeline.
PaymentsC) Block / fail safelyDuplicate charge is catastrophic. Better to delay than double-charge. Retry with circuit breaker, fail if still down.
🧠Senior thinking: Mitigate with circuit breaker around Redis, retry with backoff, alerting/monitoring, and multi-region Redis replication. For payments: bounded retries, then fail the transaction — let client retry later.

Interview Answers

▶"Why can't we guarantee exactly-once delivery?"
In distributed systems, network failures and crashes make it impossible to distinguish between a lost acknowledgment and a failed operation. Therefore, systems typically provide at-least-once delivery and rely on idempotent processing to prevent duplicate side effects.
▶"How do you handle duplicate notification events?"
We use idempotent processing by storing processed event IDs. Before sending a notification, we check whether the event ID has already been processed. If it has, we discard it to prevent duplicate delivery.
▶"Where would you store dedupe event IDs?"
I would use a distributed in-memory store like Redis with TTL support. It provides low-latency lookups, atomic operations like SETNX to prevent duplicates, and automatic expiration to prevent unbounded memory growth.
▶"If Redis is unavailable, would you still send without dedupe for payments?"
No. In payment systems, duplicate processing is unacceptable. I would prefer to block with bounded retries or fail the transaction rather than risk double charging. Payments prioritize strong consistency over availability.
▶"Where does dedupe fit in real systems?"
Dedupe is used in payment systems (critical), notification systems, webhooks (e.g. Stripe retries), Kafka consumers, and event-driven architectures. For example, if Stripe retries a webhook, your server must dedupe using the event_id.
