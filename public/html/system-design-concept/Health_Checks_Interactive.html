01 Why Health Checks Exist

In a monolith, your service is either running or it's not. In microservices with Kubernetes, a pod can be in many states between "fully dead" and "fully ready":

Starting up
Process is alive but hasn't loaded cache or connected to DB yet. If K8s sends traffic now → errors. Need: startup probe to give it time, readiness probe to block traffic until ready.
Overloaded
Alive but too busy to accept new requests. Need: readiness probe to temporarily remove from load balancer. Don't kill it — just stop sending traffic until it recovers.
Deadlocked
Process is alive (PID exists, memory allocated) but threads are stuck waiting on each other. Will never recover on its own. Need: liveness probe to detect and restart.
Partially broken
HTTP works but database connection is lost. Process is alive (liveness passes) but can't do useful work (readiness fails). Don't restart — just wait for DB to reconnect.
Gracefully shutting down
Finishing in-flight requests, shouldn't get new ones. Readiness fails → K8s stops routing traffic. Liveness still passes → K8s doesn't kill it prematurely.

K8s needs to know which state your pod is in to make the right decision. That's what the three probe types are for.

02 The 3 Probe Types

Liveness probe — "Is the process alive?"
On failure: K8s kills and restarts the pod.
Endpoint: /health/live or /healthz
What it checks: "Can the process respond to HTTP at all?" — minimal, just returns 200 OK.
Purpose: Detect deadlocks, infinite loops, memory leaks — unrecoverable situations.
Real scenario: Orders service has a thread pool. A bug causes all threads to deadlock. Process is alive (PID exists) but can't handle any request. Without liveness: K8s thinks it's fine, routes traffic to a dead pod. With liveness: probe times out → K8s kills the pod → fresh one starts → deadlock gone.
Readiness probe — "Can it handle traffic right now?"
On failure: K8s removes from Service endpoints (stops routing traffic) but does NOT kill the pod.
Endpoint: /health/ready
What it checks: "Can I do useful work?" — checks dependencies: DB, Redis, external APIs.
Purpose: Temporarily take pod out of rotation when alive but can't serve.
Real scenario: Service starts and needs to warm a cache from Redis (30 seconds). During those 30 seconds, liveness passes but readiness fails. K8s doesn't route traffic → cache loads → readiness passes → traffic flows. Users never see errors.
Startup probe — "Has it finished starting?"
On failure: K8s kills and restarts (same as liveness) — but only during startup.
Endpoint: Same as liveness, or dedicated /health/startup
What it checks: "Has the app finished initialization?" — migrations, cache loading, config.
Purpose: Protect slow-starting apps from being killed by liveness before they're ready. Once startup passes, it never runs again.
Real scenario: Service runs EF Core migrations on startup — takes 60 seconds. Liveness has 10-second timeout. Without startup: liveness fails 6 times during migration → K8s kills → restart → migration starts again → infinite CrashLoopBackOff. With startup: gets 300s to complete → migrations finish → liveness takes over.

03 All Three Working Together

Pod created
  → Startup probe begins checking
  → App is starting (migrations, cache load)... startup fails, that's OK
  → App finishes startup → startup probe passes
  → Startup probe DISABLES ITSELF (never runs again)

Liveness probe + Readiness probe take over
  → Liveness: checks every 20s — "are you alive?"
  → Readiness: checks every 10s — "can you handle traffic?"

Normal operation:
  → Both pass → pod receives traffic ✅
Database connection lost:
  → Liveness still passes (process is alive, can respond to HTTP)
  → Readiness FAILS (can't do useful work without DB)
  → K8s removes pod from Service endpoints (no new traffic)
  → Pod stays alive, keeps trying to reconnect
  → DB recovers → readiness passes → K8s adds pod back → traffic flows

Key: pod was NOT restarted. Just temporarily removed from rotation.
Thread deadlock occurs:
  → Liveness probe times out (process can't respond to HTTP)
  → First failure... second failure... third failure (failureThreshold: 3)
  → K8s kills the pod
  → Fresh pod starts → startup probe begins
  → Startup passes → liveness + readiness take over
  → New pod is healthy ✅

Key: liveness detected an UNRECOVERABLE issue and restarted.

04 Full ASP.NET Core Implementation

// Program.cs — register all health check dependencies
builder.Services.AddHealthChecks()
    // Can we reach the database?
    .AddDbContextCheck<OrdersDbContext>(
        name: "database",
        tags: new[] { "ready" })       // tagged as readiness dependency

    // Can we reach Redis?
    .AddRedis(
        builder.Configuration.GetConnectionString("Redis"),
        name: "redis",
        tags: new[] { "ready" })

    // Can we reach the payment provider?
    .AddUrlGroup(
        new Uri("https://api.stripe.com/v1"),
        name: "stripe",
        tags: new[] { "ready" })

    // Custom: is the cache warm?
    .AddCheck("cache-warmup", () =>
    {
        return CacheService.IsWarmed
            ? HealthCheckResult.Healthy("Cache is warm")
            : HealthCheckResult.Unhealthy("Cache still loading");
    }, tags: new[] { "ready" });
// Map three endpoints — one per probe type

// Liveness: MINIMAL — just "can the process respond?"
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = _ => false   // run NO checks — just return 200
});

// Readiness: FULL — checks all dependencies tagged "ready"
app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready"),
    ResponseWriter = WriteDetailedResponse  // JSON with each check
});

// Startup: same as readiness (once passes, K8s stops calling it)
app.MapHealthChecks("/health/startup", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready")
});
The key design decision: liveness has Predicate = _ => false

It runs NO checks. Just tests "can ASP.NET respond to HTTP?"

If you put database checks in liveness, a temporary DB outage would cause K8s to restart ALL pods — making things worse, not better. Database connectivity belongs in readiness (temporarily stop traffic) not liveness (kill the pod).

05 Detailed Response Writer

When readiness fails, you want to know which dependency is down:

private static async Task WriteDetailedResponse(
    HttpContext context, HealthReport report)
{
    context.Response.ContentType = "application/json";

    var result = new
    {
        status = report.Status.ToString(),
        totalDuration = report.TotalDuration.TotalMilliseconds + "ms",
        checks = report.Entries.Select(e => new
        {
            name = e.Key,
            status = e.Value.Status.ToString(),
            duration = e.Value.Duration.TotalMilliseconds + "ms",
            description = e.Value.Description,
            error = e.Value.Exception?.Message
        })
    };

    await context.Response.WriteAsJsonAsync(result);
}
{
  "status": "Unhealthy",
  "totalDuration": "234ms",
  "checks": [
    { "name": "database",     "status": "Healthy",   "duration": "12ms"   },
    { "name": "redis",        "status": "Unhealthy", "duration": "3001ms",
      "error": "Connection timeout" },
    { "name": "stripe",       "status": "Healthy",   "duration": "89ms"   },
    { "name": "cache-warmup", "status": "Healthy",   "duration": "0ms"    }
  ]
}

// One glance: Redis is down. Everything else is fine.

06 Kubernetes YAML Configuration

containers:
  - name: orders
    image: orders:v2

    # Startup probe — protect slow starters
    startupProbe:
      httpGet:
        path: /health/startup
        port: 5001
      initialDelaySeconds: 5    # wait 5s before first check
      periodSeconds: 5          # check every 5s
      failureThreshold: 60      # allow up to 60 failures
      # 60 × 5s = 300s = 5 minutes max startup time

    # Liveness probe — detect deadlocks
    livenessProbe:
      httpGet:
        path: /health/live
        port: 5001
      initialDelaySeconds: 0    # starts AFTER startup passes
      periodSeconds: 20         # check every 20s
      failureThreshold: 3       # 3 failures = kill and restart
      timeoutSeconds: 5         # each check must respond in 5s

    # Readiness probe — control traffic routing
    readinessProbe:
      httpGet:
        path: /health/ready
        port: 5001
      initialDelaySeconds: 0
      periodSeconds: 10         # check every 10s
      failureThreshold: 3       # 3 failures = remove from endpoints
      successThreshold: 1       # 1 success = add back
      timeoutSeconds: 5
FieldWhat it does
initialDelaySecondsWait this long before first probe check. Gives the app time to start before being evaluated.
periodSecondsHow often to run the probe. Lower = faster detection, higher = less overhead.
failureThresholdHow many consecutive failures before K8s acts. Filters transient issues.
successThresholdHow many consecutive successes before K8s considers the pod healthy again (readiness only).
timeoutSecondsHow long to wait for a response before considering the probe failed.

07 Who Calls These Endpoints?

All three probes are called only by Kubernetes — specifically by the kubelet (K8s agent running on each node). Not by other services. Not by users. Not by the API Gateway.

Users / other services call:
  → /api/orders          (business logic)
  → /api/orders/123      (business logic)
  → /api/payments/charge  (business logic)

Kubernetes (kubelet) calls:
  → /health/live      (liveness)
  → /health/ready     (readiness)
  → /health/startup   (startup)

These endpoints are infrastructure-level, not application-level.
Services never know or care about them.

Health endpoints are typically not exposed outside the cluster. The API Gateway doesn't route /health/* to the outside world. They exist on the internal pod IP, accessible only by the kubelet on the same node.

// Extra safety — restrict to internal port
app.MapHealthChecks("/health/live")
    .RequireHost("*:5001");   // only on internal port

// Public API on a different port or path
app.MapControllers();         // /api/* routes for real traffic

That's why liveness can be super minimal — it's just K8s whispering "you alive?" and your pod whispering back "yeah."

08 The 4 Production Mistakes

Mistake: DB check in liveness probe
Liveness checks database. DB has a 30-second blip. Liveness fails 3 times → K8s kills ALL pods → all pods restart → all reconnect simultaneously → thundering herd → DB overwhelmed → more failures → cascade outage.
Fix: DB check belongs in READINESS only
Readiness fails → K8s stops routing traffic (pod stays alive). DB recovers → readiness passes → traffic flows. No restarts, no thundering herd.
Mistake: No startup probe for slow apps
App runs EF migrations (60 seconds). Liveness starts after 10 seconds. Fails → killed → restart → migrations restart → killed again → infinite CrashLoopBackOff.
Fix: Add startup probe with generous failureThreshold
startupProbe with failureThreshold: 60, periodSeconds: 5 = 5 minutes to start. Once passes, liveness takes over. Startup never runs again.
Mistake: Checking Stripe/external APIs in liveness
Liveness checks if Stripe is reachable. Stripe has 2-minute outage. K8s kills all pods. Your service is DOWN even though your code is fine — restarting healthy pods because Stripe is slow.
Fix: External deps in readiness (or nowhere)
Stripe down? Readiness can fail (stop taking new orders). But your process is healthy — don't restart it. Or better: don't check Stripe at all — handle with circuit breaker + fallback.
Mistake: Too aggressive probe timing
periodSeconds: 1, failureThreshold: 1 → one slow response and K8s kills the pod. GC pauses (50-200ms) cause timeouts. Brief CPU spikes cause liveness to fail.
Fix: Give room for transient issues
periodSeconds: 20, failureThreshold: 3, timeoutSeconds: 5 → must fail 3 checks over 60 seconds before K8s acts. Filters GC pauses, CPU spikes, network blips.

09 The Golden Rule

ProbeWhat to checkWhat NOT to check
LivenessCan the process respond to HTTP? (minimal)Database, Redis, external APIs — NEVER
ReadinessCan I do useful work? (DB, cache, Redis)Nothing too expensive or slow
StartupSame as readiness (has init finished?)N/A — once passes, gone forever
The medical analogy:

Liveness = is the patient breathing? (If no → defibrillator / restart)
Readiness = is the patient able to work? (If no → rest at home / no traffic)
Startup = has the patient woken up from surgery? (If no → keep monitoring, don't panic)

10 Interview Q&A

Q: What's the difference between liveness and readiness probes?
Liveness checks if the process is alive — failure triggers a restart. Readiness checks if it can handle traffic — failure removes from load balancer but keeps running. A service loading its cache is alive but not ready. Critical rule: never put dependency checks (DB, Redis) in liveness — a temporary DB outage would restart all pods, making it worse.
Q: When do you need a startup probe?
When your app takes a long time to initialize — migrations, large caches, remote config. Without it, liveness would kill the pod before it finishes starting (infinite CrashLoopBackOff). Startup probe gives time (e.g., 5 minutes). Once it passes, it disables itself forever — liveness and readiness take over.
Q: What should a liveness probe check?
As little as possible. Ideally just "can the HTTP server respond with 200?" — Predicate = _ => false in ASP.NET Core, running no health checks. Only detects unrecoverable issues (deadlocks, memory corruption). Anything recoverable (DB down, cache cold) belongs in readiness.
Q: Who calls the health check endpoints?
Only Kubernetes — specifically the kubelet on each node. Never by other services, never by users, never by the API Gateway. These are infrastructure-level endpoints, not exposed outside the cluster. They exist on the internal pod IP only.
Q: What happens when readiness fails?
K8s removes the pod from Service endpoints — it stops receiving traffic. But the pod stays alive and keeps running. When readiness passes again, K8s adds it back. This is perfect for temporary issues like a DB blip or cache warming — the pod recovers without being restarted.
Q: Why is putting DB checks in liveness dangerous?
A 30-second DB blip causes liveness to fail → K8s kills all pods → all restart simultaneously → all try to reconnect to DB at once → thundering herd overwhelms the recovering DB → more failures → cascade outage. DB checks belong in readiness — stop traffic but don't restart.
