What is a Feature Flag?

A feature flag (also called a feature toggle) is a conditional in your code that controls whether a piece of functionality is active — without deploying new code. Think of it as a light switch for your features: the code ships to production but stays dormant until you flip the switch.

💡
Core insight: The new code ships to production. It's dormant until you toggle the flag. You can enable it for 1 user, 10%, or everyone — whenever you're ready, with zero redeployment.

The Simplest Possible Example

C#
// ❌ Without a feature flag — hardcoded, requires redeployment to change
public string GetDashboard()
{
    return RenderNewDashboard();
}

// ✅ With a feature flag — controlled at runtime, zero redeployment
public string GetDashboard()
{
    if (_featureFlags.IsEnabled("new-dashboard"))
        return RenderNewDashboard();   // New experience
    else
        return RenderOldDashboard();   // Safe fallback
}

Building it in C# — Layer by Layer

Let's build a production-style feature flag system from the ground up. Each layer adds a capability.

Layer 1 — The Interface

C# — Contract
// The contract: "Can this user see this feature?"
public interface IFeatureFlagService
{
    bool IsEnabled(string featureName);
    bool IsEnabledForUser(string featureName, string userId);
}

Layer 2 — The Configuration Model

C# — Model
public class FeatureFlag
{
    public string Name { get; set; }
    public bool IsEnabled { get; set; }           // Global on/off
    public int RolloutPercentage { get; set; }        // 0–100 for gradual rollout
    public List<string> AllowedUserIds { get; set; } = new();
}

Layer 3 — The Service

C# — Service Implementation
public class FeatureFlagService : IFeatureFlagService
{
    private readonly Dictionary<string, FeatureFlag> _flags;

    public FeatureFlagService(IEnumerable<FeatureFlag> flags)
    {
        // Index flags by name for O(1) lookup
        _flags = flags.ToDictionary(f => f.Name);
    }

    // 1️⃣ Global check — is the flag on at all?
    public bool IsEnabled(string featureName)
    {
        return _flags.TryGetValue(featureName, out var flag) && flag.IsEnabled;
    }

    // 2️⃣ Per-user check — considers rollout % and allow-lists
    public bool IsEnabledForUser(string featureName, string userId)
    {
        if (!_flags.TryGetValue(featureName, out var flag)) return false;
        if (!flag.IsEnabled) return false;

        // Explicitly allowed user
        if (flag.AllowedUserIds.Contains(userId)) return true;

        // Gradual rollout: hash userId to a stable 0–99 bucket
        int bucket = Math.Abs(userId.GetHashCode()) % 100;
        return bucket < flag.RolloutPercentage;
    }
}

Layer 4 — Dependency Injection

C# — Program.cs
builder.Services.AddSingleton<IFeatureFlagService>(sp =>
{
    var flags = new List<FeatureFlag>
    {
        new() { Name = "new-dashboard",  IsEnabled = true,  RolloutPercentage = 20 },
        new() { Name = "dark-mode",      IsEnabled = true,  RolloutPercentage = 100 },
        new() { Name = "ai-suggestions", IsEnabled = false, RolloutPercentage = 0 },
    };
    return new FeatureFlagService(flags);
});

// ✅ Best practice: centralize flag names as constants
public static class FeatureFlags
{
    public const string NewDashboard  = "new-dashboard";
    public const string AiSuggestions = "ai-suggestions";
    public const string DarkMode      = "dark-mode";
}

Layer 5 — Using in a Controller

C# — Controller
[ApiController]
[Route("api/[controller]")]
public class DashboardController : ControllerBase
{
    private readonly IFeatureFlagService _flags;

    public DashboardController(IFeatureFlagService flags)
    {
        _flags = flags;
    }

    [HttpGet]
    public IActionResult GetDashboard()
    {
        var userId = User.FindFirst("sub")?.Value ?? "anonymous";

        if (_flags.IsEnabledForUser(FeatureFlags.NewDashboard, userId))
            return Ok(new { version = "v2", layout = "redesigned" });

        return Ok(new { version = "v1", layout = "classic" });
    }
}
04

Gradual Rollout & Hash Bucketing

The key to a stable rollout is deterministic bucketing. The same userId must always produce the same yes/no answer — on every server, every request, forever.

⚠️
Never use Random(). new Random().Next(100) < 30 gives a different result on every request. The same user gets the feature on one request and loses it on the next. Their UI flickers. Their session breaks. Use hashing.
⚠️
In .NET 5+, string.GetHashCode() is randomized per process. Two separate app instances will produce different results for the same string. Use MD5.HashData() or SHA256 instead — they're deterministic across all machines and restarts.
C# — Stable Hash Bucket
// ✅ MD5-based stable hash — consistent across all processes and machines
private static int StableBucket(string userId)
{
    byte[] bytes = MD5.HashData(Encoding.UTF8.GetBytes(userId));
    int value = BitConverter.ToInt32(bytes, 0);  // take first 4 bytes
    return Math.Abs(value) % 100;               // map to bucket 0–99
}

// How the check works:
// bucket = StableBucket("user-042")  →  e.g. 37
// if rolloutPercentage = 40 and bucket (37) < 40  →  flag is ON
// Same user → same bucket → same result, always

Microsoft.FeatureManagement

The official first-party .NET library. Zero boilerplate — reads from appsettings.json, has built-in filters, and integrates with ASP.NET Core DI.

bash — Install
dotnet add package Microsoft.FeatureManagement.AspNetCore
C# — Program.cs
builder.Services.AddFeatureManagement()
    .WithTargeting();  // enables TargetingFilter (gradual rollout + groups)
JSON — appsettings.json
{
  "FeatureManagement": {
    "NewDashboard": {
      "EnabledFor": [
        {
          "Name": "Microsoft.Targeting",
          "Parameters": {
            "Audience": {
              "Users": ["omar", "alice"],     // always ON for these users
              "Groups": [
                { "Name": "beta-testers", "RolloutPercentage": 100 },
                { "Name": "general",      "RolloutPercentage": 20  }
              ],
              "DefaultRolloutPercentage": 5  // 5% of everyone else
            }
          }
        }
      ]
    }
  }

Built-in Filters

ITargetingContextAccessor — Telling the Library Who the User Is

C# — Targeting Context
public class HttpContextTargetingAccessor : ITargetingContextAccessor
{
    private readonly IHttpContextAccessor _http;
    public HttpContextTargetingAccessor(IHttpContextAccessor http) => _http = http;

    public ValueTask<TargetingContext> GetContextAsync()
    {
        var user = _http.HttpContext?.User;
        return ValueTask.FromResult(new TargetingContext
        {
            UserId = user?.FindFirst("sub")?.Value ?? "anonymous",
            Groups = user?.FindAll("role").Select(c => new GroupRolloutPercentage
            {
                Name = c.Value,
                RolloutPercentage = 100
            }).ToList()
        });
    }
}

// Register in Program.cs
builder.Services.AddHttpContextAccessor();
builder.Services.AddSingleton<ITargetingContextAccessor, HttpContextTargetingAccessor>();

EF Core + Database-Backed Flags

Config files require redeployment to change. A database-backed store lets you toggle flags live at runtime — no code changes, no deploys.

The Entity

C# — Entity
public class FeatureFlagEntity
{
    public int     Id                { get; set; }
    public string  Name              { get; set; } = string.Empty;
    public bool    IsEnabled         { get; set; }
    public int     RolloutPercentage { get; set; }   // 0–100
    public string? AllowedUserIds    { get; set; }   // JSON: ["user1","user2"]
    public DateTime UpdatedAt        { get; set; }
}

Database-Backed Service

C# — DbFeatureFlagService
public class DbFeatureFlagService
{
    private readonly AppDbContext _db;
    public DbFeatureFlagService(AppDbContext db) => _db = db;

    public async Task<bool> IsEnabledForUserAsync(string name, string userId)
    {
        var flag = await _db.FeatureFlags
            .AsNoTracking()          // read-only, no change tracking overhead
            .FirstOrDefaultAsync(f => f.Name == name);

        if (flag is null || !flag.IsEnabled) return false;

        // Check allow-list (JSON array stored in DB column)
        if (flag.AllowedUserIds is not null)
        {
            var allowed = JsonSerializer.Deserialize<List<string>>(flag.AllowedUserIds);
            if (allowed?.Contains(userId) == true) return true;
        }

        return StableBucket(userId) < flag.RolloutPercentage;
    }

    private static int StableBucket(string userId)
    {
        byte[] bytes = MD5.HashData(Encoding.UTF8.GetBytes(userId));
        return Math.Abs(BitConverter.ToInt32(bytes, 0)) % 100;
    }
}

Caching Layer

Every flag check hitting the DB creates massive load at scale. A caching wrapper (Decorator pattern) solves this with no changes to the underlying service.

C# — Cached Wrapper (Decorator Pattern)
public class CachedFeatureFlagService : IFeatureFlagService
{
    private readonly DbFeatureFlagService _inner;
    private readonly IMemoryCache _cache;
    private static readonly TimeSpan CacheDuration = TimeSpan.FromMinutes(1);

    public CachedFeatureFlagService(DbFeatureFlagService inner, IMemoryCache cache)
    {
        _inner = inner;
        _cache = cache;
    }

    public async Task<bool> IsEnabledForUserAsync(string name, string userId)
    {
        // Cache key: unique per flag + user combination
        string cacheKey = $"ff:{name}:{userId}";

        return await _cache.GetOrCreateAsync(cacheKey, async entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = CacheDuration;
            return await _inner.IsEnabledForUserAsync(name, userId);
        });
    }
}
✅
This is the Decorator pattern in action — CachedFeatureFlagService wraps DbFeatureFlagService and adds caching behavior without modifying the underlying class. The controller code never changes.

Testing Feature Flags

Every feature-flagged code path has three test scenarios you must cover: flag ON, flag OFF, and flag missing/default.

Mocking IFeatureManager with Moq

C# — Unit Tests with Moq
public class DashboardServiceTests
{
    private readonly Mock<IFeatureManager> _featureManagerMock;
    private readonly DashboardService _sut;  // System Under Test

    public DashboardServiceTests()
    {
        _featureManagerMock = new Mock<IFeatureManager>();
        _sut = new DashboardService(_featureManagerMock.Object);
    }

    // ✅ Test 1: Flag ON → new dashboard returned
    [Fact]
    public async Task GetDashboard_WhenFlagOn_ReturnsNewDashboard()
    {
        _featureManagerMock
            .Setup(fm => fm.IsEnabledAsync("new-dashboard"))
            .ReturnsAsync(true);

        var result = await _sut.GetDashboardAsync();

        Assert.Equal("v2", result.Version);
    }

    // ✅ Test 2: Flag OFF → old dashboard returned
    [Fact]
    public async Task GetDashboard_WhenFlagOff_ReturnsClassicDashboard()
    {
        _featureManagerMock
            .Setup(fm => fm.IsEnabledAsync("new-dashboard"))
            .ReturnsAsync(false);

        var result = await _sut.GetDashboardAsync();

        Assert.Equal("v1", result.Version);
    }

    // ✅ Test 3: Verify the flag was actually checked (not bypassed)
    [Fact]
    public async Task GetDashboard_AlwaysChecksFlag()
    {
        _featureManagerMock
            .Setup(fm => fm.IsEnabledAsync(It.IsAny<string>()))
            .ReturnsAsync(false);

        await _sut.GetDashboardAsync();

        _featureManagerMock.Verify(
            fm => fm.IsEnabledAsync("new-dashboard"),
            Times.Once);
    }
}

Test Fake — Cleaner Than Mocking

For your custom IFeatureFlagService, a test fake reads more clearly than a mock and is reusable across all test classes:

C# — Test Fake (In-Memory)
// Reusable fake — full control, zero mock setup noise
public class FakeFeatureFlagService : IFeatureFlagService
{
    private readonly Dictionary<string, bool> _flags;

    public FakeFeatureFlagService(Dictionary<string, bool> flags)
    {
        _flags = flags;
    }

    public Task<bool> IsEnabledAsync(string name) =>
        Task.FromResult(_flags.GetValueOrDefault(name, false));

    public Task<bool> IsEnabledForUserAsync(string name, string userId) =>
        Task.FromResult(_flags.GetValueOrDefault(name, false));
}

// Usage — reads like plain English
[Fact]
public async Task PlaceOrder_WithNewCheckout_UsesStripeV2()
{
    var flags = new FakeFeatureFlagService(new()
    {
        [FeatureFlags.NewCheckout]  = true,   // ON
        [FeatureFlags.LoyaltyPoints] = false  // OFF
    });

    var sut = new OrderService(flags);
    var result = await sut.PlaceOrderAsync(testOrder);

    Assert.Equal("stripe-v2", result.PaymentProvider);
}
🚫
Magic string anti-pattern. Never write IsEnabledAsync("new-dashboard") directly in 10 different files. A single typo silently disables the flag. Always use constants: FeatureFlags.NewDashboard. This also makes it easy to find all usages when it's time to delete a flag.

Flag Lifecycle

A feature flag has a defined life — from the moment code is merged to the moment the flag is deleted. Skipping the cleanup step creates flag debt.

⚠️
Flag debt is real. Flags that are never removed accumulate as dead conditional branches. They make code harder to read, test, and maintain — and can interact with each other in unexpected ways. Best practice: every flag gets a deletion date set at creation time.

Interview Q&A

What's the difference between a feature flag and an environment variable?
An env var is static — set at deploy time and needs a restart to change. A feature flag is evaluated at request time and can be toggled live, per user, with percentage rollouts — no restart or redeployment needed.
What is "flag debt" and why is it dangerous?
Flags never removed accumulate as dead conditional branches. They make code harder to read, test, and maintain — and can interact in unexpected ways. Best practice: every flag gets a deletion date at creation time.
How do you prevent inconsistent user experience with gradual rollouts?
Use deterministic hashing — hash the userId (with MD5 or SHA256, not GetHashCode()) to a bucket number 0–99. Same user → same bucket → same flag state every request, on every server, forever. Never use Random().
Why can't you use string.GetHashCode() in production for bucketing?
In .NET 5+, GetHashCode() is randomized per process startup for security reasons. Two app instances will produce different hashes for the same string, meaning the same user could get different bucket assignments across servers. Use MD5.HashData() which is deterministic across all machines and restarts.
What is a kill switch?
An ops-type flag permanently wired to disable a costly or risky feature (like a heavy 3rd-party API call) in emergencies — without deploying. You flip it via config or a dashboard and the feature goes dark instantly across all instances.
How do you test code that uses feature flags?
Two approaches: (1) Mock IFeatureManager with Moq and call Setup(...).ReturnsAsync(true/false); (2) build a FakeFeatureFlagService that takes a dictionary — reusable, cleaner to read. Always test both ON and OFF paths, and verify the flag is actually being checked.
Why wrap the DB service in a caching layer instead of caching inside it?
Separation of concerns — the DB service does one thing (query the database), the caching wrapper does another (memoize results). This is the Decorator pattern. It keeps both classes simple, independently testable, and lets you swap the caching strategy without touching DB logic.
What's the difference between PercentageFilter and TargetingFilter in Microsoft.FeatureManagement?
PercentageFilter re-evaluates randomly on every request — it's not user-sticky. The same user could see different results on consecutive requests. TargetingFilter uses deterministic hashing per userId, so a user always gets the same experience. For gradual rollouts to real users, always use TargetingFilter.
