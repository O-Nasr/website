What Is Fan-Out?

1 → N distribution: one event, many consumers.

Simple Definition

Fan-out means taking one incoming event and distributing it to many downstream consumers. The service responsible for doing this is called a fan-out service. It's a 1 → N distribution pattern.

2

Why a Separate Fan-Out Service?

Because doing this inside the main API would block requests and crash the system.

Why Not Inside the API?

This operation is write-heavy, potentially huge (millions of followers), CPU and IO intensive, and asynchronous. If done inside your main API server, the request becomes very slow, the server blocks, and the system becomes unstable. Instead, the API writes to a queue and the fan-out service processes it in the background.

3

Fan-Out on Write (Push Model)

When user posts → immediately push the post to all followers' feeds. Precompute feeds.

✅ Pros

  • Fast reads — feed is pre-computed
  • Good for heavy-read systems
  • Simple read path — just fetch from cache

❌ Cons

  • Very expensive write — push to all followers
  • Celebrity problem — 10M+ followers = massive write amplification
  • Wasted work if followers never read the feed
4

Fan-Out on Read (Pull Model)

When user opens app → system gathers posts dynamically from followed users.

✅ Pros

  • Writes are cheap — just store the post once
  • Good for celebrity users — no write amplification
  • No wasted work

❌ Cons

  • Reads become expensive — must gather from many sources
  • Slower feed loading
  • More complex read path

🧠Fan-out is a recurring distributed systems pattern — not just a one-off concept. It's about scalability, async processing, decoupling, handling traffic spikes, and preventing API blocking.

Interview-Level Definition

A fan-out service distributes a single event to multiple downstream consumers, typically asynchronously, to handle high fan-out workloads like notifications or feed updates without blocking the main request path.
▶If a celebrity with 20 million followers posts something, what problem occurs with fan-out on write?
A single post triggers 20 million write operations to individual follower feeds. This causes minutes of processing delay, overloads the fan-out workers, and delays other users' posts. The solution is a hybrid approach: use fan-out on write for regular users, but fan-out on read for celebrities — merging both at read time.
▶Why shouldn't fan-out happen inside the API server?
Because it's write-heavy, potentially millions of operations, CPU/IO intensive, and must be asynchronous. Doing it inside the API would block the request, make the server slow, and crash the system under load. Instead: API → Queue → Fan-out Service (background).
▶What's the difference between push and pull fan-out?
Push (write): Pre-compute feeds at post time. Fast reads, expensive writes. Bad for celebrities. Pull (read): Gather feeds dynamically when user opens app. Cheap writes, expensive reads. Good for celebrities. Most real systems use a hybrid of both.
