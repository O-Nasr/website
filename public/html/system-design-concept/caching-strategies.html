Three Tiers of Caching

1

Cache-Aside (Lazy Loading)

App checks cache first. On miss → load from DB → populate cache. Writes go directly to DB.
🧠Key: App controls the cache. Lazy loading — cache populated only when needed. Writes bypass cache (go straight to DB). Used with Memcached and Redis.
2

Read-Through Cache

Cache sits in-line with DB. On miss → cache itself loads from DB, populates, and returns.
📌Difference from Cache-Aside: In Cache-Aside, the app manages the cache. In Read-Through, the cache itself manages loading from DB. App only talks to the cache.
3

Write-Through Cache

Writes go to cache first → cache synchronously writes to DB. Always consistent.
4

Write-Around (Bypassed)

Writes go directly to DB, bypassing cache. Cache only populated when data is read later.
5

Write-Behind (Write-Back)

Writes go to cache → app gets ACK immediately → cache asynchronously writes to DB later.
⚡Key difference: Write-Through = sync DB write (strong consistency). Write-Behind = async DB write (eventual consistency, faster writes, risk of data loss).

When to Use What

If you need...Choose
Simple caching, app controls everythingCache-Aside (most common)
Transparent caching, app doesn't manage cacheRead-Through
Cache + DB always consistent, no stale dataWrite-Through
Write-heavy data that's rarely readWrite-Around
Maximum write speed, eventual consistency OKWrite-Behind
