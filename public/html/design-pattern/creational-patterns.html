01

🏭 Factory Method

Define an interface for creating an object, but let subclasses decide which class to instantiate.

Intent

Define an interface for creating an object, but let subclasses decide which class to instantiate. The creator defers instantiation to its subclasses.

The Problem It Solves

You are building a logistics app starting with trucks. Later you need ships, planes, drones. Hardcoding new Truck() everywhere means changing dozens of files for every new transport — that violates the Open/Closed Principle.

Step 1 — Product interface
public interface ITransport
{
    void Deliver();
}
Step 2 — Concrete products
public class Truck : ITransport
{
    public void Deliver() =>
        Console.WriteLine("Delivering by road in a truck.");
}

public class Ship : ITransport
{
    public void Deliver() =>
        Console.WriteLine("Delivering by sea on a ship.");
}
Step 3 — Abstract Creator
public abstract class Logistics
{
    // The Factory Method — subclasses decide what to create
    public abstract ITransport CreateTransport();

    // Business logic uses the product WITHOUT knowing its type
    public void PlanDelivery()
    {
        ITransport transport = CreateTransport();
        Console.WriteLine("Planning delivery...");
        transport.Deliver();
    }
}
Step 4 — Concrete creators
public class RoadLogistics : Logistics
{
    public override ITransport CreateTransport() => new Truck();
}

public class SeaLogistics : Logistics
{
    public override ITransport CreateTransport() => new Ship();
}
Step 5 — Client code
Logistics logistics = new RoadLogistics();
logistics.PlanDelivery();
// Output: Planning delivery... Delivering by road in a truck.

logistics = new SeaLogistics();
logistics.PlanDelivery();
// Output: Planning delivery... Delivering by sea on a ship.
💡Adding a new transport type? Just create AirLogistics : Logistics — zero changes to existing code. This is the Open/Closed Principle in action.
🎯Real-World Analogy: A hiring agency is a Factory Method. You ask the agency for a developer, and the agency (the subclass) decides whether to send a frontend dev, backend dev, or full-stack. You use whoever shows up — you do not know or care about the specifics.
▶What is the difference between Factory Method and just calling new?
With new, the calling class is tightly coupled to the concrete type. Factory Method isolates creation behind an abstraction so the client code never changes when new product types are added.
▶Is Factory Method the same as a static factory method like DateTime.Now?
No. A static factory is just a convenience method on the same class. Factory Method is a full pattern involving inheritance — a subclass overrides a method to decide what gets created.
▶When would you use this?
When you do not know ahead of time which class to instantiate, or when you want subclasses to control which objects get created.

✅ Use When

  • You don't know the exact type at compile time
  • Subclasses should choose the product
  • Adding new products shouldn't touch existing code

❌ Avoid When

  • You only ever have one product type
  • The creation logic is trivial — a single new call
  • Inheritance adds unnecessary complexity
02

🏭🏭 Abstract Factory

Provide an interface for creating families of related objects without specifying their concrete classes.

Intent

Provide an interface for creating families of related objects without specifying their concrete classes. Think of it as a factory of factories.

Factory Method vs Abstract Factory

Factory MethodAbstract Factory
CreatesOne productA family of related products
HowInheritance — subclass overridesComposition — factory injected
Use whenOne varying product typeMultiple products that must match
Step 1 — Product interfaces
public interface IButton { void Render(); void OnClick(); }
public interface ICheckbox { void Render(); void Toggle(); }
Step 2 — Windows family
public class WindowsButton : IButton
{
    public void Render() => Console.WriteLine("[Windows] Rendering a flat button.");
    public void OnClick() => Console.WriteLine("[Windows] Button clicked.");
}
public class WindowsCheckbox : ICheckbox
{
    public void Render() => Console.WriteLine("[Windows] Rendering a square checkbox.");
    public void Toggle() => Console.WriteLine("[Windows] Checkbox toggled.");
}
Step 3 — Mac family
public class MacButton : IButton
{
    public void Render() => Console.WriteLine("[Mac] Rendering a rounded button.");
    public void OnClick() => Console.WriteLine("[Mac] Button clicked.");
}
public class MacCheckbox : ICheckbox
{
    public void Render() => Console.WriteLine("[Mac] Rendering a rounded checkbox.");
    public void Toggle() => Console.WriteLine("[Mac] Checkbox toggled.");
}
Step 4 — Abstract Factory + Concrete Factories
public interface IUIFactory
{
    IButton CreateButton();
    ICheckbox CreateCheckbox();
}

public class WindowsFactory : IUIFactory
{
    public IButton CreateButton() => new WindowsButton();
    public ICheckbox CreateCheckbox() => new WindowsCheckbox();
}

public class MacFactory : IUIFactory
{
    public IButton CreateButton() => new MacButton();
    public ICheckbox CreateCheckbox() => new MacCheckbox();
}
Step 5 — Client depends only on interfaces
public class Application
{
    private readonly IButton _button;
    private readonly ICheckbox _checkbox;

    public Application(IUIFactory factory)
    {
        _button = factory.CreateButton();
        _checkbox = factory.CreateCheckbox();
    }
    public void RenderUI() { _button.Render(); _checkbox.Render(); }
}
Step 6 — Entry point, family decided once
IUIFactory factory = new WindowsFactory();
var app = new Application(factory);
app.RenderUI();
// [Windows] Rendering a flat button.
// [Windows] Rendering a square checkbox.
💡Want to switch to Mac? Change only new WindowsFactory() to new MacFactory() at the entry point. Every product automatically switches — the Application class remains untouched.
🎯Real-World Analogy: Think of IKEA vs. Pottery Barn. Each store produces a matching family — sofa, table, chair — all in the same style. You pick one store (factory) and everything you buy matches.
▶What is the core benefit of Abstract Factory over multiple Factory Methods?
Abstract Factory guarantees consistency across a family. It is impossible to accidentally mix products from different families because one factory is the single source for the entire set.
▶What is the downside of Abstract Factory?
Adding a new product type requires changing the abstract factory interface and every concrete factory. It scales well for new families, but poorly for new product types.
▶How is Abstract Factory related to Factory Method?
Abstract Factory is often implemented using Factory Methods internally. Each Create___() method on the concrete factory is itself a Factory Method.

✅ Use When

  • Products come in families that must stay consistent
  • You want to swap entire families at runtime
  • Isolate client code from concrete types

❌ Avoid When

  • You only have one product type
  • Your families rarely change
  • Adding new product types is frequent

🔨 Builder

Construct a complex object step by step. The same construction process can create different representations.

Intent

Construct a complex object step by step. The same construction process can produce different results depending on who executes the steps and which steps are called.

The Problem It Solves

A telescoping constructor — one constructor with too many parameters — is unreadable and error-prone:

The problem
// Telescoping constructor — what does true, false, true even mean?
var house = new House(4, true, false, true, false, "brick", 2, true, "red");
Step 1 — Product
public class House
{
    public string Walls { get; set; }
    public string Roof { get; set; }
    public bool HasPool { get; set; }
    public bool HasGarage { get; set; }
    public override string ToString() =>
        $"House: Walls={Walls}, Roof={Roof}, Pool={HasPool}, Garage={HasGarage}";
}
Step 2 — Builder interface
public interface IHouseBuilder
{
    void BuildWalls();
    void BuildRoof();
    void BuildPool();
    void BuildGarage();
    House GetResult();
}
Step 3 — Concrete Builders
public class WoodenHouseBuilder : IHouseBuilder
{
    private House _house = new House();
    public void BuildWalls() => _house.Walls = "Wooden walls";
    public void BuildRoof() => _house.Roof = "Wooden roof";
    public void BuildPool() => _house.HasPool = false;
    public void BuildGarage() => _house.HasGarage = true;
    public House GetResult() => _house;
}

public class StoneHouseBuilder : IHouseBuilder
{
    private House _house = new House();
    public void BuildWalls() => _house.Walls = "Stone walls";
    public void BuildRoof() => _house.Roof = "Slate roof";
    public void BuildPool() => _house.HasPool = true;
    public void BuildGarage() => _house.HasGarage = true;
    public House GetResult() => _house;
}
Step 4 — Director controls build order
public class ConstructionDirector
{
    private IHouseBuilder _builder;
    public ConstructionDirector(IHouseBuilder builder) => _builder = builder;

    public void BuildBasicHouse()
    {
        _builder.BuildWalls();
        _builder.BuildRoof();
        _builder.BuildGarage();
    }

    public void BuildLuxuryHouse()
    {
        _builder.BuildWalls();
        _builder.BuildRoof();
        _builder.BuildGarage();
        _builder.BuildPool();
    }
}
Step 5 — Client code
IHouseBuilder builder = new WoodenHouseBuilder();
var director = new ConstructionDirector(builder);
director.BuildBasicHouse();
Console.WriteLine(builder.GetResult());
// House: Walls=Wooden walls, Roof=Wooden roof, Pool=False, Garage=True

builder = new StoneHouseBuilder();
director = new ConstructionDirector(builder);
director.BuildLuxuryHouse();
Console.WriteLine(builder.GetResult());
// House: Walls=Stone walls, Roof=Slate roof, Pool=True, Garage=True
Fluent Builder — Common in Real C# Code
public class HouseBuilder
{
    private House _house = new House();
    public HouseBuilder WithWalls(string type) { _house.Walls = type; return this; }
    public HouseBuilder WithRoof(string type) { _house.Roof = type; return this; }
    public HouseBuilder WithPool() { _house.HasPool = true; return this; }
    public HouseBuilder WithGarage() { _house.HasGarage = true; return this; }
    public House Build() => _house;
}

House house = new HouseBuilder()
    .WithWalls("Brick")
    .WithRoof("Tile")
    .WithGarage()
    .WithPool()
    .Build();
💡The Fluent Builder is the style you will encounter most in real C# codebases. StringBuilder, IHostBuilder in ASP.NET Core, and EF Core's ModelBuilder all follow this style.
▶What is the difference between Builder and Factory patterns?
Factory patterns focus on which type to create. Builder focuses on how to construct a complex object step by step. Use Factory when the type varies, use Builder when the construction process is complex.
▶Is the Director mandatory in the Builder pattern?
No. The Director is optional. It is useful when you have predefined build sequences to reuse. Without a Director, the client drives the steps directly — which is the Fluent Builder style.
▶Why return this in a Fluent Builder?
Returning this from each method allows you to chain calls on the same object. It makes the construction read like a sentence, improving clarity and readability.

✅ Use When

  • Object construction involves many optional steps
  • You want readable, self-documenting creation
  • Same steps can produce different representations

❌ Avoid When

  • The object is simple with few fields
  • All fields are always required
  • Construction order does not matter

🧬 Prototype

Create new objects by cloning an existing object instead of building from scratch.

Intent

Create new objects by cloning an existing object — the prototype — instead of building from scratch. Use this when object creation is expensive and you need many similar objects.

Shallow Copy vs Deep Copy — Critical Concept

Shallow Copy ❌Deep Copy ✅
Copies the object but shares references to nested objects. Modifying a nested object on the clone also affects the original.Copies the object and all nested objects recursively. Clone and original are fully independent.
Step 1 — Prototype interface
public interface IPrototype<T>
{
    T Clone();
}
Step 2 — Nested object with its own copy
public class Address
{
    public string Street { get; set; }
    public string City { get; set; }
    public Address(string street, string city) { Street = street; City = city; }
    // Address clones itself — responsible for its own copy
    public Address Clone() => new Address(Street, City);
}
Step 3 — Prototype class with deep copy
public class Employee : IPrototype<Employee>
{
    public string Name { get; set; }
    public string Department { get; set; }
    public Address Address { get; set; }

    public Employee(string name, string department, Address address)
    { Name = name; Department = department; Address = address; }

    public Employee Clone()
    {
        // Deep copy — Address.Clone() creates a brand new Address object
        return new Employee(Name, Department, Address.Clone());
    }
}
Step 4 — Client code
var original = new Employee("Omar", "Engineering", new Address("Tahrir St", "Cairo"));

var clone = original.Clone();
clone.Name = "Ahmed";
clone.Address.City = "Alexandria";

Console.WriteLine(original.Name);          // Omar
Console.WriteLine(original.Address.City);  // Cairo  ← untouched
Console.WriteLine(clone.Name);             // Ahmed
Console.WriteLine(clone.Address.City);     // Alexandria
💡MemberwiseClone() built into .NET gives a shallow copy only. For deep copy you must manually clone each nested object. This is why many developers avoid ICloneable in production — it does not make the shallow/deep distinction clear to the caller.
▶When would you use Prototype over just calling new?
When object creation is expensive — involves DB calls, file I/O, or complex initialization — and you need many similar objects. Cloning is far cheaper than rebuilding from scratch each time.
▶What is the risk of shallow copy?
Shared nested object references. Modifying a nested object on the clone will silently modify the original too, causing hard-to-find bugs.
▶How is Prototype different from Builder?
Builder constructs an object step by step from scratch. Prototype skips construction entirely by copying an already-built object.

✅ Use When

  • Object creation is expensive or complex
  • You need many similar objects with small differences
  • Avoid subclassing just to vary an object

❌ Avoid When

  • Objects are simple and cheap to create
  • Deep copy logic becomes too complex
  • Objects have no meaningful copy concept

👑 Singleton

Ensure a class has only one instance and provide a global access point to it.

Intent

Ensure a class has only one instance for the entire application lifetime, and provide a single global access point to that instance.

Version 1 — Basic (not thread-safe)
public class Logger
{
    private static Logger _instance;
    private Logger() { } // Private constructor — blocks new from outside

    public static Logger GetInstance()
    {
        if (_instance == null)
            _instance = new Logger();
        return _instance;
    }
    public void Log(string message) => Console.WriteLine($"[LOG] {message}");
}
💡This basic version is not safe in multithreaded environments. Two threads could both pass the null check simultaneously and each create a separate instance.
Version 2 — Thread-safe with double-checked locking
public class Logger
{
    private static Logger _instance;
    private static readonly object _lock = new object();
    private Logger() { }

    public static Logger GetInstance()
    {
        if (_instance == null) // outer check — avoids locking every call
        {
            lock (_lock)
            {
                if (_instance == null) // inner check — handles the race condition
                    _instance = new Logger();
            }
        }
        return _instance;
    }
}
Version 3 — Modern C# using Lazy<T> (recommended)
public class Logger
{
    // Lazy<T> handles thread safety and lazy initialization automatically
    private static readonly Lazy<Logger> _instance =
        new Lazy<Logger>(() => new Logger());

    private Logger() { }
    public static Logger Instance => _instance.Value;
    public void Log(string message) => Console.WriteLine($"[LOG] {message}");
}

Logger.Instance.Log("App started");
Logger.Instance.Log("User logged in");
💡Lazy<T> is the cleanest and most idiomatic C# approach. It is thread-safe by default, lazy (created only when first accessed), and requires no manual locking code.
ASP.NET Core DI — The Modern Alternative
// Program.cs — register once
builder.Services.AddSingleton<ILogger, AppLogger>();

// Any class just declares the dependency — DI injects the single instance
public class OrderService
{
    private readonly ILogger _logger;
    public OrderService(ILogger logger) { _logger = logger; }
}
💡DI-based Singleton is preferred in real applications. The class does not need to know it is a Singleton. It also makes unit testing easier since you can inject a mock.

AddSingleton vs AddScoped vs AddTransient

LifetimeCreatedShared?
AddSingletonOnce for the app lifetimeEveryone gets the same instance
AddScopedOnce per HTTP requestShared within one request only
AddTransientEvery time it is requestedNever shared
▶Why is Singleton considered an anti-pattern by some developers?
Because it introduces hidden global state, making code hard to test. A class calling Logger.GetInstance() is tightly coupled to that implementation — you cannot swap it for a mock in tests. DI-based Singletons solve this.
▶Can a Singleton have multiple instances in some scenarios?
Yes. If your app runs across multiple servers or processes, each process gets its own Singleton. It is only guaranteed to be one instance per process, not per distributed system.
▶What is lazy initialization in Singleton?
The instance is not created when the app starts — it is created only on the first call. This saves resources if the Singleton is never actually needed at runtime.

✅ Use When

  • Exactly one shared instance is genuinely needed
  • Managing shared resources like config or logging
  • The instance is stateless or carefully thread-managed

❌ Avoid When

  • Using it just for global convenience
  • It makes unit testing harder
  • Multiple instances are actually perfectly fine
