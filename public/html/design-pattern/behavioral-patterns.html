01

⛓️ Chain of Responsibility

Pass a request along a chain of handlers until one handles it.

Intent

Pass a request along a chain of handlers. Each handler decides to process the request or pass it to the next. The sender does not know which handler will process it.

The Problem It Solves

Without this pattern, handling logic lives in one bloated method with cascading if/else blocks. Adding a new handler requires editing that method — violating OCP.

Step 1 — Abstract Handler
public abstract class SupportHandler
{
    private SupportHandler _next;

    public SupportHandler SetNext(SupportHandler next)
    {
        _next = next;
        return next; // enables: h1.SetNext(h2).SetNext(h3)
    }

    public abstract void Handle(SupportTicket ticket);

    protected void PassToNext(SupportTicket ticket)
    {
        if (_next != null) _next.Handle(ticket);
        else Console.WriteLine($"Unhandled: '{ticket.Issue}'");
    }
}
Step 2 — Concrete Handlers
public class FrontlineSupport : SupportHandler
{
    public override void Handle(SupportTicket ticket)
    {
        if (ticket.Severity == 1)
            Console.WriteLine($"Frontline handled: '{ticket.Issue}'");
        else { Console.WriteLine("Passing up."); PassToNext(ticket); }
    }
}

public class TechnicalSupport : SupportHandler
{
    public override void Handle(SupportTicket ticket)
    {
        if (ticket.Severity == 2)
            Console.WriteLine($"Technical handled: '{ticket.Issue}'");
        else { Console.WriteLine("Passing up."); PassToNext(ticket); }
    }
}
Step 3 — Build chain and run
var frontline = new FrontlineSupport();
var technical = new TechnicalSupport();
var management = new ManagementSupport();

frontline.SetNext(technical).SetNext(management);

frontline.Handle(new SupportTicket("Password reset", 1)); // Frontline handled
frontline.Handle(new SupportTicket("Server crash", 3));   // passed up to Management
💡 ASP.NET Core Middleware is a built-in example of Chain of Responsibility. Each middleware calls next() to pass the request down the chain.
▶What is the difference between Chain of Responsibility and if/else?
An if/else chain is hardcoded — adding a new condition edits existing code, violating OCP. Chain of Responsibility makes handlers independent objects that can be added, removed, or reordered without touching existing code.
▶What happens if no handler processes the request?
You can silently drop it, log it, throw an exception, or add a default catch-all handler at the end — similar to catch(Exception ex) at the bottom of a try/catch.
▶Can a handler both process AND pass the request forward?
Yes. In logging systems every handler logs at its level AND passes to the next. The pattern supports both variations — stop at first match or traverse all handlers.

✅ Use When

  • More than one object may handle a request
  • The handler is not known upfront
  • Handlers should be configurable at runtime

❌ Avoid When

  • You always know exactly which handler will process it
  • The chain is always the same — hardcoding is simpler
  • Long chains make request flow hard to trace

📋 Command

Encapsulate a request as an object — enabling undo, queue, log, and macro.

Intent

Encapsulate a request as an object. This lets you parameterize methods with different requests, queue or log operations, and support undoable actions.

Four Key Roles

RoleResponsibilityExample
InvokerTriggers commands, manages history stackRemoteControl, CommandInvoker
ICommandInterface: Execute() and Undo()ICommand interface
ConcreteCommandWraps receiver, implements Execute/UndoInsertTextCommand
ReceiverDoes the actual workTextEditor, Lights
ICommand + ConcreteCommand + Invoker
public interface ICommand
{
    void Execute();
    void Undo();
}

public class InsertTextCommand : ICommand
{
    private readonly TextEditor _editor;
    private readonly string _text;
    public InsertTextCommand(TextEditor editor, string text)
    { _editor = editor; _text = text; }
    public void Execute() => _editor.InsertText(_text);
    public void Undo() => Console.WriteLine("Undo insert.");
}

public class CommandInvoker
{
    private readonly Stack<ICommand> _history = new();
    public void ExecuteCommand(ICommand command)
    { command.Execute(); _history.Push(command); }
    public void Undo()
    {
        if (_history.Count == 0) { Console.WriteLine("Nothing to undo."); return; }
        _history.Pop().Undo();
    }
}

// Macro Command (Command + Composite)
public class MacroCommand : ICommand
{
    private readonly List<ICommand> _commands;
    public MacroCommand(List<ICommand> commands) { _commands = commands; }
    public void Execute() => _commands.ForEach(c => c.Execute());
    // Undo in REVERSE order — critical
    public void Undo() => _commands.AsEnumerable().Reverse()
        .ToList().ForEach(c => c.Undo());
}
💡MacroCommand combines Command + Composite. Undo must run in reverse order — undoing in original order can leave the system in an inconsistent state.
▶What are the four roles in the Command pattern?
Invoker triggers commands. Command is the request object with Execute() and Undo(). Receiver does the actual work. Client assembles the command, wires the receiver, and gives it to the invoker.
▶What is a Macro Command?
A composite command that holds a list of commands and executes them all as one unit. Undo reverses all sub-commands in reverse order. It combines Command with the Composite pattern.
▶What is the difference between Command and Strategy?
Both encapsulate behavior as an object. Strategy encapsulates an algorithm to vary. Command encapsulates a request — focused on invoking, queuing, logging, and undoing. Strategy has no concept of Undo.

✅ Use When

  • You need undo / redo functionality
  • You want to queue, schedule, or log operations
  • You need macro commands

❌ Avoid When

  • The operation is simple and never needs undoing
  • No need for history or logging
  • Wrapping adds unnecessary complexity

🔁 Iterator

Traverse a collection without exposing its internal structure.

Intent

Provide a way to sequentially access elements of a collection without exposing its underlying structure.

Step 1 — Iterator and Collection interfaces
public interface IIterator<T>
{
    bool HasNext();
    T Next();
    void Reset();
}

public interface IPlaylist<T>
{
    void AddSong(T song);
    IIterator<T> CreateIterator();
}
Step 2 — Concrete Iterator and Collection
public class PlaylistIterator<T> : IIterator<T>
{
    private readonly List<T> _songs;
    private int _position = 0;
    public PlaylistIterator(List<T> songs) { _songs = songs; }
    public bool HasNext() => _position < _songs.Count;
    public T Next() => _songs[_position++];
    public void Reset() => _position = 0;
}

public class MusicPlaylist<T> : IPlaylist<T>
{
    private readonly List<T> _songs = new();
    public void AddSong(T song) => _songs.Add(song);
    public IIterator<T> CreateIterator() => new PlaylistIterator<T>(_songs);
}
Step 3 — C# built-in: yield keyword
public class MusicPlaylist : IEnumerable<string>
{
    private readonly List<string> _songs = new();
    public void AddSong(string song) => _songs.Add(song);
    public IEnumerator<string> GetEnumerator()
    {
        foreach (var song in _songs)
            yield return song; // compiler generates full state machine
    }
    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}

foreach (var song in playlist) // foreach uses Iterator under the hood
    Console.WriteLine(song);
💡yield return is C#'s built-in Iterator. foreach is syntactic sugar — it calls GetEnumerator() then MoveNext() + Current in a loop.
▶Why not just expose the internal list directly?
That couples the client to the internal data structure. If you switch from List to Tree, every client breaks. Iterator decouples traversal from internal representation.
▶How does foreach relate to Iterator in C#?
foreach is syntactic sugar for the Iterator pattern. The compiler calls GetEnumerator(), then MoveNext() and Current on each iteration.
▶Can you have multiple iterators on the same collection?
Yes. Each call to CreateIterator() returns a new independent iterator with its own position. Multiple iterators traverse simultaneously without interfering.

✅ Use When

  • You want to hide collection internals
  • Multiple independent simultaneous traversals needed
  • Uniform traversal interface across different types

❌ Avoid When

  • The collection is a plain list — just use foreach
  • You never need more than one traversal
  • Lazy evaluation is not needed

🎙️ Mediator

Centralize communication between objects through one hub.

Intent

Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by preventing objects from referring to each other directly.

The Problem It Solves

With n components communicating directly, you get n × (n-1) connections. Mediator replaces this web with a star topology — n connections total.

Control Tower Mediator
public interface IControlTower
{
    void RegisterAircraft(Aircraft aircraft);
    void Notify(Aircraft sender, string message);
}

public abstract class Aircraft
{
    protected IControlTower _tower;
    public string CallSign { get; }
    protected Aircraft(string callSign, IControlTower tower)
    { CallSign = callSign; _tower = tower; _tower.RegisterAircraft(this); }
    public void Send(string msg) { Console.WriteLine($"{CallSign}: {msg}"); _tower.Notify(this, msg); }
    public abstract void Receive(string message, string from);
}

public class ControlTower : IControlTower
{
    private readonly List<Aircraft> _aircrafts = new();
    public void RegisterAircraft(Aircraft a) => _aircrafts.Add(a);
    public void Notify(Aircraft sender, string message)
    {
        foreach (var a in _aircrafts)
            if (a != sender) a.Receive(message, sender.CallSign);
    }
}
💡MediatR library in ASP.NET Core is a direct Mediator implementation. SignalR hubs are also Mediators — clients talk through the hub which routes to other clients.
▶What is the difference between Mediator and Facade?
Facade provides a simplified one-directional interface — subsystem classes do not know it exists. Mediator centralizes two-directional communication — components actively use it to communicate with each other.
▶Does Mediator violate SRP?
It can. As components grow, the mediator accumulates more coordination logic and risks becoming a God Object. Keep the mediator focused on coordination only.

Mediator vs Facade vs Observer

MediatorFacadeObserver
DirectionTwo-way coordinationOne-way simplificationOne-to-many broadcast
AwarenessComponents know mediatorSubsystem unaware of FacadePublisher unaware of observers
ControlMediator decides routingNo routing logicPublisher just broadcasts

✅ Use When

  • Many objects communicate in complex ways
  • You want to reuse components independently
  • Centralized control of interactions is desired

❌ Avoid When

  • Only two or three objects interact
  • The mediator grows too large and becomes a God Object
  • Communication is simple and stable

💾 Memento

Capture and restore an object's state without violating encapsulation.

Intent

Capture and externalize an object's internal state so it can be restored later — without violating encapsulation. The object itself decides what to save and how to restore it.

Three Roles

RoleResponsibility
OriginatorCreates mementos (Save) and restores from them (Restore). Only one that can read memento internals.
IMementoOpaque snapshot. Caretaker can only see metadata like GetDate(). Never the state inside.
CaretakerStores and retrieves mementos via a history stack. Never reads the contents.
Memento + Originator + Caretaker
public interface IMemento { string GetDate(); }

public class EditorMemento : IMemento
{
    private readonly string _content; private readonly int _cursorPos;
    private readonly string _date;
    public EditorMemento(string content, int cursorPos)
    { _content = content; _cursorPos = cursorPos; _date = DateTime.Now.ToString("HH:mm:ss"); }
    public (string, int) GetState() => (_content, _cursorPos);
    public string GetDate() => _date;
}

public class TextEditor // Originator
{
    private string _content = string.Empty; private int _cursor = 0;
    public void Type(string t) { _content += t; _cursor += t.Length; }
    public IMemento Save() => new EditorMemento(_content, _cursor);
    public void Restore(IMemento m)
    { var (c, cur) = ((EditorMemento)m).GetState(); _content = c; _cursor = cur; }
}

public class HistoryManager // Caretaker
{
    private readonly Stack<IMemento> _history = new();
    private readonly TextEditor _editor;
    public HistoryManager(TextEditor editor) { _editor = editor; }
    public void Backup() => _history.Push(_editor.Save());
    public void Undo()
    { if (_history.Count == 0) return; _editor.Restore(_history.Pop()); }
}
▶Why use IMemento instead of exposing the state directly?
If the Caretaker could read the state, the Originator's internal representation would leak out — violating encapsulation. IMemento hides everything. The Caretaker only sees metadata.
▶When would you choose Memento over Command for undo?
Use Memento when the state is complex and individual changes are hard to reverse — for example a drawing canvas. Use Command when operations are discrete and clearly reversible.

Memento vs Command for Undo

MementoCommand
What is savedFull state snapshotThe operation + data to reverse it
Memory costHigh — stores everythingLow — stores only what changed
Best forComplex state hard to reverse (canvas)Discrete reversible operations (insert, delete)

✅ Use When

  • You need undo and the state is complex
  • Save checkpoints without exposing internals
  • History of states must be maintained

❌ Avoid When

  • State is very large — snapshots are too expensive
  • Objects hold external resources that cannot be snapshotted
  • Command-based undo is sufficient

👁️ Observer

Notify many objects automatically when one object changes state.

Intent

Define a one-to-many dependency so that when one object changes state, all dependents are notified and updated automatically.

Push vs Pull Models

Push ModelPull Model
Update signatureUpdate(product, qty, ...)Update(IPublisher publisher)
Data deliveryPublisher sends all dataObserver pulls only what it needs
Use whenAll observers need same dataDifferent observers need different parts
Observer + C# events
// Manual implementation
public interface IObserver { void Update(string product, int quantity); }
public interface IPublisher
{
    void Subscribe(IObserver observer);
    void Unsubscribe(IObserver observer);
    void NotifyObservers();
}

public class Store : IPublisher
{
    private readonly List<IObserver> _observers = new();
    private string _product; private int _quantity;
    public void Subscribe(IObserver o) => _observers.Add(o);
    public void Unsubscribe(IObserver o) => _observers.Remove(o);
    public void NotifyObservers() => _observers.ForEach(o => o.Update(_product, _quantity));
    public void AddStock(string product, int qty)
    { _product = product; _quantity = qty; NotifyObservers(); }
}

// C# event = Observer built into the language
public class Store
{
    public event Action<string, int> StockArrived;
    public void AddStock(string p, int q) => StockArrived?.Invoke(p, q);
}
store.StockArrived += (p, q) => Console.WriteLine($"Email: {p}"); // Subscribe
store.StockArrived -= handler; // Unsubscribe
💡Memory leak warning: if observers subscribe but never unsubscribe, the publisher holds references preventing garbage collection. Always unsubscribe when an observer is no longer needed.
▶What is the difference between Observer and event/delegate in C#?
They are the same pattern. C# event is the language-level implementation. += is Subscribe(), -= is Unsubscribe(), and invoking the event is NotifyObservers().
▶What is the difference between push and pull models?
Push: publisher sends all data in Update(). Observer receives everything. Pull: publisher sends just a signal. Observer calls back to pull only what it needs. Push is simpler, pull gives more control.

✅ Use When

  • A change in one object needs to notify many others
  • Subscribers should register/unregister at runtime
  • Follow OCP — add subscribers without touching publisher

❌ Avoid When

  • The set of dependents is small and fixed
  • Unexpected update chains cause hard-to-trace bugs
  • Memory leaks from forgotten subscriptions are a concern

🔄 State

Change an object's behavior when its internal state changes.

Intent

Allow an object to alter its behavior when its internal state changes. The object appears to change its class. Each state becomes its own class — eliminating massive if/else chains.

Vending Machine State Pattern
public interface IState
{
    void InsertCoin(VendingMachine machine);
    void PressButton(VendingMachine machine);
}
Step 2 — Context
public class VendingMachine
{
    private IState _state;
    public IState IdleState { get; } = new IdleState();
    public IState HasMoneyState { get; } = new HasMoneyState();
    public IState DispensingState { get; } = new DispensingState();
    public VendingMachine() { _state = IdleState; }

    public void SetState(IState state)
    { Console.WriteLine($" [Transition: {_state.GetType().Name} → {state.GetType().Name}]");
      _state = state; }

    public void InsertCoin() => _state.InsertCoin(this);
    public void PressButton() => _state.PressButton(this);
}
Step 3 — Concrete States
public class IdleState : IState
{
    public void InsertCoin(VendingMachine m)
    { Console.WriteLine("Coin accepted."); m.SetState(m.HasMoneyState); }
    public void PressButton(VendingMachine m) =>
        Console.WriteLine("Insert coin first.");
}

public class HasMoneyState : IState
{
    public void InsertCoin(VendingMachine m) =>
        Console.WriteLine("Already inserted.");
    public void PressButton(VendingMachine m)
    { Console.WriteLine("Dispensing...");
      m.SetState(m.DispensingState); }
}

public class DispensingState : IState
{
    public void InsertCoin(VendingMachine m) =>
        Console.WriteLine("Please wait.");
    public void PressButton(VendingMachine m)
    { Console.WriteLine("Product dispensed."); m.SetState(m.IdleState); }
}
▶Who is responsible for state transitions — Context or State?
Either can do it. If states transition themselves they couple to each other but Context stays clean. If Context manages transitions, state classes are simpler but Context accumulates logic. Most real implementations let states handle their own transitions.
▶What is the difference between State and enum + switch?
A switch statement adds every new state to every method — violating OCP. State gives each state its own class — adding a new state means one new class, zero changes to Context or existing states.

State vs Strategy

StateStrategy
Who swapsState — internally via ContextClient — explicitly from outside
AwarenessStates know about each otherStrategies are independent
TransitionsCore part of the patternNo concept of transitions
ExampleVending machine, TCP connectionSort algorithm, payment method

✅ Use When

  • Object's behavior changes dramatically based on state
  • State-specific behavior would create massive if/else chains
  • Adding new states should not change existing code

❌ Avoid When

  • Only two or three states with simple logic
  • States rarely change — extra classes add overhead
  • A boolean flag is sufficient

♟️ Strategy

Define a family of algorithms and make them interchangeable at runtime.

Intent

Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from the clients that use it.

Route Strategy Navigator
public interface IRouteStrategy
{
    void BuildRoute(string from, string to);
}

public class FastestRoute : IRouteStrategy
{ public void BuildRoute(string f, string t) =>
    Console.WriteLine($"Fastest: {f} to {t} via highways."); }

public class ShortestRoute : IRouteStrategy
{ public void BuildRoute(string f, string t) =>
    Console.WriteLine($"Shortest: {f} to {t}."); }

public class HighwayFreeRoute : IRouteStrategy
{ public void BuildRoute(string f, string t) =>
    Console.WriteLine($"Highway-free: {f} to {t}."); }

public class Navigator
{
    private IRouteStrategy _strategy;
    public Navigator(IRouteStrategy strategy) { _strategy = strategy; }
    public void SetStrategy(IRouteStrategy s) { _strategy = s; }
    public void BuildRoute(string from, string to) =>
        _strategy.BuildRoute(from, to);
}

var nav = new Navigator(new FastestRoute());
nav.BuildRoute("Cairo", "Alexandria"); // Fastest
nav.SetStrategy(new ShortestRoute());
nav.BuildRoute("Cairo", "Alexandria"); // Shortest
💡Passing a Func<T> or Action<T> in C# is a lightweight Strategy. For simple cases a lambda is cleaner. For complex algorithms that need their own state, a full Strategy class is more appropriate.
▶What is the difference between Strategy and if/else?
An if/else hardcodes all algorithms in one place — adding a new one edits existing code, violating OCP. Strategy gives each algorithm its own class — adding one means adding one class, zero changes to existing code.
▶Can a Context have a default strategy?
Yes — and this is good practice. The Context should always have a valid strategy to prevent null reference exceptions.

✅ Use When

  • Multiple algorithm variants need to be swappable
  • Algorithms should be independently testable
  • OCP — add new algorithms without changing existing code

❌ Avoid When

  • Only one algorithm — pattern adds unnecessary complexity
  • Algorithms are simple one-liners — a lambda is cleaner
  • Clients never need to switch algorithms

📐 Template Method

Define a skeleton algorithm in a base class; subclasses fill in the steps.

Intent

Define the skeleton of an algorithm in a base class, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps without changing the overall structure.

Three Step Types

TypeDescriptionOverride?
Common stepImplemented in AbstractClass — same for all subclassesNo — sealed
Abstract stepNo implementation in base — subclass MUST overrideMandatory
HookDefault empty implementation — subclass CAN override optionallyOptional
DataMiner Template
public abstract class DataMiner
{
    // Template Method — sealed, defines the skeleton
    public sealed void Mine(string filePath)
    {
        OpenFile(filePath);       // common
        var raw = ExtractData();  // abstract
        var parsed = ParseData(raw); // abstract
        OnDataParsed(parsed);    // hook (optional)
        var report = AnalyzeData(parsed); // common
        SendReport(report);      // common
        CloseFile();             // common
    }
    private void OpenFile(string p) => Console.WriteLine($"Opening: {p}");
    private void CloseFile() => Console.WriteLine("Closing.");
    protected abstract string ExtractData();
    protected abstract string ParseData(string raw);
    protected virtual void OnDataParsed(string data) { } // hook
}

public class PDFMiner : DataMiner
{
    protected override string ExtractData() => "raw PDF";
    protected override string ParseData(string r) => $"parsed({r})";
    protected override void OnDataParsed(string d) // overrides hook
        => Console.WriteLine($"PDF Hook: Validating {d}");
}

public class CSVMiner : DataMiner
{
    protected override string ExtractData()
    { Console.WriteLine("CSV: Reading rows..."); return "raw CSV"; }
    protected override string ParseData(string r)
    { Console.WriteLine("CSV: Splitting..."); return $"parsed({r})"; }
    // No hook override — default empty is fine
}
💡The template method is sealed to prevent subclasses from changing the order of steps. Hooks give optional extension points without making them mandatory.
▶What is a hook in Template Method?
A hook is a method with a default empty implementation in the abstract class. Subclasses can override it to add optional behavior — but they are not forced to. It gives extension points without making them mandatory.
▶Why is the template method marked sealed?
To prevent subclasses from changing the order of steps. The skeleton is fixed — only the variable steps change. If subclasses could override the template method itself they could break the algorithm structure.

✅ Use When

  • Multiple classes share the same algorithm structure
  • Avoid code duplication across similar classes
  • Control extension points via hooks

❌ Avoid When

  • Steps have nothing in common — Strategy is cleaner
  • The algorithm has only one or two steps
  • Deep inheritance hierarchies make code hard to follow

🧳 Visitor

Add new operations to existing object structures without modifying them.

Intent

Let you add new operations to existing object structures without modifying the classes of those objects. The operation is separated from the objects it operates on.

Double Dispatch — The Core Mechanism

Single dispatch picks a method based on one type. Double dispatch picks based on two — the receiver AND the argument.

Double dispatch explained
// Double dispatch explained:
shape.Accept(visitor);
// Dispatch 1: 'shape is Circle'  → Circle.Accept()
// Dispatch 2: inside Accept()    → visitor.VisitCircle(this)
//   'visitor is XMLExporter'     → XMLExporter.VisitCircle()
// Result: correct method for every shape × visitor combination
Visitor with Shapes
public interface IVisitor { void VisitCircle(Circle c); void VisitRectangle(Rectangle r); }
public interface IVisitable { void Accept(IVisitor v); }
Step 2 — Elements
public class Circle : IVisitable
{
    public float Radius { get; }
    public Circle(float r) { Radius = r; }
    public void Accept(IVisitor v) => v.VisitCircle(this); // double dispatch
}

public class Rectangle : IVisitable
{
    public float Width { get; } public float Height { get; }
    public Rectangle(float w, float h) { Width = w; Height = h; }
    public void Accept(IVisitor v) => v.VisitRectangle(this);
}
Step 3 — Concrete Visitors
public class XMLExportVisitor : IVisitor
{
    public void VisitCircle(Circle c) =>
        Console.WriteLine($"<circle r=\"{c.Radius}\"/>");
    public void VisitRectangle(Rectangle r) =>
        Console.WriteLine($"<rect w=\"{r.Width}\" h=\"{r.Height}\"/>");
}

public class AreaCalculatorVisitor : IVisitor
{
    public float TotalArea { get; private set; }
    public void VisitCircle(Circle c) { TotalArea += MathF.PI * c.Radius * c.Radius; }
    public void VisitRectangle(Rectangle r) { TotalArea += r.Width * r.Height; }
}

// Adding new operation = one new class, zero changes to shape classes
Step 4 — Client code
var shapes = new List<IVisitable> { new Circle(5), new Rectangle(4,6), new Circle(3) };

shapes.ForEach(s => s.Accept(new XMLExportVisitor()));

var calc = new AreaCalculatorVisitor();
shapes.ForEach(s => s.Accept(calc));
Console.WriteLine($"Total area: {calc.TotalArea:F2}");
💡Visitor works best when the element hierarchy is stable and new operations are added frequently. Adding new element types requires updating every Visitor class — that is the main drawback.
▶What is double dispatch and why does Visitor need it?
Single dispatch picks a method based on one type. Double dispatch picks based on two. Visitor needs this because the right operation depends on both the shape type and the visitor type. Accept() resolves the shape, VisitXxx(this) resolves the visitor.
▶What is the main drawback of Visitor?
Adding a new element type requires updating every Visitor class. With 10 visitors and a new Triangle shape, you must add VisitTriangle() to all 10. Visitor scales for new operations but poorly for new elements.

✅ Use When

  • Many distinct operations on a stable object structure
  • Keep unrelated operations out of element classes
  • New operations are added frequently

❌ Avoid When

  • Element hierarchy changes frequently
  • Operations need access to private fields
  • Double dispatch adds unnecessary complexity

📖 Interpreter

Define a grammar and provide an interpreter to evaluate it.

Intent

Define a grammar for a language and provide an interpreter to evaluate sentences in that language. Each grammar rule maps to a class. Complex expressions become trees of simple expression objects.

💡The Interpreter pattern is from the original GoF book. It is NOT covered in Dive Into Design Patterns by Alexander Shvets. It appears in C# as LINQ expression trees, regex, and SQL parsers.

Step 1 — AbstractExpression
public interface IExpression { int Interpret(); }
Step 2 — Terminal and NonTerminal Expressions
// Terminal — leaf node, no children
public class NumberExpression : IExpression
{
    private readonly int _value;
    public NumberExpression(int value) { _value = value; }
    public int Interpret() => _value;
}

// NonTerminal — delegates to children
public class AddExpression : IExpression
{
    private readonly IExpression _left, _right;
    public AddExpression(IExpression l, IExpression r) { _left = l; _right = r; }
    public int Interpret() => _left.Interpret() + _right.Interpret();
}

public class SubtractExpression : IExpression
{
    private readonly IExpression _left, _right;
    public SubtractExpression(IExpression l, IExpression r) { _left = l; _right = r; }
    public int Interpret() => _left.Interpret() - _right.Interpret();
}

public class MultiplyExpression : IExpression
{
    private readonly IExpression _left, _right;
    public MultiplyExpression(IExpression l, IExpression r) { _left = l; _right = r; }
    public int Interpret() => _left.Interpret() * _right.Interpret();
}
Step 3 — Client builds the expression tree
// Expression: (5 + 3) - 2 = 6
IExpression expr = new SubtractExpression(
    new AddExpression(new NumberExpression(5), new NumberExpression(3)),
    new NumberExpression(2)
);
Console.WriteLine($"(5+3)-2 = {expr.Interpret()}"); // 6

// Expression: (4 * 2) + (10 - 3) = 15
IExpression expr2 = new AddExpression(
    new MultiplyExpression(new NumberExpression(4), new NumberExpression(2)),
    new SubtractExpression(new NumberExpression(10), new NumberExpression(3))
);
Console.WriteLine($"(4*2)+(10-3) = {expr2.Interpret()}"); // 15
▶What is the relationship between Interpreter and Composite?
Interpreter is Composite applied to a grammar. NonTerminalExpression is a Composite node, TerminalExpression is a Leaf. The expression tree is a Composite tree where each node also knows how to interpret itself.
▶When should you NOT use Interpreter?
When the grammar is complex — a SQL parser would need hundreds of classes. For complex grammars use a dedicated parser library. Interpreter works well for simple DSLs with fewer than 10-15 rules.
▶How does LINQ use Interpreter?
Every lambda like x => x > 5 compiles into a tree of BinaryExpression, ParameterExpression, and ConstantExpression nodes. Entity Framework walks this tree and translates it to SQL.

✅ Use When

  • You have a simple language or grammar to interpret
  • Grammar rules map cleanly to a class hierarchy
  • You need to build and evaluate expressions at runtime

❌ Avoid When

  • The grammar is complex — too many classes
  • Performance is critical — tree traversal is slow
  • A parser library or regex solves it more simply
