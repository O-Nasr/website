Relations Between Objects

How objects connect, collaborate, and depend on each other — from the weakest temporary use to the strongest lifecycle ownership.

⚡ Dependency 🔗 Association 🧩 Aggregation 💀 Composition

What Are Relations Between Objects?

In object-oriented design, objects don't work in isolation — they connect, collaborate, and depend on each other. A "relation" describes the nature of that connection: how tightly two objects are bound, who creates whom, and what happens when one is destroyed.

💡Key Interview Insight: The stronger the relationship, the tighter the coupling — and tight coupling makes code harder to test, change, and reuse. Design patterns exist largely to prefer weaker relations where possible.
01

⚡ Dependency

"A temporarily uses B" — the weakest relation

What It Means

One object briefly uses another, typically as a method parameter or local variable. There is no permanent link — the using object does not hold a reference to the other. This is the loosest possible connection.

C# Example
class ReportPrinter
{
    // Report is passed in — no field, no long-term link
    public void Print(Report report)
    {
        Console.WriteLine(report.GetContent());
    }
}

// Usage
var printer = new ReportPrinter();
var report = new Report("Q3 Results");
printer.Print(report); // ReportPrinter forgets report after this call
📌When to use: Default choice. Use for utility/helper methods that need an object once and discard it.
02

🔗 Association

"A knows about B" — a weak but persistent relation

What It Means

One object holds a reference to another as a field or property, but neither controls the lifecycle of the other. They can both exist independently and the link can change at runtime.

C# Example
class Driver
{
    private Car _car; // Driver keeps a reference to a Car

    public Driver(Car car)
    {
        _car = car; // Car is injected — not created here
    }

    public void Drive() => _car.Start();

    public void SwitchCar(Car newCar)
    {
        _car = newCar; // Association can change
    }
}

// Both objects are independent — neither destroys the other
var car = new Car("Tesla");
var driver = new Driver(car);
driver.Drive();
📌When to use: Use when two objects need to collaborate long-term but remain independently managed (e.g. Dependency Injection).
03

🧩 Aggregation

"A has B, but B can live without A" — medium strength

What It Means

A 'has-a' relationship where the parent holds a collection of child objects, but the children can survive independently. If the parent is destroyed, the children live on.

C# Example
class Team
{
    private List<Player> _players;

    // Players are passed in — they exist before/after the Team
    public Team(List<Player> players)
    {
        _players = players;
    }
}

// Players created independently
var p1 = new Player("Alice");
var p2 = new Player("Bob");

var team = new Team(new List<Player> { p1, p2 });

// If team is disbanded, p1 and p2 still exist and can join another team
team = null;
Console.WriteLine(p1.Name); // Still works — "Alice"
📌When to use: Use when a 'whole-part' relationship exists, but parts have an independent identity (e.g. a Playlist and its Songs).
04

💀 Composition

"A owns B — if A dies, B dies" — the strongest relation

What It Means

A strong 'has-a' relationship where the parent fully controls the child's lifecycle. The child is created inside the parent and has no meaning or identity outside of it. When the parent is destroyed, so is the child.

C# Example
class House
{
    private readonly Room _livingRoom; // Room belongs ONLY to this House
    private readonly Room _bedroom;

    public House()
    {
        // House creates its own rooms — they cannot exist elsewhere
        _livingRoom = new Room("Living Room");
        _bedroom = new Room("Bedroom");
    }
}

// You cannot get a Room reference — it's fully private
var house = new House();
// house = null → both rooms are destroyed with the house
📌When to use: Use when the part makes no sense outside the whole (e.g. a Heart inside a Human, Rooms inside a House).

Common Interview Questions

▶What is the difference between Aggregation and Composition?
Both are 'has-a' relationships, but the key difference is lifecycle ownership. In Aggregation, the child can outlive the parent (a Player can exist without a Team). In Composition, the child is owned exclusively by the parent and is destroyed with it (a Room has no identity without its House).
▶Why should we prefer weak relations?
Weak relations (Dependency, Association) produce loosely coupled code that is easier to test in isolation, swap implementations, and change without ripple effects. Tight coupling (Composition) should only be used when the lifecycle ownership is genuinely inseparable.
▶How does Dependency Injection relate to these?
Dependency Injection promotes Association over Composition — instead of a class creating its own dependencies (Composition), they are injected from outside. This makes the class easier to test because you can inject mocks or fakes.
▶What relation do most Design Patterns try to achieve?
Most patterns aim for Association or Dependency — they introduce interfaces and abstractions to replace hard Composition with loose references. For example, the Strategy pattern replaces a hard-coded algorithm (Composition) with an injected interface (Association).

Summary to Remember

🧠Dependency = use once and forget. Association = know each other, but live separately. Aggregation = whole-part, parts survive. Composition = whole-part, parts die together. Always ask: "Who creates it? Who destroys it?" — the answer tells you which relation you have.
