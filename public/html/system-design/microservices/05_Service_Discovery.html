01

Why Service Discovery Exists

In a monolith, one module calls another with a direct method call — no address needed. In microservices, every call crosses a network. And in a cloud environment, that network address is not stable.

Why addresses keep changing
Kubernetes restarts a crashed pod → new IP. Auto-scaler adds 3 more Payments instances → 3 new IPs. A deployment rolls out → old instances die, new ones appear with different addresses.
Hardcoded IP = broken
The solution
A Service Registry — a live directory that always knows which instances are running and where. Services register themselves on startup and deregister on shutdown.
Dynamic lookup = always works
⚡
Without service discovery: every service needs a config file listing every other service's current address. Updated manually every time anything changes. At 20 services, each running 5 instances, that's 100 addresses to manage — impossible at scale.
02

The Service Registry

A central database that maintains a live list of every running service instance and its current address. Three things always happen:

1
Register on startup
When a service starts, it registers itself with its current IP, port, and a health check URL. Each instance registers independently.
2
Heartbeat / health check to stay listed
Either the service sends periodic heartbeats, or the registry actively polls the service's /health endpoint. If checks fail, the instance is marked unhealthy and removed from lookup results.
3
Lookup when you need to call someone
When Orders needs to call Payments, it asks the registry "give me healthy Payments instances." The registry returns current IPs. The registry is NOT in the call path — only used for discovery.
03

Client-Side Discovery

The calling service queries the registry itself, picks an instance using its own load-balancing logic, and calls that IP directly. The calling service contains the discovery logic.

Used by: Netflix Eureka + Ribbon, Consul with client library. The caller must have the Consul client NuGet package, query the registry, and implement load balancing logic.

04

Server-Side Discovery

The calling service just uses a stable name (e.g. http://payments). A router, load balancer, or DNS in front of the services does the registry lookup and picks an instance. The caller knows nothing about instances.

✅
Kubernetes uses server-side discovery. Your code just calls http://payments. CoreDNS resolves the name. kube-proxy routes to a healthy pod. You don't need any service discovery library in your code at all.
05

Health Checks — Heartbeat vs Active Check

The registry doesn't trust that services stay healthy forever. It uses one of two mechanisms to verify. This distinction came up directly in our conversation.

Heartbeat (push)
Service pushes to registry
The service periodically sends a ping to the registry every N seconds to say "I'm still alive." If the registry stops receiving pings for 3+ intervals, it marks the instance unhealthy and removes it from lookup results.

Who calls who: Service → Registry
Detection lag: ~15–30s after crash
Used by: Eureka
Active health check (pull)
Registry polls your /health endpoint
The registry sends a GET request to your service's /health endpoint every N seconds. If it gets a 200 OK → healthy. If it gets a non-200 or the call times out → unhealthy, removed from lookups.

Who calls who: Registry → Service
Detection lag: within one interval
Used by: Consul, Kubernetes liveness probes
⚠️
Common confusion: In the Consul C# example, the HTTP field in AgentServiceCheck is an active health check — you are telling Consul "call this URL every 10 seconds." Consul is the caller, not your service. This is different from a heartbeat where your service does the calling.
06

Kubernetes — Built-In Service Discovery

In practice, most teams don't run Consul or Eureka separately. Kubernetes has service discovery built in via DNS and kube-proxy. This is server-side discovery — and it's completely invisible to your code.

C# — Consul Implementation (Corrected)

This is a client-side discovery implementation. The key correction from our discussion: the service must not hardcode its own IP. It discovers its IP dynamically at runtime. Each running instance registers with its own actual current address.

🔧
Correction from session: The original example showed Address = "10.0.1.4" — hardcoded. That defeats the entire purpose. In the real version, the service discovers its own IP dynamically using Dns.GetHostEntry or an environment variable injected by Kubernetes.
bash — Install
dotnet add package Consul

Part 1 — The Service That Registers Itself

This is the Payments service registering with Consul on startup. Each running instance discovers its own IP dynamically and registers independently.

C# — Program.cs (Payments service)
builder.Services.AddSingleton<IConsulClient>(
    new ConsulClient(cfg => cfg.Address = new Uri("http://consul:8500")));

var lifetime = app.Lifetime;
var consul   = app.Services.GetRequiredService<IConsulClient>();

lifetime.ApplicationStarted.Register(async () =>
{
    // ✅ Discover own IP dynamically — NOT hardcoded
    // Option A: Kubernetes injects POD_IP environment variable
    var ip = Environment.GetEnvironmentVariable("POD_IP")
    // Option B: Resolve hostname to IP (non-Kubernetes environments)
          ?? Dns.GetHostEntry(Dns.GetHostName())
                .AddressList
                .First(a => a.AddressFamily == AddressFamily.InterNetwork)
                .ToString();

    var port = int.Parse(Environment.GetEnvironmentVariable("APP_PORT") ?? "8002");

    var registration = new AgentServiceRegistration
    {
        // Unique per instance — e.g. "payments-10.0.1.4-8002"
        ID      = $"payments-{ip}-{port}",
        Name    = "payments",  // stable name — what callers look up
        Address = ip,           // dynamic — different for every pod
        Port    = port,
        Check   = new AgentServiceCheck
        {
            // ACTIVE HEALTH CHECK — Consul calls this URL every 10s
            // Consul is the CALLER, not this service
            // 200 OK = healthy, stays listed
            // Non-200 or timeout = unhealthy, removed from lookups
            HTTP     = $"http://{ip}:{port}/health",
            Interval = TimeSpan.FromSeconds(10),
            Timeout  = TimeSpan.FromSeconds(3)
        }
    };

    await consul.Agent.ServiceRegister(registration);
});

// Graceful deregister on shutdown — don't wait for health check to fail
lifetime.ApplicationStopping.Register(async () =>
{
    await consul.Agent.ServiceDeregister($"payments-{ip}-{port}");
});

// ── Your /health endpoint ──────────────────────────────────────────
app.MapGet("/health", () => Results.Ok(new { status = "healthy" }));

Part 2 — The Caller (Orders Service)

This is the Orders service querying Consul to find a healthy Payments instance, then calling it directly. This is client-side discovery — the lookup logic lives in the calling service.

C# — PaymentsClient.cs (Orders service)
public class PaymentsClient
{
    private readonly IConsulClient _consul;
    private readonly HttpClient    _http;

    public PaymentsClient(IConsulClient consul, HttpClient http)
    {
        _consul = consul;
        _http   = http;
    }

    public async Task<PaymentResult> ChargeAsync(PaymentRequest req)
    {
        // Ask Consul: "give me healthy Payments instances right now"
        var services  = await _consul.Health.Service("payments", passing: true);
        var instances = services.Response;

        if (!instances.Any())
            throw new ServiceUnavailableException("No healthy Payments instances found");

        // Simple random load balancing — pick one instance
        var instance = instances[Random.Shared.Next(instances.Length)];
        var url      = $"http://{instance.Service.Address}:{instance.Service.Port}";

        // Call that specific IP directly — no registry in the call path
        var response = await _http.PostAsJsonAsync($"{url}/api/payments", req);
        response.EnsureSuccessStatusCode();
        return await response.Content.ReadFromJsonAsync<PaymentResult>();
    }
}

// ⚠️ Performance note: cache lookup results for a few seconds
// Calling Consul on EVERY request adds latency unnecessarily
// Refresh the cached list in the background
💡
Why this is client-side discovery: The Orders service contains the Consul client library, makes the registry lookup call, and performs the load balancing (picks an instance). The discovery logic lives inside the calling service — not in a separate router.

Important Corrections from Discussion

🔧 Correction 1 — IP should be dynamic, not hardcoded
What was wrong: The initial example showed Address = "10.0.1.4" hardcoded.

Why it's wrong: If the IP is hardcoded, you don't need service discovery at all. The whole point is that each instance discovers its own current IP at startup and registers it. When Kubernetes restarts a pod, the new pod gets a different IP and registers that automatically.

The fix: Use Environment.GetEnvironmentVariable("POD_IP") (Kubernetes injects this) or Dns.GetHostEntry(Dns.GetHostName()) as a fallback.
🔧 Correction 2 — The /health endpoint is an active check, called by Consul
What was asked: Is HTTP = "http://ip:port/health" a heartbeat or active check, and who calls it?

Answer: It is an active health check. Consul calls this endpoint — not your service. You are telling Consul: "every 10 seconds, send a GET request to this URL. If you get 200, I'm healthy. Otherwise remove me."

The flow is: Consul registry → GET /health → your Payments service → 200 OK → stays listed
🔧 Correction 3 — The Consul example is client-side, not server-side discovery
Confirmed from discussion: The Consul C# example is indeed client-side discovery. The Orders service contains the Consul client library, explicitly queries the registry, picks an instance, and calls its IP directly.

Contrast with Kubernetes: In Kubernetes, your code just calls http://payments — no Consul library, no explicit lookup, no IP selection. That's server-side discovery. k8s handles everything invisibly.

What Happens if the Registry Goes Down?

This is a common interview follow-up. The answer:

1
Cached addresses keep working temporarily
Services that already looked up addresses and cached them locally can keep making calls for a while — the registry isn't in the call path, only in the lookup path.
2
New lookups fail
Any service that tries to discover a new dependency it hasn't cached will fail. New instances trying to register also can't register.
3
Mitigation — run the registry as a cluster
Consul, Eureka, and Kubernetes etcd all run as clustered multi-node systems with replication. A single node going down doesn't take the registry offline. This is why you never run a single-node registry in production.

Interview Q&A

What is service discovery and why do microservices need it?
Services in cloud environments have dynamic IPs that change constantly — pods restart, scale up, get replaced. Service discovery provides a registry that always has the current list of healthy instances. Without it you'd need to manually update config files every time an IP changes, which is impossible at scale.
What is the difference between client-side and server-side discovery?
Client-side: the calling service queries the registry itself, picks an instance, and calls it directly. The discovery logic lives in the calling service — it needs the registry client library. Server-side: a router or load balancer sits in front of services and does the registry lookup. The caller just uses a stable name. Kubernetes uses server-side — your code calls http://payments and k8s handles everything.
What is a health check, and what types exist?
A check the registry uses to verify a service is alive. Two types: heartbeat — the service pushes periodic pings to the registry (used by Eureka); active check — the registry calls the service's /health endpoint on an interval (used by Consul, Kubernetes liveness probes). Active checks give faster detection. Kubernetes also adds readiness probes specifically to stop routing traffic to a pod that's alive but not yet ready.
In Consul, who calls the /health endpoint?
Consul calls it — not your service. When you register with Consul and provide an HTTP check URL, you're telling Consul "call this endpoint every N seconds." If Consul gets a 200 OK, the service stays listed. If it gets a non-200 or a timeout, Consul marks it unhealthy and removes it from lookup results. This is an active health check (pull model), not a heartbeat (push model).
Should a service hardcode its own IP when registering with Consul?
Never. If the IP is hardcoded, service discovery has no value because IPs change on every pod restart. The service must discover its own current IP dynamically at startup — using the POD_IP environment variable (injected by Kubernetes) or Dns.GetHostEntry(Dns.GetHostName()) as a fallback. Each running instance registers its own actual IP independently.
Is the registry in the critical call path between services?
No — and this is important. The registry is only used for discovery (lookup phase), not for routing. After Orders service gets the Payments IP from the registry, it calls that IP directly. The registry is not involved in the actual HTTP call. This is why a brief registry outage doesn't immediately break all service communication — cached addresses still work.
What happens if the service registry goes down?
Services with cached addresses can keep making calls temporarily (registry is not in the call path). New lookups fail. New instances can't register. Mitigation: run the registry as a replicated cluster (Consul Raft, Eureka peer replication, k8s etcd cluster) so a single node failure doesn't bring down the registry. Never run a single-node registry in production.
Do you need a service discovery library when running on Kubernetes?
No. Kubernetes provides server-side discovery built-in via CoreDNS and kube-proxy. Your code just calls http://service-name — the DNS name resolves to a stable ClusterIP, and kube-proxy routes that to a healthy pod. Dead pods are automatically removed from routing within seconds. Zero library code required.
