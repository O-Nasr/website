01 The Problem — History Lost

In a traditional database, your Products table stores current state only:

Products table:
| Id       | Name         | Price  | UpdatedAt           |
|----------|--------------|--------|---------------------|
| prod-123 | Gaming Mouse | 29.99  | 2026-03-25 14:00:00 |

What was the price yesterday? Last month? Who changed it? Why? You don't know. The previous state was overwritten.

This is like editing a Word document without Track Changes — you see the final version but have no idea what changed or when.

The bridge from Topic 7 (CQRS): We asked "how do you rebuild the read model from scratch if it gets corrupted?" — without Event Sourcing the write DB only has current state. You can rebuild current views but lose all history. Event Sourcing solves this by keeping every event that ever happened.

02 Traditional vs Event Sourcing

Create: $19.99  →  Update: $24.99  →  Update: $29.99
                                          ↓
                              ┌──────────────────────┐
                              │ Database row          │
                              │ Price = $29.99        │
                              │ (only current state)  │
                              └──────────────────────┘

$19.99 and $24.99 are gone forever — history lost.
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│ Event 1          │  │ Event 2          │  │ Event 3          │
│ ProductCreated   │  │ PriceChanged     │  │ PriceChanged     │
│ price: $19.99    │  │ newPrice: $24.99 │  │ newPrice: $29.99 │
└──────────────────┘  └──────────────────┘  └──────────────────┘
                          ↓
              ┌────────────────────────────────────────┐
              │ Event store (append-only log)           │
              │ All 3 events preserved — nothing        │
              │ deleted, nothing overwritten.           │
              │ Kafka, EventStoreDB, or append-only table│
              └────────────────────────────────────────┘
                          ↓
              Current state = replay all events in order
              $19.99 → $24.99 → $29.99 = current price $29.99
Event store                       Event handler
┌────────────────────────┐        ┌────────────────────────┐
│ 1. ProductCreated $19  │        │ Processes event 1      │
│ 2. PriceChanged  $24  │─replay─→│ Processes event 2      │
│ 3. PriceChanged  $29  │        │ Processes event 3      │
└────────────────────────┘        └────────────────────────┘
                                           ↓
                                  ┌──────────────────┐
                                  │ New read DB       │
                                  │ Fully rebuilt     │
                                  │ from history      │
                                  └──────────────────┘

Read DB corrupted? Delete it, replay events → fully restored.
Need a new field? Replay events → populated with history.

03 The Git Analogy

This is exactly how Git works — Git doesn't store "the current file." It stores every commit (event). The current file is the result of replaying all commits.

Git:                Event Sourcing:
─────────────       ────────────────────────────────────────────────
commit 1            Event 1: ProductCreated { price: $19.99 }
commit 2            Event 2: PriceChanged  { price: $24.99, reason: "demand" }
commit 3            Event 3: PriceChanged  { price: $29.99, reason: "new model" }
commit 4            Event 4: ReviewPosted  { rating: 5, userId: "user-abc" }
commit 5            Event 5: StockAdjusted { delta: -1, reason: "sold" }

git log             → full history of every change
git checkout v1     → see state at any point in time
git diff v1 v3     → see exactly what changed
Cross-topic connection: You connected version vectors from Chapter 6 (Key-Value Store) to Git merge conflicts. Same concept — Git resolves conflicting commits just like version vectors resolve write conflicts in distributed KV stores. This kind of cross-chapter synthesis is gold in interviews.

04 Full C# — Events as Immutable Records

// Events are immutable facts — something that happened in the past
public record ProductCreated(
    string ProductId, string Name, decimal Price, DateTime At);

public record PriceChanged(
    string ProductId, decimal OldPrice, decimal NewPrice,
    string Reason, DateTime At);

public record ReviewPosted(
    string ProductId, string UserId, int Rating, DateTime At);

public record ProductDeactivated(
    string ProductId, string Reason, DateTime At);
Events are named in past tense — ProductCreated, PriceChanged, not CreateProduct. They describe what already happened, not what should happen. They are immutable facts that can never be changed once stored.

05 Full C# — Product Aggregate with Apply Pattern

public class Product
{
    public string Id { get; private set; }
    public string Name { get; private set; }
    public decimal Price { get; private set; }
    public bool IsActive { get; private set; }
    public int ReviewCount { get; private set; }
    public double AverageRating { get; private set; }

    // Private — state only changes through events
    private readonly List<object> _pendingEvents = new();

    // === Rebuild from history ===
    public Product(IEnumerable<object> eventHistory)
    {
        foreach (var evt in eventHistory)
            Apply(evt);   // replay each event to rebuild state
    }

    // === Business operations produce events, NOT state changes ===
    public void ChangePrice(decimal newPrice, string reason)
    {
        // Validation against current state
        if (!IsActive)
            throw new InvalidOperationException(
                "Cannot change price of inactive product");
        if (newPrice <= 0)
            throw new ValidationException("Price must be positive");
        if (newPrice == Price)
            return; // no-op

        // Don't change state directly — emit an event
        var evt = new PriceChanged(
            Id, Price, newPrice, reason, DateTime.UtcNow);
        Apply(evt);
        _pendingEvents.Add(evt);
    }

    public void Deactivate(string reason)
    {
        if (!IsActive) return;
        var evt = new ProductDeactivated(Id, reason, DateTime.UtcNow);
        Apply(evt);
        _pendingEvents.Add(evt);
    }

    // === Apply — the ONLY place state changes ===
    private void Apply(object evt)
    {
        switch (evt)
        {
            case ProductCreated e:
                Id = e.ProductId;
                Name = e.Name;
                Price = e.Price;
                IsActive = true;
                break;

            case PriceChanged e:
                Price = e.NewPrice;
                break;

            case ReviewPosted e:
                ReviewCount++;
                // Recalculate running average
                AverageRating = ((AverageRating * (ReviewCount - 1))
                                 + e.Rating) / ReviewCount;
                break;

            case ProductDeactivated e:
                IsActive = false;
                break;
        }
    }

    public IReadOnlyList<object> GetPendingEvents() => _pendingEvents;
}

Business methods never modify state directly. They validate against current state, then produce an event.

The Apply method is the only place state changes — and it's used for two purposes:

// PURPOSE 1: Replaying history (loading from event store)
var product = new Product(eventHistory);
// Calls Apply() for each stored event → rebuilds state

// PURPOSE 2: Applying new events (business operations)
product.ChangePrice(29.99m, "demand");
// Validates → creates PriceChanged → calls Apply() → adds to pending

Same Apply method, two entry points. This guarantees that the aggregate always reaches the same state whether you replay history or apply events one at a time.

06 Full C# — EventStore + ProductService

public class EventStore
{
    private readonly EventStoreDbContext _db;

    public async Task SaveEventsAsync(
        string streamId, IEnumerable<object> events)
    {
        foreach (var evt in events)
        {
            _db.Events.Add(new StoredEvent
            {
                StreamId  = streamId,           // e.g. "product-prod-123"
                EventType = evt.GetType().Name,
                Data      = JsonSerializer.Serialize(evt),
                Timestamp = DateTime.UtcNow,
                Version   = await GetNextVersion(streamId)
            });
        }
        await _db.SaveChangesAsync();
    }

    public async Task<List<object>> GetEventsAsync(string streamId)
    {
        var stored = await _db.Events
            .Where(e => e.StreamId == streamId)
            .OrderBy(e => e.Version)
            .ToListAsync();

        return stored.Select(Deserialize).ToList();
    }
}
public class ProductService
{
    private readonly EventStore _store;
    private readonly IEventBus _bus;

    public async Task ChangePriceAsync(
        string productId, decimal newPrice, string reason)
    {
        // 1. Load all events for this product
        var events = await _store.GetEventsAsync(
            $"product-{productId}");

        // 2. Rebuild current state by replaying events
        var product = new Product(events);

        // 3. Execute business logic (produces new events)
        product.ChangePrice(newPrice, reason);

        // 4. Save new events to the store
        await _store.SaveEventsAsync(
            $"product-{productId}",
            product.GetPendingEvents());

        // 5. Publish events — CQRS read model handlers pick these up
        foreach (var evt in product.GetPendingEvents())
            await _bus.PublishAsync(evt);
    }
}

The 5-step pipeline: Load events → Rebuild state → Execute logic → Save events → Publish. Steps 4 and 5 connect directly to the CQRS read model from Topic 7.

07 Projections — 3 Views from the Same Events

This is where Event Sourcing gets really powerful. The same events feed multiple projections, each optimized for a different use case:

// Projection 1: Product detail page (for users)
public class ProductDetailProjection
{
    public async Task Handle(PriceChanged evt)
    {
        var view = await _readDb.ProductDetails
            .FindAsync(evt.ProductId);
        view.Price = evt.NewPrice;
        await _readDb.SaveChangesAsync();
    }
}
// Projection 2: Price history chart (for analytics dashboard)
public class PriceHistoryProjection
{
    public async Task Handle(PriceChanged evt)
    {
        _readDb.PriceHistory.Add(new PriceHistoryEntry
        {
            ProductId = evt.ProductId,
            Price     = evt.NewPrice,
            ChangedAt = evt.At,
            Reason    = evt.Reason
        });
        await _readDb.SaveChangesAsync();
    }
}
// Projection 3: Revenue forecast (for finance team)
public class RevenueForecastProjection
{
    public async Task Handle(PriceChanged evt)
    {
        // Update projected revenue based on new price × sales velocity
        await RecalculateForecast(evt.ProductId, evt.NewPrice);
    }
}
One PriceChanged event → three projections update independently. Need a fourth projection six months later? Replay all events from the store → the new projection is fully populated with historical data. No data migration needed.

08 Snapshots — Performance Optimization

If a product has 10,000 events, replaying all of them every time is slow. The solution: snapshots.

public class ProductService
{
    public async Task<Product> LoadProductAsync(string productId)
    {
        var streamId = $"product-{productId}";

        // Try to load latest snapshot first
        var snapshot = await _store.GetSnapshotAsync(streamId);

        if (snapshot != null)
        {
            // Load only events AFTER the snapshot
            var newEvents = await _store.GetEventsAfterAsync(
                streamId, snapshot.Version);
            var product = new Product(snapshot.State);
            product.ReplayEvents(newEvents);  // apply only new events
            return product;
        }

        // No snapshot — replay everything
        var allEvents = await _store.GetEventsAsync(streamId);
        return new Product(allEvents);
    }

    // Save snapshot every N events
    public async Task SaveIfNeeded(Product product, int currentVersion)
    {
        if (currentVersion % 100 == 0)  // snapshot every 100 events
        {
            await _store.SaveSnapshotAsync(
                $"product-{product.Id}",
                currentVersion,
                product.GetState());
        }
    }
}

Instead of replaying 10,000 events, load the snapshot at event 9,900 and replay only the last 100. The event store remains the source of truth — snapshots are purely a performance optimization.

10 Event Schema Evolution (PriceChangedV2)

Once an event is stored, you can't change its structure — it's an immutable fact. If you need to add a field to PriceChanged, create a new event type:

// V1 — original event (already stored in production)
public record PriceChanged(
    string ProductId, decimal OldPrice, decimal NewPrice,
    string Reason, DateTime At);

// V2 — new version with additional field
public record PriceChangedV2(
    string ProductId, decimal OldPrice, decimal NewPrice,
    string Reason, DateTime At,
    string ChangedBy);  // NEW: who made the change

V1 events already exist in the store. You can't go back and add ChangedBy to them. So new code must handle both versions.

// Apply must handle BOTH versions
private void Apply(object evt)
{
    switch (evt)
    {
        case ProductCreated e:
            Id = e.ProductId; Name = e.Name;
            Price = e.Price; IsActive = true;
            break;

        case PriceChanged e:       // V1 — old events
            Price = e.NewPrice;
            break;

        case PriceChangedV2 e:     // V2 — new events
            Price = e.NewPrice;
            // ChangedBy available for audit projections
            break;

        case ProductDeactivated e:
            IsActive = false;
            break;
    }
}

Both V1 and V2 produce the same core state change (Price = e.NewPrice). V2 carries additional data that new projections can use.

This is the hardest part of Event Sourcing. Over time, managing multiple event versions across hundreds of event types adds real complexity. This is why event sourcing should only be used where its benefits clearly outweigh this cost.

11 When to Use / NOT Use Event Sourcing

Use Event Sourcing whenDon't use when
Audit trail required (finance, healthcare, legal)Simple CRUD — adds massive complexity
You need "what was the state at time X?"Team is small and doesn't need the power
Multiple projections from the same dataEvents would be trivial (just "FieldUpdated")
Debugging requires replaying what happenedEvent schema changes are hard to manage
Pairing with CQRS for rebuild capabilityMost microservices — this is an advanced pattern
The honest truth: Event Sourcing is powerful but complex. Most services should use a regular database with CQRS. Reserve event sourcing for domains where history is a business requirement — financial transactions, medical records, audit-heavy systems, or systems where you genuinely need to rebuild views from scratch.

12 Interview Q&A

Q: What is Event Sourcing?
Instead of storing current state, you store an append-only log of every event that changed the entity. Current state is derived by replaying those events in order. The event store is the single source of truth. This gives you a complete audit trail, the ability to rebuild any projection from history, and time-travel debugging.
Q: How does Event Sourcing relate to CQRS?
They're independent patterns but pair naturally. CQRS splits reads and writes into separate models. Event sourcing provides the write side — events are appended to the store. Those events feed projection handlers that build the read models. The key benefit of combining them: if you need a new read model or the existing one is corrupted, you replay all events from the store to rebuild it.
Q: What is a projection?
A read-optimized view built by processing events. The same events can feed multiple projections — one for a product detail page, another for an analytics dashboard, another for a search index. Each projection is independently rebuildable by replaying events from the event store.
Q: What is a snapshot and why do you need it?
When an entity has thousands of events, replaying all of them on every load is slow. A snapshot captures the entity's state at a specific event version. To load, you read the snapshot then replay only events after it. It's a performance optimization — the event store remains the source of truth.
Q: What's the hardest part of Event Sourcing?
Event schema evolution. Once an event is stored, you can't change its structure — it's an immutable fact. If you need to add a field to PriceChanged, you create PriceChangedV2 and handle both versions in your Apply method. Over time, managing multiple event versions adds complexity. This is why event sourcing should only be used where its benefits outweigh this cost.
