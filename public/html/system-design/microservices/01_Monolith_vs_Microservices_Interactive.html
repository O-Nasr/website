Monolith vs Microservices

Core tradeoff, DDD bounded contexts, decomposition strategies, data sharing between services. Interactive diagrams, all conversation depth.

01 The Core Tradeoff

Microservices are not better than monoliths. They're a tradeoff: you gain independence and scale, you pay with complexity and distributed systems problems.

Monolith
One deployable unit. All modules share the same process, memory, and database.
Deploy everything together. Simple to develop, test, and debug.
A single stack trace shows the full chain. One grep through one log file.
Microservices
Many small, independent services. Each owns its data, deploys separately.
Communicates over a network. Stack traces end at HTTP boundaries.
Complex but independently scalable. Each team owns their service.

Break the monolith when you hit concrete pain:

  • Deployment bottleneck — one team's change blocks everyone else's release
  • Scaling mismatch — search gets 100× more traffic than admin, but you scale both together
  • Technology divergence — ML team needs Python, core team uses C#
  • Team ownership — at 200 engineers, nobody knows who owns what
Martin Fowler's MonolithFirst: Start with a monolith. Migrate when you hit pain — not before. A distributed monolith (services that must deploy together) is the worst of both worlds.

02 Decomposition Strategies

Technical split — WRONG

UI serviceall screens
API serviceall controllers
DB serviceall tables

Every feature change touches all 3 services. You gained nothing — this is a distributed monolith.

Business domain split — RIGHT (DDD)

OrdersUI+logic+data
PaymentsUI+logic+data
UsersUI+logic+data
InventoryUI+logic+data

"Add a discount to orders" lives entirely inside the Orders service. That's independence.

Bounded Context — the core DDD concept

The word "Order" means different things in different contexts:

Orders context
Order = items, quantity, status, customer
Payments context
Order = amount, payment method, transaction ID
Shipping context
Order = weight, dimensions, address, carrier

Each service owns its bounded context. Same word, different models — they never share internal objects.

Subdomain decomposition — when a bounded context grows too large

After 18 months, your Orders service handles: cart logic, checkout flow, order lifecycle, returns, promotions, fraud detection, order history, subscriptions. 3 teams, 8 subdomains. Time to split:

Carthigh reads
Checkouttransactional
Lifecyclelong-running
Returnsdiff team

Split signals: different scaling needs, different teams, different change frequency.

Strangler Fig — incremental migration

Instead of rewriting the monolith, gradually route requests from old to new:

Route /api/users → new Users service
→
Route /api/orders → still monolith
Route /api/orders → new Orders service
→
Monolith shrinks
All routes migrated → monolith gone

Zero big-bang rewrite. Name comes from the strangler fig vine that grows around a tree until the tree dies.

03 Data Sharing Between Services

When two services need the same data, they each keep their own copy:

// Catalogue publishes event when price changes
await _bus.PublishAsync(new ProductPriceUpdated(productId, newPrice));

// Orders service consumes and stores a local copy
public async Task Handle(ProductPriceUpdated evt)
{
    var cache = await _db.ProductCache.FindAsync(evt.ProductId);
    cache.Price = evt.NewPrice;
    await _db.SaveChangesAsync();
    // Orders reads from its own DB — zero coupling to Catalogue
}
Data duplication is intentional — it buys independence. Orders stores product name and price at time of order. Inventory stores stock levels and warehouse location. They sync via events, never via shared database access.

04 Interview Q&A

Q: Would you start a new project with microservices?
Usually no. Start with a well-structured modular monolith. Migrate when you hit concrete pain: scaling mismatch, deployment bottleneck, or team growth. This is Martin Fowler's MonolithFirst pattern.
Q: What is a bounded context?
A DDD term — the explicit boundary within which a domain model applies. "Order" means one thing inside Orders service and something different inside Payments. Each service owns its bounded context and doesn't share its internal model.
Q: What's the biggest challenge in microservices?
Data consistency. In a monolith, one transaction wraps everything. In microservices, you handle partial failures, eventual consistency, and compensating transactions manually (Saga pattern).
Q: What's a distributed monolith?
A system that looks like microservices but behaves like a monolith — services must deploy together, share a database, or can't function independently. All the complexity, none of the independence.
Q: How do you decide service boundaries?
Follow three signals: team ownership (who changes the code together), change frequency (what changes at the same rate), and data affinity (what data is tightly coupled). Entity-based cuts (UserService, ProductService) are a trap — use capability-based cuts (Identity service, Catalogue service, Ordering service).
Q: What happens when two services need the same data?
They each keep their own copy. Orders stores product name/price at time of order. Inventory stores stock levels. They sync via events — ProductPriceUpdated published by Catalogue, consumed by whoever cares. Duplication is intentional.
