Service Discovery

Client-side vs server-side, Kubernetes DNS, Consul, health checks, liveness vs readiness probes.

01 Why Hardcoded URLs Break

You spin up 3 Inventory instances. Config only knows one IP. Two instances sit idle.

Inventory crashes, gets new IP. Config still points to dead address. All requests fail.

New deploy gets different IP. Nobody told the callers. Requests fail.

03 Health Checks

builder.Services.AddHealthChecks()
    .AddDbContextCheck<AppDbContext>()
    .AddRedis("redis-connection-string")
    .AddCheck("custom", () =>
    {
        var isReady = CheckSomething();
        return isReady ? HealthCheckResult.Healthy()
                       : HealthCheckResult.Unhealthy("Not ready");
    });

app.MapHealthChecks("/health");          // liveness
app.MapHealthChecks("/health/ready", ... // readiness
ProbeQuestionOn failure
Liveness /healthIs the process alive?K8s restarts the pod
Readiness /health/readyCan it handle requests?K8s stops routing (doesn't kill it)

A service loading cache? Alive (liveness=healthy) but not ready (readiness=unhealthy). K8s keeps it running, doesn't send traffic until cache is warm.

Without K8s — Consul dynamic registration

// Service discovers its own IP dynamically at startup
var address = Environment.GetEnvironmentVariable("HOST_IP")
              ?? GetLocalIPAddress();
var port = int.Parse(Environment.GetEnvironmentVariable("PORT") ?? "5001");

await _consul.Agent.ServiceRegister(new AgentServiceRegistration
{
    ID = $"inventory-{Guid.NewGuid()}",
    Name = "inventory-service",
    Address = address,    // dynamic, not hardcoded!
    Port = port,
    Check = new AgentServiceCheck
    {
        HTTP = $"http://{address}:{port}/health",
        Interval = TimeSpan.FromSeconds(10),
        DeregisterCriticalServiceAfter = TimeSpan.FromSeconds(30)
    }
});

04 Interview Q&A

Q: How does K8s service discovery work?
K8s Service gets a stable DNS name + virtual IP. CoreDNS resolves. kube-proxy routes via iptables to healthy pod IPs. Pods auto-register via Endpoints API. Health monitored through liveness and readiness probes.
Q: Liveness vs readiness?
Liveness = "is the process alive?" Failure = restart. Readiness = "can it handle traffic?" Failure = remove from LB but keep running. A service loading cache is alive but not ready.
