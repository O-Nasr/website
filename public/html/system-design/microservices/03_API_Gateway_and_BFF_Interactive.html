API Gateway + BFF Pattern

Routing, authentication, rate limiting, aggregation, YARP in .NET, BFF per client type. Full conversation depth.

01 Why a Gateway?

Without API Gateway
Every client talks directly to every service
Mobile app needs to know 8 service URLs
Auth implemented in every service
CORS configured everywhere
With API Gateway
One front door — clients talk to one URL
Gateway routes, authenticates, rate-limits, aggregates
Services stay hidden behind it

02 Gateway Responsibilities

Routing — maps URLs to services

// YARP reverse proxy — appsettings.json
"Routes": {
  "orders-route": {
    "ClusterId": "orders-cluster",
    "Match": { "Path": "/api/orders/{**catch-all}" }
  }
},
"Clusters": {
  "orders-cluster": {
    "Destinations": {
      "d1": { "Address": "http://orders-service:5001" }
    }
  }
}

// Program.cs
builder.Services.AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));
app.MapReverseProxy();

Authentication — verify JWT once, forward claims

// Gateway verifies JWT
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://your-identity-server";
        options.Audience  = "api-gateway";
    });

// Stamp userId into header for downstream services
app.MapReverseProxy(pipeline =>
{
    pipeline.Use(async (context, next) =>
    {
        if (context.User.Identity?.IsAuthenticated == true)
        {
            var userId = context.User.FindFirst("sub")?.Value;
            context.Request.Headers["X-User-Id"] = userId;
        }
        await next();
    });
});

// Orders service — trusts the header
var userId = Request.Headers["X-User-Id"].ToString();

Rate limiting — protect services from abuse

builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("standard", opt =>
    {
        opt.PermitLimit = 100;
        opt.Window = TimeSpan.FromMinutes(1);
        opt.QueueLimit = 10;
    });
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
});
app.UseRateLimiter();

Aggregation — combine multiple service responses

Mobile home screen needs Users + Orders + Inventory. Without aggregation: 3 API calls. With gateway aggregation: 1 call, gateway fans out and combines.

03 BFF Pattern (Backend for Frontend)

Different clients need different data. Instead of one gateway for all, create one dedicated gateway per client type:

Mobile BFF
Compact payloads, fewer fields, bandwidth-optimized
Web BFF
Rich aggregations, detailed views, multiple data sources
Partner API BFF
Strict versioning, heavy throttling, webhooks

In practice: YARP/Kong as infrastructure gateway (routing, SSL, rate limiting) + ASP.NET Core BFF behind it for aggregation logic. Two layers, two concerns.

04 Interview Q&A

Q: Gateway vs load balancer — what's the difference?
Load balancer distributes traffic across instances of the same service — purely availability. Gateway operates at application layer: understands routes, reads JWTs, applies business logic, aggregates across different services. You often have both.
Q: Risks of an API Gateway?
Single point of failure (mitigate: multiple instances behind a LB). Becoming a "God service" if too much business logic added. Gateway should only do cross-cutting concerns — routing, auth, rate limiting. Never database calls.
Q: What is the BFF pattern?
Backend for Frontend — a dedicated gateway per client type (mobile, web, partner). Use when different clients have meaningfully different data requirements. Each BFF is typically owned by the frontend team that consumes it.
