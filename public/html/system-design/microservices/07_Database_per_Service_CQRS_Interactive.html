01 Database per Service

Orders
Inventory
Payments
↓ all hit ↓
One shared database
Orders → PostgreSQL
Inventory → MongoDB
Search → Elasticsearch

Each service picks the best DB for its workload. This is polyglot persistence.

  • Tight coupling: Inventory adds column → Orders SQL breaks → forced to deploy together
  • No independent scaling: Search queries hammer DB → Orders writes slow down
  • No tech freedom: Search needs Elasticsearch but everyone's stuck on PostgreSQL
A shared database makes your microservices a distributed monolith.

02 CQRS

Writes want normalized data (ACID, constraints). Reads want denormalized data (pre-joined, no JOINs, fast). Optimizing for one hurts the other. CQRS splits into two models.

public class UpdateProductPriceHandler
{
    public async Task Handle(UpdateProductPriceCommand cmd)
    {
        var product = await _db.Products.FindAsync(cmd.ProductId);
        if (cmd.NewPrice <= 0) throw new ValidationException();
        product.Price = cmd.NewPrice;
        await _db.SaveChangesAsync();
        await _bus.PublishAsync(new ProductPriceUpdated(
            cmd.ProductId, cmd.NewPrice, product.Name));
    }
}
// Single read, no JOINs — everything pre-joined
public class ProductView
{
    public string ProductId { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public string CategoryName { get; set; }   // pre-joined!
    public double AverageRating { get; set; }  // pre-calculated!
    public int StockLevel { get; set; }        // from Inventory events!
}
// Consumes events from MULTIPLE services
public class ProductReadModelUpdater
{
    public async Task Handle(ProductPriceUpdated evt)
    {   view.Price = evt.NewPrice; }

    public async Task Handle(ReviewPosted evt)
    {   view.ReviewCount++; view.AverageRating = evt.NewAvg; }

    public async Task Handle(StockLevelChanged evt)
    {   view.StockLevel = evt.NewLevel; }
}
Don't use CQRS when: Simple CRUD. Read/write patterns are similar. Small scale. Most services don't need it — reserve for read-heavy, cross-service aggregation (product pages, dashboards, search results).

03 Interview Q&A

Q: What's the tradeoff of CQRS?
Eventual consistency — read model lags behind writes (ms to seconds). Two databases, event handlers, sync logic. Only worth it when read/write patterns differ significantly.
Q: How does CQRS connect to Saga?
Sagas handle distributed writes (coordinating across services). CQRS handles distributed reads (building views from events). They pair naturally — Saga events feed CQRS read models.
