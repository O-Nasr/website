01 Shared Database — The Trap

              Orders    Inventory    Payments    Shipping
                │           │           │           │
                └─────────┐ │ ┌─────────┘           │
                          ▼ ▼ ▼                     ▼

                │     ONE SHARED DATABASE              │
                │  Orders table, Products table,       │

Every service reads and writes every table.

  Orders         Inventory        Payments         Search

Each service picks the best database for its workload.
Schema changes in Orders DB affect nobody else.
This is called polyglot persistence.

02 Why Shared DB Breaks Microservices

1. Tight coupling
Inventory adds a column to Products table → Orders service's SQL breaks → must deploy together. You've recreated a monolith with extra network hops.
2. No independent scaling
Search queries hammer the DB → Orders writes slow down → one DB bottlenecks everyone. You can't scale the Search read path without affecting the Orders write path.
3. No technology freedom
Search needs Elasticsearch, Inventory needs a document store, Sessions need Redis — but everyone is stuck on one PostgreSQL because they share a database.
A shared database makes your microservices a distributed monolith. All the operational complexity of microservices, none of the independence benefits.

03 Database per Service + Polyglot Persistence

04 Data Sharing via Events

If Orders needs product info, it does NOT query Catalogue's database. Instead:

Catalogue updates price
→
Publishes ProductPriceUpdated
→
Orders consumes event
→
Stores local copy

Orders keeps a local copy of only the fields it needs (product name, price). This means data duplication — and that's intentional. The tradeoff: duplicate data buys full independence. Each service can deploy, scale, and change its schema without breaking anyone else.

05 The Read/Write Mismatch Problem

This is where CQRS comes from. In a typical e-commerce product page:

-- WRITE: admin updates price — 10 updates/min
UPDATE Products SET Price = 29.99 WHERE Id = 'prod-123'

→ Touches 1 table, 1 row
→ Needs normalization, constraints, ACID
→ Simple and fast
-- READ: user views product page — 100,000 reads/sec
SELECT p.Name, p.Price, p.Description,
       AVG(r.Rating), COUNT(r.Id),
       i.StockLevel, i.EstimatedDelivery,
       c.Name AS CategoryName
FROM Products p
JOIN Reviews r ON r.ProductId = p.Id
JOIN Inventory i ON i.ProductId = p.Id
JOIN Categories c ON c.Id = p.CategoryId
WHERE p.Id = 'prod-123'

→ 4 JOINs, aggregations, 100K times/second
→ SLOW at scale

The write model wants normalized data — no duplicated data, foreign keys, constraints. Clean writes.

The read model wants denormalized data — everything pre-joined, ready to serve instantly. No JOINs.

Optimizing for one hurts the other. Add indexes to speed up reads → writes slow down. Normalize for clean writes → reads need expensive JOINs.

This is NOT a replication problem (read replicas solve that). The deeper problem is that reads and writes want different data shapes. Reads want flat, pre-joined documents. Writes want normalized, constrained tables.

06 Enter CQRS — Two Separate Models

Command Query Responsibility Segregation — split your application into two models:

         Admin writes (10/min)    User reads (100K/sec)
                  \                  /
                   ▼                ▼
            ┌──────────────────────────┐
            │    One database, one     │
            │    schema for both       │
            └──────────────────────────┘
  Optimized for writes → reads suffer
  Optimized for reads  → writes suffer
  Command side (write)              Query side (read)
  UpdateProductPriceCommand         GetProductDetailsQuery
          │                                  │
          ▼                                  ▼
  ┌──────────────────┐          ┌──────────────────────┐
  │     Write DB     │──events──│      Read DB          │
  │ Normalized, ACID │          │ Denormalized, fast    │
  │   PostgreSQL     │          │ Redis / Elasticsearch │
  └──────────────────┘          └──────────────────────┘
  Scale each independently. Optimize each for its workload.
Admin UI
→
Command handler validates + writes
→
Write DB (PostgreSQL)
then →
Publishes ProductPriceUpdated event
→
Event handler updates Read DB
→
User queries Read DB — instant

Key insight: Write side = normalized, ACID, slow is OK (rare operation). Read side = denormalized, fast, eventual consistency is OK.

07 Full C# — Command Side (Write)

// === Command — represents an intention to change state ===
public record UpdateProductPriceCommand(string ProductId, decimal NewPrice);

// === Command Handler — validates and writes ===
public class UpdateProductPriceHandler
{
    private readonly WriteDbContext _db;
    private readonly IEventBus _bus;

    public async Task Handle(UpdateProductPriceCommand cmd)
    {
        var product = await _db.Products.FindAsync(cmd.ProductId);
        if (product == null) throw new NotFoundException();

        // Business validation
        if (cmd.NewPrice <= 0)
            throw new ValidationException("Price must be positive");

        // Write to normalized DB
        product.Price = cmd.NewPrice;
        product.UpdatedAt = DateTime.UtcNow;
        await _db.SaveChangesAsync();

        // Publish event — read side will pick this up
        await _bus.PublishAsync(new ProductPriceUpdated(
            cmd.ProductId, cmd.NewPrice, product.Name));
    }
}

08 Full C# — Query Side + ProductView (Read)

// === Query — represents a request for data ===
public record GetProductDetailsQuery(string ProductId);

// === Query Handler — reads from denormalized store ===
public class GetProductDetailsHandler
{
    private readonly ReadDbContext _readDb;  // DIFFERENT DB!

    public async Task<ProductDetailsDto> Handle(GetProductDetailsQuery query)
    {
        // Single read, no JOINs — everything pre-joined
        return await _readDb.ProductViews
            .FirstOrDefaultAsync(p => p.ProductId == query.ProductId);
    }
}

Compare this to the 4-JOIN SQL in Section 5. One flat read vs four JOINs. At 100K reads/sec, this is the difference between 2ms and 200ms.

// === The read model — flat, denormalized, ready to serve ===
public class ProductView  // this lives in the Read DB
{
    public string ProductId { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public string CategoryName { get; set; }     // pre-joined from Categories!
    public double AverageRating { get; set; }    // pre-calculated from Reviews!
    public int ReviewCount { get; set; }         // pre-calculated from Reviews!
    public int StockLevel { get; set; }          // from Inventory events!
    public string EstimatedDelivery { get; set; }
}

Every field that required a JOIN in traditional SQL is now pre-computed and stored flat. The query handler does zero JOINs — just one single-table lookup.

09 Full C# — Event Handler (Write → Read Bridge)

This is the glue. The ProductReadModelUpdater consumes events from multiple services and builds the denormalized read model:

public class ProductReadModelUpdater
{
    private readonly ReadDbContext _readDb;

    // Event from Catalogue service — price changed
    public async Task Handle(ProductPriceUpdated evt)
    {
        var view = await _readDb.ProductViews
            .FirstOrDefaultAsync(p => p.ProductId == evt.ProductId);

        if (view == null)
        {
            view = new ProductView { ProductId = evt.ProductId };
            _readDb.ProductViews.Add(view);
        }

        view.Price = evt.NewPrice;
        view.Name = evt.ProductName;
        await _readDb.SaveChangesAsync();
    }

    // Event from Reviews service — new review posted
    public async Task Handle(ReviewPosted evt)
    {
        var view = await _readDb.ProductViews.FindAsync(evt.ProductId);
        view.ReviewCount++;
        view.AverageRating = evt.NewAverageRating;
        await _readDb.SaveChangesAsync();
    }

    // Event from Inventory service — stock level changed
    public async Task Handle(StockLevelChanged evt)
    {
        var view = await _readDb.ProductViews.FindAsync(evt.ProductId);
        view.StockLevel = evt.NewLevel;
        await _readDb.SaveChangesAsync();
    }
}
Key insight: This one handler consumes events from 3 different services — Catalogue, Reviews, Inventory — and builds ONE flat, pre-joined view. The user's query hits this single table and gets everything in one read. No JOINs. No cross-service calls. Blazing fast.

10 When to Use / NOT Use CQRS

Use CQRS whenDon't use CQRS when
Read/write ratio is heavily skewed (100:1+)Read and write patterns are similar
Read model needs data from multiple servicesSimple CRUD with one entity
Reads need a different shape than writesSmall scale — adds unnecessary complexity
You need to scale reads independentlyTeam is small and doesn't need the separation
The honest truth: most services don't need CQRS. A simple Orders service with PostgreSQL and normal Entity Framework queries is fine. CQRS shines on read-heavy, cross-service aggregation scenarios — product pages, dashboards, search results, news feeds.

11 Connection to Saga and Event Sourcing

Sagas handle distributed writes — coordinating transactions across services.

CQRS handles distributed reads — building query-optimized views from events published by multiple services.

They work together: a Saga orchestrates the write flow, and the events produced during that flow feed the CQRS read models.

With just CQRS, the write DB stores current state only. If the read DB gets corrupted, you rebuild from current state — but you lose history.

With Event Sourcing + CQRS: the event store keeps every event that ever happened. Read DB corrupted? Replay all events → fully restored. Need a new field? Replay → populated with historical data.

Without Event Sourcing:
  Read DB dies → rebuild from write DB current state
  → ProductView gets current price ($29.99)
  → But price history ($19.99 → $24.99 → $29.99) is LOST

With Event Sourcing:
  Read DB dies → replay all events from event store
  → Event 1: ProductCreated { price: $19.99 }
  → Event 2: PriceChanged  { price: $24.99 }
  → Event 3: PriceChanged  { price: $29.99 }
  → Full history restored
  → Add "price trend" field later? No problem — replay events.

12 Interview Q&A

Q: What is Database per Service and why is it important?
Each microservice owns its database privately. No other service accesses it directly. This ensures services can deploy independently, choose the right storage engine for their workload (polyglot persistence), and evolve their schema without breaking others. Services share data through APIs or events, not shared tables.
Q: What is CQRS?
Command Query Responsibility Segregation splits your application into two models: a write model (commands) optimized for transactions and validation with normalized data, and a read model (queries) optimized for fast retrieval with denormalized, pre-joined data. The write side publishes events, and event handlers update the read model asynchronously.
Q: What's the tradeoff of CQRS?
Eventual consistency — the read model lags behind the write model by the time it takes to process events (typically milliseconds to seconds). Increased complexity: two databases, event handlers, and sync logic. Only worth it when read/write patterns are significantly different or when aggregating data from multiple services into a single view.
Q: How does CQRS connect to the Saga pattern?
Sagas handle distributed writes (coordinating transactions across services). CQRS handles distributed reads (building query-optimized views from events published by multiple services). They often work together — a Saga orchestrates the write flow, and the events produced feed the CQRS read models.
Q: What is polyglot persistence?
Each service uses the database technology best suited to its workload. Orders uses PostgreSQL for ACID transactions. Search uses Elasticsearch for full-text queries. Sessions use Redis for fast key-value lookups with TTL. This is only possible when each service owns its database independently.
Q: How do services share data without accessing each other's database?
Through events. Catalogue publishes ProductPriceUpdated. Orders consumes it and stores a local copy of only the fields it needs. This is intentional data duplication — each service stays in sync via events, not via shared database access. Duplication buys independence.
