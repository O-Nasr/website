1 What Is a Web Crawler?

A web crawler is a program that automatically discovers and downloads web pages. It starts with a small set of seed URLs and expands by following every link it finds — forever. Google uses this to index the entire internet.

4 Politeness

Without rules, a crawler with 10,000 nytimes.com URLs would send all 10,000 requests in seconds — effectively a DDoS attack. Politeness rules prevent this.

  • Never send more than N requests per second to the same domain
  • Maintain a delay between requests to the same host
  • Respect robots.txt — the file websites use to tell crawlers what not to visit

5 URL Frontier — Two Layers

The URL Frontier is not a simple FIFO queue. It has two layers working together:

Layer 1 — Prioritizer

Assigns a priority score to each URL based on PageRank, update frequency, and traffic. Higher score = crawl sooner.

Layer 2 — Per-Domain Queues

One dedicated queue per domain. Each queue has its own rate limiter. The crawler alternates between domain queues — enforcing politeness automatically.

Key Insight
Layer 1 decides WHAT to crawl (by importance). Layer 2 decides HOW FAST per domain (by politeness). Together they give both priority and safety.
6 Deduplication — Two Types

Type 1 — URL Deduplication (Bloom Filter)

A Bloom Filter uses multiple hash functions and a large bit array. It can say "definitely NOT seen" or "probably seen". False positives are acceptable — missing one URL is harmless at billion-page scale.

Interactive — Bloom Filter simulator
Bit array (20 bits):
Bloom Filter ResultMeaningAction
All bits = 1 (probably seen)URL was likely already crawledSkip — don't add to frontier
Any bit = 0 (definitely new)URL was definitely never seenAdd to frontier — crawl it
False Positives OK
Missing one URL occasionally is far less harmful than crawling duplicates at billion-page scale. We always prefer false positives over false negatives here.

Type 2 — Content Deduplication (Hash Store)

Different URLs can have identical content. Bloom Filter won't catch this — it only checks URLs. We hash the page content itself and store it in an exact key-value store.

Crawl nytimes.com/article/123: MD5(page content) = "a1b2c3" Hash store: "a1b2c3" not seen → store → index ✅ Crawl msn.com/news/nytimes-article-123: MD5(page content) = "a1b2c3" ← same hash! Hash store: "a1b2c3" already seen → DISCARD ❌
StorePurposeStructureFalse Positives OK?
Bloom FilterURL deduplicationProbabilistic bit arrayYes ✅
Hash StoreContent deduplicationExact key-value storeNo ❌
Elegant Reuse
Content hashing is already used for change detection. The same hash store solves content deduplication too — one system, two problems solved.
7 Adaptive Recrawl Scheduling

The Prioritizer learns from history. Pages that change often get recrawled frequently. Pages that never change get crawled rarely.

9 Interview Answer Template
"A web crawler starts with seed URLs fed into a two-layer URL Frontier. Layer 1 is a prioritizer that scores URLs by PageRank, update frequency, and traffic. Layer 2 is per-domain queues that enforce politeness by rate-limiting requests per host. Workers download pages, parse links, and feed new URLs back. URL deduplication uses a Bloom Filter for memory-efficient O(1) checks — false positives are acceptable since missing one URL is harmless at scale. Content deduplication uses a hash store for exact matching — false positives here would silently discard unique content. Recrawl frequency is adaptive: pages with high change rates are revisited hourly, stale pages every 30+ days, using HTTP ETags and content hashing to detect changes efficiently."
