1What Is a Notification System?

Breaking news   → "Major earthquake reported" → Push + SMS + Email

5Why Separate Queues Per Channel?

One queue = SPOF. If it goes down, all notifications stop. Separate queues = isolated failures. If email provider is down — push and SMS keep working perfectly.

BenefitExplanation
Failure isolationEmail down → push and SMS keep working
Independent scalingPush needs 100 workers, SMS needs 10 — scale each separately
Independent rate limitingEach queue throttles to its provider's limit independently
No SPOFOne queue fails → only that channel affected

8Notification Templates

Templates are reusable message blueprints with placeholders. Workers — not queues — fetch and fill them at processing time. This keeps queue payloads lean.

// Template stored in Template DB (accessed by workers)
Template: ORDER_SHIPPED
Title:    "Your order has shipped!"
Body:     "Hi {{first_name}}, order {{order_id}} arrives {{date}}"

// Lean queue payload — NOT the full message
{
  event_id:    "evt_847291",
  user_id:     "usr_123",
  template_id: "ORDER_SHIPPED",
  data: { first_name: "Omar", order_id: "847291", date: "Jan 18" }
}

// Worker: fetch template → fill placeholders → send
Reason for TemplatesExplanation
Centralized updatesChange template once — all future notifications use new version
A/B testingTest two versions without changing event code
LocalizationSame template ID, different language per user preference
ConsistencyAll notifications follow the same brand format
Reduced payload sizeQueue carries IDs not 200-char messages — 4x storage saving
Your Fix
Template DB sits with Workers — not Queues. Workers fetch at processing time. Queues carry lean payloads only. This also allows template updates without re-queuing messages already in the queue.
9Reliability — Retry + Dead Letter Queue

Failed notifications are retried with exponential backoff. After all retries are exhausted, the message moves to a Dead Letter Queue for engineer investigation.

Why Backoff?
If provider is overloaded — hammering it with immediate retries makes it worse. Increasing waits (10s → 30s → 60s) gives the provider time to recover.
10Dedupe + Idempotency

At-least-once delivery means the same notification may be processed more than once after retries. Dedupe logic prevents users receiving duplicate notifications.

// Each event has a unique event_id
event_id: "evt_847291"

// Fast path — check Redis before processing
GET "evt_847291"  → nil    → not seen → process + store ✅
GET "evt_847291"  → "1"    → seen!   → discard duplicate ❌

// Atomic set-if-not-exists with 24-hour TTL
SET "evt_847291" "1" NX EX 86400
StorePurposeWhy
Redis (fast path)Dedupe check before processingO(1), SETNX atomic, TTL auto-cleanup
Notification Log DB (slow path)Fallback if Redis is downDurable, queryable, but slower
Why Not Exactly-Once?
True exactly-once delivery is impossible in distributed systems. Network failures make it impossible to distinguish a lost acknowledgment from a failed operation. We use at-least-once + idempotency to simulate exactly-once from the user's perspective.
11Notification Log DB

Every notification processed is written to the Log DB — success or failure. Fast dedupe uses Redis; the Log DB is the durable slow path.

UseExplanation
Delivery trackingDid notification reach the user? Provider response code?
DebuggingEngineer investigates why a notification failed or was delayed
Dedupe slow pathFallback if Redis is down — slower but durable
AnalyticsOpen rate, delivery rate, failure rate per channel
Audit trailLegal/compliance — prove notification was sent at specific time
Retry decisionsHow many times attempted? DLQ threshold reached?
Two-Layer Design
Fast path → Redis (O(1), in-memory, TTL). Slow path → Log DB (durable, queryable, audit). Together they give speed AND durability. Don't collapse them into one.

13Interview Answer Template
"A notification system routes events from multiple services through a Load Balancer to multiple stateless Notification Servers. Each server validates requests, checks user preferences, applies per-user rate limiting, and routes lean payloads to separate queues per channel — Push, SMS, and Email. Separate queues isolate failures so one channel going down doesn't affect others. Workers consume from queues, check Redis for dedupe using event IDs, fetch and fill notification templates, then call third-party providers — APNs/FCM for push, Twilio for SMS, SendGrid for email — while respecting per-provider rate limits. Failed deliveries are retried with exponential backoff. After exhausting retries, messages move to a Dead Letter Queue for investigation. Every notification is logged to a Notification Log DB for debugging, analytics, and audit."
