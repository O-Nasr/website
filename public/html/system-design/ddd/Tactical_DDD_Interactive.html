01 Entity vs Value Object

The foundation of all tactical DDD. The question: how do you determine identity?

Entity: two Orders with the same Id are the same order, even if fields differ. Identity comes from the ID. You can change every field — it's still the same entity. Like a person: you change hair, weight, address — still you.

Value Object: $59.99 USD and $59.99 USD — there's no "identity" to money. If all fields match, they ARE the same. Like a $20 bill: any $20 is the same as any other $20.

// Entity — identity matters
var order1 = new Order { Id = "ord-123", Total = 59.99m };
var order2 = new Order { Id = "ord-123", Total = 79.99m }; // price changed!
// order1 == order2?  YES — same Id

// Value Object — value matters
var money1 = new Money(59.99m, "USD");
var money2 = new Money(59.99m, "USD");
// money1 == money2?  YES — same amount + currency

Entity — has identity, mutable, compared by ID

public abstract class Entity
{
    public Guid Id { get; protected set; }

    public override bool Equals(object? obj)
    {
        if (obj is not Entity other) return false;
        return Id == other.Id;  // compared by ID only
    }
    public override int GetHashCode() => Id.GetHashCode();
}

public class Order : Entity
{
    public CustomerId CustomerId { get; private set; }
    public Money TotalAmount { get; private set; }
    public OrderStatus Status { get; private set; }

    // State changes through methods, not public setters
    public void Cancel(string reason)
    {
        if (Status == OrderStatus.Shipped)
            throw new DomainException("Cannot cancel shipped order");
        Status = OrderStatus.Cancelled;
    }
}

"State changes over time" means an entity has a lifecycle: Draft → Submitted → Shipped → Delivered. Same Id, different state at each stage.

Value Object — no identity, immutable, compared by value

// C# records are PERFECT for value objects
public record Money(decimal Amount, string Currency)
{
    public Money Add(Money other)
    {
        if (Currency != other.Currency)
            throw new DomainException("Cannot add different currencies");
        return new Money(Amount + other.Amount, Currency);
    }
    public Money Multiply(int quantity)
        => new Money(Amount * quantity, Currency);
}

// Usage — immutable, compared by value
var price1 = new Money(29.99m, "USD");
var price2 = new Money(29.99m, "USD");
Console.WriteLine(price1 == price2);  // TRUE

var doubled = price1.Multiply(2);  // NEW object, price1 unchanged

No Id. Immutable — Multiply returns a NEW object. C# record gives equality + immutability for free.

02 Value Objects in Practice

Instead of passing raw strings, decimals, and Guids — wrap them in value objects. This is called avoiding primitive obsession:

// Instead of: string email
public record Email(string Value)
{
    public Email(string value) : this(value)
    {
        if (!value.Contains('@'))
            throw new DomainException("Invalid email");
    }
}
// Validation at construction — impossible to create invalid Email
// Instead of: string street, string city, string country, string zip
public record Address(
    string Street, string City, string Country, string PostalCode);
// Instead of: DateTime start, DateTime end
public record DateRange(DateTime Start, DateTime End)
{
    public DateRange(DateTime start, DateTime end) : this(start, end)
    {
        if (end < start)
            throw new DomainException("End must be after start");
    }
    public bool Contains(DateTime date) => date >= Start && date <= End;
    public int Days => (End - Start).Days;
}
// Instead of: Guid customerId (primitive obsession!)
public record CustomerId(Guid Value);
public record ProductId(Guid Value);
public record OrderId(Guid Value);

// Now you can't accidentally pass an OrderId where CustomerId is expected:
void Process(CustomerId id) { ... }
Process(new OrderId(guid));  // COMPILE ERROR — type safety!
Benefits of value objects: validation at construction (impossible to create invalid data), type safety (can't mix up CustomerId and OrderId), natural home for related behavior (Money.Add, DateRange.Contains), self-documenting code.

03 Aggregate — The Hardest Concept

A cluster of entities and value objects treated as one unit for data changes. One entry point called the Aggregate Root.

The Rules

  • Everything outside talks ONLY to the root — never reach into internal entities
  • One transaction = one aggregate — never modify two aggregates in same transaction
  • Aggregates reference each other by ID only — not by object reference
public class Order : AggregateRoot  // the root — only entry point
{
    public CustomerId CustomerId { get; private set; }  // reference by ID!
    public Address ShippingAddress { get; private set; }
    public OrderStatus Status { get; private set; }
    public Money TotalAmount { get; private set; }

    private readonly List<OrderLine> _lines = new();
    public IReadOnlyList<OrderLine> Lines => _lines.AsReadOnly();

    // All state changes go through the root
    public void AddLine(ProductId productId, string name, int qty, Money price)
    {
        if (Status != OrderStatus.Draft)
            throw new DomainException("Can only add lines to draft orders");

        var line = new OrderLine(Guid.NewGuid(), productId, name, qty, price);
        _lines.Add(line);
        RecalculateTotal();
    }

    public void Submit()
    {
        if (!_lines.Any())
            throw new DomainException("Cannot submit empty order");
        Status = OrderStatus.Submitted;
        RaiseDomainEvent(new OrderSubmitted(Id, CustomerId, TotalAmount));
    }

    private void RecalculateTotal()
    {
        TotalAmount = _lines
            .Select(l => l.UnitPrice.Multiply(l.Quantity))
            .Aggregate(Money.Zero("USD"), (sum, p) => sum.Add(p));
    }
}
// Internal entity — can't exist without an Order
public class OrderLine
{
    internal OrderLine(Guid id, ProductId productId, string name,
                       int qty, Money price)
    {
        Id = id;
        ProductId = productId;     // reference to Product aggregate by ID
        ProductName = name;         // local copy (denormalized)
        Quantity = qty;
        UnitPrice = price;          // price at time of order
    }

    public Guid Id { get; }            // has identity (entity)
    public ProductId ProductId { get; } // NOT a Product object!
    public string ProductName { get; }
    public int Quantity { get; }
    public Money UnitPrice { get; }     // value object
}
// ProductId points to the Product aggregate (in Catalogue service)
// NOT to the Order class. It's how OrderLine remembers WHICH product.

Order aggregate (in Orders service)
├── Order (root)
│   ├── CustomerId     → points to Customer (different aggregate)
│   └── OrderLines[]
│       └── OrderLine
│           ├── ProductId  → points to Product (Catalogue service!)
│           ├── ProductName (local copy — denormalized)

// 1. Product lives in different service — can't hold C# object reference
// 2. Price at time of order — if product price changes, this stays correct
// 3. Aggregate rule — reference other aggregates by ID only

04 Aggregate Boundaries — The Customer + Order Trap

The most common mistake: "Order can't exist without Customer → put them in the same aggregate." WRONG.

Wrong: Customer + all Orders in one aggregate
To change ANYTHING you must load and lock the ENTIRE aggregate. Customer with 500 orders means loading 500 orders just to update a phone number.
Customer aggregate (TOO BIG)
├── Customer (root)
│   ├── Name, Email, Phone
│   ├── Order #1 (with lines)
│   ├── Order #2 (with lines)
│   └── Order #3 (with lines)  ← load ALL just to update phone?
Right: separate aggregates, connected by ID
Customer changes phone? Load only Customer. Add item to Order #3? Load only that Order. Each small, fast, independent.
Customer aggregate          Order aggregate
├── Customer (root)         ├── Order (root)
│   ├── Name                │   ├── CustomerId → reference by ID!
│   ├── Email               │   ├── Status
│   └── Phone               │   ├── OrderLine
                             │   └── TotalAmount

Big aggregate = fake conflicts

Thread 1: Customer updates email. Thread 2: Customer adds item to Order #3.

Both load the same big aggregate. Both modify it. One save wins, the other gets a concurrency conflict. Email update fails because someone added an item to an order. These operations have NOTHING to do with each other.

With separate aggregates: Thread 1 loads Customer. Thread 2 loads Order #3. No conflict. Independent units.

The golden rule: make aggregates as SMALL as possible.

The question is NOT "does A need B?"
The question IS "must A and B change in the same transaction?"

Order + OrderLine = same aggregate (adding a line must update total atomically)
Customer + Order = different aggregates (updating email doesn't need order total to change)

Application service orchestrates between aggregates

public class PlaceOrderHandler
{
    public async Task Handle(PlaceOrderCommand cmd)
    {
        // Load each aggregate independently
        var customer = await _customerRepo.GetByIdAsync(cmd.CustomerId);
        if (customer == null) throw new NotFoundException();

        // Create order with CustomerId (not Customer object)
        var order = new Order(customerId: customer.Id, ...);
        order.AddLine(cmd.ProductId, cmd.Name, cmd.Qty, cmd.Price);
        order.Submit();

        // Save ONLY the order — customer wasn't modified
        await _orderRepo.SaveAsync(order);

        // Domain events notify other aggregates
        await _eventBus.PublishAsync(order.DomainEvents);
    }
}

05 Protecting Internal Entities

OrderLine is public but shouldn't be created directly. Three approaches to enforce this:

public class OrderLine
{
    // Only code in the SAME project can create it
    internal OrderLine(Guid id, ProductId productId, ...) { ... }
}

// Inside Order (same project) — works fine
var line = new OrderLine(Guid.NewGuid(), productId, ...);  // OK

// From API layer (different project) — BLOCKED
var line = new OrderLine(...);  // Compile error!
public class Order : AggregateRoot
{
    private readonly List<OrderLine> _lines = new();

    // Read-only — can see but not add/remove
    public IReadOnlyList<OrderLine> Lines => _lines.AsReadOnly();

    // Only way to add — through the root
    public void AddLine(...) { _lines.Add(...); RecalculateTotal(); }
}

// From outside:
order.Lines.Add(new OrderLine(...));   // Compile error!
order.AddLine(productId, name, 2, price);  // OK — root validates
// Most teams combine both + convention:
// 1. internal constructor (can't create from outside)
// 2. IReadOnlyList (can't add from outside)
// 3. Team convention: always go through the root

// The real protection is the pattern being understood.
// internal + IReadOnlyList are safety nets for larger teams.

06 Domain Events — Full Pipeline

How aggregates communicate without being in the same transaction. The full pipeline:

public abstract class AggregateRoot
{
    private readonly List<IDomainEvent> _domainEvents = new();
    public IReadOnlyList<IDomainEvent> DomainEvents => _domainEvents;
    protected void RaiseDomainEvent(IDomainEvent evt) => _domainEvents.Add(evt);
    public void ClearDomainEvents() => _domainEvents.Clear();
}

public class Order : AggregateRoot
{
    public void Submit()
    {
        Status = OrderStatus.Submitted;
        // Raised — stored in list, NOT dispatched yet
        RaiseDomainEvent(new OrderSubmitted(Id, CustomerId, TotalAmount));
    }
}
public class SubmitOrderHandler
{
    public async Task Handle(SubmitOrderCommand cmd)
    {
        var order = await _repo.GetByIdAsync(cmd.OrderId);
        order.Submit();     // raises event internally

        // 1. Save FIRST
        await _repo.SaveAsync(order);

        // 2. Dispatch AFTER save succeeds
        foreach (var evt in order.DomainEvents)
        {
            await _mediator.Publish(evt);    // in-process
            await _bus.PublishAsync(evt);     // cross-service
        }
        order.ClearDomainEvents();
    }
}
Critical ordering: save first, dispatch second. If save fails → events never dispatched, no inconsistency. If handler fails after save → order is committed, handler can retry.
// Handler 1: Loyalty (same service — MediatR)
public class UpdateLoyaltyHandler : INotificationHandler<OrderSubmitted>

public class ReserveStockHandler : IConsumer<OrderSubmitted>

07 Shopping Cart — Worked Example

A full aggregate design exercise applying everything above:

The scenario and design decisions:

Cart + CartItem = same aggregate, Cart is root. Why? Adding a CartItem MUST recalculate the total in the same transaction. They change atomically.

Checkout → domain event, not direct call. Reserving inventory is a different service (Inventory). Cart just announces "I was checked out" — Inventory reacts independently.

Direct method call (same transaction):
  cart.AddItem()    → recalculate total immediately
  cart.RemoveItem() → recalculate total immediately

Domain event (different transaction):
  cart.Checkout()   → raises CartCheckedOut
                    → Inventory handler reserves stock
                    → Orders handler creates the Order
                    → Notification handler sends email
public class Cart : AggregateRoot
{
    public CustomerId CustomerId { get; private set; }
    private readonly List<CartItem> _items = new();
    public IReadOnlyList<CartItem> Items => _items.AsReadOnly();
    public Money Total { get; private set; }

    public void AddItem(ProductId productId, string name, int qty, Money price)
    {
        var existing = _items.FirstOrDefault(i => i.ProductId == productId);
        if (existing != null)
            existing.IncreaseQuantity(qty);
        else
            _items.Add(new CartItem(productId, name, qty, price));
        RecalculateTotal();  // same transaction
    }

    public void RemoveItem(ProductId productId)
    {
        _items.RemoveAll(i => i.ProductId == productId);
        RecalculateTotal();
    }

    public void Checkout()
    {
        if (!_items.Any())
            throw new DomainException("Cannot checkout empty cart");
        Status = CartStatus.CheckedOut;

        // Event — inventory is OUTSIDE this aggregate
        RaiseDomainEvent(new CartCheckedOut(
            Id, CustomerId,
            _items.Select(i => new CheckoutItem(i.ProductId, i.Quantity)).ToList()));
    }

    private void RecalculateTotal()
    {
        Total = _items
            .Select(i => i.UnitPrice.Multiply(i.Quantity))
            .Aggregate(Money.Zero("USD"), (sum, p) => sum.Add(p));
    }
}
INSIDE aggregate (direct method, same txn)
AddItem → recalculate total
RemoveItem → recalculate total
Cart + CartItem live together
OUTSIDE aggregate (event, different txn)
Reserve inventory → CartCheckedOut event
Create order → CartCheckedOut event
Send email → CartCheckedOut event
The rule: Must be consistent immediately → direct method inside aggregate. Can be eventually consistent → domain event outside.

08 Interview Q&A

Q: What is the difference between an Entity and a Value Object?
Entity has identity (compared by ID, mutable, has a lifecycle). Value Object has no identity (compared by value, immutable, replaced entirely). In C#: entities are classes with an Id field. Value objects are records. Order is an entity (same Id even if total changes). Money is a value object ($59.99 USD = $59.99 USD regardless of which object).
Q: What is an Aggregate?
A cluster of entities and value objects treated as one unit for data changes. Has a single entry point (aggregate root). Rules: one transaction = one aggregate, reference other aggregates by ID, all access through the root. The root enforces business invariants.
Q: How do you decide aggregate boundaries?
Make aggregates as small as possible. The question is NOT "does A need B" but "must A and B change in the same transaction?" Order + OrderLine = same aggregate (total must be consistent). Customer + Order = different aggregates (updating email doesn't need order total to change). Big aggregates cause concurrency conflicts and performance issues.
Q: How do aggregates communicate?
Via domain events. The aggregate raises an event (stored in a list), the application service saves then dispatches. Handlers in the same service use MediatR (in-process). Handlers in other services use message queues (MassTransit/Kafka). Each handler runs in its own transaction. Critical: save first, dispatch second.
Q: What is a Domain Event?
Something that happened in the domain — OrderSubmitted, PaymentCompleted, CartCheckedOut. Named in past tense. Raised by aggregates, dispatched after save, handled independently. They decouple aggregates: Order doesn't know about Inventory, it just says "I was submitted" and Inventory reacts.
Q: Why reference other aggregates by ID instead of object?
Three reasons: (1) the other aggregate may be in a different service — can't hold a C# reference across boundaries, (2) you want the data at the time of the event (price at time of order, not current price), (3) holding an object reference creates tight coupling — loading one aggregate would load all referenced aggregates transitively.
Q: What is primitive obsession and how do Value Objects fix it?
Using raw strings, ints, and Guids instead of domain types. string email can be anything — no validation. Email value object validates at construction. Guid customerId and Guid orderId are interchangeable — typed CustomerId and OrderId prevent mixing them up. Value objects add validation, type safety, and a natural home for behavior.
